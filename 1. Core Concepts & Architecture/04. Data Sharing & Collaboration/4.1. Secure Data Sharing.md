# 4.1 Secure Data Sharing 

## **1. What is Secure Data Sharing in Snowflake?**
Secure Data Sharing allows you to share selected database objects (tables, schemas, databases) with other Snowflake accounts **without copying or moving the data**. Instead, Snowflake uses **metadata pointers** to grant read-only access to the original data source. This approach is efficient, scalable, and secure, as the data remains in the provider’s account and only metadata is shared with consumers [5, 10].

---

## **2. Key Concepts & Architecture**

### **A. Zero-Copy Architecture**
- **No Data Duplication:** The data stays in the provider’s account; only metadata and access permissions are shared.
- **Read-Only Access:** Consumers can query the shared data but cannot modify it.
- **Cost Efficiency:** Consumers do not incur storage costs for shared data [5, 10].

### **B. Provider vs. Consumer Roles**
- **Provider:** The Snowflake account that owns the data and creates the share.
- **Consumer:** The Snowflake account that accesses the shared data.

### **C. How Shares Work**
- A **Share** is a Snowflake object that encapsulates all the information needed to share a database or specific objects.
- Shares are created and managed by the provider, and consumers are granted access via their account identifiers [7, 10].

### **D. Supported Objects for Sharing**
- Databases, schemas, tables, views, and secure views can be shared.
- Externally managed and Iceberg tables are also supported [1, 8].

---

## **3. How to Set Up Secure Data Sharing**

### **Step-by-Step Process**
1. **Create a Share:**
   ```sql
   CREATE SHARE share_name;
   ```
2. **Grant Access to Objects:**
   ```sql
   GRANT USAGE ON DATABASE db_name TO SHARE share_name;
   GRANT USAGE ON SCHEMA schema_name TO SHARE share_name;
   GRANT SELECT ON TABLE table_name TO SHARE share_name;
   ```
3. **Add Consumer Accounts:**
   ```sql
   ALTER SHARE share_name ADD ACCOUNTS=consumer_account1, consumer_account2;
   ```
4. **Verify Grants:**
   ```sql
   SHOW GRANTS TO SHARE share_name;
   ```

### **Example Workflow**
- A provider creates a share and grants access to specific tables in a database.
- Consumers receive an invitation to access the share and can query the data directly from their own Snowflake account [7].

## **4. Secure Data Sharing Across Regions & Clouds**
- **Cross-Region Sharing:** Use Snowflake’s replication features to share data across different regions and cloud platforms (AWS, Azure, GCP).
- **Cross-Cloud Sharing:** Secure data sharing is supported across different cloud providers, enabling global collaboration [4].

Here are **step-by-step SQL commands and instructions** for the top 5 most common use cases under Snowflake Secure Data Sharing:

---

### 1. **Share a Sales Database with the Marketing Team**

| Step | Command/Action                                                                                     | Notes                                                                                     |
|------|---------------------------------------------------------------------------------------------------|-------------------------------------------------------------------------------------------|
| 1    | ```sql\nUSE ROLE accountadmin;\nCREATE SHARE sales_marketing_share;\n``` | Create a share for the marketing team.                                                     |
| 2    | ```sql\nGRANT USAGE ON DATABASE sales_db TO SHARE sales_marketing_share;\nGRANT USAGE ON SCHEMA sales_db.marketing TO SHARE sales_marketing_share;\nGRANT SELECT ON TABLE sales_db.marketing.campaign_performance TO SHARE sales_marketing_share;\n``` | Grant access to the sales database, schema, and specific table.                            |
| 3    | ```sql\nALTER SHARE sales_marketing_share ADD ACCOUNTS=MKT001, MKT002;\n``` | Add the marketing team’s account IDs to the share.                                         |
| 4    | ```sql\n-- In their own account:\nCREATE DATABASE shared_sales_data FROM SHARE <provider_account>.sales_marketing_share;\n``` | The marketing team can now query the shared tables directly in their own Snowflake account. |

**Note:** Replace `<provider_account>` with the actual provider account locator.

## **5. Benefits of Secure Data Sharing**
- **Real-Time Collaboration:** Data is always up-to-date for all consumers.
- **Governance & Compliance:** Access is controlled and auditable.
- **Cost Savings:** No need to replicate data, reducing storage and compute costs.
- **Scalability:** Easily share data with multiple consumers or across organizations [5, 6].

## **6. Use Cases**
- **Internal Collaboration:** Share data between departments or teams within the same organization.
- **External Partnerships:** Collaborate with vendors, partners, or customers without moving data.
- **Data Marketplace:** Monetize your data by listing it on the Snowflake Marketplace [6, 9].

## **7. Best Practices**
- **Use Database Roles:** Assign privileges to database roles, then grant those roles to shares for better management [7].
- **Secure Views:** For unstructured data, use secure views to share only the necessary data.
- **Monitoring:** Regularly review and audit share privileges and consumer access [2].

Here are the SQL queries for the **Best Practices** section:

---

### **1. Use Database Roles for Better Management**

| Step | Query                                                                                     | Notes                                                                                     |
|------|-------------------------------------------------------------------------------------------|-------------------------------------------------------------------------------------------|
| 1    | ```sql\n-- Step 1: Create a database role\nCREATE ROLE sales_marketing_role;\n``` | Create a role to centralize privileges.                                                   |
| 2    | ```sql\n-- Step 2: Grant privileges to the role\nGRANT USAGE ON DATABASE sales_db TO ROLE sales_marketing_role;\nGRANT USAGE ON SCHEMA sales_db.marketing TO ROLE sales_marketing_role;\nGRANT SELECT ON TABLE sales_db.marketing.campaign_performance TO ROLE sales_marketing_role;\n``` | Grant necessary privileges to the role.                                                  |
| 3    | ```sql\n-- Step 3: Grant the role to the share\nGRANT ROLE sales_marketing_role TO SHARE sales_marketing_share;\n``` | Assign the role to the share for simplified management.                                   |

**Note:** This approach centralizes privilege management and simplifies share administration.

---

### **2. Use Secure Views for Unstructured Data**

| Step | Query                                                                                     | Notes                                                                                     |
|------|-------------------------------------------------------------------------------------------|-------------------------------------------------------------------------------------------|
| 1    | ```sql\n-- Step 1: Create a secure view\nCREATE SECURE VIEW analytics.logs.logs_summary_v AS\n  SELECT id, timestamp, log_level, LEFT(message, 100) AS message_preview\n  FROM @analytics.logs.logs_stage\n  (FILE_FORMAT => 'JSON');\n``` | Create a secure view to expose only the necessary data from unstructured sources.        |
| 2    | ```sql\n-- Step 2: Grant access to the secure view\nGRANT SELECT ON VIEW analytics.logs.logs_summary_v TO ROLE analytics_team_role;\n``` | Grant access to the secure view, masking sensitive or irrelevant data.                    |

**Note:** Secure views allow you to control exactly what data is exposed.

---
### **3. Monitor and Audit Share Access**

| Step | Query                                                                                     | Notes                                                                                     |
|------|-------------------------------------------------------------------------------------------|-------------------------------------------------------------------------------------------|
| 1    | ```sql\n-- Step 1: Show all shares and their privileges\nSHOW SHARES;\n``` | List all shares and their configurations.                                                 |
| 2    | ```sql\n-- Step 2: Show accounts granted access to a specific share\nSHOW GRANTS TO SHARE sales_marketing_share;\n``` | View which accounts have access to a specific share.                                     |
| 3    | ```sql\n-- Step 3: Audit share access (requires ACCOUNTADMIN or SECURITYADMIN role)\nSELECT * FROM TABLE(INFORMATION_SCHEMA.SHARE_EVENTS_HISTORY())\n  WHERE SHARE_NAME = 'sales_marketing_share'\n  ORDER BY EVENT_TIME DESC;\n``` | Review the audit log for access events to the share.                                      |

**Note:** Regularly review these logs to ensure only authorized access.
