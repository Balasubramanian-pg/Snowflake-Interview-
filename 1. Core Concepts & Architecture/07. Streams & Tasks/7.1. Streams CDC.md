# Snowflake Streams CDC: Mastering Event-Driven Change Data Capture

## 1. Core Concepts

Snowflake **Streams** provide a native, serverless framework for **Change Data Capture (CDC)**. Instead of creating physical copies of data or relying on expensive trigger-based architectures, a Stream acts as a lightweight metadata object. It records an "offset" (a point in time) on a source table and tracks all DML operations (`INSERT`, `UPDATE`, `DELETE`) that occur after that offset. When the stream is consumed via a DML operation, the offset advances, allowing downstream data pipelines and Snowflake Tasks to process only the net new incremental changes, guaranteeing exactly-once semantics without redundant data scanning.

### 1.1. How it Works
The architecture of Snowflake Streams relies heavily on the system's foundational Time Travel and micro-partition versioning capabilities.

1. **Offset Initialization:** Upon creation, the Stream takes a zero-cost metadata snapshot of the current table version, establishing the initial offset pointer.
2. **Delta Tracking:** As DML operations alter the source table, the Stream tracks the micro-partition changes, exposing them via three virtual columns: `METADATA$ACTION`, `METADATA$ISUPDATE`, and `METADATA$ROW_ID`.
3. **Net Change Collapse:** The Stream is not an event queue; it returns the **net effect** of changes. If a row is inserted and then deleted before the stream is consumed, it will not appear in the stream at all.
4. **Offset Advancement:** When the Stream is utilized inside a committed DML transaction (e.g., `INSERT INTO target SELECT * FROM stream`), the offset pointer automatically advances to the current timestamp.

### 1.2. CDC Architecture Comparison

| Metric | Legacy Approach (Triggers & High-Water Marks) | Modern Approach (Snowflake Streams) |
| :--- | :--- | :--- |
| **Compute Overhead** | High (Triggers execute code per row/batch) | **Zero** (Metadata-driven offset tracking) |
| **Intermediate Changes** | Logs every single mutation, causing pipeline bloat | Intelligently collapses intermediate changes (Net Delta) |
| **Storage Cost** | Requires dedicated audit tables for CDC logs | Relies on existing Time Travel storage overhead |
| **Schema Evolution** | Brittle; requires manual mapping updates | Automatically tracks column additions/drops natively |
| **Concurrency** | Custom locks needed for multiple consumers | Support infinite, isolated Streams per table |

### 1.3. Key Factors
1. **Stream Types:** 
   * **Standard Streams:** Track all DML (`INSERT`, `UPDATE`, `DELETE`) on standard tables.
   * **Append-Only Streams:** Only track `INSERT` operations (ignores updates/deletes), ideal for immutable event logs.
   * **Insert-Only Streams:** Exclusively used for External Tables to track new files added to a data lake.
2. **Time Travel Dependency:** Streams rely on the source table's Time Travel retention period. If a stream is not consumed before the retention period expires (e.g., 1 day), the stream becomes **stale** and must be recreated.
3. **Transactional Boundaries:** A stream's offset only advances if the consuming DML statement successfully commits. If the transaction rolls back, the stream offset remains unchanged, ensuring zero data loss.
4. **Update Representation:** Snowflake represents an `UPDATE` in a Standard Stream as two distinct rows: a `DELETE` (the old state) and an `INSERT` (the new state), both flagged with `METADATA$ISUPDATE = TRUE`.

## 2. Basic Commands & Privileges

### 2.1. Required Privileges

| Privilege | Scope | Description |
| :--- | :--- | :--- |
| `CREATE STREAM` | Schema | Required to instantiate a new stream object in the target schema. |
| `SELECT` | Source Table/View | The executing role must have read access to the underlying table being tracked. |
| `OWNERSHIP` | Stream | Required to alter, drop, or grant usage on the stream to other roles. |

### 2.2. Create a Standard Stream
Instantiates a standard CDC stream on a table, tracking all inserts, updates, and deletes from the current moment forward.
```sql
CREATE STREAM customer_cdc_stream ON TABLE raw_customers;
```

### 2.3. Create an Append-Only Stream
Creates a streamlined CDC object that strictly tracks inserts, reducing compute overhead for massive, immutable log tables.
```sql
CREATE STREAM web_logs_append_stream ON TABLE web_logs APPEND_ONLY = TRUE;
```

### 2.4. Create a Stream on a View
Establishes a stream on top of a View, allowing downstream consumers to read pre-filtered or transformed CDC data.
```sql
CREATE STREAM secure_sales_stream ON VIEW vw_masked_sales;
```

### 2.5. Check if Stream Contains Data
Evaluates if new changes have occurred since the last consumption, used programmatically to prevent waking up Virtual Warehouses unnecessarily.
```sql
SELECT SYSTEM$STREAM_HAS_DATA('customer_cdc_stream');
```

### 2.6. Consume and Advance the Stream
Reads the net changes from the stream and inserts them into a target table, which implicitly advances the offset upon successful commit.
```sql
INSERT INTO target_customers SELECT * FROM customer_cdc_stream;
```

### 2.7. Drop a Stream
Removes the metadata pointer without affecting the underlying source table data.
```sql
DROP STREAM customer_cdc_stream;
```

## 3. Impactful Use Cases

### Use Case 1: Slowly Changing Dimensions (SCD Type 2) Processing
* **Scenario** – A `raw_employee` table receives continuous upstream updates. We need to maintain an SCD Type 2 `dim_employee` table, tracking historical changes via effective dates without scanning the entire source table.
* ```sql
  -- The stream exposes METADATA columns to identify exactly what changed
  MERGE INTO dim_employee target
  USING (
      SELECT 
          employee_id, name, department,
          METADATA$ACTION as action,
          METADATA$ISUPDATE as is_update
      FROM employee_stream
  ) source ON target.employee_id = source.employee_id 
           AND target.is_current = TRUE
  -- Handle Updates (Close old record)
  WHEN MATCHED AND source.action = 'DELETE' AND source.is_update = TRUE THEN
      UPDATE SET is_current = FALSE, end_date = CURRENT_TIMESTAMP()
  -- Handle Inserts (New hires or the new version of an updated record)
  WHEN NOT MATCHED AND source.action = 'INSERT' THEN
      INSERT (employee_id, name, department, is_current, start_date)
      VALUES (source.employee_id, source.name, source.department, TRUE, CURRENT_TIMESTAMP());
  ```
* **Explanation** – Standard streams treat an `UPDATE` as two records: a `DELETE` of the old state and an `INSERT` of the new state, both marked with `METADATA$ISUPDATE = TRUE`. The `MERGE` statement leverages these metadata flags to perform precise SCD Type 2 logic. It expires the active record when it sees the `DELETE` action, and inserts the fresh row when it sees the `INSERT` action, processing only the exact rows that changed.

### Use Case 2: Fan-Out Architecture for Multiple Downstream Consumers
* **Scenario** – A central `transactions` table feeds three independent downstream models: Fraud Detection, Accounting, and Marketing. If they share one stream, the first to read it advances the offset, breaking the pipeline for the others.
* ```sql
  -- Step 1: Create dedicated streams for every isolated pipeline
  CREATE STREAM stream_fraud ON TABLE transactions;
  CREATE STREAM stream_accounting ON TABLE transactions;
  CREATE STREAM stream_marketing ON TABLE transactions;

  -- Step 2: Consumers run independently on their own schedules
  -- Fraud pipeline runs every 5 minutes
  INSERT INTO fraud_features SELECT * FROM stream_fraud WHERE amount > 10000;
  
  -- Accounting pipeline runs daily
  INSERT INTO daily_ledger SELECT * FROM stream_accounting;
  ```
* **Explanation** – A stream's offset is destructively advanced for the caller once consumed in a DML statement. To achieve an event-driven "fan-out" (publish-subscribe) architecture, you must create a dedicated stream for each consuming pipeline. Because streams are purely metadata offsets, creating 3 or 300 streams on a single table incurs zero additional storage cost, allowing infinite, decoupled downstream scalability.

### Use Case 3: Optimizing High-Volume Ingestion with Append-Only Streams
* **Scenario** – A massive `clickstream` table ingests 10 billion rows daily. Standard CDC is too slow because it evaluates micro-partitions for updates and deletes, but this table is immutable (inserts only).
* ```sql
  -- Create an optimized stream that completely ignores update/delete overhead
  CREATE STREAM clickstream_fast_cdc 
  ON TABLE raw_clickstream 
  APPEND_ONLY = TRUE;

  -- Consume strictly the net-new inserts into the aggregate layer
  INSERT INTO clickstream_daily_aggs (click_date, total_clicks)
  SELECT DATE(click_timestamp), COUNT(*)
  FROM clickstream_fast_cdc
  GROUP BY DATE(click_timestamp);
  ```
* **Explanation** – By specifying `APPEND_ONLY = TRUE`, Snowflake skips the computational overhead of calculating the net-change delta (matching deletes to inserts). The stream blindly and instantly yields newly appended rows. This is structurally necessary for massive time-series, log, or IoT event tables where maintaining standard CDC tracking would cause unnecessary metadata compilation delays.

### Use Case 4: External Table CDC for Data Lake Synchronization
* **Scenario** – Upstream systems drop Parquet files into Amazon S3. Snowflake queries these via an External Table. We need to incrementally load only the new files into a native Snowflake table.
* ```sql
  -- Create an Insert-Only stream directly on the External Table
  CREATE STREAM s3_parquet_stream 
  ON EXTERNAL TABLE ext_s3_events 
  INSERT_ONLY = TRUE;

  -- The stream detects when new Parquet files are registered and returns their rows
  INSERT INTO native_events_table
  SELECT 
      value:event_id::VARCHAR,
      value:event_type::VARCHAR,
      value:timestamp::TIMESTAMP
  FROM s3_parquet_stream;
  ```
* **Explanation** – External Tables do not support Time Travel or updates. Therefore, Snowflake offers an `INSERT_ONLY` stream exclusively for this object type. When the external table's metadata is refreshed (e.g., via AWS SQS auto-ingest notifications), the stream captures the contents of the newly discovered files. This transforms a static data lake into an event-driven streaming source without requiring heavy ETL orchestration tools.

### Use Case 5: View-Based CDC for PII Abstraction
* **Scenario** – Data Scientists need a real-time feed of new user registrations, but the source table contains plain-text Passwords and SSNs. Giving them a stream on the raw table violates security compliance.
* ```sql
  -- Step 1: Create a secure view abstracting the PII
  CREATE SECURE VIEW vw_safe_users AS
  SELECT user_id, 
         signup_date, 
         region, 
         SHA2(email) as hashed_email
  FROM raw_users;

  -- Step 2: Create a stream ON THE VIEW, not the table
  CREATE STREAM safe_users_stream ON VIEW vw_safe_users;

  -- Data Scientists consume from the abstracted stream
  SELECT * FROM safe_users_stream;
  ```
* **Explanation** – Creating a stream on a view dynamically applies the view's SQL transformations to the underlying table's CDC delta. When the Data Scientist consumes `safe_users_stream`, Snowflake calculates the net changes on `raw_users`, immediately hashes the emails, drops the PII columns, and delivers a compliant delta payload. This guarantees real-time data access without compromising Row Access Policies or column-level security.

## 4. Best Practices Summary

| Practice | Why |
| :--- | :--- |
| **Use `SYSTEM$STREAM_HAS_DATA`** | Before executing a heavy transformation task, check this function. If it returns false, skip the task to save Virtual Warehouse compute credits. |
| **Match Stream Types to Workloads** | Use `APPEND_ONLY` for log tables. Standard streams waste compute trying to resolve net-changes on datasets that never experience updates or deletes. |
| **Never Share Streams** | A stream's offset advances upon consumption. Multiple pipelines reading the same stream will steal data from one another. Create 1 stream per consumer. |
| **Align Time Travel with Task Schedules** | If a table's Time Travel is 1 Day, the consuming task must run at least once every 24 hours, or the stream becomes **stale** and permanently unusable. |
| **Wrap Multi-Table Consumption in Transactions** | If a stream populates Table A and Table B, wrap both `INSERT` statements in `BEGIN; ... COMMIT;` to ensure the offset advances only if *both* succeed. |
| **Do Not `CREATE OR REPLACE` Source Tables** | Dropping or replacing a base table instantly breaks all streams attached to it. Use `TRUNCATE` or `INSERT OVERWRITE` to maintain the object's internal ID. |
| **Understand Net-Collapse Logic** | Streams are not audit logs. If an application inserts a row and deletes it 5 minutes later, a stream consumed hourly will show zero trace of that row. |
| **Utilize `METADATA$ISUPDATE` Carefully** | Differentiating between a true `DELETE` and the delete-half of an `UPDATE` requires filtering explicitly on the `METADATA$ISUPDATE` flag. |

## 5. Important Limitations & Considerations

| Limitation | Mitigation |
| :--- | :--- |
| **Stream Staleness** | A stream goes stale if the offset falls behind the table's Time Travel retention (e.g., 14 days max for extended streams). Monitor and consume streams regularly. Recreate if stale. |
| **No Schema Evolution on View Streams** | Streams on Views track the exact columns defined at creation. If the base table adds a column, the View Stream will not capture it until recreated. |
| **Data Sharing Restrictions** | You cannot natively create a stream on a table shared from another Snowflake account if the provider has not explicitly granted Time Travel privileges on the share. |
| **Transactional Latency** | The offset strictly tracks committed transactions. Long-running, uncommitted transactions on the source table will not appear in the stream until finalized. |
| **Zero-Copy Clone Inheritance** | Cloning a schema or database clones the streams, but their offsets are reset to the exact moment of the clone. They do not inherit the original stream's unconsumed backlog. |

## 6. Auditing & Monitoring Queries

### Identify Stale Streams
```sql
SELECT 
    table_catalog, 
    table_schema, 
    name AS stream_name,
    stale, 
    mode AS stream_type
FROM snowflake.information_schema.streams
WHERE stale = TRUE;
```

### Check Unconsumed Data Status (Programmatic Control)
```sql
-- Often injected into stored procedures or Task conditions
SELECT SYSTEM$STREAM_HAS_DATA('PROD_DB.ANALYTICS.CUSTOMER_STREAM');
```

### List All Active Streams and Their Source Objects
```sql
SELECT 
    name, 
    table_name AS source_object, 
    type AS source_type, 
    mode, 
    created_on
FROM snowflake.information_schema.streams
ORDER BY created_on DESC;
```

### Audit Stream Consumption via Task History
```sql
SELECT 
    name AS task_name,
    state,
    error_message,
    scheduled_time,
    completed_time
FROM snowflake.information_schema.task_history
WHERE query_text ILIKE '%CUSTOMER_STREAM%'
ORDER BY scheduled_time DESC
LIMIT 50;
```

## 7. Key Takeaways
* Snowflake Streams are **serverless metadata pointers** that track the offset of DML changes, consuming **zero compute and zero storage** on their own.
* They guarantee **exactly-once semantics**; the offset only advances when the stream is consumed within a successfully committed DML transaction.
* Streams do not act as sequential event logs; they yield the **net effect** of changes, intelligently collapsing intermediate inserts, updates, and deletes to save processing time.
* For massive, immutable datasets, always use **Append-Only streams** to bypass the computational overhead of delta-collapse matching.
* Streams rely entirely on the source table's **Time Travel retention**. If the unconsumed data ages past the retention period, the stream becomes irreparably **stale** and must be recreated.
* To feed multiple downstream models from a single source, implement a **fan-out architecture** by creating an independent, dedicated stream for each consuming pipeline.
