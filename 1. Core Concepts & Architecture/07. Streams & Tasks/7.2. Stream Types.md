# Snowflake Stream Types: Tailoring CDC Mechanics for Specialized Workloads

## 1. Core Concepts

While a Snowflake Stream inherently acts as a Change Data Capture (CDC) offset pointer, treating all data ingestion patterns equally is computationally inefficient. To optimize performance, Snowflake offers distinct **Stream Types**: **Standard**, **Append-Only**, and **Insert-Only**. Each type is architected to handle specific object models (Native Tables, External Tables, Views, Directory Tables) and data mutability profiles (Mutable vs. Immutable). Selecting the correct stream type prevents the Snowflake engine from performing unnecessary metadata calculations—such as matching deletes to inserts for net-change collapse—drastically reducing serverless compute overhead on high-velocity data pipelines.

### 1.1. How it Works
When a stream is created, its type dictates how the Cloud Services layer evaluates the underlying micro-partition changes.

1. **Type Assignment:** The engineer explicitly defines the stream type (`APPEND_ONLY`, `INSERT_ONLY`) or defaults to a Standard Stream during the DDL `CREATE` statement.
2. **Metadata Evaluation:** As DML operations occur, the stream reads the micro-partition version history. 
3. **Delta Processing:** 
   * A *Standard Stream* executes a complex algorithm to calculate the **net effect** (collapsing intermediate `UPDATE`s and `DELETE`s).
   * An *Append-Only* or *Insert-Only Stream* completely bypasses this algorithm, operating as a lightweight pass-through filter.
4. **Offset Advancement:** Regardless of type, the stream's offset pointer advances exclusively when its data is successfully consumed within a committed DML transaction.

### 1.2. Stream Type Optimization Comparison

| Metric | Standard Stream | Append-Only Stream | Insert-Only Stream |
| :--- | :--- | :--- | :--- |
| **Supported Objects** | Native Tables, Views, Directory Tables | Native Tables, Views | External Tables |
| **Tracked DML Operations** | `INSERT`, `UPDATE`, `DELETE` | `INSERT` only | `INSERT` (New Cloud Files) only |
| **Delta Computation Overhead** | **Moderate:** Must calculate net-change collapse | **Zero:** Blindly passes appended rows | **Minimal:** Scans external stage metadata |
| **Micro-Partition Logic** | Reads both added and removed partition metadata | Reads only newly added partitions | Relies on cloud storage event notifications |
| **Primary Architectural Use Case** | SCD Type 1/2, transactional replication | High-velocity logs, IoT telemetry, immutable events | Data Lake synchronization (S3/GCS/Azure) |

### 1.3. Key Factors
1. **Data Immutability:** Using a Standard Stream on an immutable (insert-only) log table is an anti-pattern. The engine wastes compute verifying that no updates or deletes occurred. Append-Only streams eliminate this check.
2. **Object Compatibility:** You cannot create a Standard Stream on an External Table because Snowflake does not control the granular block-level storage of the external data lake; you are strictly forced to use Insert-Only streams.
3. **Implicit Deletes:** In an Append-Only stream, if a row is inserted and subsequently deleted before the stream is consumed, the row **will still be returned** by the stream, because the stream intentionally ignores the `DELETE` operation.
4. **View Resolution:** When creating a stream on a View, the stream type is restricted by the underlying base tables. If the base table is External, the View stream must act as Insert-Only.

## 2. Basic Commands & Privileges

### 2.1. Required Privileges

| Privilege | Scope | Description |
| :--- | :--- | :--- |
| `CREATE STREAM` | Schema | Required to define the stream object in the specific schema. |
| `SELECT` | Source Object | The executing role must have read access to the base Table, View, or External Table. |
| `OWNERSHIP` | Stream | Required to alter, grant permissions on, or drop the stream object. |

### 2.2. Create a Standard Stream (Default)
Initializes a full CDC stream that tracks inserts, updates, and deletes, applying the net-change collapse algorithm automatically.
```sql
CREATE STREAM customer_master_stream ON TABLE raw_customers;
```

### 2.3. Create an Append-Only Stream
Creates a highly optimized stream for immutable datasets by explicitly instructing the engine to track only inserts and ignore all other DML.
```sql
CREATE STREAM web_traffic_append_stream ON TABLE web_traffic APPEND_ONLY = TRUE;
```

### 2.4. Create an Insert-Only Stream (External Tables)
Establishes a stream on a data lake architecture, tracking newly registered files in an External Table.
```sql
CREATE STREAM datalake_insert_stream ON EXTERNAL TABLE ext_s3_events INSERT_ONLY = TRUE;
```

### 2.5. Create a Stream on a Directory Table
Enables unstructured data CDC, tracking the addition or removal of files (PDFs, images) on a Snowflake Stage.
```sql
CREATE STREAM stage_file_stream ON STAGE @my_unstructured_stage;
```

### 2.6. Create a Stream on a View
Abstracts the CDC tracking through a logical view layer, ensuring transformations or security filters are applied to the delta payload.
```sql
CREATE STREAM secure_sales_stream ON VIEW vw_masked_sales;
```

### 2.7. Validate Stream Properties
Queries the metadata dictionary to programmatically verify the `mode` (type) and status of existing streams.
```sql
SHOW STREAMS LIKE '%stream';
```

## 3. Impactful Use Cases

### Use Case 1: Standard Stream for Dimensional Merging (SCD2)
* **Scenario** – A transactional database syncs its `orders` table to Snowflake. The pipeline must capture updates (e.g., status changing to 'SHIPPED') and new orders, merging them into a dimensional model.
* ```sql
  -- Standard stream automatically collapses multiple intra-batch updates into the final net state
  MERGE INTO dim_orders tgt
  USING (
      SELECT * FROM order_standard_stream
  ) src ON tgt.order_id = src.order_id
  
  -- Handle updates (Old record is marked as deleted by the stream)
  WHEN MATCHED AND src.METADATA$ACTION = 'DELETE' THEN
      UPDATE SET is_active = FALSE
      
  -- Handle new inserts or the new version of an updated record
  WHEN NOT MATCHED AND src.METADATA$ACTION = 'INSERT' THEN
      INSERT (order_id, status, is_active) 
      VALUES (src.order_id, src.status, TRUE);
  ```
* **Explanation** – Standard streams are strictly required when the source data is mutable. If an order goes from 'PENDING' to 'PROCESSING' to 'SHIPPED' between task runs, the Standard Stream's net-collapse algorithm ensures only the final 'SHIPPED' state (paired with a `DELETE` of the 'PENDING' state) is returned. This prevents the `MERGE` statement from failing due to duplicate keys in the source payload.

### Use Case 2: Append-Only Stream for Massive Telemetry Ingestion
* **Scenario** – IoT sensors stream 5 billion temperature readings a day into a `sensor_logs` table. This data is strictly immutable. Standard CDC processing is causing warehouse delays.
* ```sql
  -- Step 1: Create the optimized Append-Only stream
  CREATE STREAM iot_append_stream ON TABLE sensor_logs APPEND_ONLY = TRUE;

  -- Step 2: Consume directly into time-series aggregates
  INSERT INTO daily_temperature_aggs (sensor_id, avg_temp, log_date)
  SELECT 
      sensor_id, 
      AVG(temperature), 
      DATE(log_timestamp)
  FROM iot_append_stream
  GROUP BY sensor_id, DATE(log_timestamp);
  ```
* **Explanation** – An `APPEND_ONLY = TRUE` stream bypasses the Cloud Services metadata checks for updates and deletes. For a 5-billion row table, attempting to calculate a net-change delta (even if no deletes occurred) requires immense memory and CPU. The Append-Only stream acts as a pure, lightweight pointer to newly written micro-partitions, delivering zero-latency, zero-overhead ingestion for aggregate modeling.

### Use Case 3: Insert-Only Stream for External Data Lake Synchronization
* **Scenario** – Upstream applications dump Parquet files into an Amazon S3 bucket. Snowflake queries this bucket via an External Table, but you need to incrementally load these records into a high-performance native Snowflake table.
* ```sql
  -- Step 1: Create an Insert-Only stream on the External Table
  CREATE STREAM s3_external_stream ON EXTERNAL TABLE ext_app_events INSERT_ONLY = TRUE;

  -- Step 2: Task consumes new Parquet rows natively into Snowflake
  INSERT INTO native_app_events (event_id, event_payload, loaded_at)
  SELECT 
      value:id::VARCHAR,
      value::VARIANT,
      CURRENT_TIMESTAMP()
  FROM s3_external_stream;
  ```
* **Explanation** – External Tables do not utilize Snowflake's micro-partition Time Travel. Therefore, they cannot support Standard or Append-Only streams. The `INSERT_ONLY` stream relies on the cloud provider's event notifications (e.g., SQS/SNS) updating the External Table metadata. When a new Parquet file is registered, the stream flags its rows as new, enabling a seamless, event-driven bridge between an external Data Lake and native Snowflake storage.

### Use Case 4: Directory Table Stream for Unstructured File Pipelines
* **Scenario** – Users upload PDF invoices to an internal Snowflake stage. A pipeline must process ONLY new PDFs, passing them to a Snowflake Python UDF for OCR (Optical Character Recognition) data extraction.
* ```sql
  -- Step 1: Create a stream directly on the Stage (Directory Table)
  CREATE STREAM invoice_stage_stream ON STAGE @invoice_landing_stage;

  -- Step 2: Process only the newly landed files using a Python UDF
  INSERT INTO parsed_invoices (file_name, extracted_text)
  SELECT 
      RELATIVE_PATH,
      extract_text_from_pdf(BUILD_SCOPED_FILE_URL(@invoice_landing_stage, RELATIVE_PATH))
  FROM invoice_stage_stream
  WHERE METADATA$ACTION = 'INSERT';
  ```
* **Explanation** – Directory Tables expose unstructured staging environments as relational objects. By placing a stream on the stage, the stream tracks the underlying cloud storage metadata. When a new PDF is PUT into the stage, the stream captures its `RELATIVE_PATH`. This allows data engineers to build continuous, incremental unstructured data pipelines (like ML inference or OCR) without building custom high-water-mark logic to track which files have already been processed.

### Use Case 5: Standard Stream on a View to Enforce CDC Security
* **Scenario** – A downstream analytics team needs real-time CDC from the `hr_employees` table, but the table contains sensitive PII (Social Security Numbers). Giving them a stream on the raw table violates compliance.
* ```sql
  -- Step 1: Create a view that masks or drops the PII
  CREATE SECURE VIEW vw_safe_employees AS
  SELECT employee_id, department, region
  FROM hr_employees;

  -- Step 2: Create a Standard Stream on the View
  CREATE STREAM safe_emp_stream ON VIEW vw_safe_employees;

  -- Step 3: Analytics team consumes the stream safely
  SELECT * FROM safe_emp_stream;
  ```
* **Explanation** – Streams track the base table's partitions, but when applied to a View, Snowflake dynamically injects the View's SQL logic *on top* of the underlying delta at runtime. The `safe_emp_stream` calculates the net changes of `hr_employees`, instantly applies the column-dropping logic of `vw_safe_employees`, and delivers a fully compliant, secure CDC payload to the analytics team without ever persisting a duplicate copy of the data.

## 4. Best Practices Summary

| Practice | Why |
| :--- | :--- |
| **Match Stream Type to Data Profile** | Never use a Standard stream on an immutable log table. Always use `APPEND_ONLY = TRUE` to eliminate unnecessary delta-computation overhead. |
| **Understand External Table Limits** | You must use `INSERT_ONLY = TRUE` for External Tables. Understand that file overwrites in S3/GCS will not be tracked as updates. |
| **Filter on `METADATA$ACTION`** | In Standard Streams, always filter by `METADATA$ACTION = 'INSERT'` or `'DELETE'` in your `MERGE` statements to properly handle the two-row representation of an `UPDATE`. |
| **Use Directory Table Streams for AI/ML** | Unstructured data processing pipelines should rely on Stage streams rather than manual `LIST @stage` polling to guarantee exactly-once file processing. |
| **Do Not Mutate Append-Only Sources** | If you define a stream as `APPEND_ONLY`, any accidental `UPDATE` or `DELETE` on the base table will be permanently and silently ignored by the stream. |
| **Monitor View Stream Performance** | A stream on a View executes the View's logic against the delta. If the View contains heavy `JOIN`s, consuming the stream will require significant Virtual Warehouse compute. |
| **Align Time Travel Configurations** | All stream types rely on the base object's Time Travel. Ensure the consuming task runs well within the `DATA_RETENTION_TIME_IN_DAYS` window to prevent staleness. |
| **Limit Concurrent Stream Consumption** | A single stream should only have one consumer (e.g., one Task). Multiple consumers will advance the offset unpredictably, causing data loss for each other. |

## 5. Important Limitations & Considerations

| Limitation | Mitigation |
| :--- | :--- |
| **Append-Only Ignores Deletes Natively** | If a row is inserted, then deleted, an Append-Only stream still outputs the insertion. Ensure downstream logic can handle or deduplicate this if accidental inserts occur. |
| **External Table Overwrites** | `INSERT_ONLY` streams track new files. If an existing file in the data lake is overwritten or modified, the stream **will not** capture the changes. Write pipelines must be append-only. |
| **View Streams Cannot Track All Views** | You cannot create a stream on a View that utilizes Window Functions (e.g., `ROW_NUMBER()`), `GROUP BY`, `LIMIT`, or non-deterministic functions (e.g., `CURRENT_DATE()`). |
| **Staleness is Irreversible** | If a stream's offset falls behind the table's Time Travel window, it becomes completely stale and cannot be resumed. You must `CREATE OR REPLACE` the stream, resulting in the loss of the unconsumed delta. |
| **Data Sharing Restrictions** | You cannot create a stream on an External Table or View that was shared to you from another Snowflake account via Secure Data Sharing. |

## 6. Auditing & Monitoring Queries

### Identify the Type (Mode) of All Active Streams
```sql
SELECT 
    table_catalog,
    table_schema,
    name AS stream_name,
    table_name AS base_object,
    mode AS stream_type,
    stale
FROM snowflake.information_schema.streams
ORDER BY mode, stream_name;
```

### Detect Stale Streams Requiring Recreation
```sql
SELECT 
    name AS stale_stream_name,
    table_name AS source_object,
    mode AS stream_type,
    created_on
FROM snowflake.information_schema.streams
WHERE stale = TRUE;
```

### Check if an Append-Only Stream Has Pending Data
```sql
-- Used inside Tasks to avoid waking up a warehouse if the stream is empty
SELECT SYSTEM$STREAM_HAS_DATA('PROD_DB.LOGS.IOT_APPEND_STREAM');
```

### Monitor Task Consumption of Specific Stream Types
```sql
SELECT 
    name AS task_name,
    state,
    query_text,
    completed_time
FROM snowflake.information_schema.task_history
WHERE query_text ILIKE '%APPEND_ONLY%' 
   OR query_text ILIKE '%INSERT_ONLY%'
ORDER BY completed_time DESC;
```

## 7. Key Takeaways
* **Standard Streams** evaluate historical micro-partitions to execute a complex **net-change collapse**, returning the exact delta of inserts, updates, and deletes for mutable dimension/fact tables.
* **Append-Only Streams** (`APPEND_ONLY = TRUE`) bypass delta-collapse calculations entirely, providing massive compute savings for high-velocity, **immutable log and telemetry datasets**.
* **Insert-Only Streams** (`INSERT_ONLY = TRUE`) are strictly required when tracking **External Tables**, operating purely on cloud-storage file registration events rather than native micro-partition versioning.
* **Directory Table Streams** natively track unstructured file storage (Stages), acting as the ultimate event-driven trigger for **unstructured data pipelines** and ML/OCR UDFs.
* Streams placed on **Views** dynamically apply the View's SQL logic (such as masking or filtering) to the underlying table's CDC delta, enabling highly secure, abstracted data distribution.
* Choosing the incorrect stream type (e.g., placing a Standard Stream on an immutable table) will not break the pipeline, but it will cause **unnecessary serverless compute overhead** as the engine endlessly checks for updates that mathematically do not exist.
