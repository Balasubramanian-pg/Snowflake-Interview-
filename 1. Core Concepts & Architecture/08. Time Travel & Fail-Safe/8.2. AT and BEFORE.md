# Snowflake AT and BEFORE: Precision Time Travel Querying

## 1. Core Concepts

The `AT` and `BEFORE` clauses in Snowflake Time Travel allow **surgical reconstruction of past data states** using timestamps, offsets, or query lineage.

They are not syntactic sugar. They are **direct entry points into Snowflake’s metadata timeline**, letting you choose *when* in history to materialize a dataset.

* `AT` → “Give me the table exactly at this moment”
* `BEFORE` → “Give me the table just before this event occurred”

---

### 1.1. How it Works

**Architecture Overview:**

* Every DML operation generates a new version of micro-partitions
* Metadata tracks **validity intervals** for each partition
* `AT` resolves to a specific timestamp or offset
* `BEFORE` resolves to a **statement boundary**, not a timestamp

**Process Flow:**

1. A query or DML operation modifies table state
2. Snowflake records metadata including:

   * Timestamp
   * Query ID
   * Partition lineage
3. When `AT` is used:

   * System reconstructs partitions active at that timestamp
4. When `BEFORE` is used:

   * System identifies partitions active just before a given query ID
5. Query executes against reconstructed snapshot

**Primary Use Cases:**

* Recover from bad updates or deletes
* Debug pipelines using query IDs
* Perform temporal comparisons
* Reproduce historical reports exactly

---

### 1.2. AT vs BEFORE vs Traditional Recovery

| Metric                 | Traditional Systems | AT Clause          | BEFORE Clause           |
| ---------------------- | ------------------- | ------------------ | ----------------------- |
| Reference Type         | Backup snapshot     | Timestamp / Offset | Query ID                |
| Precision              | Low                 | High (time-based)  | Very High (event-based) |
| Debugging Support      | Limited             | Moderate           | Excellent               |
| Recovery Granularity   | Full restore        | Table-level        | Statement-level         |
| Operational Complexity | High                | Low                | Low                     |

---

### 1.3. Key Factors

1. **Timestamp vs Statement Semantics**
   `AT` uses time-based resolution, `BEFORE` uses execution lineage

2. **Clock Precision Matters**
   Timestamp queries depend on system time accuracy

3. **Query ID Availability**
   `BEFORE` requires capturing query IDs from history

4. **Retention Window Constraint**
   Both clauses operate only within Time Travel retention

5. **Offset Simplicity vs Timestamp Accuracy**
   OFFSET is faster to use but less precise than exact timestamps

6. **Metadata Reconstruction Cost**
   Deep historical queries may increase planning overhead

## 2. Basic Commands & Privileges

### 2.1. Required Privileges

| Privilege | Scope           | Description                      |
| --------- | --------------- | -------------------------------- |
| SELECT    | Table/View      | Query historical data            |
| USAGE     | Database/Schema | Access objects                   |
| MONITOR   | Account         | Retrieve query IDs               |
| OWNERSHIP | Object          | Required for recovery operations |

---

### 2.2. Query Using AT with Timestamp

```sql
-- Retrieve table state at exact timestamp
SELECT *
FROM sales_data
AT (TIMESTAMP => '2026-04-04 10:00:00');
```

Fetches data as it existed at a precise moment.

---

### 2.3. Query Using AT with OFFSET

```sql
-- Retrieve table state from 10 minutes ago
SELECT *
FROM sales_data
AT (OFFSET => -600);
```

Uses relative time in seconds for quick recovery.

---

### 2.4. Query Using BEFORE with Statement ID

```sql
-- Retrieve state before a specific query executed
SELECT *
FROM sales_data
BEFORE (STATEMENT => '01a1b2c3-0001-xyz');
```

Targets the exact state before a known operation.

---

### 2.5. Retrieve Query IDs

```sql
-- Get query history to identify statement IDs
SELECT query_id, query_text, start_time
FROM SNOWFLAKE.ACCOUNT_USAGE.QUERY_HISTORY
ORDER BY start_time DESC;
```

Required for `BEFORE` clause usage.

---

### 2.6. Combine AT with CLONE

```sql
-- Clone table at historical point
CREATE TABLE sales_clone
CLONE sales_data
AT (TIMESTAMP => '2026-04-04 09:00:00');
```

Creates a persistent snapshot from a past state.

---

### 2.7. Validate Historical Access

```sql
-- Check retention window
SHOW PARAMETERS LIKE 'DATA_RETENTION_TIME_IN_DAYS' IN TABLE sales_data;
```

Ensures historical queries are still valid.

## 3. Impactful Use Cases

### Use Case 1: Recover from Accidental DELETE

**Scenario** – A DELETE statement removed critical rows.

```sql
-- Step 1: Find the DELETE query
SELECT query_id, query_text
FROM SNOWFLAKE.ACCOUNT_USAGE.QUERY_HISTORY
WHERE query_text ILIKE '%DELETE%'
ORDER BY start_time DESC;

-- Step 2: Query data before deletion
SELECT *
FROM sales_data
BEFORE (STATEMENT => 'identified_query_id');
```

**Explanation** – `BEFORE` provides exact rollback point tied to execution, eliminating guesswork with timestamps.

---

### Use Case 2: Compare Before and After States

**Scenario** – Validate impact of a transformation job.

```sql
-- Current state
SELECT COUNT(*) AS current_count FROM sales_data;

-- Historical state
SELECT COUNT(*) AS previous_count
FROM sales_data
AT (OFFSET => -3600);
```

**Explanation** – Enables quick validation without maintaining audit tables or snapshots.

---

### Use Case 3: Reproduce Historical Report

**Scenario** – Finance needs exact dataset used in a past report.

```sql
SELECT *
FROM financial_data
AT (TIMESTAMP => '2026-03-31 23:59:59');
```

**Explanation** – Guarantees reproducibility of reporting datasets without versioning pipelines.

---

### Use Case 4: Debug ETL Pipeline

**Scenario** – A transformation step introduced anomalies.

```sql
-- Inspect data before ETL step
SELECT *
FROM staging_data
BEFORE (STATEMENT => 'etl_query_id');
```

**Explanation** – Statement-level debugging isolates root cause precisely.

---

### Use Case 5: Create Historical Snapshot for Testing

**Scenario** – QA needs dataset from a specific point in time.

```sql
CREATE TABLE test_snapshot
CLONE sales_data
AT (OFFSET => -7200); -- 2 hours ago
```

**Explanation** – Combines Time Travel with zero-copy cloning for efficient test environments.

---

### Use Case 6: Investigate Data Drift

**Scenario** – Metrics changed unexpectedly over time.

```sql
-- Compare aggregates across time
SELECT SUM(revenue) AS current_revenue FROM sales_data;

SELECT SUM(revenue) AS past_revenue
FROM sales_data
AT (OFFSET => -86400);
```

**Explanation** – Helps identify when and how drift occurred without external tracking systems.

## 4. Best Practices Summary

| Practice                                  | Why                             |
| ----------------------------------------- | ------------------------------- |
| Use BEFORE for destructive query recovery | Highest precision rollback      |
| Capture query IDs in pipelines            | Enables deterministic debugging |
| Prefer TIMESTAMP for reproducibility      | More precise than OFFSET        |
| Use OFFSET for quick recovery             | Faster and simpler              |
| Combine with CLONE for persistence        | Preserve historical states      |
| Limit retention appropriately             | Balance cost vs utility         |
| Monitor query history regularly           | Ensure traceability             |
| Avoid reliance on approximate timestamps  | Prevent incorrect snapshots     |

---

## 5. Important Limitations & Considerations

| Limitation                           | Mitigation                            |
| ------------------------------------ | ------------------------------------- |
| Limited to retention window          | Increase retention or archive         |
| BEFORE requires query ID             | Log query IDs in pipelines            |
| OFFSET is approximate                | Use TIMESTAMP for accuracy            |
| Deep history increases planning time | Limit retention for high-churn tables |
| Cannot query beyond Fail-safe        | Use cloning before expiry             |

## 6. Auditing & Monitoring Queries

### 6.1. Retrieve Recent Query IDs

```sql
-- Fetch recent queries for BEFORE usage
SELECT query_id, query_text, start_time
FROM SNOWFLAKE.ACCOUNT_USAGE.QUERY_HISTORY
ORDER BY start_time DESC;
```

---

### 6.2. Identify Destructive Queries

```sql
-- Detect DELETE/UPDATE operations
SELECT query_id, query_text
FROM SNOWFLAKE.ACCOUNT_USAGE.QUERY_HISTORY
WHERE query_text ILIKE '%DELETE%'
   OR query_text ILIKE '%UPDATE%';
```

---

### 6.3. Validate Retention Window

```sql
-- Check retention configuration
SHOW PARAMETERS LIKE 'DATA_RETENTION_TIME_IN_DAYS' IN ACCOUNT;
```

---

### 6.4. Monitor Historical Query Usage

```sql
-- Identify Time Travel queries
SELECT query_text, start_time
FROM SNOWFLAKE.ACCOUNT_USAGE.QUERY_HISTORY
WHERE query_text ILIKE '%AT (%'
   OR query_text ILIKE '%BEFORE (%';
```

---

### 6.5. Track Query Execution Timeline

```sql
-- Analyze execution order for debugging
SELECT query_id, start_time, end_time
FROM SNOWFLAKE.ACCOUNT_USAGE.QUERY_HISTORY
ORDER BY start_time DESC;
```

---

## 7. Key Takeaways

* **AT = time-based snapshot reconstruction**
* **BEFORE = statement-level precision recovery**
* **BEFORE is superior for debugging destructive operations**
* **OFFSET is fast but less precise than TIMESTAMP**
* **Query IDs are critical for deterministic recovery workflows**
* **All operations are bounded by retention window**
* **Combining Time Travel with CLONE unlocks powerful recovery patterns**
