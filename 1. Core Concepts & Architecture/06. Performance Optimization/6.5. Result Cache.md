# Snowflake Result Cache: Maximizing Zero-Compute Query Performance

## 1. Core Concepts

The Snowflake **Result Cache** (often referred to as Persisted Query Results) is a powerful mechanism located within the **Cloud Services Layer** that stores the output of every executed query for 24 hours. When a user or application submits a query, Snowflake checks if the exact same query has been run recently. If the semantic context matches and the underlying data has not changed, Snowflake instantly returns the cached result without waking up or utilizing a Virtual Warehouse. This delivers extremely fast, **zero-compute** responses, making it highly valuable for dashboarding, repetitive exploratory analysis, and multi-step data pipelines.

### 1.1. How it Works
The architecture of the Result Cache operates entirely independently of Virtual Warehouses, relying on Snowflake's metadata management.

1. **Query Submission & Hashing:** An incoming query is parsed, and its syntax and execution context (Role, Database, Schema) are evaluated and hashed.
2. **Data State Validation:** The Cloud Services Layer checks the table's metadata to ensure no DML (`INSERT`, `UPDATE`, `DELETE`) operations have altered the micro-partitions since the cached query was originally executed.
3. **Cache Hit & Retrieval:** If the hash matches and the underlying data is static, Snowflake bypasses the compute engine and retrieves the result directly from the distributed storage cache.
4. **Retention Reset:** The 24-hour retention timer for that specific result is reset (up to a hard limit of 31 days from the initial execution).

**Primary Use Cases:**
* BI Dashboards that auto-refresh frequently but query data that only updates daily.
* Iterative data science exploration where engineers repeatedly filter or aggregate a massive, static dataset.
* Application pagination and post-processing of expensive CTEs (Common Table Expressions) using the `RESULT_SCAN` function.

### 1.2. Cache Architecture Comparison

| Metric | Traditional RDBMS Buffer Cache | Snowflake Result Cache |
| :--- | :--- | :--- |
| **Storage Location** | Memory (RAM) of the active database server | Snowflake Cloud Services Layer (Persisted Storage) |
| **Compute Requirement** | Requires the database engine to be active | **Zero compute**; Virtual Warehouse can be suspended |
| **Data Scope** | Caches raw disk blocks/pages | Caches the exact final, formatted result set |
| **Retention Trigger** | Evicted based on LRU (Least Recently Used) memory limits | Guaranteed 24-hour retention per query, resetting on use |
| **Sharing** | Shared across concurrent transactions | Shared across users in the account executing the exact query |

### 1.3. Key Factors
1. **Syntactic & Contextual Match:** The query must match exactly. Differences in whitespace or aliases can cause a cache miss. The user's role and search path must also align.
2. **Data Volatility:** Any DML operation (even if it doesn't affect the exact rows returned) updates the table's micro-partition version, immediately invalidating the result cache for all queries touching that table.
3. **Non-Deterministic Functions:** Queries utilizing functions like `CURRENT_TIMESTAMP()`, `RANDOM()`, or `UUID_STRING()` will **never** use the result cache because their outputs are inherently dynamic.
4. **Session Configuration:** The session parameter `USE_CACHED_RESULT` must be enabled (it is `TRUE` by default).

## 2. Basic Commands & Privileges

### 2.1. Required Privileges

| Privilege | Scope | Description |
| :--- | :--- | :--- |
| `USAGE` | Virtual Warehouse | Required to submit the query initially. If a cache hit occurs, the warehouse is not resumed or billed. |
| `SELECT` | Table / View | The executing role must retain the underlying data access privileges to view the cached results. |
| `USAGE` | Database & Schema | Required to establish the execution context that matches the original query. |

### 2.2. Enable Result Cache (Default)
Ensures the current session will attempt to retrieve results from the Cloud Services Layer before utilizing warehouse compute.
```sql
ALTER SESSION SET USE_CACHED_RESULT = TRUE;
```

### 2.3. Disable Result Cache
Forces the query engine to bypass the result cache and execute using a Virtual Warehouse, which is critical for performance benchmarking.
```sql
ALTER SESSION SET USE_CACHED_RESULT = FALSE;
```

### 2.4. Query Previous Results via RESULT_SCAN
Retrieves the output of a specific previous query using its Query ID, treating the cached result set as a temporary table.
```sql
SELECT * FROM TABLE(RESULT_SCAN('01ab2c3d-0000-1111-2222-333344445555'));
```

### 2.5. Retrieve the Last Executed Query Result
Automatically fetches the result of the most recently executed query in the current session without needing the explicit Query ID.
```sql
SELECT * FROM TABLE(RESULT_SCAN(LAST_QUERY_ID()));
```

### 2.6. Check if Result Cache was Used
Queries the information schema to verify if a specific query was fulfilled by the result cache (`BYTES_SCANNED` will typically be 0, and `PERCENTAGE_SCANNED_FROM_CACHE` will be irrelevant as no warehouse cache was used).
```sql
SELECT query_id, execution_status, bytes_scanned 
FROM table(information_schema.query_history())
WHERE query_id = 'your_query_id_here';
```

## 3. Impactful Use Cases

### Use Case 1: Decoupling BI Dashboards from Compute Costs
* **Scenario** – A Tableau dashboard is configured to refresh every 15 minutes, but the underlying `fact_sales` table is only updated via a batch ETL job once at midnight.
* ```sql
  -- Application generates the exact same query every 15 minutes
  SELECT 
      region, 
      SUM(sales_amount) as total_revenue, 
      COUNT(DISTINCT customer_id) as unique_customers
  FROM prod_db.analytics.fact_sales
  WHERE order_date >= '2024-01-01'
  GROUP BY region;
  ```
* **Explanation** – Because the underlying data is entirely static throughout the day, the first dashboard user at 12:01 AM triggers the warehouse to execute the query. For the next 23 hours and 59 minutes, every subsequent refresh by any user hits the Result Cache. The warehouse remains completely suspended, resulting in 100% cost avoidance for the BI workload during the day while delivering millisecond response times.

### Use Case 2: Post-Processing Expensive Transformations
* **Scenario** – A Data Scientist runs a massive, 20-minute aggregation query on 50 billion rows. They realize they forgot to filter the final output for a specific product category.
* ```sql
  -- Step 1: Original expensive query (already executed)
  SELECT product_id, customer_id, SUM(spend) as ltv ... (takes 20 mins)
  
  -- Step 2: Use RESULT_SCAN to filter without re-running the aggregation
  SELECT * 
  FROM TABLE(RESULT_SCAN(LAST_QUERY_ID()))
  WHERE product_id IN ('PROD_A', 'PROD_B')
  ORDER BY ltv DESC;
  ```
* **Explanation** – `RESULT_SCAN` treats the finalized, cached output of the previous query as a temporary, fully-materialized table. Instead of turning the warehouse back on for another 20-minute scan of the raw data, Snowflake executes the `WHERE` filter directly against the highly-compressed cached result set. This saves immense compute time and drastically accelerates iterative development.

### Use Case 3: Application Pagination
* **Scenario** – A web application needs to display a heavily computed customer behavior report, but the UI can only display 50 rows per page.
* ```sql
  -- Step 1: Execute the core analytical query (App caches the Query ID)
  SELECT customer_name, risk_score, last_login 
  FROM ml_predictions 
  ORDER BY risk_score DESC; -- Returns 1,000,000 rows
  
  -- Step 2: Page 1 Request
  SELECT * FROM TABLE(RESULT_SCAN('query_id_from_step_1')) LIMIT 50 OFFSET 0;
  
  -- Step 3: Page 2 Request
  SELECT * FROM TABLE(RESULT_SCAN('query_id_from_step_1')) LIMIT 50 OFFSET 50;
  ```
* **Explanation** – RDBMS pagination using `OFFSET` forces the database to re-evaluate the entire query and discard rows up to the offset value, wasting massive compute. By passing the `query_id` to the front end, subsequent page requests query the cached `RESULT_SCAN`. The heavy lifting is done exactly once, making pagination instantly responsive and highly cost-efficient.

### Use Case 4: Performance Benchmarking & Query Tuning
* **Scenario** – A Data Engineer is rewriting a poorly performing query. They need to test multiple join strategies to see which is fastest, but the Result Cache keeps returning artificial 0.1-second times.
* ```sql
  -- Disable cache to ensure the virtual warehouse performs the actual work
  ALTER SESSION SET USE_CACHED_RESULT = FALSE;
  
  -- Execute tuning variation 1
  SELECT /* test_v1 */ ... ;
  
  -- Execute tuning variation 2
  SELECT /* test_v2 */ ... ;
  
  -- Re-enable when testing is complete
  ALTER SESSION SET USE_CACHED_RESULT = TRUE;
  ```
* **Explanation** – During active query tuning, the Result Cache is an obstacle because it masks the true execution time of the Virtual Warehouse. By explicitly disabling `USE_CACHED_RESULT` at the session level, the engineer forces Snowflake to parse, plan, and execute the physical data scan every time, allowing for an accurate baseline and comparison of different query structures via the Query Profile.

### Use Case 5: Avoiding Non-Deterministic Function Invalidation
* **Scenario** – A daily operational report consistently wakes up an XL warehouse because the `WHERE` clause dynamically calculates the current date, invalidating the cache.
* ```sql
  -- Anti-pattern (Breaks Cache):
  SELECT * FROM daily_sales WHERE sale_date = CURRENT_DATE();
  
  -- Best Practice (Enables Cache):
  -- The BI tool or application passes the literal date at execution time
  SELECT * FROM daily_sales WHERE sale_date = '2024-10-25'::DATE;
  ```
* **Explanation** – Functions like `CURRENT_DATE()` or `CURRENT_TIMESTAMP()` are evaluated at runtime and make a query fundamentally unique, preventing the Result Cache from ever being utilized. By shifting the temporal logic to the application or BI layer (e.g., using Tableau's `TODAY()` parameter to inject a string literal into the SQL), the generated query becomes static for the entire day, enabling a 100% cache hit rate.

## 4. Best Practices Summary

| Practice | Why |
| :--- | :--- |
| **Batch DML over Trickle-Feed** | Continuous micro-batch ingestion (e.g., Snowpipe every 1 min) constantly invalidates the cache. Batch loading aligns better with high cache hit rates. |
| **Inject Literals for Dates** | Avoid `CURRENT_DATE()` in BI queries. Pass exact date strings from the front end to maintain a static SQL signature. |
| **Standardize Query Formatting** | Ensure applications construct SQL strings identically (casing, whitespace, aliases) so hashes match perfectly across different users. |
| **Utilize RESULT_SCAN for Iteration** | Never rerun a heavy query just to change an `ORDER BY` or add a `LIMIT`. Use `RESULT_SCAN` on the `LAST_QUERY_ID()`. |
| **Align BI Refresh with ETL Schedules** | Do not refresh dashboards faster than the underlying data changes, unless relying on the Result Cache to absorb the zero-compute hits. |
| **Disable Cache During Benchmarking** | Always `ALTER SESSION SET USE_CACHED_RESULT = FALSE` when actively tuning to measure true warehouse compilation and execution times. |
| **Maintain Context/Roles** | A query run under `ANALYST_ROLE` might not serve a cache hit to `SALES_ROLE` if row-level security or search paths differ. Standardize service accounts. |
| **Monitor Warehouse Auto-Suspend** | If the Result Cache handles 90% of queries, aggressive auto-suspend (e.g., 60 seconds) ensures the warehouse scales down efficiently between the 10% of cache misses. |

## 5. Important Limitations & Considerations

| Limitation | Mitigation |
| :--- | :--- |
| **24-Hour Expiration** | The cache natively expires after 24 hours. If a result is needed longer, schedule a lightweight task to execute the query every 23 hours to reset the 24-hour timer (up to 31 days). |
| **Underlying Data Invalidation** | Even a single row updated via `MERGE` invalidates the cache for the *entire* table. Isolate heavily modified operational data from static analytical views if caching is critical. |
| **Row-Level Security (RLS)** | Queries against tables with Dynamic Data Masking or Row Access Policies often bypass the cache because the result depends strictly on the executing user's context. |
| **RESULT_SCAN 24-Hour Limit** | You cannot run `RESULT_SCAN` on a `query_id` older than 24 hours. If long-term persistence is needed, use `CREATE TABLE AS SELECT (CTAS)` instead. |
| **Dropped/Recreated Objects** | If a table or view is dropped and recreated (even with the exact same data), all prior query results referencing that object are immediately purged from the cache. |

## 6. Auditing & Monitoring Queries

### Identify Queries Resolved by the Result Cache
```sql
SELECT 
    query_id, 
    query_text, 
    user_name, 
    role_name, 
    total_elapsed_time
FROM snowflake.account_usage.query_history
WHERE start_time >= DATEADD(day, -7, CURRENT_TIMESTAMP())
  AND bytes_scanned = 0 
  AND execution_status = 'SUCCESS'
  -- Queries with no execution time on a warehouse typically hit the Cloud Services cache
  AND cluster_number IS NULL 
ORDER BY total_elapsed_time DESC;
```

### Calculate Result Cache Hit Ratio by Warehouse
```sql
SELECT 
    warehouse_name,
    COUNT(*) as total_queries,
    SUM(IFF(cluster_number IS NULL AND bytes_scanned = 0, 1, 0)) as cache_hits,
    (SUM(IFF(cluster_number IS NULL AND bytes_scanned = 0, 1, 0)) / COUNT(*)) * 100 as cache_hit_percentage
FROM snowflake.account_usage.query_history
WHERE start_time >= DATEADD(day, -30, CURRENT_TIMESTAMP())
  AND query_type = 'SELECT'
GROUP BY warehouse_name
ORDER BY cache_hit_percentage DESC;
```

### Monitor `RESULT_SCAN` Usage
```sql
SELECT 
    query_id, 
    query_text, 
    user_name, 
    start_time
FROM snowflake.account_usage.query_history
WHERE start_time >= DATEADD(day, -7, CURRENT_TIMESTAMP())
  AND query_text ILIKE '%RESULT_SCAN%'
ORDER BY start_time DESC;
```

### Find Top Users Bypassing Compute (Cache Champions)
```sql
SELECT 
    user_name,
    COUNT(query_id) as cached_queries_executed
FROM snowflake.account_usage.query_history
WHERE start_time >= DATEADD(day, -30, CURRENT_TIMESTAMP())
  AND cluster_number IS NULL 
  AND bytes_scanned = 0
  AND query_type = 'SELECT'
GROUP BY user_name
ORDER BY cached_queries_executed DESC;
```

## 7. Key Takeaways
* The Result Cache resides entirely in the **Cloud Services Layer**, meaning returning cached results consumes **zero virtual warehouse compute credits**.
* Results are guaranteed to persist for **24 hours**, and the retention clock automatically resets every time the exact query is successfully re-executed.
* **Any DML operation** (Insert, Update, Delete) on the underlying tables will immediately alter the micro-partition state, permanently **invalidating** the cache for all dependent queries.
* **Non-deterministic functions** like `CURRENT_DATE()` bypass the cache natively. Replace them with front-end **literal parameters** to force query consistency and maximize cache hits.
* The **`RESULT_SCAN`** function is a critical Data Engineering tool, allowing teams to instantly paginate, filter, or re-sort heavy datasets without re-evaluating the foundational query.
