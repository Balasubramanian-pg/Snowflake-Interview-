# Snowflake Multi-Factor Authentication (MFA): Through and Through

## 1. Core Concepts

Multi-Factor Authentication (MFA) in Snowflake reduces the security risks associated with password authentication by requiring users to present two or more proofs (factors) of their identity[reference:0]. MFA is intended for **human users** who authenticate with a password; service users must use other forms of authentication such as key-pair authentication, OAuth, or workload identity federation[reference:1].

### 1.1. How MFA Works in Snowflake

When a password user is enrolled in MFA, they enter their password and then use a **second factor of authentication** to sign in to Snowflake[reference:2]. Snowflake's MFA feature is powered by the **Duo Security service**, which is fully managed by Snowflake[reference:3]. Users do not need a separate Duo account; they simply install the Duo Mobile application on their smartphone (iOS, Android, Windows, etc.).

### 1.2. Supported MFA Methods

Snowflake allows the following MFA methods as the second factor of authentication[reference:4]:

| Method | Description |
|--------|-------------|
| **Passkey** | Based on FIDO industry standards, allows authentication using biometrics, PIN, or security keys (e.g., laptop fingerprint sensor, hardware security key)[reference:5] |
| **Authenticator App (TOTP)** | Time-Based One-Time Password (TOTP) authenticator apps (e.g., Microsoft, Google Authenticator)[reference:6] |
| **Duo** | Duo Push, passcode, or phone call via the Duo Mobile app (default method)[reference:7] |

> **Tip**: Passkeys are recommended due to their strong security properties and phishing resistance[reference:8].

### 1.3. MFA Enrollment Enforcement

MFA enrollment requirements vary depending on when the account was created[reference:9]:

| Account Creation | Default MFA Requirement |
|------------------|------------------------|
| **Before 2024_08 bundle** | No default enforcement; must be configured by administrator |
| **After 2024_08 bundle** | All human users who authenticate with a password must enroll in MFA by default. Service users are exempt. Trial and reader accounts are exempt[reference:10] |

> **Important**: The ability to opt out of mandatory MFA is **temporary**. Snowflake is rolling out changes to deprecate single-factor password sign-ins for all human users. The updated timeline (as of October 2025) consolidates phases and extends final enforcement from August 2026 to **October 2026**[reference:11].

### 1.4. The Duo Universal Prompt

Snowflake uses the **Duo Universal Prompt** interface for MFA enrollment and authentication[reference:12]. This interface may require configuration adjustments if:
- Your account name contains an underscore **and** you use private connectivity (add a DNS entry with a dash instead of an underscore)[reference:13]
- Your corporate firewall or proxy blocks `api-*.duosecurity.com`, `*.devicemanagement.duosecurity.com`, or `*.duosecurity.com:443` (allow these domains)[reference:14]

### 1.5. MFA vs Other Authentication Methods

| Authentication Method | Use Case | MFA Required? |
|-----------------------|----------|---------------|
| **Password + MFA** | Human users (recommended) | Yes |
| **Key-Pair Authentication** | Service accounts, automated tools | No (stronger than password) |
| **SAML SSO** | Federated identity (e.g., Azure AD, Okta) | Managed by IdP |
| **OAuth** | Programmatic access, drivers | Managed by OAuth provider |
| **Programmatic Access Tokens (PATs)** | Applications that only support passwords | No (tied to roles, expires, network policy required)[reference:15] |

## 2. Basic Commands & Privileges

### 2.1. Required Privileges

| Privilege | Scope | Description |
|-----------|-------|-------------|
| `CREATE AUTHENTICATION POLICY` | Schema | Allows creation of authentication policies[reference:16] |
| `OWNERSHIP` on user | User | Required to modify user MFA properties using `ALTER USER`[reference:17] |
| `ALTER USER` | User | Modify MFA settings for a user |

### 2.2. Create an Authentication Policy (MFA Enforcement)

```sql
CREATE OR REPLACE AUTHENTICATION POLICY enforce_mfa_policy
  MFA_ENROLLMENT = REQUIRED
  CLIENT_TYPES = ('SNOWFLAKE_UI');
```

> **Note**: If you set `MFA_ENROLLMENT = REQUIRED`, you **must** include `SNOWFLAKE_UI` in the `CLIENT_TYPES` list to allow users to enroll in MFA[reference:18].

### 2.3. Apply Authentication Policy to Account

```sql
ALTER ACCOUNT SET AUTHENTICATION POLICY enforce_mfa_policy;
```

### 2.4. Disable MFA for a User (Emergency/Service Account)

```sql
ALTER USER <username> SET DISABLE_MFA = TRUE;
```

### 2.5. Temporarily Bypass MFA for a User

```sql
ALTER USER <username> SET MINS_TO_BYPASS_MFA = 30;  -- Bypass for 30 minutes
```

### 2.6. Restrict Which MFA Methods Are Available

```sql
CREATE OR REPLACE AUTHENTICATION POLICY mfa_methods_policy
  MFA_ENROLLMENT = REQUIRED
  MFA_POLICY = ( ALLOWED_METHODS = ('PASSKEY', 'TOTP') )  -- Only passkey and TOTP
  CLIENT_TYPES = ('SNOWFLAKE_UI');
```

### 2.7. Show MFA Status for Users

```sql
SELECT user_name, has_mfa, type
FROM snowflake.account_usage.users
ORDER BY user_name;
```

### 2.8. View Authentication Policies

```sql
SHOW AUTHENTICATION POLICIES;
```

## 3. Impactful Use Cases (4–8 with Queries & Explanations)

### Use Case 1: Enforce MFA for All Human Password Users (Account‑Level)

**Scenario** – Your organisation wants to enforce MFA for all human users who authenticate with a password, but service accounts should be exempt. This aligns with Snowflake's security recommendations.

```sql
-- Step 1: Create an authentication policy
CREATE OR REPLACE AUTHENTICATION POLICY require_mfa_for_humans
  MFA_ENROLLMENT = REQUIRED
  CLIENT_TYPES = ('SNOWFLAKE_UI', 'SNOWSQL', 'ODBC', 'JDBC');

-- Step 2: Apply to the account
ALTER ACCOUNT SET AUTHENTICATION POLICY require_mfa_for_humans;

-- Step 3: Verify which users are not yet enrolled
SELECT user_name, has_mfa, type
FROM snowflake.account_usage.users
WHERE type = 'PERSON' AND has_mfa = FALSE;

-- Step 4: Send reminder to unenrolled users (manual process)
-- Users will be prompted to enroll via Duo on next Snowsight login
```

**Explanation** – This policy applies to **all human users** (`TYPE = PERSON` or `NULL`). Service users (`TYPE = SERVICE`) are not affected[reference:19]. When a user with a password logs in, they will be prompted to enroll in MFA via Duo before they can proceed. The ability to disable this requirement is temporary[reference:20].

---

### Use Case 2: Create a Custom Authentication Policy with Optional MFA (Transition Period)

**Scenario** – During a transition period, you want to encourage MFA enrollment but not require it for all users. This allows users to enroll at their own pace while preparing for future mandatory enforcement.

```sql
-- Step 1: Create a policy with OPTIONAL enrollment
CREATE OR REPLACE AUTHENTICATION POLICY optional_mfa_policy
  MFA_ENROLLMENT = OPTIONAL
  CLIENT_TYPES = ('SNOWFLAKE_UI', 'SNOWSQL', 'ODBC', 'JDBC');

-- Step 2: Apply to account
ALTER ACCOUNT SET AUTHENTICATION POLICY optional_mfa_policy;

-- Step 3: Monitor enrollment progress
SELECT 
    COUNT(*) AS total_human_users,
    SUM(CASE WHEN has_mfa = TRUE THEN 1 ELSE 0 END) AS enrolled_in_mfa,
    ROUND(100 * SUM(CASE WHEN has_mfa = TRUE THEN 1 ELSE 0 END) / COUNT(*), 2) AS pct_enrolled
FROM snowflake.account_usage.users
WHERE type = 'PERSON';
```

**Explanation** – With `MFA_ENROLLMENT = OPTIONAL`, users **can** but are **not required** to enroll in MFA[reference:21]. This is useful for phased rollouts. The built‑in authentication policy for new accounts (after 2024_08) enforces MFA by default; to make it optional, you must create a custom policy with `OPTIONAL`[reference:22].

---

### Use Case 3: Restrict MFA Methods – Only Allow Passkeys and TOTP

**Scenario** – Your security policy requires the use of phishing‑resistant MFA methods. You want to allow only **passkeys** (FIDO2) and **TOTP authenticator apps**, and block the default Duo method for certain high‑security groups.

```sql
-- Step 1: Create a policy restricting allowed MFA methods
CREATE OR REPLACE AUTHENTICATION POLICY high_security_mfa_policy
  MFA_ENROLLMENT = REQUIRED
  MFA_POLICY = ( ALLOWED_METHODS = ('PASSKEY', 'TOTP') )  -- Duo is excluded
  CLIENT_TYPES = ('SNOWFLAKE_UI');

-- Step 2: Apply to a specific user (not account-wide)
ALTER USER security_officer SET AUTHENTICATION POLICY high_security_mfa_policy;

-- Step 3: Verify the policy assignment
SELECT user_name, authentication_policy
FROM snowflake.account_usage.users
WHERE user_name = 'SECURITY_OFFICER';
```

**Explanation** – The `MFA_POLICY` parameter with `ALLOWED_METHODS` allows administrators to restrict which MFA methods users can use for their second factor of authentication[reference:23]. This is useful for enforcing organisational security standards. You can apply policies at **account level** (affects all users) or **user level** (overrides account policy)[reference:24].

---

### Use Case 4: Bypass MFA Temporarily for Emergency Access (Break‑Glass)

**Scenario** – During an emergency (e.g., primary MFA device lost or unavailable), a trusted administrator needs temporary access to Snowflake without MFA.

```sql
-- Step 1: As ACCOUNTADMIN, temporarily bypass MFA for the user
ALTER USER emergency_admin SET MINS_TO_BYPASS_MFA = 60;  -- Bypass for 60 minutes

-- Step 2: After emergency access, verify the user has re‑enabled MFA
SELECT user_name, has_mfa, mins_to_bypass_mfa
FROM snowflake.account_usage.users
WHERE user_name = 'EMERGENCY_ADMIN';

-- Step 3: Audit who used bypass and when
SELECT 
    user_name,
    query_text,
    start_time,
    role_name
FROM snowflake.account_usage.query_history
WHERE user_name = 'EMERGENCY_ADMIN'
  AND start_time >= CURRENT_TIMESTAMP() - INTERVAL '2 hours'
ORDER BY start_time DESC;

-- Step 4: Reset MFA requirement (bypass automatically expires after the specified minutes)
-- No action needed – bypass expires automatically. For manual reset:
ALTER USER emergency_admin SET MINS_TO_BYPASS_MFA = 0;
```

**Explanation** – `MINS_TO_BYPASS_MFA` allows temporary bypass of MFA for a specified number of minutes[reference:25]. This is a **break‑glass mechanism** for emergencies and should be:
- Used **sparingly** (only when primary MFA is unavailable)
- **Audited immediately** after use
- **Logged** for compliance review
- **Automatically expiring** (no manual cleanup needed)

> **Important**: For service accounts requiring long‑term password‑only access, use **Programmatic Access Tokens (PATs)** instead of MFA bypass. PATs are tied to specific roles, have expiration dates, and must be used with network policies[reference:26].

---

### Use Case 5: Enforce MFA for ACCOUNTADMIN During Account Creation

**Scenario** – You are creating a new Snowflake account, and a human will directly use the `ACCOUNTADMIN` role. You want to enforce MFA enrollment during account creation to secure the account from day one.

```sql
-- During account creation (as organisation administrator)
CREATE ACCOUNT my_company_account
  ADMIN_NAME = 'first_admin'
  ADMIN_PASSWORD = 'InitialComplexP@ssw0rd'
  ADMIN_USER_TYPE = PERSON;   -- Specifies human user, MFA enrollment required
```

**Explanation** – When creating a new Snowflake account after the 2024_08 bundle, you must specify `ADMIN_USER_TYPE = PERSON` to enforce MFA enrollment for the human `ACCOUNTADMIN`. If you do not specify this (or set `ADMIN_USER_TYPE = SERVICE`), the account may be locked out or the service may fail to run[reference:27][reference:28].

> **Caution**: If the `ACCOUNTADMIN` is a **service** (not a human), you must set `ADMIN_USER_TYPE = SERVICE` and provide an `ADMIN_RSA_PUBLIC_KEY` to prevent MFA enforcement from locking out the service[reference:29].

### Use Case 6: Monitor MFA Compliance Across the Account (Security Audit)

**Scenario** – Your security team needs a dashboard of all human users, their MFA enrollment status, and when they last logged in. This is required for compliance reporting (e.g., SOC2, ISO 27001).

```sql
-- Comprehensive MFA compliance report
SELECT 
    u.user_name,
    u.type AS user_type,
    CASE WHEN u.has_mfa = TRUE THEN 'ENROLLED' ELSE 'NOT ENROLLED' END AS mfa_status,
    u.created_on AS user_created,
    MAX(qh.start_time) AS last_login_time,
    DATEDIFF('day', MAX(qh.start_time), CURRENT_TIMESTAMP()) AS days_since_last_login
FROM snowflake.account_usage.users u
LEFT JOIN snowflake.account_usage.query_history qh 
    ON u.user_name = qh.user_name
    AND qh.start_time >= DATEADD('month', -3, CURRENT_TIMESTAMP())
WHERE u.deleted_on IS NULL
  AND u.type = 'PERSON'  -- Only human users
GROUP BY 1, 2, 3, 4
ORDER BY mfa_status DESC, days_since_last_login DESC;

-- List users who have MFA disabled (excluding service accounts)
SELECT user_name, type, has_mfa, disabled
FROM snowflake.account_usage.users
WHERE type = 'PERSON' AND has_mfa = FALSE AND disabled = FALSE;

-- Alert on ACCOUNTADMIN users without MFA
SELECT u.user_name, u.has_mfa, u.type
FROM snowflake.account_usage.users u
WHERE u.has_mfa = FALSE
  AND u.user_name IN (
    SELECT grantee_name 
    FROM snowflake.account_usage.grants_to_users 
    WHERE role = 'ACCOUNTADMIN'
  );
```

**Explanation** – The `snowflake.account_usage.users` view contains the `has_mfa` column, which indicates whether a user is enrolled in MFA[reference:30]. This query helps identify:
- Users who have not yet enrolled in MFA
- Stale accounts that may no longer need access
- High‑privilege users (especially `ACCOUNTADMIN`) without MFA protection

Snowflake strongly recommends that **all users with the `ACCOUNTADMIN` role** be required to use MFA[reference:31].

### Use Case 7: Programmatic Access Tokens (PATs) for Applications Without MFA Support

**Scenario** – Your business intelligence application (e.g., Power BI Desktop) does not currently support MFA prompts. You need a secure way to connect without disabling MFA for human users.

```sql
-- Step 1: Create a programmatic access token for the service account
CREATE USER bi_service_user
  PASSWORD = 'InitialComplexP@ssw0rd'
  TYPE = SERVICE;  -- Service user type – not subject to MFA

-- Step 2: Generate a PAT (via Snowsight or SQL)
-- (PAT creation requires Snowsight UI or ALTER USER command)

-- Step 3: Configure the PAT with network policy and expiration
ALTER USER bi_service_user 
  SET PROGRAMMATIC_ACCESS_TOKEN = '<token>'
  MINS_TO_BYPASS_NETWORK_POLICY = 0;  -- Enforce network policy

-- Step 4: Grant necessary roles to the service user
GRANT ROLE BI_READONLY TO USER bi_service_user;
```

**Explanation** – Programmatic Access Tokens (PATs) are a drop‑in replacement for passwords for applications that only support username and password authentication[reference:32]. PATs **raise the security bar** because:
- They are tied to specific roles (least privilege)
- They have an expiration date
- They must be used in tandem with a network policy
- Separate PATs should be created for different use cases to minimise blast radius in case of compromise[reference:33]

> **Note**: PATs are designed for programmatic access to Snowpark Container Services and Snowflake REST APIs. For human users, MFA remains the recommended authentication method[reference:34].

### Use Case 8: Phase Out Legacy Service Users (Deprecation Preparation)

**Scenario** – Snowflake is deprecating password‑based authentication for service users. You need to identify and migrate all `LEGACY_SERVICE` users to a more secure authentication method (key-pair, OAuth, or PATs) before the October 2026 deadline.

```sql
-- Step 1: Identify all legacy service users using passwords
SELECT 
    user_name,
    type,
    has_mfa,
    created_on,
    last_successful_login
FROM snowflake.account_usage.users
WHERE type IN ('LEGACY_SERVICE', 'SERVICE')
  AND deleted_on IS NULL
ORDER BY user_name;

-- Step 2: Generate key-pair for each service user (external process)
-- Step 3: Set RSA public key for service user
ALTER USER legacy_etl_user SET RSA_PUBLIC_KEY = 'MIIBIjANBgkqhkiG9w0BAQEFAAOCAQ8AMIIBCgKCAQEA...';

-- Step 4: Change user type to SERVICE (if currently LEGACY_SERVICE)
ALTER USER legacy_etl_user SET TYPE = SERVICE;

-- Step 5: Test authentication with key-pair
-- (using SnowSQL: snowsql -a account -u legacy_etl_user --private-key-path /path/to/key.p8)

-- Step 6: Remove legacy service type after verification
-- (Manual verification – no SQL command to revert)
```

**Explanation** – The deprecation timeline for single‑factor password sign-ins includes[reference:35]:

| Phase | Affected Users | Timeline |
|-------|----------------|----------|
| Phase 1 (BCR) | Human users (Snowsight UI) | Sep 2025 – Jan 2026 |
| Phase 2 | Human users (all clients) | May 2026 – Jul 2026 |
| Phase 3 | Legacy service users | Aug 2026 – Oct 2026 |

The `LEGACY_SERVICE` user type is a **temporary** workaround that allows services to continue using passwords[reference:36]. These users **must** be migrated to:
- **Key-pair authentication** (recommended for services)
- **OAuth** (for federated identity)
- **Workload identity federation** (secretless authentication)[reference:37]

## 4. Best Practices Summary

| Practice | Why |
|----------|-----|
| **Enforce MFA for all human users, especially ACCOUNTADMIN** | Prevents credential theft and unauthorised access; ACCOUNTADMIN MFA is strongly recommended by Snowflake[reference:38] |
| **Use passkeys or TOTP over Duo when possible** | Passkeys are phishing‑resistant and follow FIDO industry standards[reference:39] |
| **Never share MFA bypass as a permanent solution** | `MINS_TO_BYPASS_MFA` is for break‑glass emergencies only; audit every use |
| **Migrate LEGACY_SERVICE users before the October 2026 deadline** | Single‑factor password sign-ins are being deprecated; prepare migration plans now[reference:40] |
| **Use Programmatic Access Tokens (PATs) for applications** | PATs are more secure than passwords: role‑bound, expiring, and network‑policy enforced[reference:41] |
| **Monitor MFA compliance regularly** | Use `snowflake.account_usage.users` to track enrollment and identify gaps |
| **Configure network policies to restrict access** | Combine MFA with IP allowlists for defence in depth[reference:42] |
| **Allow Duo domains in corporate firewalls** | Ensure `api-*.duosecurity.com` and `*.duosecurity.com:443` are whitelisted[reference:43] |

## 5. Important Limitations & Considerations

| Limitation | Mitigation |
|------------|------------|
| **MFA enrollment is user‑initiated** | Administrators cannot enrol users automatically. Users must log in to Snowsight to complete enrollment[reference:44] |
| **Service accounts cannot use MFA** | Use key‑pair authentication, OAuth, or PATs instead of passwords + MFA[reference:45] |
| **MFA bypass is time‑limited only** | `MINS_TO_BYPASS_MFA` cannot be removed except by expiration; use PATs for long‑term service access |
| **Some third‑party tools lack MFA support** | Use PATs or OAuth as a secure alternative for BI tools and connectors[reference:46] |
| **Duo Universal Prompt may require network configuration** | Allow Duo domains and handle underscores in account names for private connectivity[reference:47] |
| **Trial and reader accounts are exempt from MFA enforcement** | Be aware of security implications when promoting trial accounts to paid[reference:48] |
| **MFA enforcement timeline is evolving** | Monitor Snowflake documentation for updated deprecation dates. As of October 2025, final enforcement is **October 2026**[reference:49] |

## 6. Auditing & Monitoring Queries

### List all users with MFA status
```sql
SELECT user_name, type, has_mfa, disabled, last_successful_login
FROM snowflake.account_usage.users
WHERE deleted_on IS NULL
ORDER BY has_mfa DESC, user_name;
```

### Find users who have bypassed MFA recently
```sql
SELECT user_name, mins_to_bypass_mfa
FROM snowflake.account_usage.users
WHERE mins_to_bypass_mfa > 0;
```

### Show authentication policies and their settings
```sql
SHOW AUTHENTICATION POLICIES;
DESCRIBE AUTHENTICATION POLICY enforce_mfa_policy;
```

### Track MFA enrollment events
```sql
SELECT 
    query_id,
    query_text,
    user_name,
    role_name,
    start_time
FROM snowflake.account_usage.query_history
WHERE query_text ILIKE '%MFA%'
  OR query_text ILIKE '%AUTHENTICATION POLICY%'
ORDER BY start_time DESC
LIMIT 50;
```

## 7. Key Takeaways

- **MFA in Snowflake is powered by Duo Security** and is fully managed by Snowflake. Users install the Duo Mobile app; no separate Duo account is required[reference:50].
- **New Snowflake accounts enforce MFA by default** for all human password users. This can be modified with custom authentication policies[reference:51].
- **Supported MFA methods** include passkeys (recommended), TOTP authenticator apps, and Duo Push/passcode[reference:52].
- **Authentication policies** control MFA enrollment (`REQUIRED`/`OPTIONAL`) and allowed methods (`MFA_POLICY`) at account or user level[reference:53].
- **Temporary bypass** (`MINS_TO_BYPASS_MFA`) is available for break‑glass emergencies and must be audited[reference:54].
- **Programmatic Access Tokens (PATs)** provide a secure, MFA‑free authentication method for applications that do not support MFA prompts[reference:55].
- **Single‑factor password sign-ins are being deprecated**. All human users must transition to MFA, and legacy service users to key‑pair/OAuth/PATs, by **October 2026**[reference:56].
- **Audit regularly** using `snowflake.account_usage.users` and `snowflake.account_usage.query_history` to track MFA compliance.

This completes the thorough explanation of Snowflake's Multi-Factor Authentication model with practical, actionable use cases.
