# Snowflake RBAC Model: Through and Through

## 1. Core Concepts

Snowflake implements **Role-Based Access Control (RBAC)** as its primary security model. Permissions are never granted directly to users; instead:

- **Users** are assigned **Roles**
- **Roles** are granted **Privileges** on **Securable Objects** (databases, schemas, tables, views, warehouses, etc.)
- Privileges can be inherited through **role hierarchies** (a role can be granted to another role)

### 1.1. Key Objects in RBAC

| Object        | Description                                                                 |
|---------------|-----------------------------------------------------------------------------|
| **User**      | A security principal that authenticates to Snowflake.                      |
| **Role**      | A named collection of privileges. Roles can be granted to users or other roles. |
| **Securable** | Any object on which privileges can be granted (e.g., DATABASE, SCHEMA, TABLE, WAREHOUSE). |
| **Privilege** | A specific right to perform an action (e.g., SELECT, USAGE, CREATE TABLE). |
| **Grant**     | The assignment of a privilege on a securable to a role.                    |

### 1.2. Standard Role Types

Snowflake provides several default roles:

- **ACCOUNTADMIN** – Super‑administrator for account‑level tasks (billing, role management, etc.)
- **SECURITYADMIN** – Manages users, roles, and grants (often delegated from ACCOUNTADMIN)
- **SYSADMIN** – Creates databases, schemas, tables, and warehouses; owns most data objects
- **USERADMIN** – Manages users and roles (subset of SECURITYADMIN)
- **PUBLIC** – Automatically granted to every user; can be used for system‑wide read‑only access

Best practice: Build a **custom hierarchy** below SYSADMIN.

### 1.3. Role Hierarchy Example

```
ACCOUNTADMIN
  │
  └─ SECURITYADMIN
        │
        └─ USERADMIN
  │
  └─ SYSADMIN
        │
        └─ ANALYST_ROLE
              │
              └─ JUNIOR_ANALYST_ROLE
```

A role inherits all privileges of any role granted to it. For example, `JUNIOR_ANALYST_ROLE` automatically gets all privileges from `ANALYST_ROLE` and `SYSADMIN`.

## 2. Basic RBAC Commands

### Create a role
```sql
CREATE ROLE analyst_role;
```

### Grant a role to another role (hierarchy)
```sql
GRANT ROLE analyst_role TO ROLE sysadmin;
```

### Grant a role to a user
```sql
GRANT ROLE analyst_role TO USER alice;
```

### Grant a privilege on a securable
```sql
GRANT SELECT ON DATABASE sales_db TO ROLE analyst_role;
```

### Grant a privilege on all tables in a schema
```sql
GRANT SELECT ON ALL TABLES IN SCHEMA sales_db.raw TO ROLE analyst_role;
```

### Grant future privileges (for objects created later)
```sql
GRANT SELECT ON FUTURE TABLES IN SCHEMA sales_db.raw TO ROLE analyst_role;
```

### Revoke a privilege
```sql
REVOKE SELECT ON DATABASE sales_db FROM ROLE analyst_role;
```

## 3. Impactful Use Cases (4–8 with Queries & Explanations)

### Use Case 1: Separation of Data Warehouse Admin from Security Admin

**Scenario** – You want `SYSADMIN` to create and manage objects but not create users or assign roles. `SECURITYADMIN` handles user/role management.

```sql
-- As ACCOUNTADMIN
CREATE ROLE sysadmin;
CREATE ROLE securityadmin;

-- Grant SYSADMIN the ability to manage warehouses and databases
GRANT CREATE WAREHOUSE ON ACCOUNT TO ROLE sysadmin;
GRANT CREATE DATABASE ON ACCOUNT TO ROLE sysadmin;

-- Grant SECURITYADMIN the ability to manage users, roles, and grants
GRANT CREATE USER ON ACCOUNT TO ROLE securityadmin;
GRANT CREATE ROLE ON ACCOUNT TO ROLE securityadmin;
GRANT MANAGE GRANTS ON ACCOUNT TO ROLE securityadmin;

-- Assign the default system roles to these custom roles (optional)
GRANT ROLE sysadmin TO ROLE SYSADMIN;        -- SYSADMIN is a system role
GRANT ROLE securityadmin TO ROLE SECURITYADMIN;
```

**Explanation** – This enforces duty separation. `SYSADMIN` cannot create new users or assign roles; `SECURITYADMIN` cannot create warehouses or databases unless explicitly granted. Auditing becomes cleaner.

---

### Use Case 2: Column-Level Security with RBAC + Masking Policy

**Scenario** – Only `hr_manager` role can see salary column; other roles see `NULL`.

```sql
-- Create a masking policy
CREATE OR REPLACE MASKING POLICY salary_mask AS (val NUMBER) RETURNS NUMBER ->
  CASE WHEN CURRENT_ROLE() IN ('HR_MANAGER', 'ACCOUNTADMIN') THEN val
       ELSE NULL
  END;

-- Apply policy to a table
CREATE OR REPLACE TABLE employees (
  id INT,
  name STRING,
  salary NUMBER
);

ALTER TABLE employees MODIFY COLUMN salary SET MASKING POLICY salary_mask;

-- Test: As a regular analyst
USE ROLE analyst_role;
SELECT * FROM employees;   -- salary shows NULL

-- Test: As HR manager
USE ROLE hr_manager;
SELECT * FROM employees;   -- salary shows actual values
```

**Explanation** – Row‑based RBAC alone cannot hide columns. Here a **Dynamic Data Masking** policy uses `CURRENT_ROLE()` to decide whether to reveal the column. Works seamlessly with existing RBAC.

### Use Case 3: Future Grants for Automated Object Provisioning

**Scenario** – Every week a new table is created in `sales_db.staging`. All analysts should always have `SELECT` on any new table.

```sql
-- Grant future SELECT on all tables in the staging schema to analyst_role
GRANT SELECT ON FUTURE TABLES IN SCHEMA sales_db.staging TO ROLE analyst_role;

-- Also grant USAGE on schema and database (required to access any table)
GRANT USAGE ON DATABASE sales_db TO ROLE analyst_role;
GRANT USAGE ON SCHEMA sales_db.staging TO ROLE analyst_role;

-- Now, when a new table is created:
CREATE TABLE sales_db.staging.new_sales (id INT, amount NUMBER);
-- analyst_role automatically has SELECT on new_sales
```

**Explanation** – Without future grants, you would need to run `GRANT SELECT` after every table creation. Future grants eliminate this operational overhead and enforce consistency.

### Use Case 4: Managed Access Schema – Prevent Owner from Granting

**Scenario** – You want a schema where only `SECURITYADMIN` (or a custom admin role) can grant privileges, not the object owner.

```sql
-- Create a managed access schema
CREATE SCHEMA secure_schema WITH MANAGED ACCESS;
GRANT USAGE ON SCHEMA secure_schema TO ROLE analyst_role;

-- Create a table in that schema
CREATE TABLE secure_schema.customer_data (id INT, email STRING);

-- Even the table owner (currently SYSADMIN) cannot grant SELECT on this table
-- Only roles with MANAGE GRANTS privilege (like SECURITYADMIN) can grant
```

**Explanation** – In a standard schema, the object owner can grant privileges to anyone. Managed access schemas centralize grant management, preventing accidental or malicious privilege escalation.

### Use Case 5: Row-Level Security via a Secure View

**Scenario** – Sales reps should only see rows belonging to their region. Use a **secure view** that filters based on `CURRENT_ROLE()` and a mapping table.

```sql
-- Create a mapping of roles to regions
CREATE TABLE region_access (role_name STRING, region STRING);
INSERT INTO region_access VALUES ('SALES_EMEA', 'EMEA'), ('SALES_NA', 'NA');

-- Create a secure view (definition hidden from non‑owners)
CREATE SECURE VIEW sales_view AS
SELECT s.*
FROM sales_table s
JOIN region_access r ON r.region = s.region
WHERE r.role_name = CURRENT_ROLE();

-- Grant SELECT on the secure view to the sales roles
GRANT SELECT ON sales_view TO ROLE sales_emea;
GRANT SELECT ON sales_view TO ROLE sales_na;
```

**Explanation** – The secure view hides the underlying join logic and row‑filtering condition. Users can only see rows their role is allowed to see. This is a lightweight alternative to row‑access policies.

### Use Case 6: Hierarchical Role for Multi‑Team Projects

**Scenario** – Finance team has `finance_lead` and `finance_junior`. The lead can insert/update; juniors can only select. Build hierarchy with two roles.

```sql
-- Create roles
CREATE ROLE finance_lead;
CREATE ROLE finance_junior;

-- Grant junior to lead (so lead inherits junior’s privileges, but we will give extra to lead)
GRANT ROLE finance_junior TO ROLE finance_lead;

-- Grant minimal schema usage
GRANT USAGE ON DATABASE finance_db TO ROLE finance_junior;
GRANT USAGE ON SCHEMA finance_db.reports TO ROLE finance_junior;

-- Grant SELECT to both (junior gets it, lead inherits it)
GRANT SELECT ON ALL TABLES IN SCHEMA finance_db.reports TO ROLE finance_junior;

-- Grant INSERT/UPDATE only to lead
GRANT INSERT, UPDATE ON ALL TABLES IN SCHEMA finance_db.reports TO ROLE finance_lead;

-- Assign users
GRANT ROLE finance_junior TO USER bob;
GRANT ROLE finance_lead TO USER alice;
```

**Explanation** – Role hierarchy reduces duplication. `finance_lead` automatically gets all `finance_junior` privileges plus extra write capabilities. This follows the principle of least privilege – juniors have only read, leads have read+write.

### Use Case 7: Restricting Warehouse Usage by Role

**Scenario** – Analysts should use only the `analytics_wh` warehouse, not the larger `etl_wh`. Prevent accidental large credit consumption.

```sql
-- Create warehouses
CREATE WAREHOUSE analytics_wh WAREHOUSE_SIZE = 'XSMALL';
CREATE WAREHOUSE etl_wh WAREHOUSE_SIZE = 'LARGE';

-- Grant USAGE on analytics_wh to analyst_role
GRANT USAGE ON WAREHOUSE analytics_wh TO ROLE analyst_role;

-- Do NOT grant USAGE on etl_wh to analyst_role
-- Optionally, grant MONITOR to see but not use
GRANT MONITOR ON WAREHOUSE etl_wh TO ROLE analyst_role;

-- Analyst can see etl_wh but cannot start/resume or run queries on it
USE ROLE analyst_role;
USE WAREHOUSE etl_wh;   -- Error: Insufficient privileges
```

**Explanation** – By selectively granting `USAGE`, you control which roles can consume compute credits. This prevents runaway costs and ensures appropriate warehouse sizes per workload.

### Use Case 8: Time‑Limited Role Activation (Temporary Role via Custom Logic)

**Scenario** – You want a `temp_data_loader` role that can be activated only for a specific time window (e.g., ETL window). Snowflake doesn’t have built‑in time‑based activation, but you can combine with a **custom session variable** or **external OAuth** mapping. Simplified example using a **secure function**:

```sql
-- Create a helper function that checks time
CREATE OR REPLACE FUNCTION is_etl_window() RETURNS BOOLEAN
AS $$
  SELECT HOUR(CURRENT_TIMESTAMP()) BETWEEN 1 AND 4   -- 1 AM to 4 AM
$$;

-- Create a role
CREATE ROLE temp_data_loader;

-- Create a secure view that only returns data if is_etl_window() is TRUE
CREATE SECURE VIEW restricted_table AS
SELECT *
FROM base_table
WHERE is_etl_window() = TRUE;

-- Grant SELECT on the secure view, not the base table
GRANT SELECT ON restricted_table TO ROLE temp_data_loader;
```

**Explanation** – The role has permanent `SELECT` on the view, but the view’s logic hides data outside the allowed window. This enforces time‑based access without needing to reassign roles. For strict time‑limited role activation, use a scheduled task that grants/revokes the role to users.

## 4. Best Practices Summary

| Practice                                 | Why                                                                 |
|------------------------------------------|---------------------------------------------------------------------|
| Never grant privileges directly to users | Always use roles as the only interface for privilege assignment.    |
| Build a role hierarchy under `SYSADMIN`  | Avoid using `ACCOUNTADMIN` for daily operations.                   |
| Use `SECURITYADMIN` for grants           | Keep object ownership and access control separation.               |
| Apply future grants                      | Reduce maintenance and ensure consistent access.                   |
| Prefer managed access schemas            | Prevent object owners from bypassing central access control.       |
| Combine RBAC with dynamic masking/views  | Achieve column and row-level security.                             |
| Regularly audit with `SHOW GRANTS`       | Detect privilege creep and enforce least privilege.                |

## 5. Auditing Queries (Bonus)

```sql
-- Show all grants for a specific role
SHOW GRANTS TO ROLE analyst_role;

-- Show all roles granted to a user
SHOW GRANTS OF ROLE TO USER alice;

-- List all privileges on a table
SHOW GRANTS ON TABLE sales_db.raw.sales;

-- Find roles that have been granted to other roles (hierarchy)
SHOW GRANTS OF ROLE;
```
