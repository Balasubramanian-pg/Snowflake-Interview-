# Snowflake Micro-Partition Pruning: Eliminating Data Before It Exists

## 1. Core Concepts

Snowflake’s **micro-partition pruning** is the primary mechanism that reduces data scanned during query execution. Instead of relying on user-defined indexes, Snowflake automatically organizes data into **immutable micro-partitions (~16MB compressed)** and maintains rich metadata (min/max values, distinct counts, null counts). Query pruning leverages this metadata to **skip entire partitions without reading them**.

### 1.1. How it Works

**Architecture + Flow**

1. **Data Ingestion**

   * Data is written into **columnar micro-partitions**.
   * Each partition stores metadata: min/max per column, bloom filters, null counts.

2. **Metadata Cataloging**

   * Metadata is stored in the **Cloud Services Layer**.
   * This allows pruning decisions **without scanning actual data**.

3. **Query Compilation**

   * The optimizer analyzes query predicates (WHERE, JOIN filters).
   * It identifies **prunable conditions**.

4. **Partition Elimination**

   * Micro-partitions that **cannot satisfy predicates** are skipped.
   * Only relevant partitions are scanned.

5. **Execution**

   * Reduced I/O → lower cost and faster performance.

**Primary Use Cases**

* Large fact tables (billions of rows)
* Time-series filtering (date-based pruning)
* Selective dimension joins
* Semi-structured data filtering (VARIANT pruning)

### 1.2. Pruning Effectiveness Comparison

| Metric              | Legacy Systems (Index-Based) | Snowflake (Pruning-Based)   |
| ------------------- | ---------------------------- | --------------------------- |
| Data Skipping       | Index lookup required        | Automatic via metadata      |
| Maintenance         | High (rebuild indexes)       | None                        |
| Storage Overhead    | Additional index storage     | Embedded metadata           |
| Query Planning      | Index selection complexity   | Metadata-driven pruning     |
| Performance Scaling | Degrades with fragmentation  | Improves with clustering    |
| Cost Model          | Compute + index overhead     | Pure compute (reduced scan) |

---

### 1.3. Key Factors

1. **Clustering Depth**

   * Poor clustering reduces pruning efficiency.
   * Measured via `SYSTEM$CLUSTERING_INFORMATION`.

2. **Predicate Selectivity**

   * Highly selective filters improve pruning.
   * Non-deterministic filters reduce effectiveness.

3. **Column Cardinality**

   * High cardinality columns benefit more.
   * Low cardinality reduces pruning precision.

4. **Data Distribution**

   * Skewed distributions create overlapping partitions.

5. **Query Patterns**

   * Consistent filtering columns improve clustering effectiveness.

6. **Partition Overlap**

   * High overlap = low pruning efficiency.

## 3. Impactful Use Cases

### Use Case 1: Time-Based Fact Table Pruning

**Scenario** – A pharma claims table with billions of records needs fast filtering by claim date.

```sql
-- Query only relevant time range
SELECT SUM(CLAIM_AMOUNT)
FROM CLAIMS_FACT
WHERE CLAIM_DATE BETWEEN '2025-01-01' AND '2025-01-31';
```

**Explanation** – Since micro-partitions store min/max date ranges, Snowflake eliminates partitions outside January 2025. Proper clustering on `CLAIM_DATE` ensures minimal overlap and maximum pruning.

---

### Use Case 2: Multi-Dimensional Filtering

**Scenario** – Analysts filter by region and product simultaneously.

```sql
SELECT PRODUCT_ID, SUM(SALES)
FROM SALES_FACT
WHERE REGION = 'WEST'
  AND PRODUCT_CATEGORY = 'ONCOLOGY'
GROUP BY PRODUCT_ID;
```

**Explanation** – Composite clustering `(REGION, PRODUCT_CATEGORY)` allows pruning across both dimensions. Without clustering, partitions overlap heavily, reducing pruning efficiency.

---

### Use Case 3: Join Pruning Optimization

**Scenario** – Large join between fact and dimension tables.

```sql
SELECT f.*
FROM SALES_FACT f
JOIN PRODUCT_DIM d
  ON f.PRODUCT_ID = d.PRODUCT_ID
WHERE d.CATEGORY = 'RARE_DISEASE';
```

**Explanation** – Snowflake pushes down filters from dimension table to fact table. If clustering aligns with `PRODUCT_ID`, pruning eliminates irrelevant partitions before join execution.

---

### Use Case 4: Semi-Structured Data Pruning

**Scenario** – JSON data stored in VARIANT column.

```sql
SELECT *
FROM EVENTS
WHERE DATA:event_type = 'CLICK';
```

**Explanation** – Snowflake maintains metadata even for VARIANT fields. If event_type distribution is well-clustered, partitions without "CLICK" events are skipped entirely.

---

### Use Case 5: Incremental Data Load Optimization

**Scenario** – Daily ingestion pipeline querying only recent data.

```sql
SELECT *
FROM TRANSACTIONS
WHERE LOAD_DATE >= CURRENT_DATE - 7;
```

**Explanation** – Recent partitions are isolated if ingestion is append-only and ordered. This ensures near-perfect pruning for incremental pipelines.

---

### Use Case 6: Detecting Poor Pruning

**Scenario** – Query scans too much data despite filters.

```sql
SELECT QUERY_ID, BYTES_SCANNED, PARTITIONS_SCANNED
FROM SNOWFLAKE.ACCOUNT_USAGE.QUERY_HISTORY
WHERE QUERY_TEXT ILIKE '%SALES_FACT%'
ORDER BY START_TIME DESC;
```

**Explanation** – High partitions scanned indicates poor clustering or overlapping partitions. This guides reclustering or schema redesign decisions.

## 4. Best Practices Summary

| Practice                                  | Why                                  |
| ----------------------------------------- | ------------------------------------ |
| Cluster on frequently filtered columns    | Maximizes pruning effectiveness      |
| Use low-to-moderate cardinality keys      | Avoids excessive partition overlap   |
| Prefer range filters over functions       | Enables metadata-based pruning       |
| Avoid wrapping columns in functions       | Prevents predicate pushdown          |
| Monitor clustering depth regularly        | Detects degradation early            |
| Use auto-clustering for dynamic tables    | Reduces manual overhead              |
| Align ingestion order with query patterns | Improves natural clustering          |
| Minimize partition overlap                | Key to pruning efficiency            |
| Avoid over-clustering                     | Increases compute cost unnecessarily |
| Use query profile to validate pruning     | Ground truth for optimization        |

---

## 5. Important Limitations & Considerations

| Limitation                                | Mitigation                     |
| ----------------------------------------- | ------------------------------ |
| High partition overlap reduces pruning    | Define clustering keys         |
| Frequent updates fragment clustering      | Use append-only patterns       |
| Over-clustering increases cost            | Balance clustering depth       |
| Non-deterministic filters disable pruning | Use direct column filters      |
| Functions on columns break pruning        | Rewrite predicates             |
| Small tables gain little benefit          | Avoid clustering overhead      |
| VARIANT pruning depends on structure      | Normalize critical fields      |
| Auto-clustering cost overhead             | Monitor credits usage          |
| No manual partition control               | Design via clustering strategy |

## 7. Key Takeaways

* **Micro-partition pruning** is Snowflake’s core performance lever, replacing traditional indexing.
* **Metadata-driven elimination** allows skipping data without scanning it.
* **Clustering strategy** directly determines pruning efficiency and query cost.
* **Predicate design** matters as much as data design for performance.
* **Partition overlap** is the silent performance killer.
* **Auto-clustering** simplifies maintenance but must be cost-monitored.
* **Query profile metrics** provide the only reliable signal of pruning effectiveness.
