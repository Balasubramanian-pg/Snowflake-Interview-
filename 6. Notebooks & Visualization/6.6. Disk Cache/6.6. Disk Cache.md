# Snowflake Disk Cache: Accelerating I/O with Local Virtual Warehouse Storage

## 1. Core Concepts

The Snowflake **Disk Cache** (often referred to as the Virtual Warehouse Cache or Local Data Cache) is an intermediate storage layer comprised of fast SSDs located directly on the compute nodes of an active Virtual Warehouse. Whenever a query reads data from Snowflake's centralized remote storage (e.g., Amazon S3, Google Cloud Storage), the retrieved **micro-partitions** are temporarily cached on these local disks. Subsequent queries that require the same micro-partitions can read them directly from the local SSDs, bypassing the network and drastically reducing I/O latency. Unlike the Result Cache, which stores finalized query outputs, the Disk Cache stores raw, encrypted data blocks, making it usable by entirely different queries that happen to scan the same underlying table data.

### 1.1. How it Works
The Disk Cache operates transparently as part of Snowflake's compute architecture.

1. **Remote Read:** A query is executed, and the required micro-partitions are read from the remote storage layer (Cloud Storage).
2. **Local Caching:** As the data is streamed into memory for processing, a copy of the raw micro-partitions is written to the Virtual Warehouse's local SSDs.
3. **Subsequent Access:** If a new query (even with different `SELECT` or `WHERE` clauses) requires the same micro-partitions, the warehouse engine intercepts the request and reads the data directly from the SSDs.
4. **LRU Eviction:** The local SSDs have limited capacity based on the Virtual Warehouse size. When the disk fills up, Snowflake uses a **Least Recently Used (LRU)** algorithm to evict older micro-partitions to make room for new ones.
5. **Lifecycle Reset:** If the Virtual Warehouse is suspended or resized, the localized Disk Cache is entirely dropped (or rebalanced) and must be rebuilt upon resumption.

### 1.2. Storage Layer Comparison

| Metric | Remote Storage (Centralized Cloud Storage) | Snowflake Disk Cache (Local SSDs) |
| :--- | :--- | :--- |
| **Physical Medium** | S3 / GCS / Azure Blob Storage | Local SSDs attached to compute nodes |
| **Latency/Speed** | Milliseconds (Network-bound) | Microseconds (Disk-bound) |
| **Data Scope** | The entire Snowflake database account | Only the most recently accessed micro-partitions |
| **Lifecycle** | Persistent (until dropped or Time Travel expires) | Ephemeral (tied to the Virtual Warehouse's uptime) |
| **Sharing** | Shared across all warehouses globally | Isolated to the specific Virtual Warehouse |

### 1.3. Key Factors
1. **Warehouse Size:** Larger Virtual Warehouses (e.g., L, XL) not only have more compute cores but also possess proportionally larger SSD storage arrays, allowing them to cache significantly more micro-partitions before LRU eviction occurs.
2. **Auto-Suspend Timing:** An aggressive auto-suspend setting (e.g., 60 seconds) saves compute credits but frequently clears the Disk Cache. A longer auto-suspend retains the cache for subsequent workloads at the cost of idle compute time.
3. **Workload Concurrency & Thrashing:** If a single warehouse is used to scan massive, unrelated tables (e.g., heavy ETL mixing with BI), the constant influx of new micro-partitions will thrash the cache, evicting useful data before it can be reused.
4. **Partition Pruning:** Efficiently clustered tables read fewer micro-partitions. Better pruning means fewer blocks are written to the local SSD, preserving cache space and reducing LRU eviction rates.

## 2. Basic Commands & Privileges

### 2.1. Required Privileges

| Privilege | Scope | Description |
| :--- | :--- | :--- |
| `OPERATE` | Virtual Warehouse | Required to start, suspend, or alter the settings (like Auto-Suspend) that directly dictate the Disk Cache lifecycle. |
| `USAGE` | Virtual Warehouse | Required to execute queries on the warehouse and consequently read from or write to the local SSDs. |
| `MONITOR` | Virtual Warehouse | Required to view query profiles and cache hit statistics on a given warehouse. |

### 2.2. Extend Auto-Suspend to Preserve Cache
Modifies the warehouse to stay active longer after the last query completes, ensuring the Disk Cache remains intact for clustered or intermittent workloads.
```sql
ALTER WAREHOUSE bi_reporting_wh SET AUTO_SUSPEND = 600;
```

### 2.3. Flush the Disk Cache via Suspension
Manually suspends the warehouse. Because local SSDs are ephemeral, this operation instantly clears the Disk Cache.
```sql
ALTER WAREHOUSE dev_wh SUSPEND;
```

### 2.4. Scale Up to Increase Cache Capacity
Increases the compute size, which simultaneously provisions larger underlying SSDs capable of caching more micro-partitions without eviction.
```sql
ALTER WAREHOUSE heavy_etl_wh SET WAREHOUSE_SIZE = 'XLARGE';
```

### 2.5. Alter Multi-Cluster Scaling Policy
Sets the scaling policy to `ECONOMY`, which forces the system to queue queries slightly longer to maximize the warm Disk Cache on existing nodes rather than immediately spinning up cold clusters.
```sql
ALTER WAREHOUSE auto_scale_wh SET SCALING_POLICY = 'ECONOMY';
```

### 2.6. Analyze Cache Hit Ratio in Query History
Retrieves the percentage of data scanned directly from the local Disk Cache versus remote storage for a specific query.
```sql
SELECT query_id, percentage_scanned_from_cache 
FROM table(information_schema.query_history())
WHERE query_id = 'your_query_id_here';
```

## 3. Impactful Use Cases

### Use Case 1: Cache Warming for Executive Dashboards
* **Scenario** – The CEO's financial dashboard queries massive, freshly loaded tables at 8:00 AM, experiencing slow remote I/O latency.
* ```sql
  -- Create a task that runs at 7:50 AM on the BI Warehouse
  CREATE OR REPLACE TASK warm_ceo_dashboard_cache
    WAREHOUSE = executive_bi_wh
    SCHEDULE = 'USING CRON 50 7 * * 1-5 UTC'
  AS 
    -- Execute a broad scan of the required table to pull micro-partitions into the local SSD
    SELECT region, SUM(revenue) 
    FROM finance.public.daily_revenue 
    WHERE report_date = CURRENT_DATE()
    GROUP BY region;
  ```
* **Explanation** – By executing a programmatic "cache-warming" query on the dedicated BI warehouse just before peak usage, the necessary micro-partitions are pulled from remote storage and loaded onto the local SSDs. When the executives log in at 8:00 AM, their dashboard queries (even if structurally different) will scan the same raw data. The warehouse processes this via the local Disk Cache, delivering near-instant I/O performance.

### Use Case 2: Tuning Auto-Suspend for Cache-Dependent Workloads
* **Scenario** – A team of Data Scientists runs ad-hoc iterative machine learning queries. Their warehouse auto-suspends every 60 seconds, forcing a cold remote storage scan every time they step away to review code.
* ```sql
  -- Alter the warehouse to a 15-minute auto-suspend
  ALTER WAREHOUSE data_science_wh 
    SET AUTO_SUSPEND = 900
    AUTO_RESUME = TRUE;
  ```
* **Explanation** – While a 60-second auto-suspend is a standard cost-saving best practice, it is detrimental to highly iterative, interactive workloads. By extending the auto-suspend to 15 minutes (900 seconds), the warehouse stays "warm." The cost of 15 minutes of idle compute is offset by the immense time saved by not repeatedly pulling terabytes of data over the network from S3/GCS.

### Use Case 3: Workload Isolation to Prevent Cache Thrashing
* **Scenario** – A single `COMPUTE_WH` is handling a massive data lake ingestion job and answering front-end BI queries concurrently, causing both to run slowly due to Disk Cache LRU eviction.
* ```sql
  -- Step 1: Create a dedicated warehouse for heavy I/O ingestion
  CREATE WAREHOUSE ingestion_wh WITH WAREHOUSE_SIZE = 'LARGE' AUTO_SUSPEND = 60;
  
  -- Step 2: Route all batch ELT workloads to this new warehouse
  USE WAREHOUSE ingestion_wh;
  COPY INTO raw_data_lake FROM @my_stage;

  -- Step 3: Retain the original warehouse strictly for BI
  -- BI micro-partitions will no longer be overwritten by ETL data blocks
  ALTER USER bi_service_account SET DEFAULT_WAREHOUSE = compute_wh;
  ```
* **Explanation** – If heavy batch operations and analytical queries share a warehouse, the massive data volumes scanned by the ETL processes will constantly force the LRU algorithm to evict the BI data from the SSDs (Cache Thrashing). By physically isolating the workloads into separate Virtual Warehouses, the BI warehouse maintains a highly localized, perfectly tuned Disk Cache consisting only of reporting data.

### Use Case 4: Multi-Cluster Warehouse (MCW) Routing Strategy
* **Scenario** – A heavily used web application triggers hundreds of concurrent queries. Standard auto-scaling rapidly spins up new clusters, but queries on new clusters run slowly because their SSDs are cold.
* ```sql
  -- Switch the scaling policy to prioritize existing cached nodes
  ALTER WAREHOUSE web_app_mcw 
    SET SCALING_POLICY = 'ECONOMY'
    MIN_CLUSTER_COUNT = 1
    MAX_CLUSTER_COUNT = 5;
  ```
* **Explanation** – The default `STANDARD` scaling policy aggressively spins up a new compute cluster the moment queuing occurs. However, this new cluster has an empty Disk Cache. The `ECONOMY` policy forces the system to tolerate up to 6 minutes of queuing before spinning up a new cluster. This deliberately routes more queries to the existing, warm cluster, maximizing Disk Cache utilization and improving overall median query performance.

### Use Case 5: Benchmarking True Query Performance
* **Scenario** – A Data Engineer is testing different clustering keys on a 10TB table. Repeated query tests are running artificially fast because they are hitting the Disk Cache, hiding the true I/O cost of scanning.
* ```sql
  -- Step 1: Execute test on unoptimized table
  SELECT /* Baseline Test */ count(*) FROM massive_table WHERE col_a = 'X';

  -- Step 2: Flush the Disk Cache by suspending and resuming
  ALTER WAREHOUSE test_wh SUSPEND;
  ALTER WAREHOUSE test_wh RESUME;

  -- Step 3: Execute test on the clustered table variant
  SELECT /* Clustered Test */ count(*) FROM clustered_table WHERE col_a = 'X';
  ```
* **Explanation** – To accurately benchmark the raw I/O performance of different table architectures, engineers must ensure a "cold cache" environment. Because there is no session parameter to disable the Disk Cache (unlike the Result Cache), the warehouse must be forced to suspend, wiping the ephemeral SSDs. Resuming it guarantees that the subsequent query is forced to perform a full remote read, providing an accurate, comparable execution time.

## 4. Best Practices Summary

| Practice | Why |
| :--- | :--- |
| **Isolate Workloads by Warehouse** | Prevent "Cache Thrashing." Keep heavy ETL operations off your BI warehouses so massive data scans don't evict localized reporting micro-partitions. |
| **Balance Auto-Suspend Timers** | Weigh compute costs against latency. Use 60 seconds for batch ETL, but consider 300–600 seconds for interactive BI or Data Science to keep SSDs warm. |
| **Scale Up for Cache Capacity** | If your query relies on a 500GB working dataset but you use an X-Small WH, the SSD will constantly overwrite. Scale up to increase local disk capacity. |
| **Pre-Warm Critical Dashboards** | Use scheduled Snowflake Tasks to run generic, wide `SELECT` queries on key tables 15 minutes before peak login times to load the SSDs. |
| **Optimize Table Clustering** | Efficient clustering reduces the number of micro-partitions read. Fewer partitions read means less disk space used, maximizing overall cache efficiency. |
| **Leverage Economy Scaling Policy** | For concurrent workloads where a warm cache is more valuable than instant execution, the `ECONOMY` policy routes queries to existing warm nodes instead of cold spin-ups. |
| **Do Not Resize Randomly** | Resizing a warehouse clears or re-allocates the cache. Avoid dynamically scaling up and down multiple times an hour if cache retention is critical. |

## 5. Important Limitations & Considerations

| Limitation | Mitigation |
| :--- | :--- |
| **Cache is Ephemeral** | The cache is permanently destroyed upon warehouse suspension. Configure `AUTO_SUSPEND` carefully if cache persistence is vital to your workload SLAs. |
| **No Cross-Warehouse Sharing** | Disk Cache is strictly isolated to the compute nodes of a specific warehouse. Data cached on `WH_A` cannot be read by `WH_B`. Unify identical workloads onto a single Multi-Cluster Warehouse. |
| **Black Box Management** | There are no SQL commands to view exactly what tables/partitions are currently residing on the local SSDs. Rely on `PERCENTAGE_SCANNED_FROM_CACHE` metrics instead. |
| **LRU Thrashing on Small WHs** | Running massive table scans on an `X-Small` warehouse will rapidly fill the cache, evicting the beginning of the table before the end is finished. Scale up to provide adequate SSD space. |
| **MCW Cold Starts** | When a Multi-Cluster Warehouse scales out, the new clusters boot with an empty cache. Mitigate this by maintaining a higher `MIN_CLUSTER_COUNT` during known peak hours to maintain baseline warmth. |

## 6. Auditing & Monitoring Queries

### Identify High Disk Cache Usage Queries
```sql
SELECT 
    query_id, 
    query_text, 
    warehouse_name,
    bytes_scanned,
    percentage_scanned_from_cache,
    total_elapsed_time
FROM snowflake.account_usage.query_history
WHERE start_time >= DATEADD(day, -1, CURRENT_TIMESTAMP())
  AND percentage_scanned_from_cache > 80
  AND bytes_scanned > 1000000000 -- Scanned more than 1 GB
ORDER BY percentage_scanned_from_cache DESC;
```

### Calculate Warehouse Cache Hit Efficiency
```sql
SELECT 
    warehouse_name,
    COUNT(query_id) as total_queries,
    AVG(percentage_scanned_from_cache) as avg_cache_hit_ratio,
    SUM(bytes_scanned) as total_bytes_processed
FROM snowflake.account_usage.query_history
WHERE start_time >= DATEADD(day, -7, CURRENT_TIMESTAMP())
  AND warehouse_name IS NOT NULL
GROUP BY warehouse_name
ORDER BY avg_cache_hit_ratio DESC;
```

### Identify Potential Cache Thrashing (Heavy Remote Reads)
```sql
SELECT 
    query_id, 
    query_text, 
    warehouse_size,
    (bytes_scanned * (1 - (percentage_scanned_from_cache/100))) as remote_bytes_read
FROM snowflake.account_usage.query_history
WHERE start_time >= DATEADD(day, -1, CURRENT_TIMESTAMP())
  AND percentage_scanned_from_cache < 10
  AND bytes_scanned > 10000000000 -- Scanned more than 10 GB
ORDER BY remote_bytes_read DESC;
```

### Monitor Impact of Auto-Suspend Configuration
```sql
SELECT 
    warehouse_name,
    DATE_TRUNC('HOUR', start_time) as query_hour,
    AVG(percentage_scanned_from_cache) as avg_cache_ratio,
    COUNT(*) as query_volume
FROM snowflake.account_usage.query_history
WHERE start_time >= DATEADD(day, -3, CURRENT_TIMESTAMP())
GROUP BY warehouse_name, query_hour
ORDER BY warehouse_name, query_hour;
```

## 7. Key Takeaways
* The Disk Cache consists of fast **Local SSDs** provisioned on the compute nodes of an active Virtual Warehouse, storing raw **micro-partitions** retrieved from remote storage.
* Disk Cache relies on a **Least Recently Used (LRU)** eviction algorithm; larger warehouse sizes naturally provide larger SSD capacities, decreasing the rate of eviction.
* The cache is strictly **ephemeral**. Any action that suspends or resizes the Virtual Warehouse immediately flushes the localized Disk Cache.
* **Workload Isolation** is critical. Prevent **cache thrashing** by dedicating separate warehouses to heavy ETL ingestion and fast-paced, repetitive BI reporting.
* Strategically tuning your **`AUTO_SUSPEND`** configuration and utilizing Snowflake **Tasks for cache warming** can guarantee zero-latency remote I/O for mission-critical dashboards.
