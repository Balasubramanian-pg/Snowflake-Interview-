# Snowflake Query Profile: Through and Through

## 1. Core Concepts

The Snowflake Query Profile is a diagnostic tool that provides a granular, visual breakdown of how a specific query was executed. Unlike traditional databases where you might need to manually interpret an `EXPLAIN` plan, Snowflake automatically generates a rich, interactive profile for every executed query, allowing engineers and DBAs to pinpoint performance bottlenecks and resource consumption with precision.

### 1.1. How It Works

Snowflake executes queries through a sophisticated multi-layer architecture that separates storage, compute, and services. When you submit a query, Snowflake’s cloud services layer parses and optimizes it, generating an **execution plan** composed of individual **operators**. These operators form a **Directed Acyclic Graph (DAG)** showing how data flows through transformations to produce results[reference:0].

The query execution engine coordinates warehouse compute nodes to process operators in parallel. Each operator performs specific data manipulation tasks – scanning micro-partitions, filtering rows, joining datasets, aggregating values, or sorting results. Snowflake’s architecture enables massive parallelism across warehouse nodes, processing operators concurrently when possible and sequencing dependent operations appropriately[reference:1].

### 1.2. Key Principles

- **Automatic generation**: Snowflake generates a query profile automatically after the execution of every query, allowing DBAs and engineers granular insight into its performance[reference:2].
- **DAG-based visualization**: The execution plan is represented as a DAG, highlighting the relationships and data flow between operators[reference:3].
- **Operator-level granularity**: Profiles break down time and resource consumption by individual operator[reference:4].
- **Programmatic access**: Beyond the UI, Snowflake provides `GET_QUERY_OPERATOR_STATS` for programmatic analysis at scale[reference:5].
- **No manual tuning required**: Snowflake’s engine automatically optimizes queries; the Query Profile is a **visibility tool**, not a tuning mechanism you must operate.

### 1.3. Key Components of the Query Profile

| Component | Description |
|-----------|-------------|
| **Query metadata** | Query ID, Session ID, Query Tag, user details, warehouse details, SQL text[reference:6] |
| **Execution Plan** | Detailed map of the query execution showing every operation such as scans, joins, filters, and aggregations. Outlines the sequence and type of operations performed, providing insight into how data moves and transforms across the system[reference:7] |
| **Query Plan Visualization** | A graphical representation of the query execution plan as a DAG. Particularly useful for quickly identifying performance bottlenecks such as long-running operations or large data scans[reference:8] |
| **Performance Statistics** | Quantitative analysis including execution time, processing time, amount of data scanned, rows processed, number of micro‑partitions scanned out of total available, and more[reference:9] |
| **Resource usage** | Insights into CPU processing, disk I/O, synchronization and initialization, network transfer, and more[reference:10] |
| **Error and warning messages** | Any errors or warnings encountered during query execution[reference:11] |

### 1.4. EXPLAIN vs. Query Profile

It is important to distinguish between Snowflake’s two related but distinct features:

| Aspect | EXPLAIN | Query Profile |
|--------|---------|---------------|
| **What it shows** | **Logical** execution plan (what Snowflake *would* do) | **Actual** execution details (what Snowflake *did*) |
| **Execution required** | No – warehouse not needed | Yes – query must complete or terminate |
| **Statistics included** | No runtime statistics (e.g., execution time, rows processed) | Yes – full operator‑level runtime statistics |
| **Use case** | Understand query structure before execution | Diagnose performance issues after execution |

`EXPLAIN` compiles the SQL statement but does not execute it, so it does not require a running warehouse. The EXPLAIN plan might differ depending on the size of the current warehouse; if you run EXPLAIN outside of a current warehouse, Snowflake constructs the plan based on the capacity of an XSMALL warehouse[reference:12].

## 2. Basic Commands & Privileges

### 2.1. Required Privileges

| Privilege | Scope | Description |
|-----------|-------|-------------|
| `MONITOR` | Warehouse | Required to view query profiles for queries executed on a specific warehouse |
| `ACCOUNTADMIN` or `OBJECTADMIN` | Account | Broader access to `SNOWFLAKE.ACCOUNT_USAGE` views for cross‑warehouse analysis |

### 2.2. Generate an EXPLAIN Plan (Logical)

```sql
-- Tabular format (default, human‑readable)
EXPLAIN SELECT * FROM orders WHERE order_date > '2024-01-01';

-- JSON format (easier to store and query programmatically)
EXPLAIN USING JSON SELECT * FROM orders WHERE order_date > '2024-01-01';

-- TEXT format
EXPLAIN USING TEXT SELECT * FROM orders WHERE order_date > '2024-01-01';
```

The output contains the following columns[reference:13]:
- `step`: Most queries contain a single step, but some are executed as multiple distinct steps.
- `id`: Unique identifier assigned to each operation in the query plan.
- `parentOperators`: Array of identifiers for the operation’s parent nodes.
- `operation`: Name of the operation (e.g., `Result`, `Filter`, `TableScan`, `Join`).
- `objects`: Name of the object referenced by a table scan operation.
- `alias`: Alias of a referenced object, if the object has been given an alias.
- `expressions`: List of expressions relevant to the current operation.
- `partitionsTotal`: Total number of micro‑partitions in the referenced object.
- `partitionsAssigned`: Number of partitions left after compile‑time pruning (i.e., partitions that might be scanned).
- `bytesAssigned`: Number of bytes contained in the partitionsAssigned.

### 2.3. Retrieve Query Profile Programmatically

```sql
-- Get operator‑level statistics for a specific query
SELECT * FROM TABLE(GET_QUERY_OPERATOR_STATS('query_id_here'));
```

This table function returns rows with statistics about each query operator in the query, including:
- `operator_id`: Unique identifier for the operator
- `operator_type`: Type of operator (e.g., `TableScan`, `Join`, `Filter`, `Aggregate`)
- `execution_time_ms`: Time spent executing this operator
- `partitions_scanned` / `partitions_total`: Pruning efficiency metrics
- `bytes_processed`, `rows_produced`, and more

### 2.4. Access Query Profile in Snowsight UI

1. Navigate to **Activity → Query History**.
2. Find the query you want to analyze (use filters for time range, user, warehouse, etc.).
3. Click on the **Query ID** to open the query details.
4. Open the **Profile** tab to view the detailed graphical query profile[reference:14].

### 2.5. Find Queries with Performance Issues (Spilling)

```sql
-- Top 10 queries with most spillage in the last 45 days
SELECT 
    query_id, 
    SUBSTR(query_text, 1, 50) partial_query_text, 
    user_name, 
    warehouse_name, 
    bytes_spilled_to_local_storage, 
    bytes_spilled_to_remote_storage 
FROM snowflake.account_usage.query_history 
WHERE (bytes_spilled_to_local_storage > 0 OR bytes_spilled_to_remote_storage > 0) 
  AND start_time::date > DATEADD('days', -45, CURRENT_DATE) 
ORDER BY bytes_spilled_to_remote_storage, bytes_spilled_to_local_storage DESC 
LIMIT 10;
```

This query identifies queries suffering from memory spillage – one of the most common causes of slow performance[reference:15].

### 2.6. Use `RESULT_SCAN` with EXPLAIN for Post‑Processing

```sql
-- Store EXPLAIN output in a table for analysis
CREATE TEMP TABLE explain_results AS 
SELECT * FROM TABLE(RESULT_SCAN(LAST_QUERY_ID()));

-- Query the stored plan
SELECT * FROM explain_results WHERE operation = 'TableScan';
```

## 3. Impactful Use Cases (4–8 with Queries & Explanations)

### Use Case 1: Identify Inefficient Partition Pruning (TableScan Bottleneck)

**Scenario** – A query is scanning far more data than expected. You suspect partition pruning is not working effectively.

```sql
-- Step 1: Identify the query in Query History
-- Step 2: Open the Query Profile and locate the TableScan operator(s)

-- Step 3: Programmatically check pruning efficiency
SELECT 
    operator_type,
    partitions_total,
    partitions_scanned,
    bytes_scanned,
    rows_scanned,
    execution_time_ms
FROM TABLE(GET_QUERY_OPERATOR_STATS('your_query_id'))
WHERE operator_type = 'TableScan';
```

**Explanation** – Inefficient pruning occurs when Snowflake cannot eliminate micro‑partitions at compile time. Compare `partitions_scanned` with `partitions_total`. If they are nearly equal, the query is performing a full table scan. Common causes include:

- Using functions in `WHERE` clauses that prevent pruning (e.g., `WHERE DATE_TRUNC('day', order_date) = ...`)
- No clustering key on frequently filtered columns
- Missing Search Optimization Service on large tables

**Remediation**: 
- Add a clustering key on the filtered column(s)
- Rewrite predicates to avoid wrapping columns in functions
- Enable Search Optimization Service for point lookup queries

### Use Case 2: Detect and Diagnose “Exploding” Joins

**Scenario** – A join operation is producing an unexpectedly large number of output rows, causing performance degradation and potential memory spill.

```sql
-- Step 1: Locate the Join operator in the Query Profile
-- Step 2: Examine the operator statistics

SELECT 
    operator_type,
    rows_produced,
    rows_scanned,
    bytes_spilled_to_local_storage,
    bytes_spilled_to_remote_storage,
    execution_time_ms
FROM TABLE(GET_QUERY_OPERATOR_STATS('query_id'))
WHERE operator_type LIKE '%Join%';
```

**Explanation** – An “exploding” join happens when rows from one table match multiple rows from another table, causing the output to exceed millions of rows. This can also occur if a join condition is omitted entirely (producing a Cartesian product)[reference:16].

**Signs to look for in the Query Profile**:
- `rows_produced` is many times larger than the input row counts
- Significant spillage to remote storage (indicated by `bytes_spilled_to_remote_storage`)
- Join operator shows disproportionately high execution time

**Remediation**:
- Verify join conditions are correct and not generating Cartesian products
- Add missing join predicates
- Consider using a `LEFT SEMI JOIN` instead of a full `JOIN` when only existence matters
- For heavily skewed keys, consider breaking the query into multiple steps

### Use Case 3: Identify Memory Spillage (Local and Remote)

**Scenario** – A query is running slowly, and you suspect it is spilling to disk due to insufficient memory in the warehouse.

```sql
-- Step 1: Find queries with spillage
SELECT 
    query_id,
    query_text,
    user_name,
    warehouse_name,
    bytes_spilled_to_local_storage,
    bytes_spilled_to_remote_storage,
    execution_time / 1000 AS exec_seconds
FROM snowflake.account_usage.query_history
WHERE (bytes_spilled_to_local_storage > 0 OR bytes_spilled_to_remote_storage > 0)
  AND start_time >= DATEADD('day', -7, CURRENT_TIMESTAMP())
ORDER BY bytes_spilled_to_remote_storage DESC
LIMIT 20;

-- Step 2: For a specific query, drill into operator details
SELECT 
    operator_type,
    bytes_spilled_to_local_storage,
    bytes_spilled_to_remote_storage,
    execution_time_ms,
    rows_produced
FROM TABLE(GET_QUERY_OPERATOR_STATS('query_id'));
```

**Explanation** – When a warehouse runs out of memory during query execution, Snowflake writes data to local disk (local spill). If the query requires even more memory, it spills onto remote cloud storage (remote spill), resulting in even worse performance[reference:17]. Spills slow queries roughly 10×[reference:18].

**Signs in Query Profile**:
- `bytes_spilled_to_local_storage` > 0: Data spilled to local disk
- `bytes_spilled_to_remote_storage` > 0: Data spilled to remote storage (severe impact)
- High proportion of time spent on disk I/O vs. CPU in the Statistics pane[reference:19]

**Remediation**:
- Use a larger warehouse (effectively increasing available memory/local storage space)[reference:20]
- Process data in smaller batches
- Optimize the query to reduce memory footprint (e.g., add filters, break into multiple queries)

### Use Case 4: Compare Query Versions Before and After Optimization

**Scenario** – You have rewritten a query and want to confirm that performance improved. Use the Query Profile to compare the two versions.

```sql
-- Step 1: Run both queries and capture their Query IDs
-- Query A (original): query_id = 'original_query_id'
-- Query B (optimized): query_id = 'optimized_query_id'

-- Step 2: Compare overall execution
SELECT 
    query_id,
    execution_time / 1000 AS exec_seconds,
    bytes_scanned,
    bytes_written,
    bytes_spilled_to_local_storage,
    bytes_spilled_to_remote_storage
FROM snowflake.account_usage.query_history
WHERE query_id IN ('original_query_id', 'optimized_query_id');

-- Step 3: Compare operator‑level changes
SELECT 
    query_id,
    operator_type,
    execution_time_ms,
    rows_produced,
    bytes_spilled_to_local_storage
FROM TABLE(GET_QUERY_OPERATOR_STATS('original_query_id'))
UNION ALL
SELECT 
    'optimized',
    operator_type,
    execution_time_ms,
    rows_produced,
    bytes_spilled_to_local_storage
FROM TABLE(GET_QUERY_OPERATOR_STATS('optimized_query_id'));
```

**Explanation** – Comparing query profiles side‑by‑side allows you to verify that optimisations had the intended effect. Look for reductions in:
- Total execution time
- Bytes scanned (indicates better pruning)
- Spill metrics (indicates better memory efficiency)
- Rows processed at each operator

### Use Case 5: Distinguish Queue Time from Execution Time

**Scenario** – A user reports a “slow query,” but upon investigation, the query itself executed quickly. The delay was due to queuing.

```sql
-- Query from ACCOUNT_USAGE to see queue vs execution time
SELECT 
    query_id,
    query_text,
    user_name,
    warehouse_name,
    (queued_overload_time + queued_provisioning_time + queued_repair_time) AS total_queue_time_ms,
    execution_time / 1000 AS exec_seconds,
    (total_queue_time_ms / 1000) AS queue_seconds
FROM snowflake.account_usage.query_history
WHERE query_id = 'your_query_id';
```

**Explanation** – The distinction between queue time and execution time saves significant debugging time. A query that spent 45 seconds queued and 8 seconds executing isn’t a query optimization problem – it’s a warehouse sizing or concurrency problem[reference:21]. The Query Profile UI also displays this distinction clearly: queue time appears before execution begins in the visual timeline.

**Signs in Query Profile**:
- Long gap between query submission and the start of the first operator
- Warehouse shows high concurrency or undersized for the workload

**Remediation**:
- Increase warehouse size or add multi‑cluster scaling
- Reschedule heavy queries during off‑peak hours
- Use separate warehouses for different workloads (ETL vs. analytics)

### Use Case 6: Identify Late‑Stage Filters (Filter Pushdown Opportunities)

**Scenario** – A filter appears late in the execution plan, after large amounts of data have already been processed.

```sql
-- Examine the execution plan to locate Filter operators
SELECT 
    operator_id,
    operator_type,
    parent_operators,
    execution_time_ms,
    rows_produced
FROM TABLE(GET_QUERY_OPERATOR_STATS('query_id'))
WHERE operator_type = 'Filter'
ORDER BY operator_id;
```

**Explanation** – Filters applied after significant data processing can lead to inefficiencies; moving these earlier in the query can reduce the workload[reference:22]. In the Query Profile visualization, look for Filter operators that appear **after** expensive operations (e.g., large joins or table scans).

**Remediation**:
- Push filters as early as possible in the query (apply the most restrictive filters first)
- Move filter conditions from outer queries into subqueries or CTEs
- Use `QUALIFY` for window function filters only when necessary

### Use Case 7: Validate Clustering Key Effectiveness

**Scenario** – You have defined a clustering key on a table, but queries against it still scan excessive partitions.

```sql
-- Step 1: Identify queries that access the table
SELECT 
    query_id,
    query_text,
    start_time,
    execution_time / 1000 AS exec_seconds
FROM snowflake.account_usage.query_history
WHERE query_text ILIKE '%my_table%'
ORDER BY start_time DESC
LIMIT 50;

-- Step 2: For a specific query, check pruning effectiveness
SELECT 
    operator_type,
    partitions_total,
    partitions_scanned,
    bytes_scanned,
    rows_scanned
FROM TABLE(GET_QUERY_OPERATOR_STATS('query_id'))
WHERE operator_type = 'TableScan';
```

**Explanation** – A well‑chosen clustering key significantly reduces the number of micro‑partitions scanned. In the Query Profile, the TableScan operator shows `partitions_scanned` vs. `partitions_total`. If these numbers are close, the clustering key may be ineffective for the query’s filter patterns.

**Signs of poor clustering**:
- `partitions_scanned` is close to `partitions_total` despite selective filters
- Query scans large bytes despite returning few rows

**Remediation**:
- Review and potentially redefine the clustering key on columns most frequently used in filters
- Consider Search Optimization Service for point lookup queries
- Use `SYSTEM$CLUSTERING_INFORMATION` to assess clustering depth

### Use Case 8: Monitor Long‑Running Operations and Bottlenecks

**Scenario** – Your production environment has several queries that consistently run slowly. You need to identify the most expensive nodes and prioritise optimisation efforts.

```sql
-- Step 1: Find the most expensive queries from the last day
SELECT 
    query_id,
    query_text,
    user_name,
    warehouse_name,
    execution_time / 1000 AS exec_seconds,
    bytes_scanned / 1024 / 1024 AS mb_scanned
FROM snowflake.account_usage.query_history
WHERE start_time >= DATEADD('day', -1, CURRENT_TIMESTAMP())
ORDER BY execution_time DESC
LIMIT 20;

-- Step 2: For each candidate query, identify the most expensive operator
SELECT 
    operator_type,
    execution_time_ms,
    rows_produced,
    bytes_scanned,
    ROUND(100 * execution_time_ms / SUM(execution_time_ms) OVER (), 2) AS pct_of_total
FROM TABLE(GET_QUERY_OPERATOR_STATS('candidate_query_id'))
ORDER BY execution_time_ms DESC
LIMIT 5;
```

**Explanation** – The Query Profile includes a **Most Expensive Nodes** pane that identifies the operator nodes taking the longest to execute. You can drill down even further to understand why those nodes are slow[reference:23]. Use this to prioritise optimisation efforts on the operators that consume the most time.

**Common expensive operator patterns**:
- `TableScan` with high `partitions_scanned` → pruning issue
- `Join` with high spillage → memory issue or skewed data
- `Aggregate` with high cardinality → consider pre‑aggregation
- `WindowFunction` with large partitions → revisit window definition

## 4. Best Practices Summary

| Practice | Why |
|----------|-----|
| **Start with the most expensive queries** | Use `QUERY_HISTORY` to rank by execution time or spillage, then drill into Query Profile[reference:24] |
| **Use programmatic access for scale** | `GET_QUERY_OPERATOR_STATS` allows automated analysis across hundreds of queries[reference:25] |
| **Distinguish queue time from execution time** | A query queued for 45 seconds is a warehouse problem, not a query problem[reference:26] |
| **Check pruning efficiency on every TableScan** | Compare `partitions_scanned` to `partitions_total` – aim for orders of magnitude difference |
| **Monitor spill metrics aggressively** | Remote spill slows queries ~10×; investigate any non‑zero remote spill values[reference:27] |
| **Push filters as early as possible** | Late‑stage filters process unnecessary data – restructure queries to filter early |
| **Use clustering keys on frequently filtered columns** | Improves partition pruning and reduces scan costs |
| **Version control query changes** | Compare Query Profiles before and after optimisation to validate improvements |
| **Combine UI exploration with SQL analysis** | Use UI for visual intuition, SQL for bulk analysis and trend detection |

## 5. Important Limitations & Considerations

| Limitation | Mitigation |
|------------|------------|
| **Query Profile is available only after query execution** | For planning before execution, use `EXPLAIN` to see the logical plan |
| **Graphical plan cannot be exported programmatically** | At this time, it is not possible to capture the graphical query execution plan programmatically. What you can get is the logical execution plan using EXPLAIN[reference:28] |
| **Profile data retention** | Query profiles are available for as long as the query history is retained (typically up to 365 days in `ACCOUNT_USAGE`) |
| **EXPLAIN plan depends on current warehouse** | If you run EXPLAIN outside a warehouse, Snowflake assumes XSMALL capacity[reference:29]. For accurate planning, run EXPLAIN with the intended warehouse size |
| **QAS causes non‑zero spill bytes** | When Query Acceleration Service is enabled, Snowflake writes a small amount of data to remote storage for each eligible query, even if QAS isn’t used for that query. Don’t be concerned by nonzero `bytes_spilled_to_remote_storage` when QAS is enabled[reference:30] |
| **Shared database restrictions** | Query profiles for queries on shared databases (Snowflake Marketplace) may have redacted information to protect provider intellectual property[reference:31] |
| **DML operator display changes** | Insert, Update, and Delete operators display `Full table name` attributes (as of 2023_03 bundle)[reference:32] |

## 6. Auditing & Monitoring Queries

### Track overall query performance trends

```sql
-- Daily summary of query performance by warehouse
SELECT 
    DATE_TRUNC('day', start_time) AS query_day,
    warehouse_name,
    COUNT(*) AS query_count,
    AVG(execution_time) / 1000 AS avg_exec_seconds,
    SUM(bytes_scanned) / 1024 / 1024 / 1024 AS total_gb_scanned,
    SUM(bytes_spilled_to_remote_storage) AS total_remote_spill_bytes
FROM snowflake.account_usage.query_history
WHERE start_time >= DATEADD('day', -30, CURRENT_TIMESTAMP())
GROUP BY 1, 2
ORDER BY query_day DESC, avg_exec_seconds DESC;
```

### Find queries with poor pruning

```sql
-- Requires GET_QUERY_OPERATOR_STATS for multiple queries
-- This is a placeholder pattern; implement programmatically
SELECT 
    query_id,
    operator_type,
    partitions_total,
    partitions_scanned,
    ROUND(100 * partitions_scanned / NULLIF(partitions_total, 0), 2) AS pct_scanned
FROM TABLE(GET_QUERY_OPERATOR_STATS('query_id'))
WHERE operator_type = 'TableScan'
  AND partitions_scanned > 1000
  AND partitions_scanned / NULLIF(partitions_total, 0) > 0.5;
```

### Monitor queries that spilled to remote storage

```sql
SELECT 
    query_id,
    user_name,
    warehouse_name,
    execution_time / 1000 AS exec_seconds,
    bytes_spilled_to_local_storage,
    bytes_spilled_to_remote_storage,
    bytes_scanned / 1024 / 1024 AS mb_scanned
FROM snowflake.account_usage.query_history
WHERE bytes_spilled_to_remote_storage > 0
  AND start_time >= DATEADD('day', -7, CURRENT_TIMESTAMP())
ORDER BY bytes_spilled_to_remote_storage DESC;
```

### Track queue time vs execution time by warehouse

```sql
SELECT 
    warehouse_name,
    COUNT(*) AS query_count,
    AVG(queued_overload_time + queued_provisioning_time + queued_repair_time) / 1000 AS avg_queue_seconds,
    AVG(execution_time) / 1000 AS avg_exec_seconds,
    ROUND(100 * AVG(queued_overload_time + queued_provisioning_time + queued_repair_time) / NULLIF(AVG(execution_time), 0), 2) AS queue_to_exec_pct
FROM snowflake.account_usage.query_history
WHERE start_time >= DATEADD('hour', -24, CURRENT_TIMESTAMP())
GROUP BY warehouse_name
ORDER BY avg_queue_seconds DESC;
```

## 7. Key Takeaways

- **The Query Profile is Snowflake’s primary diagnostic tool** for understanding query execution after a query has run, offering operator‑level granularity that is unavailable in traditional databases.
- **Every query automatically generates a profile** – no special configuration is required.
- **The profile is a Directed Acyclic Graph (DAG)** showing the sequence and structure of operations (or nodes) performed during execution[reference:33].
- **Two access methods**: The **Snowsight UI** provides a rich visual exploration experience; **`GET_QUERY_OPERATOR_STATS`** provides programmatic access for analysis at scale.
- **`EXPLAIN` shows the plan before execution** – useful for understanding query structure, but it does not include runtime statistics.
- **Common performance issues** detectable via Query Profile include: inefficient partition pruning, exploding joins, memory spillage, late‑stage filters, and queue congestion.
- **Spill to remote storage** is one of the most severe performance killers – investigate any query with non‑zero remote spill bytes (excluding QAS).
- **Always distinguish queue time from execution time** – misdiagnosing a warehouse concurrency problem as a query logic problem wastes time and credits.
- **Compare profiles before and after optimisation** to validate that changes had the intended effect.

This completes the thorough explanation of Snowflake’s Query Profile with practical, actionable use cases.
