# What is Cross-Region Sharing?

Traditionally, sharing data across different geographic regions or cloud providers (like AWS US East to Azure Europe, or GCP Asia to AWS US West) has been a complex, costly, and often slow process. It typically involved moving large volumes of data, which brings challenges with security, latency, and egress fees.

Snowflake's **Cross-Region Sharing** capability is designed to overcome these hurdles. It allows data providers to share data with consumers whose Snowflake accounts are in a different region or even on a different cloud platform than the provider's account. This is a huge enabler for global data collaboration and breaking down geographical data silos.

### How Does it Work? The Magic of Replication

The core mechanism that enables cross-region (and cross-cloud) sharing is **database replication**. Snowflake uses a robust, built-in replication feature to create and maintain a synchronized, read-only copy of a database in one or more target regions/clouds. This replicated database then becomes the source for sharing in that specific region.

Here's the simplified flow:

1.  **Provider's Primary Account:** The provider has their original, primary database in their home region/cloud (e.g., AWS US East).
2.  **Database Replication:** The provider configures replication for their database to a target region/cloud where their consumer resides (e.g., Azure Europe). Snowflake automatically creates and keeps synchronized a read-only copy of the database in this target region.
3.  **Share Creation from Replicated Database:** In the target region, the provider then creates a standard Snowflake `SHARE` object *from the replicated database*.
4.  **Consumer Access:** The consumer, whose account is in that target region, can then access this share just like any other share in their local region.

Crucially, because the consumer is accessing a local replica, they get:
*   **Low Latency:** Queries against the shared data are executed in their local region.
*   **No Egress Fees (for the consumer):** Data transfer happens within Snowflake's internal network, and the consumer is not billed for data egress from the provider's primary region.
*   **Data Freshness:** The replicated database is kept in sync with the primary database, so consumers always get access to the latest data based on the replication schedule.

### Key Players and Concepts:

*   **Primary Database:** The original, writable database in the provider's home account.
*   **Secondary (Replicated) Database:** A read-only copy of the primary database in a different region/cloud, maintained by Snowflake's replication feature.
*   **Replication Group:** A collection of databases that are replicated together.
*   **Failover/Failback:** While primarily used for sharing, the replication mechanism also supports failover capabilities for disaster recovery.
*   **Cross-Cloud/Cross-Region Sharing:** The actual act of making data available to accounts in different clouds or regions.

### Steps to Implement Cross-Region Sharing (with Example Queries)

Let's imagine "Global Marketing Insights" (provider, AWS US East) wants to share campaign performance data with "European Retail Group" (consumer, Azure Europe).

#### Step 1: Enable Replication for the Database (Provider)

First, the provider needs to enable the database for replication and add the target region/account where the secondary database will reside.

```sql
-- As Global Marketing Insights (Provider Account - AWS US East)

-- Step 1.1: Create a database to be replicated (if not already existing)
CREATE DATABASE CAMPAIGN_PERFORMANCE_DB;

-- ... Populate CAMPAIGN_PERFORMANCE_DB with data ...
USE DATABASE CAMPAIGN_PERFORMANCE_DB;
CREATE SCHEMA PUBLIC;
CREATE TABLE PUBLIC.DAILY_CAMPAIGN_STATS (
    DATE DATE,
    CAMPAIGN_ID VARCHAR,
    IMPRESSIONS NUMBER,
    CLICKS NUMBER,
    SPEND DECIMAL(10,2)
);
INSERT INTO PUBLIC.DAILY_CAMPAIGN_STATS VALUES
('2026-03-01', 'CAMP_001', 100000, 5000, 150.00),
('2026-03-01', 'CAMP_002', 80000, 4000, 120.00);

-- Step 1.2: Enable the database for replication and add the target region/account
-- Find the account URL/region for the target (e.g., 'azure_europe_west_2.myorg.eu-west-2')
-- (This part usually requires finding the account_locator for the region or a full account identifier.)
-- Example: 'AZURE_EUROPE_WEST_2' refers to the region.
ALTER DATABASE CAMPAIGN_PERFORMANCE_DB ENABLE REPLICATION TO ACCOUNTS AZURE_EUROPE_WEST_2;
```
*   **Note:** The exact syntax for specifying accounts/regions in `ENABLE REPLICATION TO ACCOUNTS` might vary based on your cloud provider configuration and whether it's within the same organization or cross-organization. Often, it involves a combination of cloud provider, region, and organization name.

#### Step 2: Create a Secondary Database from the Primary (Provider)

Now, in the *target region/cloud*, the provider needs to create a secondary database by "promoting" the replicated database.

```sql
-- As Global Marketing Insights (Provider Account - Azure Europe)
-- Logged into the account in the Azure Europe region.

-- Step 2.1: Create the secondary database from the primary, read-only.
CREATE DATABASE CAMPAIGN_PERFORMANCE_DB_EUROPE
  AS REPLICA OF GLOBAL_MARKETING_INSIGHTS_AWS_US_EAST.CAMPAIGN_PERFORMANCE_DB;
-- Replace 'GLOBAL_MARKETING_INSIGHTS_AWS_US_EAST' with your actual provider account identifier from the primary region.
```
*   This creates a read-only copy of `CAMPAIGN_PERFORMANCE_DB` in the Azure Europe region.

#### Step 3: Configure Refresh Schedules (Provider)

The provider sets up how often the secondary database in Azure Europe should refresh from the primary database in AWS US East.

```sql
-- As Global Marketing Insights (Provider Account - Azure Europe)
-- This command is run on the secondary database.
ALTER DATABASE CAMPAIGN_PERFORMANCE_DB_EUROPE REFRESH; -- Manual refresh

-- To set up scheduled refreshes (e.g., every hour)
ALTER DATABASE CAMPAIGN_PERFORMANCE_DB_EUROPE
  SET REPLICATION_SCHEDULE = '1 HOUR';
```

#### Step 4: Create a Share from the Secondary Database (Provider)

Now that you have a synchronized database in the target region, you create a standard share *from that secondary database*.

```sql
-- As Global Marketing Insights (Provider Account - Azure Europe)

-- Step 4.1: Create a secure view if needed (recommended practice)
USE DATABASE CAMPAIGN_PERFORMANCE_DB_EUROPE;
CREATE SECURE VIEW PUBLIC.EUROPE_CAMPAIGN_SUMMARY AS
SELECT
    DATE,
    CAMPAIGN_ID,
    SUM(IMPRESSIONS) AS TOTAL_IMPRESSIONS,
    SUM(CLICKS) AS TOTAL_CLICKS
FROM PUBLIC.DAILY_CAMPAIGN_STATS
GROUP BY DATE, CAMPAIGN_ID;

-- Step 4.2: Create the SHARE object
CREATE SHARE EUROPEAN_CAMPAIGN_SHARE
COMMENT = 'Campaign performance data for European Retail Group.';

-- Step 4.3: Grant privileges to the Share
GRANT USAGE ON DATABASE CAMPAIGN_PERFORMANCE_DB_EUROPE TO SHARE EUROPEAN_CAMPAIGN_SHARE;
GRANT USAGE ON SCHEMA CAMPAIGN_PERFORMANCE_DB_EUROPE.PUBLIC TO SHARE EUROPEAN_CAMPAIGN_SHARE;
GRANT SELECT ON VIEW CAMPAIGN_PERFORMANCE_DB_EUROPE.PUBLIC.EUROPE_CAMPAIGN_SUMMARY TO SHARE EUROPEAN_CAMPAIGN_SHARE;

-- Step 4.4: Add the Consumer Account (European Retail Group's account locator in Azure Europe)
-- Let's assume European Retail Group's account locator is 'ABC12345' in Azure Europe.
ALTER SHARE EUROPEAN_CAMPAIGN_SHARE ADD ACCOUNTS = ABC12345;
```

---

**Steps for the Data Consumer ("European Retail Group")**:

The consumer's experience is largely the same as for any other direct share, but they benefit from accessing data locally.

```sql
-- As European Retail Group (Consumer Account - Azure Europe)

-- Step 1: Show available shares
SHOW SHARES;
-- This will list 'EUROPEAN_CAMPAIGN_SHARE' from 'GLOBAL_MARKETING_INSIGHTS_AZURE_EUROPE'

-- Step 2: Create a database from the inbound share
CREATE DATABASE GLOBAL_MARKETING_CAMPAIGNS
  FROM SHARE GLOBAL_MARKETING_INSIGHTS_AZURE_EUROPE.EUROPEAN_CAMPAIGN_SHARE;
-- Replace 'GLOBAL_MARKETING_INSIGHTS_AZURE_EUROPE' with the provider's account identifier in that region.

-- Step 3: Query the data
USE DATABASE GLOBAL_MARKETING_CAMPAIGNS;
USE SCHEMA CAMPAIGN_PERFORMANCE_DB_EUROPE.PUBLIC;

SELECT *
FROM EUROPE_CAMPAIGN_SUMMARY
WHERE DATE = CURRENT_DATE() - INTERVAL '1 DAY'
ORDER BY CAMPAIGN_ID;
```

### Benefits of Cross-Region Sharing:

*   **Global Collaboration:** Enables seamless data sharing across global teams, partners, and customers regardless of their physical location or cloud provider.
*   **Reduced Latency:** Consumers query a local copy of the data, resulting in faster query performance.
*   **Cost Efficiency:** Minimizes expensive cross-cloud or cross-region data transfer costs (egress fees) for consumers.
*   **Enhanced Performance:** Queries hit a local replica, reducing network overhead and improving overall user experience.
*   **Disaster Recovery Preparedness:** The underlying replication mechanism inherently supports robust disaster recovery strategies.
*   **Simplified Data Governance:** Centralized control from the primary provider account over what is replicated and shared.

### Important Considerations:

*   **Cost for Provider:** The provider pays for the storage of the replicated database(s) and any compute used for the replication process.
*   **Replication Lag:** There will always be a slight delay between updates to the primary database and their appearance in the secondary database, depending on your refresh schedule. This is typically measured in minutes.
*   **Read-Only Replicas:** The secondary databases are read-only. Consumers cannot write to them, nor can the provider make changes directly to the secondary database. All changes must originate from the primary database.
*   **Account Identifiers:** Accurately identifying the provider's source account and the consumer's target account (including region and cloud provider) is crucial during setup.

Cross-Region Sharing is a testament to Snowflake's architecture, providing a highly efficient and flexible way to extend the reach of your data assets across the globe, facilitating truly distributed data ecosystems.
