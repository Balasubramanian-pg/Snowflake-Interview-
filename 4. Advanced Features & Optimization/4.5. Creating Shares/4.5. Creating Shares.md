# What is a Snowflake Share?

At its heart, a **Share** is a named Snowflake object that encapsulates a collection of database objects (like databases, schemas, tables, views, secure views, and even user-defined functions or stored procedures) that you want to make available to other Snowflake accounts.

Think of a `SHARE` object as a permission-controlled "bundle" or "package" of data that you can then distribute. When you create a share, you're essentially defining *what* data you're making available. You then grant this `SHARE` to specific consumer accounts, which dictates *who* gets access to that data.

The magic of Snowflake's sharing is that when a consumer receives a share, they get **live, read-only access** to your data. The data itself never moves, is never copied, and remains stored in your (the provider's) account. This ensures data freshness, reduces storage costs, and simplifies governance for both parties.

### Why Do We Create Shares?

We create shares to facilitate secure, governed, and real-time data collaboration. Here’s why they are so important:

1.  **Live Data Access:** Consumers always see the most current version of the data, instantly reflecting any changes in your account.
2.  **No Data Movement:** Eliminates the need for traditional ETL/ELT processes, FTP transfers, or APIs for data exchange. This reduces complexity, cost, and latency.
3.  **Enhanced Security:** You retain full control over your data. You define exactly what objects are included in the share, and access is read-only.
4.  **Simplified Governance:** Centralized management of shared data and clear auditing trails for all consumer activity.
5.  **Cost Efficiency:** For you (the provider), you pay for the storage of the original data. For consumers (in most cases of account-to-account sharing), they use their own compute resources to query your shared data.

### Anatomy of a Share: What's Inside?

A share object doesn't *contain* data itself; it's a pointer to your existing data. When you define a share, you're essentially specifying:

*   **Databases:** You grant `USAGE` on the database(s) that contain the objects you wish to share.
*   **Schemas:** You grant `USAGE` on the specific schema(s) within those databases.
*   **Tables/Views/Secure Views:** You grant `SELECT` privileges on the individual tables, views, or secure views that hold the actual data you want to share. **Secure views are highly recommended** as they allow you to control column-level and row-level access, and hide the underlying table structure from consumers.
*   **User-Defined Functions (UDFs)/Stored Procedures:** You can also share UDFs and stored procedures.

### Steps to Create and Configure a Share (with Example Queries)

Let's walk through the process of creating a share from the perspective of a data provider.

Imagine you are "Global Analytics Corp." and you want to share a dataset of market trends with a partner, "Market Insights Inc."

**Prerequisites:**

1.  You have data in your Snowflake account.
2.  You have appropriate roles and permissions to create shares and grant privileges (typically `ACCOUNTADMIN` or a custom role with `CREATE SHARE` and `MANAGE GRANTS` privileges).

---

#### Step 1: Prepare Your Data (Optional but Recommended)

Before sharing, it's good practice to organize your data and create secure views to abstract and protect your raw tables. This gives you fine-grained control over what columns/rows are exposed.

```sql
-- As Global Analytics Corp. (Provider Account)

-- Create a database and schema for your data (if not already existing)
CREATE DATABASE GLOBAL_ANALYTICS_DB;
CREATE SCHEMA GLOBAL_ANALYTICS_DB.MARKET_DATA;

-- Create a raw table with sensitive info (e.g., internal IDs, full timestamps)
CREATE TABLE GLOBAL_ANALYTICS_DB.MARKET_DATA.RAW_MARKET_TRENDS (
    TREND_ID NUMBER,
    TREND_DATE DATE,
    MARKET_SEGMENT VARCHAR,
    KEYWORD_PHRASE VARCHAR,
    SEARCH_VOLUME NUMBER,
    COMPETITION_SCORE NUMBER,
    INTERNAL_ANALYST_NOTES VARCHAR -- Sensitive/internal only
);

-- Insert some sample data
INSERT INTO GLOBAL_ANALYTICS_DB.MARKET_DATA.RAW_MARKET_TRENDS (TREND_ID, TREND_DATE, MARKET_SEGMENT, KEYWORD_PHRASE, SEARCH_VOLUME, COMPETITION_SCORE, INTERNAL_ANALYST_NOTES) VALUES
(101, '2026-03-01', 'Tech', 'AI ethics', 50000, 0.7, 'Growing concern, monitor closely'),
(102, '2026-03-01', 'Finance', 'ESG investing', 75000, 0.6, 'Strong institutional interest'),
(103, '2026-03-02', 'Healthcare', 'telemedicine growth', 60000, 0.8, 'Post-pandemic sustained demand');

-- Create a SECURE VIEW to share only relevant, non-sensitive columns and perhaps aggregate/filter.
-- This view will be the actual object shared.
CREATE SECURE VIEW GLOBAL_ANALYTICS_DB.MARKET_DATA.PUBLIC_MARKET_TRENDS AS
SELECT
    TREND_DATE,
    MARKET_SEGMENT,
    KEYWORD_PHRASE,
    SEARCH_VOLUME,
    COMPETITION_SCORE
FROM GLOBAL_ANALYTICS_DB.MARKET_DATA.RAW_MARKET_TRENDS
WHERE TREND_DATE >= '2026-03-01'; -- Only share recent data
```

---

#### Step 2: Create the Share Object

This is where you define the share itself.

```sql
-- Create the SHARE object
CREATE SHARE MARKET_TRENDS_SHARE
COMMENT = 'Share for public market trends data with partners.';
```

---

#### Step 3: Grant Privileges to the Share

This is the most critical step. You grant specific usage and select privileges to the `SHARE` object. Remember, these grants are *on* the share, meaning they define what the *share* can access in your account, which then dictates what consumers can see.

```sql
-- Grant USAGE on the database that contains the objects to be shared
GRANT USAGE ON DATABASE GLOBAL_ANALYTICS_DB TO SHARE MARKET_TRENDS_SHARE;

-- Grant USAGE on the schema within that database
GRANT USAGE ON SCHEMA GLOBAL_ANALYTICS_DB.MARKET_DATA TO SHARE MARKET_TRENDS_SHARE;

-- Grant SELECT on the specific secure view that holds the data you want to expose
GRANT SELECT ON VIEW GLOBAL_ANALYTICS_DB.MARKET_DATA.PUBLIC_MARKET_TRENDS TO SHARE MARKET_TRENDS_SHARE;

-- If you wanted to share a UDF, you'd grant USAGE ON FUNCTION:
-- GRANT USAGE ON FUNCTION GLOBAL_ANALYTICS_DB.MARKET_DATA.MY_UDF(VARCHAR) TO SHARE MARKET_TRENDS_SHARE;
```
**Important Note:** You grant `USAGE` on databases and schemas, but `SELECT` on tables/views. You cannot grant `SELECT` on a database or schema directly to a share.

---

#### Step 4: Add Consumer Accounts to the Share

Finally, you specify which consumer Snowflake accounts can access this share. You'll need their Snowflake account locator.

Let's assume "Market Insights Inc." has a Snowflake account locator `ABC78901`.

```sql
-- Add the specific consumer account(s) to the share
ALTER SHARE MARKET_TRENDS_SHARE ADD ACCOUNTS = ABC78901;

-- You can add multiple accounts by comma-separating them:
-- ALTER SHARE MARKET_TRENDS_SHARE ADD ACCOUNTS = ABC78901, XYZ34567;

-- To remove an account:
-- ALTER SHARE MARKET_TRENDS_SHARE REMOVE ACCOUNTS = ABC78901;
```

**For Reader Accounts / Marketplace / Data Exchange:**
*   **Reader Accounts:** Instead of an account locator, you'd add the specific managed account you created (e.g., `ALTER SHARE MY_SHARE ADD ACCOUNT = MY_READER_ACCOUNT;`).
*   **Marketplace/Data Exchange:** When you create a listing in the UI, Snowflake handles adding the necessary consumer accounts to the underlying share object automatically upon subscription. You generally don't run `ALTER SHARE ... ADD ACCOUNTS` directly for these.

---

#### Step 5: Verify the Share (Optional)

You can check the details of your share.

```sql
-- Show all shares you own
SHOW SHARES;

-- Describe a specific share to see what objects are included and to whom it's granted
DESCRIBE SHARE MARKET_TRENDS_SHARE;
```

---

### What Happens on the Consumer Side?

Once you (the provider) have performed these steps, "Market Insights Inc." (the consumer) will see your share as an "inbound share" in their account.

They will then perform these actions:

1.  **Show Shares:**
    ```sql
    -- As Market Insights Inc. (Consumer Account)
    SHOW SHARES;
    -- This would show MARKET_TRENDS_SHARE from GLOBAL_ANALYTICS_CORP
    ```

2.  **Create Database from Share:**
    ```sql
    CREATE DATABASE GLOBAL_ANALYTICS_MARKET_DATA
    FROM SHARE GLOBAL_ANALYTICS_CORP.MARKET_TRENDS_SHARE;
    -- Note: GLOBAL_ANALYTICS_CORP is the provider's account name
    ```

3.  **Query the Data:**
    ```sql
    USE DATABASE GLOBAL_ANALYTICS_MARKET_DATA;
    USE SCHEMA GLOBAL_ANALYTICS_DB.MARKET_DATA; -- Consumer uses the provider's schema path

    SELECT *
    FROM PUBLIC_MARKET_TRENDS
    WHERE MARKET_SEGMENT = 'Tech'
    ORDER BY TREND_DATE DESC;
    ```

---

### Best Practices for Creating Shares:

*   **Least Privilege:** Only share the absolute minimum data required. Use secure views to filter rows and columns, or to obfuscate sensitive information.
*   **Secure Views:** Always prefer sharing secure views over raw tables. They abstract the underlying data structure and protect against schema changes in the raw tables breaking consumer queries.
*   **Clear Naming:** Use descriptive names for your shares, databases, and views so consumers can easily understand the data they are accessing.
*   **Documentation:** Provide clear documentation to your consumers about the shared schema, data definitions, and any usage guidelines.
*   **Monitoring:** Regularly monitor the usage of your shares in your account to understand consumer activity and performance implications.
*   **Role-Based Access:** Create dedicated roles for managing shares and grant only necessary privileges to those roles, rather than using `ACCOUNTADMIN` directly for day-to-day operations.

By following these steps and best practices, you can effectively leverage Snowflake's sharing capabilities to securely and efficiently collaborate on data.
