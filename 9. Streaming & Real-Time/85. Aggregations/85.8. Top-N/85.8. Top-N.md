# 85.8. Top N

> [!Note]
> The top N problem involves retrieving the top N items from a dataset based on a specific criterion, such as value or rank, which is crucial for data analysis and decision-making.

  I'll create a comprehensive guide to explain **Top N** analysis in Snowflake using **pharmaceutical data** (claims, formulary, LAD, retail sales, 840/852/837 EDI transactions). Top N analysis identifies the highest or lowest ranked items based on specific criteria—critical for prioritization, outlier detection, and resource allocation.

## What is Top N Analysis in Pharma Data?

In pharmaceutical analytics, **Top N** queries identify the most significant records within large datasets—such as the top 10 most expensive drugs, top 5 highest-utilizing patients, or top 20 fastest-processing pharmacies. Unlike filtering which excludes data, Top N preserves context by ranking all records while highlighting the extremes.

**Key Ranking Functions**:
- **ROW_NUMBER()**: Unique sequential rank (no ties)
- **RANK()**: Same rank for ties, skips next ranks (1,1,3)
- **DENSE_RANK()**: Same rank for ties, no skip (1,1,2)
- **NTILE(n)**: Percentile buckets (quartiles, deciles)

('00078-0364-02', 'Januvia 50mg', 'Sitagliptin', 'Merck', 'DPP-4 Inhibitors', 14.80, 12.60, 17.00, 30, 800000, FALSE);

-- TOP N QUERY: Multiple ranking methods for expensive drugs
WITH ranked_drugs AS (
    SELECT 
        ndc_code,
        drug_name,
        manufacturer,
        therapeutic_class,
        wac_per_unit,
        annual_volume_units,
        (wac_per_unit * annual_volume_units) AS annual_revenue_potential,
        is_limited_access,
        
        -- ROW_NUMBER: Strictly sequential, no ties (1,2,3,4...)
        ROW_NUMBER() OVER (ORDER BY wac_per_unit DESC) AS row_num_rank,
        
        -- RANK: Allows ties, skips numbers (1,1,3,4...)
        RANK() OVER (ORDER BY wac_per_unit DESC) AS rank_with_ties,
        
        -- DENSE_RANK: Allows ties, no skips (1,1,2,3...)
        DENSE_RANK() OVER (ORDER BY wac_per_unit DESC) AS dense_rank_ties,
        
        -- NTILE: Quartiles (top 25%, 50%, etc.)
        NTILE(4) OVER (ORDER BY wac_per_unit DESC) AS cost_quartile
        
    FROM formulary_extended
),
top_n_analysis AS (
    SELECT 
        *,
        
        -- Top 10 by row number (exactly 10 items, arbitrary tie-breaker)
        CASE WHEN row_num_rank <= 10 THEN 'TOP_10_STRICT' ELSE 'OUTSIDE_TOP_10' END AS top_10_strict,
        
        -- Top 10 by rank (could be more than 10 if ties exist)
        CASE WHEN rank_with_ties <= 10 THEN 'TOP_10_WITH_TIES' ELSE 'OUTSIDE_TOP_10' END AS top_10_with_ties,
        
        -- Top 25% (quartile)
        CASE WHEN cost_quartile = 1 THEN 'TOP_QUARTILE_25PCT' ELSE 'OTHER' END AS top_quartile,
        
        -- Tied rank detection
        CASE 
            WHEN rank_with_ties != row_num_rank THEN 'HAS_TIES_AT_THIS_LEVEL'
            ELSE 'NO_TIES'
        END AS tie_status
        
    FROM ranked_drugs
)
SELECT 
    row_num_rank AS sequential_position,
    rank_with_ties AS rank_position,
    dense_rank_ties AS dense_position,
    ndc_code,
    drug_name,
    manufacturer,
    therapeutic_class,
    wac_per_unit AS cost_per_unit,
    annual_volume_units,
    ROUND((wac_per_unit * annual_volume_units), 2) AS annual_dollar_volume,
    is_limited_access AS limited_access_drug,
    
    -- Classification
    CASE 
        WHEN row_num_rank = 1 THEN 'HIGHEST_COST'
        WHEN row_num_rank <= 5 THEN 'TOP_5_CRITICAL'
        WHEN row_num_rank <= 10 THEN 'TOP_10_HIGH_PRIORITY'
        WHEN cost_quartile = 1 THEN 'TOP_25PCT_TIER'
        ELSE 'STANDARD_COST'
    END AS negotiation_priority,
    
    -- Tie information
    tie_status,
    COUNT(*) OVER (PARTITION BY rank_with_ties) AS items_at_this_rank,
    
    -- Business insight
    CASE 
        WHEN wac_per_unit > 1000 AND is_limited_access THEN 'SPECIALTY_LAD_HIGH_IMPACT'
        WHEN wac_per_unit > 100 AND annual_volume_units > 1000000 THEN 'HIGH_VOLUME_HIGH_COST_FOCUS'
        ELSE 'EVALUATE_FORMULARY_STATUS'
    END AS contract_strategy
    
FROM top_n_analysis

WHERE row_num_rank <= 20  -- Return top 20 regardless of ties

ORDER BY wac_per_unit DESC, ndc_code;
```

**Workflow Explanation**:

1. **ROW_NUMBER()** - Assigns unique sequential integers (1,2,3...). Use when you need exactly N items, even if arbitrary tie-breaking occurs.
2. **RANK()** - Assigns same number to ties, skips next numbers (1,1,3). Use when ties are meaningful and should share the same position.
3. **DENSE_RANK()** - Assigns same number to ties, no skip (1,1,2). Use when you want consecutive ranking despite ties.
4. **NTILE(4)** - Divides data into 4 equal buckets (quartiles). Use for percentile-based analysis (top 25%).
5. **Tie Detection** - Compares ROW_NUMBER vs RANK to identify when multiple drugs share the same cost.

    DATEADD(DAY, -seq, CURRENT_DATE()) AS service_date,

    SELECT 
        claim_id,
        patient_id,
        drug_name,
        pharmacy_npi,
        payer_id,
        service_date,
        billed_amount,
        allowed_amount,
        is_specialty_pharmacy,
        
        -- Standard row number
        ROW_NUMBER() OVER (ORDER BY allowed_amount DESC) AS strict_rank,
        
        -- Rank with ties
        RANK() OVER (ORDER BY allowed_amount DESC) AS rank_with_ties,
        
        -- Count how many share this rank
        COUNT(*) OVER (PARTITION BY allowed_amount) AS tie_count
        
    FROM high_value_claims
),
top_50_threshold AS (
    -- Find the value at rank 50 to include all ties
    SELECT 
        MAX(allowed_amount) AS threshold_value
    FROM ranked_claims
    WHERE rank_with_ties <= 50
),
final_selection AS (
    SELECT 
        r.*,
        
        -- Include all claims at the 50th rank value (WITH TIES equivalent)
        CASE 
            WHEN r.rank_with_ties <= 50 THEN 'INCLUDED_TOP_50'
            WHEN r.allowed_amount = (SELECT threshold_value FROM top_50_threshold) THEN 'INCLUDED_TIE_AT_50'
            ELSE 'EXCLUDED'
        END AS inclusion_status,
        
        -- Percentile rank
        PERCENT_RANK() OVER (ORDER BY allowed_amount DESC) AS percentile_rank
        
    FROM ranked_claims r
)
SELECT 
    strict_rank AS row_number,
    rank_with_ties AS claim_rank,
    inclusion_status,
    claim_id,
    patient_id,
    drug_name,
    pharmacy_npi,
    payer_id,
    service_date,
    billed_amount AS claim_billed,
    allowed_amount AS claim_allowed,
    is_specialty_pharmacy,
    
    -- Tie information
    tie_count AS other_claims_at_this_amount,
    
    -- Business classification
    CASE 
        WHEN rank_with_ties <= 10 THEN 'TIER_1_CRITICAL_REVIEW'
        WHEN rank_with_ties <= 25 THEN 'TIER_2_HIGH_VALUE_MONITOR'
        WHEN rank_with_ties <= 50 THEN 'TIER_3_STANDARD_HIGH'
        ELSE 'STANDARD_CLAIM'
    END AS audit_priority,
    
    -- Financial impact
    SUM(allowed_amount) OVER (ORDER BY allowed_amount DESC ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS running_total_allowed,
    
    -- Pareto analysis (80/20 rule)
    ROUND(100.0 * SUM(allowed_amount) OVER (ORDER BY allowed_amount DESC ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) / 
          SUM(allowed_amount) OVER (), 2) AS cumulative_pct_of_total
    
FROM final_selection

WHERE inclusion_status LIKE 'INCLUDED%'

ORDER BY allowed_amount DESC, claim_id;
```

**Workflow Explanation**:

1. **RANK() for Ties** - Uses `RANK()` instead of `ROW_NUMBER()` to identify claims with identical allowed amounts.
2. **Threshold Detection** - Subquery finds the dollar value at the 50th rank to include all ties.
3. **WITH TIES Logic** - Includes claims beyond position 50 if they share the same value as #50.
4. **Pareto Analysis** - Running total shows what percentage of total value the Top N represents (typically 80% of value in 20% of claims).
5. **Audit Prioritization** - Stratifies into tiers (Top 10, Top 25, Top 50) for resource allocation.

## Use Case 3: Top N Fastest/Slowest Processing Pharmacies (Performance)

**Scenario**: Rank pharmacies by claims processing speed to identify top performers and laggards.

```sql
-- Create pharmacy performance data
CREATE OR REPLACE TABLE pharmacy_performance (
    pharmacy_npi VARCHAR(10),
    pharmacy_name VARCHAR(100),
    chain_affiliation VARCHAR(50),
    state VARCHAR(2),
    total_claims_30d INTEGER,
    avg_processing_time_min DECIMAL(8,2),
    median_processing_time_min DECIMAL(8,2),
    p95_processing_time_min DECIMAL(8,2),
    error_rate_pct DECIMAL(5,2),
    first_pass_resolution_rate DECIMAL(5,2),
    total_revenue_30d DECIMAL(12,2)
);

-- Insert performance data
INSERT INTO pharmacy_performance VALUES
('PHARM001', 'CVS Pharmacy #1234', 'CVS Health', 'CA', 12500, 2.5, 2.1, 4.5, 0.5, 98.5, 450000.00),
('PHARM002', 'Walgreens #5678', 'Walgreens', 'TX', 11800, 3.2, 2.8, 6.2, 1.2, 96.0, 420000.00),
('PHARM003', 'Rite Aid #9012', 'Rite Aid', 'NY', 8200, 4.8, 4.2, 9.5, 2.5, 92.0, 310000.00),
('PHARM004', 'Walmart Pharmacy', 'Walmart', 'FL', 15000, 2.1, 1.9, 3.8, 0.8, 97.8, 380000.00),
('PHARM005', 'Kroger Pharmacy', 'Kroger', 'OH', 9500, 3.5, 3.1, 7.0, 1.5, 94.5, 290000.00),
('SP_001', 'Accredo Specialty', 'CVS Specialty', 'TN', 3200, 8.5, 7.2, 15.0, 0.3, 99.2, 2800000.00),
('SP_002', 'BioPlus Specialty', 'BioPlus', 'FL', 2800, 12.3, 10.5, 22.0, 0.8, 98.0, 2200000.00),
('SP_003', 'CVS Caremark Specialty', 'CVS Health', 'AZ', 4100, 6.2, 5.5, 11.5, 0.4, 98.8, 3100000.00);

-- TOP N QUERY: Top and Bottom performers
WITH performance_ranked AS (
    SELECT 
        *,
        
        -- Top N fastest (ascending time)
        ROW_NUMBER() OVER (ORDER BY avg_processing_time_min ASC) AS fastest_rank,
        RANK() OVER (ORDER BY avg_processing_time_min ASC) AS fastest_rank_ties,
        
        -- Top N slowest (descending time)
        ROW_NUMBER() OVER (ORDER BY avg_processing_time_min DESC) AS slowest_rank,
        
        -- Top N by revenue efficiency (revenue per minute of processing)
        ROW_NUMBER() OVER (ORDER BY (total_revenue_30d / NULLIF(avg_processing_time_min, 0) / total_claims_30d) DESC) AS efficiency_rank,
        
        -- Top N by quality (first pass resolution)
        ROW_NUMBER() OVER (ORDER BY first_pass_resolution_rate DESC, error_rate_pct ASC) AS quality_rank,
        
        -- Composite score (weighted: speed 40%, quality 40%, volume 20%)
        (0.4 * (100 - avg_processing_time_min * 10) + 
         0.4 * first_pass_resolution_rate + 
         0.2 * LEAST(total_claims_30d / 1000.0, 10)) AS composite_score
        
    FROM pharmacy_performance
),
top_bottom_analysis AS (
    SELECT 
        *,
        
        -- Top 5 fastest
        CASE WHEN fastest_rank <= 5 THEN 'TOP_5_FASTEST' END AS speed_category,
        
        -- Bottom 5 slowest (out of total count)
        CASE WHEN fastest_rank >= (SELECT COUNT(*) FROM pharmacy_performance) - 4 THEN 'BOTTOM_5_SLOWEST' END AS slow_category,
        
        -- Top 5 by composite score
        ROW_NUMBER() OVER (ORDER BY composite_score DESC) AS composite_rank
        
    FROM performance_ranked
)
SELECT 
    pharmacy_npi,
    pharmacy_name,
    chain_affiliation,
    state,
    total_claims_30d AS monthly_volume,
    avg_processing_time_min AS avg_processing_min,
    median_processing_time_min AS median_processing_min,
    first_pass_resolution_rate AS fpr_pct,
    error_rate_pct AS error_rate,
    total_revenue_30d AS monthly_revenue,
    
    -- Rankings
    fastest_rank AS speed_rank_asc,
    slowest_rank AS speed_rank_desc,
    quality_rank,
    composite_rank,
    
    -- Categorization
    COALESCE(speed_category, slow_category, 'MIDDLE_TIER') AS performance_tier,
    
    -- Benchmark comparison vs median
    ROUND(avg_processing_time_min - (SELECT MEDIAN(avg_processing_time_min) FROM pharmacy_performance), 2) AS min_vs_median,
    
    -- Best in class flags
    CASE WHEN fastest_rank = 1 THEN 'FASTEST_NATIONAL' END AS national_speed_leader,
    CASE WHEN quality_rank = 1 THEN 'HIGHEST_QUALITY' END AS quality_leader,
    
    -- Improvement opportunity (slow but high volume = high impact if improved)
    CASE 
        WHEN slowest_rank <= 5 AND total_claims_30d > 10000 
        THEN 'HIGH_IMPACT_OPTIMIZATION_TARGET'
        WHEN slowest_rank <= 5 
        THEN 'LOW_PRIORITY_OPTIMIZATION'
        ELSE 'MAINTAIN_PERFORMANCE'
    END AS improvement_priority
    
FROM top_bottom_analysis

WHERE fastest_rank <= 5  -- Top 5 fastest
   OR slowest_rank <= 5  -- Top 5 slowest (which are bottom 5)
   OR composite_rank <= 5  -- Top 5 overall

ORDER BY fastest_rank;
```

**Workflow Explanation**:

1. **Dual Ranking** - Creates both ascending (fastest) and descending (slowest) rankings from same data.
2. **Composite Scoring** - Weights speed (40%), quality (40%), and volume (20%) for overall ranking.
3. **Dynamic Bottom N** - Calculates "bottom 5" as `COUNT(*) - 4` to get last 5 regardless of dataset size.
4. **Benchmarking** - Compares each pharmacy to median processing time (contextual performance).
5. **Impact Prioritization** - Flags slow pharmacies with high volume as "high impact" optimization targets.

## Use Case 4: Top N Patients by Utilization (Risk Stratification)

**Scenario**: Identify top 1% of patients by medication utilization for care management intervention.

```sql
-- Create patient utilization data
CREATE OR REPLACE TABLE patient_utilization (
    patient_id VARCHAR(30),
    age INTEGER,
    gender CHAR(1),
    zip_code VARCHAR(5),
    primary_payer VARCHAR(20),
    
    -- Utilization metrics
    unique_drugs_12m INTEGER,
    total_claims_12m INTEGER,
    total_cost_12m DECIMAL(12,2),
    total_days_supply_12m INTEGER,
    unique_pharmacies_12m INTEGER,
    unique_prescribers_12m INTEGER,
    
    -- High-risk indicators
    has_opioid BOOLEAN,
    has_antipsychotic BOOLEAN,
    has_diabetes_meds BOOLEAN,
    emergency_visits_12m INTEGER,
    hospitalizations_12m INTEGER
);

-- Insert patient data with wide distribution
INSERT INTO patient_utilization 
SELECT 
    'PT_' || LPAD(seq::STRING, 6, '0') AS patient_id,
    25 + (seq % 70) AS age,
    CASE WHEN seq % 2 = 0 THEN 'M' ELSE 'F' END AS gender,
    LPAD((seq % 99999)::STRING, 5, '0') AS zip_code,
    CASE 
        WHEN seq % 4 = 0 THEN 'Medicare'
        WHEN seq % 4 = 1 THEN 'Medicaid'
        WHEN seq % 4 = 2 THEN 'Commercial'
        ELSE 'Self-Pay'
    END AS primary_payer,
    
    -- Utilization follows power law (few high users, many low)
    CASE 
        WHEN seq % 100 = 0 THEN 50 + (seq % 20)  -- Top 1%
        WHEN seq % 20 = 0 THEN 15 + (seq % 10)    -- Top 5%
        ELSE 3 + (seq % 5)                         -- Rest
    END AS unique_drugs_12m,
    
    CASE 
        WHEN seq % 100 = 0 THEN 120 + (seq % 50)
        WHEN seq % 20 = 0 THEN 40 + (seq % 20)
        ELSE 8 + (seq % 5)
    END AS total_claims_12m,
    
    CASE 
        WHEN seq % 100 = 0 THEN 50000 + (seq * 100)
        WHEN seq % 20 = 0 THEN 15000 + (seq * 10)
        ELSE 1200 + (seq * 2)
    END AS total_cost_12m,
    
    CASE 
        WHEN seq % 100 = 0 THEN 8000
        WHEN seq % 20 = 0 THEN 2500
        ELSE 365
    END AS total_days_supply_12m,
    
    1 + (seq % 5) AS unique_pharmacies_12m,
    1 + (seq % 8) AS unique_prescribers_12m,
    
    seq % 10 = 0 AS has_opioid,
    seq % 15 = 0 AS has_antipsychotic,
    seq % 5 = 0 AS has_diabetes_meds,
    seq % 100 AS emergency_visits_12m,
    seq % 200 AS hospitalizations_12m
FROM TABLE(GENERATOR(ROWCOUNT => 10000)) AS gen(seq);

-- TOP N QUERY: Top 1% and 5% by cost (Pareto analysis)
WITH utilization_ranked AS (
    SELECT 
        patient_id,
        age,
        primary_payer,
        unique_drugs_12m,
        total_claims_12m,
        total_cost_12m,
        total_days_supply_12m,
        unique_pharmacies_12m,
        unique_prescribers_12m,
        has_opioid,
        emergency_visits_12m,
        
        -- Percentile ranking (0.0 to 1.0, where 0.0 is highest cost)
        PERCENT_RANK() OVER (ORDER BY total_cost_12m DESC) AS cost_percentile,
        
        -- Cumulative distribution
        CUME_DIST() OVER (ORDER BY total_cost_12m DESC) AS cumulative_pct,
        
        -- NTILE for deciles
        NTILE(10) OVER (ORDER BY total_cost_12m DESC) AS cost_decile,
        
        -- Row number for strict top N
        ROW_NUMBER() OVER (ORDER BY total_cost_12m DESC) AS cost_rank
        
    FROM patient_utilization
),
population_stats AS (
    SELECT 
        COUNT(*) AS total_patients,
        SUM(total_cost_12m) AS total_population_cost,
        AVG(total_cost_12m) AS avg_patient_cost,
        PERCENTILE_CONT(0.99) WITHIN GROUP (ORDER BY total_cost_12m) AS p99_cost_threshold,
        PERCENTILE_CONT(0.95) WITHIN GROUP (ORDER BY total_cost_12m) AS p95_cost_threshold
    FROM patient_utilization
),
stratified_patients AS (
    SELECT 
        ur.*,
        ps.total_patients,
        ps.total_population_cost,
        
        -- Stratification logic
        CASE 
            WHEN cost_percentile <= 0.01 THEN 'TOP_1PCT_SUPER_UTILIZER'
            WHEN cost_percentile <= 0.05 THEN 'TOP_5PCT_HIGH_UTILIZER'
            WHEN cost_percentile <= 0.10 THEN 'TOP_10PCT_ELEVATED'
            ELSE 'STANDARD_UTILIZATION'
        END AS risk_stratification,
        
        -- Running totals for Pareto
        SUM(total_cost_12m) OVER (ORDER BY total_cost_12m DESC ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS cumulative_cost,
        
        -- Cost concentration
        ROUND(100.0 * SUM(total_cost_12m) OVER (ORDER BY total_cost_12m DESC ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) / ps.total_population_cost, 2) AS pct_of_total_spend
        
    FROM utilization_ranked ur
    CROSS JOIN population_stats ps
)
SELECT 
    cost_rank AS patient_cost_rank,
    patient_id,
    age,
    primary_payer,
    total_cost_12m AS annual_medication_cost,
    total_claims_12m AS annual_claim_count,
    unique_drugs_12m AS unique_medications,
    
    -- Risk indicators
    has_opioid AS opioid_user,
    emergency_visits_12m AS ed_visits,
    hospitalizations_12m AS hospital_stays,
    
    -- Stratification
    risk_stratification,
    cost_decile,
    ROUND(cost_percentile * 100, 2) AS percentile_rank,
    
    -- Pareto concentration
    cumulative_cost AS running_total_cost,
    pct_of_total_spend AS cumulative_pct_spend,
    
    -- Intervention priority
    CASE 
        WHEN cost_percentile <= 0.01 AND has_opioid THEN 'CRITICAL_CARE_MANAGEMENT_OPIOID'
        WHEN cost_percentile <= 0.01 AND emergency_visits_12m > 2 THEN 'CRITICAL_FREQUENT_FLYER'
        WHEN cost_percentile <= 0.01 THEN 'TIER_1_CARE_MANAGEMENT'
        WHEN cost_percentile <= 0.05 AND unique_pharmacies_12m > 2 THEN 'TIER_2_PHARMACY_SHOPPING'
        WHEN cost_percentile <= 0.05 THEN 'TIER_2_CARE_COORDINATION'
        ELSE 'ROUTINE_MONITORING'
    END AS intervention_strategy,
    
    -- Provider action
    CASE 
        WHEN unique_prescribers_12m > 3 THEN 'MULTIPLE_PRESCRIBERS_REVIEW_DUPLICATION'
        WHEN unique_pharmacies_12m > 2 THEN 'MULTIPLE_PHARMACIES_REVIEW_FRAUD'
        WHEN total_days_supply_12m > 365 THEN 'EXCESSIVE_DAYS_SUPPLY_HOARDING_RISK'
        ELSE 'STANDARD_CARE'
    END AS provider_alert
    
FROM stratified_patients

WHERE cost_percentile <= 0.05  -- Top 5% only for detailed review

ORDER BY total_cost_12m DESC;
```

**Workflow Explanation**:

1. **PERCENT_RANK()** - Calculates relative standing (0.01 = top 1%) for risk stratification.
2. **NTILE(10)** - Creates deciles for population segmentation.
3. **CUME_DIST()** - Shows cumulative distribution (what % of population has higher cost).
4. **Pareto Analysis** - Running sum shows if top 5% account for 50% of costs (typical in healthcare).
5. **Risk Stratification** - Combines cost percentile with clinical indicators (opioid use, ED visits) for intervention prioritization.

```sql

INSERT INTO drug_safety_surveillance VALUES

('Revlimid 10mg', '61755-001-02', 25000, 625, 250, 35, 450, 85, 180, 120, 95, 0.015, 0.004),
('Harvoni', '00078-0429-61', 8000, 120, 45, 5, 68, 12, 25, 15, 45, 0.012, 0.003),
('Metformin 500mg', '00093-2748-01', 8000000, 8000, 400, 15, 1200, 850, 4500, 280, 450, 0.008, 0.001),
('Lisinopril 10mg', '00093-1058-01', 5000000, 3500, 175, 45, 800, 520, 450, 180, 120, 0.006, 0.0008),
('Atorvastatin 20mg', '00555-0808-02', 3500000, 2800, 140, 20, 600, 180, 320, 150, 85, 0.007, 0.001);

-- TOP N QUERY: Safety signal detection (Top N highest risk drugs)
WITH safety_metrics AS (
    SELECT 
        drug_name,
        ndc_code,
        total_dispensed_12m,
        total_adverse_events,
        serious_adverse_events,
        deaths,
        
        -- Observed rates
        total_adverse_events * 1.0 / total_dispensed_12m AS observed_ae_rate,
        serious_adverse_events * 1.0 / total_dispensed_12m AS observed_serious_rate,
        deaths * 1.0 / total_dispensed_12m AS observed_death_rate,
        
        -- Expected counts based on baseline
        total_dispensed_12m * expected_ae_rate AS expected_ae_count,
        total_dispensed_12m * expected_serious_rate AS expected_serious_count,
        
        -- Reporting Odds Ratio (ROR) approximation
        (total_adverse_events * 1.0 / (total_dispensed_12m - total_adverse_events)) / 
        (expected_ae_rate / (1 - expected_ae_rate)) AS reporting_odds_ratio,
        
        -- Proportional Reporting Ratio (PRR)
        (total_adverse_events / total_dispensed_12m) / NULLIF(expected_ae_rate, 0) AS prr
        
    FROM drug_safety_surveillance
),
signal_detection AS (
    SELECT 
        *,
        
        -- Signal strength classification
        CASE 
            WHEN prr >= 5 AND total_adverse_events >= 3 THEN 'STRONG_SIGNAL'
            WHEN prr >= 2 AND total_adverse_events >= 3 THEN 'MODERATE_SIGNAL'
            WHEN prr >= 1.5 AND total_adverse_events >= 3 THEN 'WEAK_SIGNAL'
            ELSE 'NO_SIGNAL'
        END AS signal_strength,
        
        -- Excess events vs expected
        total_adverse_events - expected_ae_count AS excess_events,
        
        -- Seriousness ratio
        serious_adverse_events * 1.0 / NULLIF(total_adverse_events, 0) AS pct_serious

),

        -- Composite safety score

         (deaths * 1000.0 / total_dispensed_12m) * 0.3) AS composite_risk_score
        
    FROM signal_detection
    WHERE signal_strength != 'NO_SIGNAL'  -- Only ranked if signal detected
)
SELECT 
    prr_rank AS risk_rank,
    drug_name,
    ndc_code,
    total_dispensed_12m AS annual_dispensing_volume,
    
    -- Event counts
    total_adverse_events AS total_aes,
    serious_adverse_events AS serious_aes,
    deaths AS fatalities,
    
    -- Rates per 1000 dispensed
    ROUND(observed_ae_rate * 1000, 2) AS ae_rate_per_1000,
    ROUND(observed_serious_rate * 1000, 2) AS serious_rate_per_1000,
    ROUND(observed_death_rate * 1000, 4) AS death_rate_per_1000,
    
    -- Signal metrics
    ROUND(prr, 2) AS proportional_reporting_ratio,
    signal_strength,
    ROUND(excess_events, 0) AS excess_events_vs_expected,
    
    -- Risk categorization
    CASE 
        WHEN prr >= 5 AND deaths > 10 THEN 'CRITICAL_BLACK_BOX_REVIEW'
        WHEN prr >= 3 AND serious_adverse_events > 100 THEN 'HIGH_RISK_REMS_EVALUATION'
        WHEN prr >= 2 THEN 'MODERATE_RISK_ENHANCED_MONITORING'
        ELSE 'STANDARD_MONITORING'
    END AS regulatory_action,
    
    -- Clinical priority
    mortality_rank AS fatality_rank,
    volume_impact_rank AS excess_volume_rank,
    
    -- Provider communication
    CASE 
        WHEN prr_rank <= 3 THEN 'URGENT_PROVIDER_COMMUNICATION_REQUIRED'
        WHEN prr_rank <= 10 THEN 'ADD_TO_FORMULARY_ALERTS'
        ELSE 'ROUTINE_SURVEILLANCE'
    END AS communication_priority
    
FROM ranked_signals

WHERE prr_rank <= 20  -- Top 20 safety signals

ORDER BY composite_risk_score DESC;
```

**Workflow Explanation**:

1. **Disproportionality Analysis** - Calculates PRR (Proportional Reporting Ratio) comparing observed vs. expected adverse events.
2. **Signal Strength** - Classifies signals as Strong (PRR≥5), Moderate (PRR≥2), or Weak (PRR≥1.5).
3. **Excess Events** - Quantifies actual harm above baseline expectations (public health impact).
4. **Multi-metric Ranking** - Ranks by PRR, mortality rate, and total excess events (different perspectives).

## Key Top N Concepts Summary

| Function | Behavior | Use Case | Pharma Example |
|----------|----------|----------|----------------|
| **ROW_NUMBER()** | Unique, no ties | Exact counts needed | "Exactly 10 drugs for formulary review" |
| **RANK()** | Ties allowed, skips | Fair ranking with ties | "Top 50 claims including ties at cutoff" |
| **DENSE_RANK()** | Ties allowed, no skips | Consecutive ranking | "Severity levels without gaps" |
| **NTILE(n)** | Equal buckets | Percentile groups | "Top 25% (quartile) high-utilizers" |
| **PERCENT_RANK()** | 0-1 relative position | Risk stratification | "Top 1% super-utilizers" |
| **CUME_DIST()** | Cumulative probability | Pareto analysis | "What % of patients account for 80% cost" |

**Critical Top N Validation Rules**:

1. **Tie Handling** - Document whether ties are included or excluded (affects patient safety if rationing care).
2. **Dynamic Thresholds** - Use subqueries to find cutoff values rather than hard limits (includes all relevant ties).
3. **Stratification** - Use `PARTITION BY` for category-specific rankings (e.g., top drug per therapeutic class).
4. **Context Preservation** - Always include percentile or score, not just rank (rank 50/100 vs rank 50/50,000).
5. **Composite Scoring** - Weight multiple factors rather than single metric (prevents gaming the system).

**Silent Failure Modes in Top N**:

| Failure | Symptom | Detection |
|---------|---------|-----------|
| **Arbitrary Tie-Breaking** | Similar values get different ranks | Compare ROW_NUMBER vs RANK |
| **Small Sample Bias** | Top N in small group looks significant | Include confidence intervals |
| **Ranking Inflation** | Everyone is "Top 10" in some category | Validate partition sizes |
| **Temporal Blindness** | All-time Top N ignores recent decline | Include time-decay weights |
| **Survivorship Bias** | Only successful drugs ranked (failed ones excluded) | Audit exclusion criteria |

**Remember**: Just like the e-commerce example where Top 10 by revenue differs from Top 10 by profit margin, pharmaceutical Top N analysis must explicitly define the criterion (cost, utilization, risk, speed) to ensure the "right" items are prioritized for patient care and financial optimization!

> [!Tip]
> To find the top N items, try using a combination of sorting, filtering, and ranking techniques, such as using the `ORDER BY` and `LIMIT` clauses in SQL or the `sort` and `head` functions in Python.

## Key Points
- The top N problem can be solved using various algorithms, including sorting, hashing, and heap-based approaches.
- The choice of algorithm depends on the size of the dataset, the available computational resources, and the desired level of accuracy.
- The top N items can be retrieved based on a single criterion or multiple criteria, such as value, rank, or frequency.
- The result can be a list, a table, or a graph, depending on the desired output format.

> [!important]
> The one rule that must not be violated is that the top N items must be determined based on a well-defined and consistent criterion to ensure accurate and reliable results.

## Mental Model
- Step 1: Define the criterion for determining the top N items, such as value or rank.
- Step 2: Collect and preprocess the data to ensure it is clean, complete, and consistent.
- Step 3: Apply the chosen algorithm to retrieve the top N items based on the defined criterion.

## Observe
- A common mistake people make is assuming that the top N items are always the ones with the highest values, without considering other factors such as frequency, rank, or context.
- A silent failure mode is when the algorithm used to retrieve the top N items is not scalable or efficient, leading to performance issues or incorrect results.
- A misleading assumption is that the top N items are always representative of the entire dataset, without considering potential biases or outliers.

> [!Important]
> For example, in a e-commerce platform, finding the top 10 best-selling products can help inform inventory management and marketing strategies, but it requires careful consideration of the criterion used to determine the top products, such as sales revenue, customer reviews, or profit margins.
