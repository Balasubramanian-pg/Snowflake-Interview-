# 85.2. Hopping

> [!Note]
> Hopping is a complex movement pattern that involves the coordinated contraction of multiple muscle groups to achieve explosive propulsion.

## What is Hopping in Data Engineering?

In stream processing, **Hopping Windows** (also called **Sliding Windows with Hop Size**) are fixed-size windows that advance by a specified "hop" interval. Unlike tumbling windows, **hopping windows overlap**—just like how each hop in a basketball player's movement overlaps with their previous position.

**Key Formula**:
- Window Size = How long each window lasts (e.g., 10 minutes)
- Hop Size = How much the window moves forward (e.g., 5 minutes)
- **Overlap = Window Size - Hop Size** (e.g., 5 minutes overlap)

## Use Case 1: Basketball Player Movement Analytics

**Scenario**: Track player performance metrics using 30-second windows hopping every 10 seconds (20-second overlap).

```sql
-- Create sample data: Basketball player sensor data
CREATE OR REPLACE TABLE player_movements (
    player_id VARCHAR(30),
    team VARCHAR(20),
    action VARCHAR(30), -- 'hop', 'sprint', 'jump', 'land'
    force_impact DECIMAL(6,2), -- Newtons
    ankle_angle DECIMAL(5,2), -- degrees
    knee_flexion DECIMAL(5,2), -- degrees
    recorded_at TIMESTAMP
);

-- Insert sample data simulating a basketball game
INSERT INTO player_movements VALUES
('PLAYER_23', 'LAKERS', 'hop_takeoff', 850.00, 75.0, 45.0, '2024-08-01 19:30:05'),
('PLAYER_23', 'LAKERS', 'hop_airborne', 0.00, 90.0, 60.0, '2024-08-01 19:30:06'),
('PLAYER_23', 'LAKERS', 'hop_landing', 1200.00, 85.0, 55.0, '2024-08-01 19:30:07'),
('PLAYER_23', 'LAKERS', 'hop_takeoff', 900.00, 78.0, 48.0, '2024-08-01 19:30:15'),
('PLAYER_23', 'LAKERS', 'hop_landing', 1350.00, 88.0, 58.0, '2024-08-01 19:30:17'),
('PLAYER_23', 'LAKERS', 'direction_change', 600.00, 80.0, 50.0, '2024-08-01 19:30:25'),
('PLAYER_24', 'LAKERS', 'hop_takeoff', 820.00, 76.0, 46.0, '2024-08-01 19:30:08');

-- HOPPING WINDOW QUERY: Analyze movement patterns with overlap
WITH hop_parameters AS (
    -- Define hopping window configuration (like setting hop height and distance)
    SELECT 
        30 AS window_size_seconds,  -- Window duration (like total hop time)
        10 AS hop_size_seconds        -- Advance interval (like distance between hops)
),
time_series AS (
    -- Generate hopping window start times (like each hop's takeoff point)
    SELECT 
        DATEADD(SECOND, 
            seq * (SELECT hop_size_seconds FROM hop_parameters), 
            DATE_TRUNC('MINUTE', MIN(recorded_at))
        ) AS window_start
    FROM player_movements,
    TABLE(GENERATOR(ROWCOUNT => 20)) AS gen(seq)
    WHERE seq * (SELECT hop_size_seconds FROM hop_parameters) <= 60
),
hopping_windows AS (
    SELECT 
        ts.window_start,
        DATEADD(SECOND, 
            (SELECT window_size_seconds FROM hop_parameters), 
            ts.window_start
        ) AS window_end,
        p.player_id,
        p.action,
        p.force_impact,
        p.ankle_angle,
        p.knee_flexion,
        p.recorded_at
    FROM time_series ts
    LEFT JOIN player_movements p
        ON p.recorded_at >= ts.window_start 
        AND p.recorded_at < DATEADD(SECOND, 
            (SELECT window_size_seconds FROM hop_parameters), 
            ts.window_start
        )
)
SELECT 
    window_start,
    window_end,
    player_id,
    
    -- Hopping mechanics analysis (like analyzing each hop's phases)
    COUNT(CASE WHEN action LIKE '%takeoff%' THEN 1 END) AS takeoffs,
    COUNT(CASE WHEN action LIKE '%landing%' THEN 1 END) AS landings,
    COUNT(CASE WHEN action LIKE '%airborne%' THEN 1 END) AS airborne_phases,
    
    -- Force analysis (propulsion vs absorption like hopping mechanics)
    AVG(CASE WHEN action LIKE '%takeoff%' THEN force_impact END) AS avg_propulsion_force,
    AVG(CASE WHEN action LIKE '%landing%' THEN force_impact END) AS avg_landing_force,
    MAX(force_impact) AS peak_impact,
    
    -- Joint stability (ankle and knee control like core stability in hopping)
    AVG(ankle_angle) AS avg_ankle_angle,
    STDDEV(knee_flexion) AS knee_stability_variation, -- Lower is more stable
    
    -- Injury risk assessment (the "one rule" - proper landing technique)
    CASE 
        WHEN AVG(CASE WHEN action LIKE '%landing%' THEN force_impact END) > 1300 
             AND STDDEV(knee_flexion) > 10
        THEN 'HIGH_INJURY_RISK'
        WHEN AVG(CASE WHEN action LIKE '%landing%' THEN force_impact END) > 1100 
        THEN 'MODERATE_RISK'
        ELSE 'HEALTHY_HOPPING_PATTERN'
    END AS injury_risk_assessment
    
FROM hopping_windows

WHERE player_id IS NOT NULL

GROUP BY window_start, window_end, player_id

ORDER BY window_start, player_id;
```

**Workflow Explanation**:

1. **Hop Parameters** - Define window size (30s) and hop size (10s) like setting how high and far a basketball player hops
2. **Time Series Generation** - Creates overlapping window start times every 10 seconds (19:30:00, 19:30:10, 19:30:20...)
3. **Window Expansion** - Each 30-second window overlaps with the previous by 20 seconds (like overlapping hop positions)
4. **LEFT JOIN** - Events can appear in multiple windows (an event at 19:30:15 appears in windows starting at 19:30:00 and 19:30:10)
5. **Landing Analysis** - Evaluates landing technique across overlapping windows for continuous monitoring

## Use Case 2: Athletic Training Load Monitoring

**Scenario**: Monitor training intensity using 1-hour windows hopping every 15 minutes (45-minute overlap).

```sql
CREATE OR REPLACE TABLE training_sessions (
    athlete_id VARCHAR(30),
    exercise_type VARCHAR(30),
    rep_count INTEGER,
    intensity_score INTEGER, -- 1-10 scale
    impact_force DECIMAL(6,2),
    session_time TIMESTAMP
);

INSERT INTO training_sessions VALUES
('ATHLETE_001', 'box_hops', 10, 8, 850.50, '2024-08-01 06:15:00'),
('ATHLETE_001', 'box_hops', 12, 9, 920.00, '2024-08-01 06:30:00'),
('ATHLETE_001', 'lateral_hops', 15, 7, 780.25, '2024-08-01 06:45:00'),
('ATHLETE_001', 'depth_jumps', 8, 10, 1100.00, '2024-08-01 07:00:00'),
('ATHLETE_001', 'recovery_hops', 20, 4, 450.00, '2024-08-01 07:15:00'),
('ATHLETE_002', 'box_hops', 15, 8, 880.00, '2024-08-01 06:20:00');

-- HOPPING WINDOW: Progressive overload monitoring
WITH RECURSIVE hop_starts AS (
    -- Anchor: First window start
    SELECT MIN(DATE_TRUNC('HOUR', session_time)) AS hop_start
    FROM training_sessions
    
    UNION ALL
    
    -- Recursive: Add 15-minute hops (like successive training intervals)
    SELECT DATEADD(MINUTE, 15, hop_start)
    FROM hop_starts
    WHERE hop_start < (SELECT MAX(session_time) FROM training_sessions)
),
hopping_analysis AS (
    SELECT 
        hs.hop_start AS window_start,
        DATEADD(HOUR, 1, hs.hop_start) AS window_end,
        t.athlete_id,
        t.exercise_type,
        t.rep_count,
        t.intensity_score,
        t.impact_force,
        t.session_time
    FROM hop_starts hs
    LEFT JOIN training_sessions t
        ON t.session_time >= hs.hop_start 
        AND t.session_time < DATEADD(HOUR, 1, hs.hop_start)
)
SELECT 
    window_start,
    window_end,
    athlete_id,
    
    -- Progressive overload metrics (like building hopping ability gradually)
    SUM(rep_count) AS total_reps_in_window,
    AVG(intensity_score) AS avg_intensity,
    MAX(intensity_score) AS peak_intensity,
    SUM(impact_force * rep_count) AS total_volume_load,
    
    -- Training balance (like proper technique in hopping)
    COUNT(DISTINCT exercise_type) AS exercise_variety,
    STDDEV(intensity_score) AS intensity_variation,
    
    -- Recovery assessment (like landing phase importance)
    CASE 
        WHEN AVG(intensity_score) > 8 AND SUM(rep_count) > 40 
        THEN 'OVERREACHING_RISK'
        WHEN AVG(intensity_score) > 6 AND SUM(rep_count) > 30 
        THEN 'OPTIMAL_PROGRESSIVE_OVERLOAD'
        ELSE 'RECOVERY_OR_BASE_BUILDING'
    END AS training_phase,
    
    -- Window overlap indicator (shows hopping nature)
    LAG(SUM(rep_count)) OVER (PARTITION BY athlete_id ORDER BY window_start) AS prev_window_reps,
    SUM(rep_count) - LAG(SUM(rep_count)) OVER (PARTITION BY athlete_id ORDER BY window_start) AS load_change
    
FROM hopping_analysis

WHERE athlete_id IS NOT NULL

GROUP BY window_start, window_end, athlete_id

ORDER BY athlete_id, window_start;
```

**Workflow Explanation**:

1. **Recursive CTE** - Generates hop start times every 15 minutes (like successive hops in training)
2. **1-Hour Window** - Each window looks back 1 hour from hop start (like analyzing recent training history)
3. **Overlap Effect** - Windows overlap by 45 minutes, providing smooth trend detection (like continuous movement monitoring)
4. **Progressive Overload** - Tracks intensity changes across overlapping windows (like gradually increasing hop height)
5. **Load Change Calculation** - Compares current window to previous overlapping window for trend analysis

## Use Case 3: Injury Prevention Monitoring (Landing Technique)

**Scenario**: Real-time landing analysis using 10-second windows hopping every 2 seconds (8-second overlap) - critical for the "one rule" of proper landing.

```sql
CREATE OR REPLACE TABLE landing_analysis (
    sensor_id VARCHAR(30),
    athlete_id VARCHAR(30),
    landing_phase VARCHAR(20), -- 'initial_contact', 'loading', 'stabilization'
    ground_reaction_force DECIMAL(7,2), -- Newtons
    knee_valgus_angle DECIMAL(5,2), -- degrees (inward collapse = injury risk)
    hip_flexion DECIMAL(5,2),
    ankle_inversion DECIMAL(5,2),
    timestamp_ms INTEGER -- milliseconds for precision
);

INSERT INTO landing_analysis VALUES
('SENSOR_01', 'ATHLETE_001', 'initial_contact', 2500.00, 5.0, 45.0, 2.0, 1000),
('SENSOR_01', 'ATHLETE_001', 'loading', 1800.00, 8.0, 55.0, 5.0, 1100),
('SENSOR_01', 'ATHLETE_001', 'stabilization', 1200.00, 4.0, 60.0, 3.0, 1200),
('SENSOR_01', 'ATHLETE_001', 'initial_contact', 2800.00, 12.0, 42.0, 8.0, 3500), -- Risky landing
('SENSOR_01', 'ATHLETE_001', 'loading', 2200.00, 15.0, 50.0, 10.0, 3600),
('SENSOR_02', 'ATHLETE_002', 'initial_contact', 2300.00, 6.0, 48.0, 3.0, 2000);

-- HOPPING WINDOW: Micro-analysis for injury prevention
WITH hop_windows AS (
    SELECT 
        -- Generate 2-second hop intervals (like rapid successive hops)
        FLOOR(timestamp_ms / 2000) * 2000 AS hop_start_ms,
        
        -- 10-second window with 8-second overlap
        FLOOR(timestamp_ms / 2000) * 2000 AS window_start_ms,
        FLOOR(timestamp_ms / 2000) * 2000 + 10000 AS window_end_ms,
        
        sensor_id,
        athlete_id,
        landing_phase,
        ground_reaction_force,
        knee_valgus_angle,
        hip_flexion,
        ankle_inversion
    FROM landing_analysis
),
detailed_analysis AS (
    SELECT 
        window_start_ms,
        window_end_ms,
        sensor_id,
        athlete_id,
        
        -- Landing technique analysis (the critical "one rule")
        AVG(CASE WHEN landing_phase = 'initial_contact' THEN ground_reaction_force END) AS avg_impact_force,
        MAX(ground_reaction_force) AS peak_force,
        
        -- Knee valgus monitoring (silent failure mode detection)
        MAX(knee_valgus_angle) AS max_knee_collapse,
        AVG(knee_valgus_angle) AS avg_knee_stability,
        
        -- Multi-joint coordination (like coordinated muscle contraction)
        STDDEV(hip_flexion) AS hip_control_consistency,
        MAX(ankle_inversion) AS max_ankle_roll,
        
        -- Landing quality score (proper technique assessment)
        CASE 
            WHEN MAX(knee_valgus_angle) > 10 OR MAX(ankle_inversion) > 7 
            THEN 'POOR_TECHNIQUE_ALERT'
            WHEN AVG(CASE WHEN landing_phase = 'initial_contact' THEN ground_reaction_force END) > 2500 
            THEN 'HIGH_IMPACT_WARNING'
            WHEN STDDEV(hip_flexion) > 8 
            THEN 'INSTABILITY_DETECTED'
            ELSE 'PROPER_LANDING_TECHNIQUE'
        END AS landing_assessment
        
    FROM hop_windows
    
    GROUP BY window_start_ms, window_end_ms, sensor_id, athlete_id
)
SELECT 
    window_start_ms / 1000.0 AS window_start_sec,
    window_end_ms / 1000.0 AS window_end_sec,
    athlete_id,
    
    -- Injury risk indicators
    avg_impact_force,
    max_knee_collapse,
    max_ankle_roll,
    landing_assessment,
    
    -- Overlap analysis (hopping window characteristic)
    COUNT(*) OVER (PARTITION BY athlete_id ORDER BY window_start_ms 
                   ROWS BETWEEN 1 PRECEDING AND 1 PRECEDING) AS overlapping_windows_count,
    
    -- Trend detection across overlapping windows
    LAG(landing_assessment) OVER (PARTITION BY athlete_id ORDER BY window_start_ms) AS prev_assessment,
    
    CASE 
        WHEN landing_assessment != LAG(landing_assessment) OVER (PARTITION BY athlete_id ORDER BY window_start_ms)
        THEN 'STATUS_CHANGE_DETECTED'
        ELSE 'STABLE'
    END AS trend_change
    
FROM detailed_analysis

ORDER BY athlete_id, window_start_ms;
```

**Workflow Explanation**:

1. **2-Second Hops** - Windows advance every 2 seconds for real-time monitoring (like rapid landing adjustments)
2. **10-Second Window** - Analyzes 10 seconds of data with 8-second overlap (continuous coverage like overlapping hops)
3. **Knee Valgus Detection** - Monitors inward knee collapse (silent failure mode in hopping)
4. **Status Change Detection** - Uses LAG function to detect changes across overlapping windows
5. **Continuous Assessment** - Each overlapping window provides updated injury risk status

## Use Case 4: E-commerce Clickstream Analysis

**Scenario**: User behavior analysis using 5-minute windows hopping every 1 minute (4-minute overlap).

```sql
CREATE OR REPLACE TABLE clickstream_events (
    user_id VARCHAR(30),
    session_id VARCHAR(40),
    event_type VARCHAR(20), -- 'page_view', 'hop_to_product', 'add_cart', 'purchase'
    product_category VARCHAR(30),
    time_on_page_seconds INTEGER,
    event_timestamp TIMESTAMP
);

INSERT INTO clickstream_events VALUES
('USER_001', 'SESSION_A', 'page_view', 'sports_shoes', 45, '2024-08-01 14:05:10'),
('USER_001', 'SESSION_A', 'hop_to_product', 'sports_shoes', 30, '2024-08-01 14:05:55'),
('USER_001', 'SESSION_A', 'add_cart', 'sports_shoes', 0, '2024-08-01 14:06:30'),
('USER_002', 'SESSION_B', 'page_view', 'basketball_gear', 120, '2024-08-01 14:06:15'),
('USER_002', 'SESSION_B', 'hop_to_product', 'basketball_gear', 45, '2024-08-01 14:08:20'),
('USER_001', 'SESSION_A', 'purchase', 'sports_shoes', 0, '2024-08-01 14:09:45');

-- HOPPING WINDOW: User journey analysis with overlap
WITH minute_sequence AS (
    -- Generate 1-minute hop intervals
    SELECT 
        DATEADD(MINUTE, seq, DATE_TRUNC('HOUR', MIN(event_timestamp))) AS hop_start
    FROM clickstream_events,
    TABLE(GENERATOR(ROWCOUNT => 15)) AS gen(seq)
    WHERE DATEADD(MINUTE, seq, DATE_TRUNC('HOUR', MIN(event_timestamp))) <= 
          (SELECT MAX(event_timestamp) FROM clickstream_events)
),
hopping_journey AS (
    SELECT 
        m.hop_start AS window_start,
        DATEADD(MINUTE, 5, m.hop_start) AS window_end,
        c.user_id,
        c.session_id,
        c.event_type,
        c.product_category,
        c.time_on_page_seconds,
        c.event_timestamp
    FROM minute_sequence m
    LEFT JOIN clickstream_events c
        ON c.event_timestamp >= m.hop_start 
        AND c.event_timestamp < DATEADD(MINUTE, 5, m.hop_start)
)
SELECT 
    window_start,
    window_end,
    
    -- User behavior metrics (like explosive propulsion in hopping)
    COUNT(DISTINCT user_id) AS active_users,
    COUNT(DISTINCT session_id) AS active_sessions,
    
    -- Conversion funnel (takeoff, airborne, landing phases)
    COUNT(CASE WHEN event_type = 'page_view' THEN 1 END) AS page_views,
    COUNT(CASE WHEN event_type = 'hop_to_product' THEN 1 END) AS product_hops,
    COUNT(CASE WHEN event_type = 'add_cart' THEN 1 END) AS cart_adds,
    COUNT(CASE WHEN event_type = 'purchase' THEN 1 END) AS purchases,
    
    -- Conversion rates (like hopping efficiency)
    CASE 
        WHEN COUNT(CASE WHEN event_type = 'page_view' THEN 1 END) > 0 
        THEN COUNT(CASE WHEN event_type = 'purchase' THEN 1 END) * 100.0 / 
             COUNT(CASE WHEN event_type = 'page_view' THEN 1 END)
        ELSE 0 
    END AS view_to_purchase_rate,
    
    -- Engagement depth (like core stability in hopping)
    AVG(time_on_page_seconds) AS avg_engagement_time,
    PERCENTILE_CONT(0.9) WITHIN GROUP (ORDER BY time_on_page_seconds) AS p90_engagement,
    
    -- Category performance (like different hopping techniques)
    MODE(product_category) AS top_category,
    
    -- Hopping window overlap benefit: Smooth trend detection
    AVG(COUNT(CASE WHEN event_type = 'purchase' THEN 1 END)) OVER (
        ORDER BY window_start 
        ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
    ) AS purchases_3_window_avg
    
FROM hopping_journey

WHERE user_id IS NOT NULL

GROUP BY window_start, window_end

ORDER BY window_start;
```

**Workflow Explanation**:

1. **1-Minute Hops** - Windows advance every minute (like continuous small hops through user journey)
2. **5-Minute Window** - Each window captures 5 minutes of behavior with 4-minute overlap
3. **Smooth Trends** - Moving average across overlapping windows shows true trends without spikes
4. **Funnel Analysis** - Tracks user progression through phases (like hop preparation, explosion, landing)
5. **Overlap Benefit** - Events appear in multiple windows, providing context continuity

## Use Case 5: Network Latency Monitoring

**Scenario**: Monitor API response times using 1-minute windows hopping every 10 seconds (50-second overlap).

```sql
CREATE OR REPLACE TABLE api_requests (
    request_id VARCHAR(40),
    endpoint VARCHAR(50),
    response_time_ms INTEGER,
    status_code INTEGER,
    server_region VARCHAR(20),
    request_time TIMESTAMP
);

INSERT INTO api_requests VALUES
('REQ_001', '/api/hop-data', 45, 200, 'us-east-1', '2024-08-01 10:15:05'),
('REQ_002', '/api/landing-stats', 120, 200, 'us-east-1', '2024-08-01 10:15:12'),
('REQ_003', '/api/hop-data', 2000, 503, 'us-east-1', '2024-08-01 10:15:18'), -- Spike
('REQ_004', '/api/propulsion-metrics', 55, 200, 'us-east-1', '2024-08-01 10:15:25'),
('REQ_005', '/api/hop-data', 60, 200, 'us-east-1', '2024-08-01 10:15:35'),
('REQ_006', '/api/landing-stats', 150, 200, 'us-west-2', '2024-08-01 10:15:42'),
('REQ_007', '/api/hop-data', 48, 200, 'us-east-1', '2024-08-01 10:15:55');

-- HOPPING WINDOW: Latency analysis with high overlap for anomaly detection
WITH second_hops AS (
    SELECT 
        DATEADD(SECOND, seq * 10, DATE_TRUNC('MINUTE', MIN(request_time))) AS hop_start
    FROM api_requests,
    TABLE(GENERATOR(ROWCOUNT => 12)) AS gen(seq)
    WHERE DATEADD(SECOND, seq * 10, DATE_TRUNC('MINUTE', MIN(request_time))) <= 
          (SELECT MAX(request_time) FROM api_requests)
),
latency_windows AS (
    SELECT 
        s.hop_start AS window_start,
        DATEADD(MINUTE, 1, s.hop_start) AS window_end,
        a.request_id,
        a.endpoint,
        a.response_time_ms,
        a.status_code,
        a.server_region,
        a.request_time
    FROM second_hops s
    LEFT JOIN api_requests a
        ON a.request_time >= s.hop_start 
        AND a.request_time < DATEADD(MINUTE, 1, s.hop_start)
)
SELECT 
    window_start,
    window_end,
    server_region,
    
    -- Latency metrics (like measuring hop height consistency)
    COUNT(*) AS total_requests,
    AVG(response_time_ms) AS avg_latency,
    PERCENTILE_CONT(0.95) WITHIN GROUP (ORDER BY response_time_ms) AS p95_latency,
    PERCENTILE_CONT(0.99) WITHIN GROUP (ORDER BY response_time_ms) AS p99_latency,
    MAX(response_time_ms) AS max_latency,
    
    -- Error analysis (like poor landing detection)
    COUNT(CASE WHEN status_code >= 500 THEN 1 END) AS server_errors,
    COUNT(CASE WHEN status_code >= 400 THEN 1 END) AS client_errors,
    
    -- Stability metrics (like core stability in hopping)
    STDDEV(response_time_ms) AS latency_jitter,
    
    -- Performance grade (like hopping technique rating)
    CASE 
        WHEN AVG(response_time_ms) < 100 AND MAX(response_time_ms) < 500 
        THEN 'EXCELLENT_STABILITY'
        WHEN AVG(response_time_ms) < 200 AND MAX(response_time_ms) < 1000 
        THEN 'GOOD_CONTROL'
        WHEN MAX(response_time_ms) > 2000 OR COUNT(CASE WHEN status_code >= 500 THEN 1 END) > 0 
        THEN 'POOR_TECHNIQUE_CRITICAL'
        ELSE 'NEEDS_OPTIMIZATION'
    END AS performance_grade,
    
    -- Hopping window advantage: Early anomaly detection
    AVG(response_time_ms) OVER (
        PARTITION BY server_region 
        ORDER BY window_start 
        ROWS BETWEEN 2 PRECEDING AND 1 PRECEDING
    ) AS prev_avg_latency,
    
    AVG(response_time_ms) - AVG(response_time_ms) OVER (
        PARTITION BY server_region 
        ORDER BY window_start 
        ROWS BETWEEN 2 PRECEDING AND 1 PRECEDING
    ) AS latency_trend_change
    
FROM latency_windows

WHERE request_id IS NOT NULL

GROUP BY window_start, window_end, server_region

ORDER BY server_region, window_start;
```

**Workflow Explanation**:

1. **10-Second Hops** - High-frequency hopping for real-time monitoring (like rapid adjustments in hopping)
2. **1-Minute Window** - 50-second overlap provides continuous coverage (like maintaining balance between hops)
3. **Trend Change Calculation** - Compares current window to previous overlapping windows for anomaly detection
4. **Jitter Analysis** - Standard deviation measures stability (like consistent hopping rhythm)
5. **Early Warning** - Overlapping windows detect issues faster than tumbling windows

## Use Case 6: Financial Trading Volume Analysis

**Scenario**: Analyze trading patterns using 15-minute windows hopping every 5 minutes (10-minute overlap).

```sql
CREATE OR REPLACE TABLE stock_trades (
    trade_id VARCHAR(30),
    symbol VARCHAR(10),
    trade_type VARCHAR(10), -- 'buy', 'sell'
    volume INTEGER,
    price DECIMAL(10,2),
    trade_time TIMESTAMP
);

INSERT INTO stock_trades VALUES
('T_001', 'HOP', 'buy', 1000, 45.50, '2024-08-01 09:30:15'),
('T_002', 'HOP', 'buy', 1500, 45.75, '2024-08-01 09:35:20'),
('T_003', 'HOP', 'sell', 800, 46.00, '2024-08-01 09:40:10'),
('T_004', 'HOP', 'buy', 2000, 46.25, '2024-08-01 09:45:30'),
('T_005', 'HOP', 'sell', 1200, 46.50, '2024-08-01 09:50:00'),
('T_006', 'HOP', 'buy', 3000, 47.00, '2024-08-01 10:00:00'),
('T_007', 'JUMP', 'buy', 500, 120.00, '2024-08-01 09:32:00');

-- HOPPING WINDOW: Trading momentum analysis
WITH five_min_hops AS (
    SELECT 
        DATEADD(MINUTE, seq * 5, DATE_TRUNC('HOUR', MIN(trade_time))) AS hop_start
    FROM stock_trades,
    TABLE(GENERATOR(ROWCOUNT => 12)) AS gen(seq)
    WHERE DATEADD(MINUTE, seq * 5, DATE_TRUNC('HOUR', MIN(trade_time))) <= 
          (SELECT MAX(trade_time) FROM stock_trades)
),
trading_windows AS (
    SELECT 
        h.hop_start AS window_start,
        DATEADD(MINUTE, 15, h.hop_start) AS window_end,
        t.symbol,
        t.trade_type,
        t.volume,
        t.price,
        t.trade_time
    FROM five_min_hops h
    LEFT JOIN stock_trades t
        ON t.trade_time >= h.hop_start 
        AND t.trade_time < DATEADD(MINUTE, 15, h.hop_start)
)
SELECT 
    window_start,
    window_end,
    symbol,
    
    -- Volume analysis (like explosive propulsion force)
    SUM(volume) AS total_volume,
    SUM(CASE WHEN trade_type = 'buy' THEN volume ELSE 0 END) AS buy_volume,
    SUM(CASE WHEN trade_type = 'sell' THEN volume ELSE 0 END) AS sell_volume,
    
    -- Price momentum (like hopping height progression)
    MIN(price) AS low_price,
    MAX(price) AS high_price,
    AVG(price) AS vwap, -- Volume-weighted average price
    MAX(price) - MIN(price) AS price_range,
    
    -- Market pressure (like landing force absorption)
    CASE 
        WHEN SUM(CASE WHEN trade_type = 'buy' THEN volume ELSE 0 END) > 
             SUM(CASE WHEN trade_type = 'sell' THEN volume ELSE 0 END) * 1.5 
        THEN 'STRONG_BUYING_PRESSURE'
        WHEN SUM(CASE WHEN trade_type = 'sell' THEN volume ELSE 0 END) > 
             SUM(CASE WHEN trade_type = 'buy' THEN volume ELSE 0 END) * 1.5 
        THEN 'STRONG_SELLING_PRESSURE'
        ELSE 'BALANCED'
    END AS market_pressure,
    
    -- Hopping window advantage: Momentum continuity
    AVG(price) OVER (
        PARTITION BY symbol 
        ORDER BY window_start 
        ROWS BETWEEN 1 PRECEDING AND 1 PRECEDING
    ) AS prev_window_avg_price,
    
    (AVG(price) - AVG(price) OVER (
        PARTITION BY symbol 
        ORDER BY window_start 
        ROWS BETWEEN 1 PRECEDING AND 1 PRECEDING
    )) / AVG(price) OVER (
        PARTITION BY symbol 
        ORDER BY window_start 
        ROWS BETWEEN 1 PRECEDING AND 1 PRECEDING
    ) * 100 AS price_momentum_percent,
    
    -- Overlap coverage indicator
    COUNT(*) OVER (PARTITION BY symbol ORDER BY window_start) AS cumulative_windows
    
FROM trading_windows

WHERE symbol IS NOT NULL

GROUP BY window_start, window_end, symbol

ORDER BY symbol, window_start;
```

**Workflow Explanation**:

1. **5-Minute Hops** - Trading windows advance every 5 minutes (like strategic position adjustments)
2. **15-Minute Window** - 10-minute overlap ensures no trading activity is missed between windows
3. **VWAP Calculation** - Volume-weighted average price across overlapping windows (like balanced force distribution)
4. **Momentum Tracking** - Compares current window to previous overlapping window for trend detection
5. **Pressure Analysis** - Buy vs sell volume ratio indicates market direction (like preparing for next hop)

## Use Case 7: Smart Home Energy Consumption

**Scenario**: Monitor energy usage using 1-hour windows hopping every 15 minutes (45-minute overlap).

```sql
CREATE OR REPLACE TABLE energy_readings (
    device_id VARCHAR(30),
    room VARCHAR(20),
    appliance_type VARCHAR(30),
    power_consumption_watts DECIMAL(8,2),
    voltage DECIMAL(5,2),
    reading_time TIMESTAMP
);

INSERT INTO energy_readings VALUES
('DEV_001', 'kitchen', 'refrigerator', 150.00, 120.0, '2024-08-01 08:05:00'),
('DEV_002', 'living_room', 'ac_unit', 3500.00, 120.0, '2024-08-01 08:15:30'),
('DEV_003', 'kitchen', 'microwave', 1200.00, 120.0, '2024-08-01 08:25:00'),
('DEV_002', 'living_room', 'ac_unit', 3400.00, 119.5, '2024-08-01 08:35:00'),
('DEV_004', 'bedroom', 'space_heater', 2000.00, 118.0, '2024-08-01 08:45:00'),
('DEV_001', 'kitchen', 'refrigerator', 155.00, 120.0, '2024-08-01 09:05:00'),
('DEV_002', 'living_room', 'ac_unit', 3600.00, 119.0, '2024-08-01 09:15:00');

-- HOPPING WINDOW: Energy consumption pattern analysis
WITH quarter_hour_hops AS (
    SELECT 
        DATEADD(MINUTE, seq * 15, DATE_TRUNC('HOUR', MIN(reading_time))) AS hop_start
    FROM energy_readings,
    TABLE(GENERATOR(ROWCOUNT => 20)) AS gen(seq)
    WHERE DATEADD(MINUTE, seq * 15, DATE_TRUNC('HOUR', MIN(reading_time))) <= 
          (SELECT MAX(reading_time) FROM energy_readings)
),
energy_hopping AS (
    SELECT 
        h.hop_start AS window_start,
        DATEADD(HOUR, 1, h.hop_start) AS window_end,
        e.device_id,
        e.room,
        e.appliance_type,
        e.power_consumption_watts,
        e.voltage,
        e.reading_time
    FROM quarter_hour_hops h
    LEFT JOIN energy_readings e
        ON e.reading_time >= h.hop_start 
        AND e.reading_time < DATEADD(HOUR, 1, h.hop_start)
)
SELECT 
    window_start,
    window_end,
    
    -- Consumption metrics (like measuring hop energy expenditure)
    SUM(power_consumption_watts) / 1000 AS total_kwh_consumed,
    AVG(power_consumption_watts) AS avg_power_demand,
    MAX(power_consumption_watts) AS peak_demand,
    
    -- Appliance breakdown (like different muscle groups in hopping)
    SUM(CASE WHEN appliance_type = 'ac_unit' THEN power_consumption_watts ELSE 0 END) / 1000 AS hvac_consumption,
    SUM(CASE WHEN appliance_type = 'refrigerator' THEN power_consumption_watts ELSE 0 END) / 1000 AS refrigeration_consumption,
    SUM(CASE WHEN appliance_type IN ('microwave', 'space_heater') THEN power_consumption_watts ELSE 0 END) / 1000 AS high_demand_consumption,
    
    -- Grid stability (like core stability in hopping)
    AVG(voltage) AS avg_voltage,
    MIN(voltage) AS min_voltage,
    CASE 
        WHEN MIN(voltage) < 115 THEN 'VOLTAGE_SAG_WARNING'
        WHEN MAX(power_consumption_watts) > 4000 THEN 'PEAK_DEMAND_ALERT'
        ELSE 'STABLE_CONSUMPTION'
    END AS grid_status,
    
    -- Efficiency trend (like improving hopping technique over time)
    AVG(SUM(power_consumption_watts)) OVER (
        ORDER BY window_start 
        ROWS BETWEEN 3 PRECEDING AND CURRENT ROW
    ) / 1000 AS rolling_4_window_avg_kwh,
    
    -- Overlap analysis: How much data is shared between consecutive windows
    LAG(SUM(power_consumption_watts)) OVER (ORDER BY window_start) AS prev_window_total,
    
    (SUM(power_consumption_watts) - LAG(SUM(power_consumption_watts)) OVER (ORDER BY window_start)) / 
    NULLIF(LAG(SUM(power_consumption_watts)) OVER (ORDER BY window_start), 0) * 100 AS consumption_change_percent
    
FROM energy_hopping

WHERE device_id IS NOT NULL

GROUP BY window_start, window_end

ORDER BY window_start;
```

**Workflow Explanation**:

1. **15-Minute Hops** - Windows advance every 15 minutes (like periodic energy assessments)
2. **1-Hour Window** - 45-minute overlap provides smooth consumption curves (like maintaining rhythm in hopping)
3. **Rolling Average** - 4-window moving average across overlapping windows shows true consumption trends
4. **Change Detection** - Compares current window to previous for usage pattern changes
5. **Grid Stability** - Voltage monitoring acts like balance maintenance in hopping

## Key Hopping Window Concepts Summary

| Concept | Basketball/Hopping Analogy | SQL Implementation |
|---------|---------------------------|-------------------|
| **Window Size** | Total hop duration (air + ground time) | `DATEADD` to define window end |
| **Hop Size** | Distance between successive hops | Interval between window starts |
| **Overlap** | Time when both feet are on ground between hops | `Window Size - Hop Size` |
| **Smooth Trends** | Continuous movement without jerky stops | Events appear in multiple windows |
| **Early Detection** | Quick adjustment during landing phase | Anomalies caught in overlapping windows |

**Key Differences: Tumbling vs Hopping Windows**

| Feature | Tumbling Windows | Hopping Windows |
|---------|---------------|-----------------|
| **Overlap** | None (0%) | Configurable (e.g., 75%) |
| **Use Case** | Discrete events, complete cycles | Continuous monitoring, trend detection |
| **SQL Complexity** | Simple `GROUP BY` | Requires window generation logic |
| **Event Duplication** | Events in exactly 1 window | Events in multiple windows |
| **Analogy** | Gymnastics passes (distinct) | Basketball hops (overlapping) |

**Remember**: Hopping windows in SQL are like a basketball player's defensive shuffles—each position overlaps with the previous, providing continuous coverage and smooth transitions. The overlap ensures you never miss critical data between windows, just like a player never leaves a gap in defense!

## Key Points
- The ankle joint plays a critical role in hopping, as it is responsible for generating the majority of the propulsive force.
- The hip and knee joints also contribute to hopping mechanics, as they help to regulate movement and maintain balance.
- Core strength and stability are essential for maintaining control and generating force during hopping movements.
- The landing phase of hopping is just as important as the takeoff phase, as it requires careful control to absorb force and prepare for the next hop.

> [!important]
> The one rule that must not be violated when hopping is to maintain proper landing technique, as poor landing mechanics can lead to serious injury.

## Mental Model
- Step 1: Preparation - The individual prepares to hop by assuming a squatting position and loading the muscles of the lower extremities.
- Step 2: Explosion - The individual rapidly extends the ankles, knees, and hips to generate propulsive force and leave the ground.
- Step 3: Landing - The individual lands on both feet, absorbing the force of the landing by bending the knees and hips, and preparing to repeat the process.

## Observe
- A common mistake people make when hopping is to overlook the importance of proper landing technique, which can lead to injuries such as ankle sprains or knee strains.
- A silent failure mode of hopping is the gradual development of poor movement patterns, which can lead to chronic injuries or decreased athletic performance over time.
- A misleading assumption about hopping is that it is a simple movement pattern that requires minimal technique or training, when in fact it requires careful practice and attention to detail to master.
