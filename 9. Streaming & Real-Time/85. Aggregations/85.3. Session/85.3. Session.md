# 85.3. Session

> [!Note]
> Effective session management is crucial for maintaining user data integrity and security, ensuring a seamless user experience across interactions with a system or application.

## What is a Session Window in Pharma Data?

In stream processing, **Session Windows** dynamically group events into sessions based on activity. A session **starts** with an event and **ends** after a specified **idle timeout** (gap of inactivity)—just like a user login session that expires after 30 minutes of inactivity. Perfect for analyzing patient journeys, prescription fill patterns, or EDI transaction flows.

## Use Case 1: Patient Prescription Fill Journey (Retail Sales)

**Scenario**: Track patient medication adherence using session windows with 90-day inactivity timeout (standard refill window).

```sql
-- Create retail pharmacy sales data (NDC, Rx, patient, date)
CREATE OR REPLACE TABLE retail_sales (
    transaction_id VARCHAR(30),
    patient_id VARCHAR(30),
    ndc_code VARCHAR(11), -- National Drug Code
    drug_name VARCHAR(100),
    quantity_dispensed INTEGER,
    days_supply INTEGER,
    pharmacy_npi VARCHAR(10),
    prescriber_npi VARCHAR(10),
    copay_amount DECIMAL(8,2),
    total_paid DECIMAL(10,2),
    sale_date DATE
);

-- Insert sample data: Patient with chronic condition (diabetes)
INSERT INTO retail_sales VALUES
('RX_001', 'PT_10001', '00093-2748-01', 'Metformin 500mg', 60, 30, 'PHARM001', 'DR_001', 10.00, 45.00, '2024-01-15'),
('RX_002', 'PT_10001', '00093-2748-01', 'Metformin 500mg', 60, 30, 'PHARM001', 'DR_001', 10.00, 45.00, '2024-02-14'), -- 30 days later (adherent)
('RX_003', 'PT_10001', '00093-2748-01', 'Metformin 500mg', 60, 30, 'PHARM001', 'DR_001', 10.00, 45.00, '2024-03-16'), -- 30 days later (adherent)
('RX_004', 'PT_10001', '00093-2748-01', 'Metformin 500mg', 60, 30, 'PHARM001', 'DR_001', 10.00, 45.00, '2024-07-20'), -- 4+ months gap (new session - gap)
('RX_005', 'PT_10002', '00555-0808-02', 'Atorvastatin 20mg', 30, 30, 'PHARM002', 'DR_002', 5.00, 150.00, '2024-01-10'),
('RX_006', 'PT_10002', '00555-0808-02', 'Atorvastatin 20mg', 30, 30, 'PHARM002', 'DR_002', 5.00, 150.00, '2024-02-05'); -- 26 days (same session)

-- SESSION WINDOW QUERY: Patient medication adherence sessions
WITH patient_events AS (
    -- Sort patient events chronologically (like session start sequencing)
    SELECT 
        patient_id,
        ndc_code,
        drug_name,
        transaction_id,
        sale_date,
        days_supply,
        quantity_dispensed,
        total_paid,
        -- Calculate days since previous fill (session inactivity detector)
        LAG(sale_date) OVER (PARTITION BY patient_id, ndc_code ORDER BY sale_date) AS prev_fill_date,
        DATEDIFF(DAY, LAG(sale_date) OVER (PARTITION BY patient_id, ndc_code ORDER BY sale_date), sale_date) AS days_since_prev_fill
    FROM retail_sales
),
session_detection AS (
    SELECT 
        *,
        -- Define session boundary: 90-day inactivity timeout (pharmacy standard)
        CASE 
            WHEN days_since_prev_fill IS NULL THEN 1 -- First fill = new session
            WHEN days_since_prev_fill > 90 THEN 1 -- Gap > 90 days = new session (like session timeout)
            ELSE 0 -- Same session (active medication period)
        END AS is_new_session
    FROM patient_events
),
session_ids AS (
    SELECT 
        *,
        -- Generate session ID using cumulative sum (like session ID generation)
        SUM(is_new_session) OVER (PARTITION BY patient_id, ndc_code ORDER BY sale_date) AS session_number,
        CONCAT(patient_id, '_', ndc_code, '_', 
            SUM(is_new_session) OVER (PARTITION BY patient_id, ndc_code ORDER BY sale_date)) AS session_id
    FROM session_detection
)
SELECT 
    session_id,
    patient_id,
    ndc_code,
    drug_name,
    
    -- Session metadata (like session start/end timestamps)
    MIN(sale_date) AS session_start_date,
    MAX(sale_date) AS session_end_date,
    DATEDIFF(DAY, MIN(sale_date), MAX(sale_date)) AS session_duration_days,
    
    -- Medication adherence metrics (session activity analysis)
    COUNT(*) AS fills_in_session,
    SUM(days_supply) AS total_days_covered,
    AVG(days_since_prev_fill) AS avg_gap_between_fills,
    MAX(days_since_prev_fill) AS max_gap_between_fills,
    
    -- Financial metrics (session value)
    SUM(total_paid) AS total_session_cost,
    AVG(total_paid) AS avg_fill_cost,
    
    -- Adherence quality score (like session health check)
    CASE 
        WHEN COUNT(*) >= 3 AND AVG(days_since_prev_fill) <= 35 THEN 'HIGH_ADHERENCE'
        WHEN COUNT(*) >= 2 AND AVG(days_since_prev_fill) <= 45 THEN 'MODERATE_ADHERENCE'
        WHEN MAX(days_since_prev_fill) > 90 THEN 'GAP_IN_THERAPY'
        ELSE 'LOW_ADHERENCE'
    END AS adherence_classification,
    
    -- Session termination reason (like session logout reason)
    CASE 
        WHEN DATEDIFF(DAY, MAX(sale_date), CURRENT_DATE()) > 90 THEN 'EXPIRED_NO_RECENT_FILL'
        WHEN AVG(days_since_prev_fill) > 45 THEN 'IRREGULAR_FILL_PATTERN'
        ELSE 'ACTIVE_OR_RECENTLY_CLOSED'
    END AS session_status
    
FROM session_ids

GROUP BY session_id, patient_id, ndc_code, drug_name

ORDER BY patient_id, ndc_code, session_start_date;
```

**Workflow Explanation**:

1. **Event Sequencing** - `LAG()` calculates days between fills (detecting session inactivity like idle timeout)
2. **Session Boundary Detection** - 90-day gap triggers new session (standard pharmacy refill window)
3. **Cumulative Sum** - Generates unique session IDs by counting session starts (like incrementing session counters)
4. **Session Metrics** - Analyzes fills within each therapy session (like tracking user actions in a web session)
5. **Adherence Classification** - Evaluates session quality based on fill patterns (like session health monitoring)

**Scenario**: Track EDI 837 claim submission sessions with 24-hour inactivity timeout (batch processing windows).

```sql

('EDI_003', 'PHARM001', 'PAYER_A', 'CLM_1003', 'PT_5003', 'P', 245.00, 'original', '2024-03-15', '2024-03-15 09:45:00', 5),
('EDI_004', 'PHARM001', 'PAYER_A', 'CLM_1004', 'PT_5004', 'P', 67.00, 'original', '2024-03-16', '2024-03-16 10:30:00', 1), -- Next day (new session)
('EDI_005', 'PHARM001', 'PAYER_A', 'CLM_1005', 'PT_5005', 'P', 156.00, 'original', '2024-03-16', '2024-03-16 10:35:00', 3),
('EDI_006', 'PHARM002', 'PAYER_B', 'CLM_2001', 'PT_6001', 'P', 340.00, 'original', '2024-03-15', '2024-03-15 14:20:00', 4);

-- SESSION WINDOW: EDI submission batch analysis
WITH claim_events AS (
    SELECT 
        submitter_id,
        receiver_id,
        transaction_id,
        claim_id,
        patient_id,
        total_charge_amount,
        line_item_count,
        submission_timestamp,
        -- Calculate time since previous submission (session idle detection)
        LAG(submission_timestamp) OVER (PARTITION BY submitter_id, receiver_id ORDER BY submission_timestamp) AS prev_submission_time,
        DATEDIFF(MINUTE, LAG(submission_timestamp) OVER (PARTITION BY submitter_id, receiver_id ORDER BY submission_timestamp), submission_timestamp) AS minutes_since_prev
    FROM edi_837_claims
),
session_boundaries AS (
    SELECT 
        *,
        -- 60-minute inactivity = new session (batch processing timeout)
        CASE 
            WHEN minutes_since_prev IS NULL THEN 1 -- First claim = new session
            WHEN minutes_since_prev > 60 THEN 1 -- Gap > 60 min = new batch session
            ELSE 0 -- Same submission session
        END AS is_new_session_flag
    FROM claim_events
),
session_grouping AS (
    SELECT 
        *,
        -- Generate session ID (like session cookie generation)
        CONCAT(submitter_id, '_', receiver_id, '_', DATE(submission_timestamp), '_', 
            SUM(is_new_session_flag) OVER (PARTITION BY submitter_id, receiver_id ORDER BY submission_timestamp)) AS batch_session_id,
        SUM(is_new_session_flag) OVER (PARTITION BY submitter_id, receiver_id ORDER BY submission_timestamp) AS session_sequence
    FROM session_boundaries
)
SELECT 
    batch_session_id,
    submitter_id,
    receiver_id,
    
    -- Session timing (like session start/end)
    MIN(submission_timestamp) AS session_start_time,
    MAX(submission_timestamp) AS session_end_time,
    DATEDIFF(MINUTE, MIN(submission_timestamp), MAX(submission_timestamp)) AS session_duration_minutes,
    
    -- Claim batch metrics (session activity)
    COUNT(*) AS claims_in_batch,
    SUM(total_charge_amount) AS total_batch_value,
    AVG(total_charge_amount) AS avg_claim_value,
    SUM(line_item_count) AS total_line_items,
    
    -- Processing efficiency (like session performance)
    COUNT(*) / NULLIF(DATEDIFF(MINUTE, MIN(submission_timestamp), MAX(submission_timestamp)), 0) AS claims_per_minute,
    
    -- Batch characteristics (like session classification)
    CASE 
        WHEN COUNT(*) >= 10 THEN 'HIGH_VOLUME_BATCH'
        WHEN COUNT(*) >= 5 THEN 'MEDIUM_VOLUME_BATCH'
        ELSE 'LOW_VOLUME_BATCH'
    END AS batch_size_category,
    
    -- Revenue concentration (like session value analysis)
    SUM(total_charge_amount) / SUM(SUM(total_charge_amount)) OVER (PARTITION BY submitter_id, receiver_id) * 100 AS pct_of_total_revenue,
    
    -- Session quality indicators (like security audit)
    CASE 
        WHEN AVG(minutes_since_prev) < 5 THEN 'RAPID_FIRE_SUBMISSION'
        WHEN MAX(minutes_since_prev) > 30 THEN 'INTERRUPTED_BATCH'
        ELSE 'NORMAL_PROCESSING_PATTERN'
    END AS submission_pattern
    
FROM session_grouping

GROUP BY batch_session_id, submitter_id, receiver_id

ORDER BY submitter_id, receiver_id, session_start_time;
```

**Workflow Explanation**:

1. **Timestamp Sequencing** - `LAG()` captures previous submission time (like tracking last user activity)
2. **Idle Detection** - 60-minute gap defines new batch session (like web session timeout)
3. **Session ID Generation** - Combines submitter, receiver, date, and sequence (like secure session cookie)
4. **Batch Metrics** - Analyzes claims per session (like page views per user session)
5. **Pattern Detection** - Identifies rapid submissions or interrupted batches (like detecting bot behavior)

**Scenario**: Track LAD authorization workflows with 30-day inactivity timeout (prior authorization lifecycle).

```sql

('AUTH_004', 'PT_3002', '61755-001-01', 'Revlimid (Lenalidomide)', 'DR_102', 'PAYER_COMM', 'approved', '2024-02-01', '2024-08-01', 21, 11, '2024-01-28', '2024-02-01', 4),

('AUTH_006', 'PT_3002', '61755-001-01', 'Revlimid (Lenalidomide)', 'DR_102', 'PAYER_COMM', 'approved', '2024-04-01', '2024-10-01', 21, 11, '2024-03-28', '2024-04-01', 4); -- Continuous therapy

-- SESSION WINDOW: LAD therapy authorization journey
WITH auth_events AS (
    SELECT 
        patient_id,
        ndc_code,
        drug_name,
        payer_id,
        auth_id,
        submission_date,
        decision_date,
        auth_status,
        auth_start_date,
        auth_end_date,
        days_to_decision,
        -- Calculate gap between authorizations (session timeout detection)
        LAG(auth_end_date) OVER (PARTITION BY patient_id, ndc_code ORDER BY auth_start_date) AS prev_auth_end,
        DATEDIFF(DAY, LAG(auth_end_date) OVER (PARTITION BY patient_id, ndc_code ORDER BY auth_start_date), auth_start_date) AS gap_from_prev_auth
    FROM lad_authorizations
),
therapy_sessions AS (
    SELECT 
        *,
        -- 30-day gap = new therapy session (LAD standard renewal window)
        CASE 
            WHEN gap_from_prev_auth IS NULL THEN 1 -- First auth = new session

            ELSE 0 -- Same therapy session (continuous treatment)
        END AS new_session_indicator
    FROM auth_events
),
session_mapping AS (
    SELECT 
        *,
        -- Generate therapy session ID (like secure session management)
        CONCAT(patient_id, '_', ndc_code, '_', 
            SUM(new_session_indicator) OVER (PARTITION BY patient_id, ndc_code ORDER BY auth_start_date)) AS therapy_session_id,
        SUM(new_session_indicator) OVER (PARTITION BY patient_id, ndc_code ORDER BY auth_start_date) AS course_number
    FROM therapy_sessions
)
SELECT 
    therapy_session_id,
    patient_id,
    ndc_code,
    drug_name,
    payer_id,
    
    -- Session lifecycle (like user session duration)
    MIN(submission_date) AS session_start_date,
    MAX(auth_end_date) AS session_end_date,
    DATEDIFF(DAY, MIN(submission_date), MAX(auth_end_date)) AS total_therapy_days,
    
    -- Authorization metrics (session activity)
    COUNT(*) AS authorizations_in_session,
    SUM(CASE WHEN auth_status = 'approved' THEN 1 ELSE 0 END) AS approved_auths,
    SUM(CASE WHEN auth_status = 'denied' THEN 1 ELSE 0 END) AS denied_auths,
    
    -- Efficiency metrics (like session performance)
    AVG(days_to_decision) AS avg_decision_time,
    MAX(days_to_decision) AS max_decision_time,
    
    -- Continuity metrics (like session stability)
    AVG(DATEDIFF(DAY, auth_start_date, auth_end_date)) AS avg_auth_duration_days,
    SUM(DATEDIFF(DAY, auth_start_date, auth_end_date)) AS total_authorized_days,
    
    -- Gap analysis (like session interruption detection)
    AVG(gap_from_prev_auth) AS avg_gap_between_auths,
    MAX(gap_from_prev_auth) AS max_gap_between_auths,
    
    -- Therapy outcome classification (like session success rate)
    CASE 
        WHEN COUNT(*) >= 3 AND SUM(CASE WHEN auth_status = 'denied' THEN 1 ELSE 0 END) = 0 
        THEN 'SUCCESSFUL_LONG_TERM_THERAPY'
        WHEN SUM(CASE WHEN auth_status = 'denied' THEN 1 ELSE 0 END) > 0 
        THEN 'THERAPY_INTERRUPTED'
        WHEN COUNT(*) = 1 AND MAX(gap_from_prev_auth) > 90 
        THEN 'SINGLE_COURSE_TREATMENT'
        ELSE 'ONGOING_THERAPY'
    END AS therapy_outcome,
    
    -- Security/Compliance check (like session security audit)
    CASE 
        WHEN AVG(days_to_decision) > 10 THEN 'SLOW_AUTH_RISK'
        WHEN MAX(gap_from_prev_auth) > 60 THEN 'THERAPY_GAP_RISK'
        ELSE 'COMPLIANT'
    END AS risk_flag
    
FROM session_mapping

GROUP BY therapy_session_id, patient_id, ndc_code, drug_name, payer_id

ORDER BY patient_id, ndc_code, session_start_date;
```

**Workflow Explanation**:

1. **Authorization Sequencing** - `LAG()` tracks previous authorization end dates (like session last activity)
2. **Gap Detection** - 30-day gap triggers new therapy session (LAD renewal standard)
3. **Session Continuity** - Groups continuous authorizations into treatment courses (like persistent user sessions)
4. **Outcome Classification** - Categorizes therapy success based on session patterns (like conversion tracking)
5. **Risk Monitoring** - Identifies gaps and delays (like session security monitoring)

```sql
-- SESSION WINDOW: Inventory depletion analysis
WITH inventory_events AS (
    SELECT 
        sender_id,
        receiver_id,
        ndc_code,
        product_name,
        transaction_id,
        activity_type,
        quantity_change,
        current_inventory,
        activity_date,
        -- Calculate days since previous activity (session idle detection)
        LAG(activity_date) OVER (PARTITION BY sender_id, receiver_id, ndc_code ORDER BY activity_date) AS prev_activity_date,
        DATEDIFF(DAY, LAG(activity_date) OVER (PARTITION BY sender_id, receiver_id, ndc_code ORDER BY activity_date), activity_date) AS days_since_prev_activity
    FROM edi_852_inventory
),
depletion_sessions AS (
    SELECT 
        *,
        -- 7-day gap = new inventory session (weekly monitoring cycle)
        CASE 
            WHEN days_since_prev_activity IS NULL THEN 1 -- First activity = new session
            WHEN days_since_prev_activity > 7 THEN 1 -- Gap > 7 days = new depletion cycle
            ELSE 0 -- Same depletion session
        END AS new_inventory_session
    FROM inventory_events
),
session_identifiers AS (
    SELECT 
        *,
        -- Generate inventory session ID (like session ID with security)
        CONCAT(sender_id, '_', receiver_id, '_', ndc_code, '_', DATE_FORMAT(activity_date, 'YYYY-MM'), '_',
            SUM(new_inventory_session) OVER (PARTITION BY sender_id, receiver_id, ndc_code ORDER BY activity_date)) AS inventory_session_id
    FROM depletion_sessions
)
SELECT 
    inventory_session_id,
    sender_id,
    receiver_id,
    ndc_code,
    product_name,
    
    -- Session timeline (like user session start/end)
    MIN(activity_date) AS session_start_date,
    MAX(activity_date) AS session_end_date,
    DATEDIFF(DAY, MIN(activity_date), MAX(activity_date)) AS session_span_days,
    
    -- Inventory movement metrics (session activity)
    COUNT(*) AS transactions_in_session,
    SUM(CASE WHEN activity_type = 'sale' THEN ABS(quantity_change) ELSE 0 END) AS total_units_sold,
    SUM(CASE WHEN activity_type = 'return' THEN quantity_change ELSE 0 END) AS total_units_returned,
    SUM(quantity_change) AS net_inventory_change,
    
    -- Depletion rate (like session engagement rate)
    CASE 
        WHEN DATEDIFF(DAY, MIN(activity_date), MAX(activity_date)) > 0 
        THEN ABS(SUM(quantity_change)) / DATEDIFF(DAY, MIN(activity_date), MAX(activity_date))
        ELSE ABS(SUM(quantity_change))
    END AS daily_depletion_rate,
    
    -- Inventory health (like session health check)
    AVG(current_inventory) AS avg_inventory_level,
    MIN(current_inventory) AS lowest_inventory_point,
    
    -- Session characteristics (like session classification)
    CASE 
        WHEN SUM(CASE WHEN activity_type = 'sale' THEN 1 ELSE 0 END) >= 5 THEN 'HIGH_VELOCITY_PRODUCT'
        WHEN SUM(CASE WHEN activity_type = 'return' THEN 1 ELSE 0 END) > 0 THEN 'RETURN_ADJUSTMENT_CYCLE'
        WHEN DATEDIFF(DAY, MIN(activity_date), MAX(activity_date)) = 0 THEN 'SINGLE_DAY_BURST'
        ELSE 'STEADY_DEPLETION'
    END AS depletion_pattern,
    
    -- Supply chain risk (like session security alert)
    CASE 
        WHEN MIN(current_inventory) < 1000 THEN 'LOW_INVENTORY_ALERT'
        WHEN AVG(days_since_prev_activity) > 3 THEN 'IRREGULAR_MOVEMENT_PATTERN'
        WHEN SUM(CASE WHEN activity_type = 'return' THEN 1 ELSE 0 END) > 2 THEN 'HIGH_RETURN_RATE'
        ELSE 'NORMAL_FLOW'
    END AS supply_chain_status,
    
    -- Data integrity check (like session validation)
    CASE 
        WHEN MAX(current_inventory) - MIN(current_inventory) != SUM(quantity_change) 
        THEN 'DATA_INTEGRITY_ISSUE'
        ELSE 'BALANCED'
    END AS inventory_reconciliation
    
FROM session_identifiers

GROUP BY inventory_session_id, sender_id, receiver_id, ndc_code, product_name

ORDER BY sender_id, receiver_id, ndc_code, session_start_date;
```

**Workflow Explanation**:

1. **Activity Sequencing** - `LAG()` tracks previous inventory movement (like session activity timestamps)
2. **7-Day Timeout** - Gap > 7 days triggers new depletion session (weekly inventory cycle)
3. **Session Grouping** - Groups continuous movements into supply chain sessions (like user browsing sessions)
4. **Depletion Analysis** - Calculates daily depletion rates within sessions (like session engagement metrics)
5. **Integrity Check** - Validates inventory math (like session data consistency checks)

## Use Case 5: EDI 840 Product Transfer (Resale)

**Scenario**: Track EDI 840 quote-to-order sessions with 14-day inactivity timeout (sales cycle management).

```sql

    SELECT 
        *,
        -- 14-day gap = new sales session (quote expiration standard)
        CASE 
            WHEN days_since_prev_txn IS NULL THEN 1 -- First transaction = new session
            WHEN days_since_prev_txn > 14 THEN 1 -- Gap > 14 days = new sales cycle
            ELSE 0 -- Same sales session (quote-to-order flow)
        END AS new_sales_session
    FROM transfer_events
),
session_enrichment AS (
    SELECT 
        *,
        -- Generate sales session ID (like secure session token)
        CONCAT(sender_id, '_', receiver_id, '_', 
            SUM(new_sales_session) OVER (PARTITION BY sender_id, receiver_id ORDER BY transaction_date)) AS sales_session_id,
        SUM(new_sales_session) OVER (PARTITION BY sender_id, receiver_id ORDER BY transaction_date) AS session_number
    FROM sales_sessions
)

    sales_session_id,
    sender_id,
    receiver_id,

    MAX(transaction_date) AS session_end_date,

    COUNT(*) AS total_transactions,
    COUNT(CASE WHEN transaction_type = 'quote' THEN 1 END) AS quotes_sent,
    COUNT(CASE WHEN transaction_type = 'order' THEN 1 END) AS orders_placed,
    COUNT(CASE WHEN transaction_type = 'acknowledgment' THEN 1 END) AS acknowledgments,
    
    -- Conversion metrics (like session conversion rate)
    CASE 
        WHEN COUNT(CASE WHEN transaction_type = 'quote' THEN 1 END) > 0 
        THEN COUNT(CASE WHEN transaction_type = 'order' THEN 1 END) * 100.0 / 
             COUNT(CASE WHEN transaction_type = 'quote' THEN 1 END)
        ELSE 0 
    END AS quote_to_order_conversion_pct,
    
    -- Financial metrics (session value)
    SUM(CASE WHEN transaction_type = 'order' THEN total_amount ELSE 0 END) AS total_revenue,
    AVG(CASE WHEN transaction_type = 'order' THEN total_amount END) AS avg_order_value,
    
    -- Velocity metrics (like session engagement speed)
    CASE 
        WHEN DATEDIFF(DAY, MIN(transaction_date), MAX(transaction_date)) > 0 
        THEN COUNT(*) * 1.0 / DATEDIFF(DAY, MIN(transaction_date), MAX(transaction_date))
        ELSE COUNT(*)
    END AS transactions_per_day,
    
    -- Session outcome classification (like session success/failure)
    CASE 
        WHEN COUNT(CASE WHEN transaction_type = 'order' THEN 1 END) > 0 
             AND COUNT(CASE WHEN transaction_type = 'quote' THEN 1 END) = COUNT(CASE WHEN transaction_type = 'order' THEN 1 END)
        THEN 'PERFECT_CONVERSION'
        WHEN COUNT(CASE WHEN transaction_type = 'order' THEN 1 END) > 0 
        THEN 'PARTIAL_CONVERSION'
        WHEN COUNT(CASE WHEN transaction_type = 'quote' THEN 1 END) > 0 
             AND DATEDIFF(DAY, MAX(transaction_date), CURRENT_DATE()) > 30
        THEN 'EXPIRED_QUOTE_NO_ORDER'
        ELSE 'PENDING_NEGOTIATION'
    END AS session_outcome,
    
    -- Contract compliance (like session security validation)
    COUNT(DISTINCT contract_number) AS contracts_used,
    CASE 
        WHEN COUNT(DISTINCT contract_number) > 1 THEN 'MULTI_CONTRACT_SESSION'
        WHEN COUNT(DISTINCT ndc_code) > 3 THEN 'MULTI_PRODUCT_NEGOTIATION'
        ELSE 'SINGLE_PURPOSE_SESSION'
    END AS session_complexity
    
FROM session_enrichment

GROUP BY sales_session_id, sender_id, receiver_id

ORDER BY sender_id, receiver_id, session_start_date;
```

1. **Transaction Sequencing** - `LAG()` tracks previous transaction dates (like session activity timestamps)

3. **Funnel Analysis** - Tracks quote → order → acknowledgment flow (like login → browse → checkout)
4. **Conversion Tracking** - Calculates quote-to-order conversion rates (like session conversion rates)
5. **Outcome Classification** - Categorizes session success (like completed vs abandoned sessions)

```sql
             COUNT(CASE WHEN transaction_type = '837' THEN 1 END)
        ELSE 0

    COUNT(DISTINCT denial_code) AS unique_denial_reasons,
    MODE(denial_code) AS most_common_denial,

        WHEN COUNT(CASE WHEN transaction_type = '837' THEN 1 END) = COUNT(CASE WHEN transaction_type = '835' THEN 1 END)

        WHEN COUNT(CASE WHEN transaction_type = '837' THEN 1 END) > COUNT(CASE WHEN transaction_type = '835' THEN 1 END)
        THEN 'SUBMISSION_HEAVY'
        ELSE 'RESPONSE_HEAVY'
    END AS session_type,

    CASE 
        WHEN COUNT(CASE WHEN transaction_type = '837' THEN 1 END) != COUNT(CASE WHEN transaction_type = '835' THEN 1 END)
        THEN 'UNMATCHED_CLAIMS_ALERT'
        ELSE 'BALANCED'
    END AS reconciliation_status
    
FROM session_tracking

GROUP BY adjudication_session_id, submitter_id, receiver_id

ORDER BY submitter_id, receiver_id, session_start;
```

## Key Session Window Concepts Summary

| Concept | Pharma/Healthcare Analogy | SQL Implementation |
|---------|--------------------------|-------------------|
| **Session Start** | First prescription fill, claim submission | First event or gap > timeout threshold |
| **Idle Timeout** | Days between refills, claim processing time | `DATEDIFF` > threshold (90 days, 24 hours, etc.) |
| **Session ID** | Patient therapy course, claim batch ID | `CONCAT` + `SUM` of session flags |
| **Session Data** | Medication history, claims in batch | `GROUP BY` session_id with aggregates |
| **Session Security** | HIPAA compliance, data integrity | Validation checks, reconciliation |

**Session vs Tumbling vs Hopping Windows**

| Feature | Session Windows | Tumbling Windows | Hopping Windows |
|---------|---------------|------------------|---------------|
| **Trigger** | Activity-based (dynamic) | Time-based (fixed) | Time-based (overlapping) |
| **Size** | Variable (depends on activity) | Fixed | Fixed |
| **Overlap** | None (distinct sessions) | None | Configurable |
| **Pharma Use** | Patient journeys, claim lifecycles | Daily reporting, batch processing | Trend monitoring, anomaly detection |
| **Timeout** | Yes (idle gap defines end) | No (fixed boundaries) | No (fixed advance) |

**Remember**: Session windows in SQL are like patient medication adherence—each therapy session has a natural lifecycle with gaps indicating new treatment courses. The session dynamically groups related events until an inactivity timeout occurs, just like a user session expires after idle time!

> [!Tip]
> Implementing secure and efficient session management can significantly enhance user experience and protect against common web application vulnerabilities by using techniques like HTTPS and secure cookie flags.

## Key Points
- Sessions are used to store user data between requests, allowing for personalized and interactive web applications.
- Each session is uniquely identified, typically by a session ID, which is stored on the user's browser as a cookie or passed in the URL.
- Sessions have a limited lifespan and can be terminated by the server or the user, requiring mechanisms for renewal or graceful termination.
- Session data should be stored securely on the server-side, using encryption and access controls to protect sensitive information.

> [!important]
> The session ID must be kept secret to prevent session hijacking, where an attacker obtains the session ID to impersonate the legitimate user.

## Mental Model
1. **User Authentication**: The user provides credentials to access the system.
2. **Session Creation**: Upon successful authentication, a new session is created, and a unique session ID is generated and stored on the user's browser.
3. **Session Utilization**: The user interacts with the application, with the session ID being passed with each request to access session data.
4. **Session Termination**: The session is closed, either by the user logging out or by the session timing out due to inactivity.

## Observe
- A common mistake is using insecure protocols (like HTTP) to transmit session IDs, making them vulnerable to interception.
- A silent failure mode occurs when sessions are not properly synchronized across load-balanced servers, leading to unexpected user logout or data loss.
- A misleading assumption is that sessions are inherently secure; in reality, sessions require careful management to prevent security breaches.

> [!Important]
> For example, in an e-commerce application, a user logs in and adds items to their cart. The session ensures that the cart contents are remembered across different pages and interactions, until the user checks out or logs out, demonstrating the critical role of session management in maintaining a seamless user experience.
