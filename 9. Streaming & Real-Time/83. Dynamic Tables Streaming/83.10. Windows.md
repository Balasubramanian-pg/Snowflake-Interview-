# 83.10. Windows

> [!Note]
> Understanding Windows fundamentals is crucial for optimizing computer performance, ensuring data security, and streamlining workflow efficiency.

Window functions analyze data across "windows" or sets of rows related to the current row—essential for pharma analytics like tracking patient treatment sequences, calculating running inventory, or ranking clinical trial sites.

Here are **6 use cases** with detailed beginner workflows using Snowflake and pharma data.

---

## Use Case 1: Patient Treatment Timeline (LAG/LEAD)  
**Scenario:** Track the progression of diabetes patients through different medication therapies to see what they were prescribed before/after their current drug.

```sql
WITH patient_prescriptions AS (
    SELECT 
        patient_id,
        prescription_date,
        drug_name,
        dosage_mg,
        therapeutic_class,
        LAG(drug_name) OVER (
            PARTITION BY patient_id 
            ORDER BY prescription_date
        ) as previous_drug,
        LEAD(drug_name) OVER (
            PARTITION BY patient_id 
            ORDER BY prescription_date
        ) as next_drug,
        LAG(prescription_date) OVER (
            PARTITION BY patient_id 
            ORDER BY prescription_date
        ) as previous_rx_date
    FROM pharma.patient_prescriptions
    WHERE therapeutic_class = 'Anti-Diabetic Agents'
      AND prescription_date >= '2025-01-01'
)
SELECT 
    patient_id,
    prescription_date,
    drug_name,
    previous_drug,
    next_drug,
    DATEDIFF(day, previous_rx_date, prescription_date) as days_between_switches
FROM patient_prescriptions
WHERE previous_drug IS NOT NULL  -- Exclude first prescriptions (no "window" before)
ORDER BY 
    patient_id, 
    prescription_date;
```

### Detailed Workflow (How the Window Moves):

**Step 1: PARTITION BY - Creating Patient-Specific Windows**
- `PARTITION BY patient_id` tells Snowflake: "Create a separate sliding window for each unique patient."
- Think of it like giving each patient their own personal film strip. The window function only looks at frames (rows) from that specific patient's history, never mixing Patient A's data with Patient B's data.

**Step 2: ORDER BY - Sequencing the Window**
- `ORDER BY prescription_date` arranges each patient's film strip chronologically.
- **Critical for beginners:** Without this order, "previous" and "next" have no meaning. The window slides along this time axis.

**Step 3: LAG() - Looking Backward**
- As Snowflake scans each row, `LAG(drug_name)` peeks at the row immediately before the current row (within that patient's partition).
- For the first prescription of each patient, there's no previous row, so it returns `NULL`.
- **Pharma Context:** This shows if the patient switched from Metformin to Insulin, for example.

**Step 4: LEAD() - Looking Forward**
- `LEAD(drug_name)` looks at the next row in the sequence.
- On the patient's most recent prescription, this returns `NULL` (no future data yet).

**Step 5: Calculating Intervals**
- We use the `LAG` of the date to calculate `days_between_switches`, revealing treatment adherence patterns or rapid therapeutic failures.

## Use Case 2: Running Inventory Levels by Distribution Center  
**Scenario:** Track cumulative drug inventory across multiple warehouses to flag when stock drops below safety thresholds.

```sql
WITH daily_inventory AS (
    SELECT 
        distribution_center_id,
        drug_sku,
        inventory_date,
        units_received,
        units_dispensed,
        (units_received - units_dispensed) as daily_net_change
    FROM pharma.inventory_transactions
    WHERE drug_sku IN ('RX-INSULIN-001', 'RX-EPINEPH-002')
      AND inventory_date BETWEEN '2026-01-01' AND '2026-03-31'
)
SELECT 
    distribution_center_id,
    drug_sku,
    inventory_date,
    units_received,
    units_dispensed,
    SUM(daily_net_change) OVER (
        PARTITION BY distribution_center_id, drug_sku 
        ORDER BY inventory_date
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) as running_inventory_balance,
    AVG(units_dispensed) OVER (
        PARTITION BY distribution_center_id, drug_sku
        ORDER BY inventory_date
        ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
    ) as seven_day_avg_demand,
    CASE 
        WHEN running_inventory_balance < (seven_day_avg_demand * 3) 
        THEN 'CRITICAL: < 3 days supply'
        ELSE 'ADEQUATE'
    END as stock_status
FROM daily_inventory
ORDER BY 
    distribution_center_id, 
    drug_sku, 
    inventory_date;
```

### Detailed Workflow:

**Step 1: Row-Based Window Frame (`ROWS BETWEEN`)**
- `ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW` defines the window as "from the very first row of this partition up to and including today."
- **Visual:** Imagine a calculator that adds today's net change to yesterday's running total, then carries that new total forward.

**Step 2: Running Sum Logic**
- For each row, Snowflake sums all `daily_net_change` values from the start of the partition (specific DC + specific drug) up to the current date.
- **Pharma Integrity:** This preserves the cumulative nature of inventory—unlike `GROUP BY` which would collapse rows, we keep daily granularity while adding the running context.

**Step 3: Moving Average Window (`ROWS BETWEEN 6 PRECEDING`)**
- This creates a rolling 7-day window (6 previous days + today).
- **Why:** It smooths out daily dispensing volatility to predict realistic demand (e.g., clinics order more on Mondays).

**Step 4: Safety Stock Calculation**
- We compare current running balance against 3x the recent average demand—a pharmaceutical supply chain best practice.
- The query flags when inventory falls below a 3-day supply, triggering reorder workflows.

## Use Case 3: Clinical Trial Site Performance Ranking  
**Scenario:** Rank hospital sites by patient enrollment speed to identify top performers and underperformers in a Phase III trial.

```sql
WITH site_enrollment AS (
    SELECT 
        site_id,
        site_name,
        country,
        patient_id,
        enrollment_date,
        screening_to_enrollment_days,
        COUNT(*) OVER (
            PARTITION BY site_id 
            ORDER BY enrollment_date
            RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
        ) as cumulative_enrolled,
        RANK() OVER (
            PARTITION BY country 
            ORDER BY COUNT(*) DESC
        ) as country_rank,
        PERCENT_RANK() OVER (
            ORDER BY COUNT(*) DESC
        ) as global_percentile,
        FIRST_VALUE(enrollment_date) OVER (
            PARTITION BY site_id 
            ORDER BY enrollment_date
        ) as first_enrollment_date,
        LAST_VALUE(enrollment_date) OVER (
            PARTITION BY site_id 
            ORDER BY enrollment_date
            ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
        ) as latest_enrollment_date
    FROM pharma.clinical_trials
    WHERE study_id = 'PH3-ONCOLOGY-2026'
      AND enrollment_status = 'Enrolled'
    GROUP BY 
        site_id, site_name, country, patient_id, 
        enrollment_date, screening_to_enrollment_days
)
SELECT 
    site_id,
    site_name,
    country,
    COUNT(patient_id) as total_patients,
    AVG(screening_to_enrollment_days) as avg_screening_time,
    MAX(cumulative_enrolled) as final_enrollment_count,
    MAX(country_rank) as rank_within_country,
    MAX(global_percentile) as performance_percentile,
    DATEDIFF(day, MAX(first_enrollment_date), MAX(latest_enrollment_date)) as enrollment_duration_days
FROM site_enrollment
GROUP BY 
    site_id, site_name, country
ORDER BY 
    total_patients DESC;
```

### Detailed Workflow:

**Step 1: Cumulative Count with `RANGE`**
- `RANGE BETWEEN UNBOUNDED PRECEDING` counts all prior enrollments at that site up to the current patient's enrollment date.
- **Difference from `ROWS`:** If two patients enroll on the same day (tie in ordering), `RANGE` includes both as "previous," while `ROWS` would treat them sequentially.

**Step 2: Ranking Functions**
- `RANK()` assigns competitive positions within each country. If two sites tie for 10 patients, both get rank #1, next gets #3.
- `PERCENT_RANK()` calculates where a site falls globally (0.0 = worst, 1.0 = best), enabling cross-country comparisons despite different market sizes.

**Step 3: Boundary Functions (`FIRST_VALUE`/`LAST_VALUE`)**
- `FIRST_VALUE` captures the date of the site's very first enrollment (the "anchor").
- **Important:** `LAST_VALUE` requires explicit window bounds (`UNBOUNDED FOLLOWING`) to see future rows; otherwise, it defaults to current row only—a common beginner error.

**Step 4: Aggregation of Window Results**
- Since window functions calculate per-row, we `GROUP BY` site to collapse the detailed rows while keeping the calculated max values (the final cumulative count is the max of the running count).

## Use Case 4: Drug Side Effect Signal Detection (Moving Average)  
**Scenario:** Monitor adverse events (AEs) for new vaccines to detect safety signals that exceed baseline rates using rolling time windows.

```sql
WITH ae_reports AS (
    SELECT 
        report_date,
        drug_name,
        adverse_event_term,
        severity_grade,
        patient_count,
        SUM(patient_count) OVER (
            PARTITION BY drug_name, adverse_event_term 
            ORDER BY report_date
            ROWS BETWEEN 29 PRECEDING AND CURRENT ROW
        ) as rolling_30_day_sum,
        AVG(patient_count) OVER (
            PARTITION BY drug_name, adverse_event_term 
            ORDER BY report_date
            ROWS BETWEEN 89 PRECEDING AND 30 PRECEDING
        ) as baseline_60_day_avg,  -- Days 90-30 ago (excluding recent 30)
        STDDEV(patient_count) OVER (
            PARTITION BY drug_name, adverse_event_term 
            ORDER BY report_date
            ROWS BETWEEN 89 PRECEDING AND 30 PRECEDING
        ) as baseline_stddev
    FROM pharma.adverse_events
    WHERE report_date >= CURRENT_DATE - 120
)
SELECT 
    report_date,
    drug_name,
    adverse_event_term,
    patient_count,
    rolling_30_day_sum,
    baseline_60_day_avg,
    CASE 
        WHEN baseline_stddev > 0 
        THEN (rolling_30_day_sum - (baseline_60_day_avg * 30)) / (baseline_stddev * SQRT(30))
        ELSE 0 
    END as z_score_signal,
    CASE 
        WHEN z_score_signal > 3 THEN 'STATISTICAL SIGNAL: Investigate immediately'
        WHEN z_score_signal > 2 THEN 'WATCH: Elevated trend'
        ELSE 'Normal variation'
    END as regulatory_alert_status
FROM ae_reports
WHERE report_date >= CURRENT_DATE - 30  -- Only recent alerts
ORDER BY 
    z_score_signal DESC;
```

### Detailed Workflow:

**Step 1: Establishing Baseline Windows**
- We create two distinct windows:
  - **Recent (Signal):** Last 30 days (rows 0-29 preceding)
  - **Historical (Baseline):** Days 90-30 ago (rows 89-30 preceding)
- **Pharma Context:** This compares "recent vaccine events" against "established historical rate" excluding the recent window to avoid circular logic.

**Step 2: Statistical Process Control**
- We calculate a Z-score: (Observed - Expected) / Standard Deviation.
- **Window Magic:** The `baseline_stddev` measures volatility from the historical window, while `rolling_30_day_sum` captures recent activity.

**Step 3: Signal Thresholds**
- Z-score > 3 indicates a statistically significant safety signal (p < 0.003), triggering FDA reporting requirements.
- The query automates pharmacovigilance monitoring that would otherwise require manual chart reviews.

## Use Case 5: Dose Escalation Cohorts in Oncology Trials  
**Scenario:** Analyze dose-limiting toxicities (DLTs) across cohorts to determine Maximum Tolerated Dose (MTD) using window frames.

```sql
WITH dose_cohorts AS (
    SELECT 
        cohort_id,
        dose_level_mg,
        patient_id,
        cycle_number,
        has_dlt,  -- 1 if dose-limiting toxicity occurred, 0 otherwise
        SUM(has_dlt) OVER (
            PARTITION BY cohort_id, dose_level_mg 
            ORDER BY cycle_number, patient_id
            ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
        ) as cumulative_dlts,
        COUNT(*) OVER (
            PARTITION BY cohort_id, dose_level_mg 
            ORDER BY cycle_number, patient_id
            ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
        ) as patients_evaluated,
        cumulative_dlts / patients_evaluated as dlt_rate,
        NTH_VALUE(patient_id, 3) OVER (
            PARTITION BY cohort_id 
            ORDER BY cycle_number
            ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
        ) as third_patient_id  -- Regulatory milestone marker
    FROM pharma.oncology_dosing
    WHERE study_phase = 'Phase I'
)
SELECT 
    cohort_id,
    dose_level_mg,
    MAX(cumulative_dlts) as total_dlts,
    MAX(patients_evaluated) as cohort_size,
    MAX(dlt_rate) as final_dlt_rate,
    CASE 
        WHEN MAX(dlt_rate) >= 0.33 THEN 'EXCEEDS MTD THRESHOLD (33%)'
        ELSE 'Acceptable toxicity profile'
    END as dose_escalation_decision,
    MAX(third_patient_id) as safety_review_trigger_patient
FROM dose_cohorts
GROUP BY 
    cohort_id, dose_level_mg
ORDER BY 
    dose_level_mg;
```

### Detailed Workflow:

**Step 1: Cohort Partitioning**
- `PARTITION BY cohort_id, dose_level_mg` treats each dose level as an isolated experiment.
- **Clinical Context:** In Phase I oncology trials, patients are grouped into cohorts (3-6 patients) at specific doses. We track toxicities cumulatively as each patient completes evaluation.

**Step 2: Rolling Toxicity Rate**
- As each patient completes their cycle, we recalculate `dlt_rate` (cumulative DLTs / patients evaluated).
- **Regulatory Logic:** If at any point DLT rate ≥ 33%, dose escalation stops (classic "3+3" trial design).

**Step 3: NTH_VALUE for Milestones**
- `NTH_VALUE(..., 3)` identifies the third patient in each cohort—a regulatory checkpoint where Data Safety Monitoring Boards (DSMB) must review before enrolling patient 4.

## Use Case 6: Pharmacy Claims Pre-authorization Sequence  
**Scenario:** Identify patients requiring step therapy (fail first on generic before brand is approved) using conditional window logic.

```sql
WITH claims_sequence AS (
    SELECT 
        patient_id,
        claim_date,
        drug_name,
        drug_type,  -- 'Generic' or 'Brand'
        insurance_status,
        SUM(CASE WHEN drug_type = 'Generic' THEN 1 ELSE 0 END) OVER (
            PARTITION BY patient_id 
            ORDER BY claim_date
            ROWS BETWEEN UNBOUNDED PRECEDING AND 1 PRECEDING
        ) as prior_generic_count,
        LAST_VALUE(drug_name) OVER (
            PARTITION BY patient_id 
            ORDER BY claim_date
            ROWS BETWEEN 1 PRECEDING AND 1 PRECEDING
        ) as immediately_prior_drug,
        ROW_NUMBER() OVER (
            PARTITION BY patient_id 
            ORDER BY claim_date
        ) as claim_sequence_number
    FROM pharma.insurance_claims
    WHERE therapeutic_class = 'Statins'
)
SELECT 
    patient_id,
    claim_date,
    drug_name,
    drug_type,
    CASE 
        WHEN drug_type = 'Brand' 
             AND prior_generic_count = 0 
             AND claim_sequence_number > 1 
        THEN 'STEP_THERAPY_VIOLATION: Generic not tried first'
        WHEN drug_type = 'Brand' 
             AND prior_generic_count >= 1 
        THEN 'APPROVED: Step therapy satisfied'
        ELSE 'STANDARD_CLAIM'
    END as prior_authorization_status
FROM claims_sequence
WHERE insurance_status = 'Pending Review'
ORDER BY 
    patient_id, 
    claim_date;
```

### Detailed Workflow:

**Step 1: Conditional Window Aggregation**
- The `SUM(CASE...)` counts only generic drugs in the patient's history *before* the current claim.
- **The Frame:** `ROWS BETWEEN UNBOUNDED PRECEDING AND 1 PRECEDING` excludes the current row from the count—critical for "prior history" logic.

**Step 2: Offset Window (`LAST_VALUE` with Specific Offset)**
- `ROWS BETWEEN 1 PRECEDING AND 1 PRECEDING` isolates exactly one row: the immediate previous claim.
- This shows what drug the patient used right before attempting to switch.

**Step 3: Business Rule Application**
- If patient tries Brand name (e.g., Lipitor) with zero prior generics, and it's not their first-ever claim → Insurance denial.
- If they've tried at least one generic (prior_generic_count ≥ 1) → Brand is approved.

## Summary: Window Functions as "Operating Systems" for Pharma Data

Just as the Windows OS manages application windows and memory partitions, SQL Window Functions manage **logical partitions** of pharmaceutical data:

| OS Windows Concept | SQL Window Function | Pharma Example |
|-------------------|---------------------|----------------|
| **Process Isolation** | `PARTITION BY` | Separate patient histories so Patient A's insulin data never mixes with Patient B's |
| **Task Scheduling** | `ORDER BY` | Chronological sequencing of treatment events |
| **Memory Buffer** | `ROWS/RANGE BETWEEN` | Rolling 30-day safety monitoring window |
| **Context Switching** | `LAG/LEAD` | Looking at previous/next medication in a sequence |
| **Priority Ranking** | `RANK/PERCENT_RANK` | Identifying top-performing clinical trial sites |

Each query maintains **data integrity** (the critical rule from your Reduction concept) by using `PARTITION BY` to ensure calculations stay within clinically relevant boundaries (patient, site, cohort) while allowing cross-row analysis that `GROUP BY` cannot provide.
