# 83.1. Reduction

> [!Note]
> Reduction is the process of breaking down complex systems into simpler, more manageable components to understand, analyze, and solve problems more effectively.

## Use Case 1: Financial Data Aggregation  
**The Complex System:** Millions of raw transaction records with timestamps, user IDs, product details, and amounts.  
**The Reduction:** Compress into daily performance metrics by category—identifying the "minimum set of elements necessary" for business decisions.

```sql
WITH raw_transactions AS (
    SELECT 
        transaction_id,
        DATE(transaction_timestamp) as transaction_date,
        product_category,
        sale_amount,
        customer_id
    FROM sales_db.transactions
    WHERE transaction_timestamp >= '2026-01-01'
)
SELECT 
    transaction_date,
    product_category,
    COUNT(DISTINCT customer_id) as unique_customers,  -- Reduced: many rows → one number
    COUNT(*) as total_transactions,                   -- Reduced: detail → count
    SUM(sale_amount) as daily_revenue,               -- Reduced: individual amounts → sum
    AVG(sale_amount) as avg_transaction_size         -- Reduced: distribution → average
FROM raw_transactions
GROUP BY 
    transaction_date, 
    product_category
ORDER BY 
    transaction_date DESC, 
    daily_revenue DESC;
```

### Detailed Workflow (How the Query Runs Step-by-Step):

**Step 1: The CTE (`WITH` block) – Isolating Components**
- Snowflake first creates a temporary view called `raw_transactions` in memory.
- **Reduction in action:** We're decomposing the massive table into only 5 essential columns (ignoring unnecessary data like customer addresses, payment method details, or IP addresses).
- The `WHERE` clause applies a boundary—further reducing the dataset to just this year's data.

**Step 2: GROUP BY – Creating Buckets**
- Snowflake sorts the filtered data into "buckets"—one bucket for each unique combination of `transaction_date` and `product_category`.
- Think of this like organizing papers into folders: one folder for "Electronics on Jan 15," another for "Clothing on Jan 15," etc.

**Step 3: Aggregation – Collapsing Complexity**
- Within each bucket, Snowflake performs mathematical reduction:
  - `COUNT(DISTINCT customer_id)`: Reduces thousands of customer IDs down to a single number (unique count).
  - `SUM(sale_amount)`: Adds up all individual transaction values into one daily total.
  - `AVG(sale_amount)`: Calculates the central tendency of the distribution.
- **Maintaining Integrity:** While we lose individual transaction details, we preserve the mathematical relationships (totals still sum correctly, averages represent true centers).

**Step 4: ORDER BY – Prioritizing Insights**
- Finally, results are sorted so the highest revenue categories appear first, reducing cognitive load when reading the report.

## Use Case 2: Hierarchical Organization Decomposition  
**The Complex System:** A recursive employee-manager hierarchy (org chart) where each person reports to someone else, creating nested layers of complexity.  
**The Reduction:** Flatten the tree structure into level-based layers—making the "interactions between components" visible as simple numbers.

```sql
WITH RECURSIVE org_structure AS (
    -- Anchor Member: Start with the top (CEO/Root)
    SELECT 
        employee_id,
        full_name,
        manager_id,
        department,
        0 as hierarchy_level,  -- CEO is level 0
        full_name as hierarchy_path  -- Start the path with root name
    FROM hr.employees
    WHERE manager_id IS NULL  -- The reduction starts from the top
    
    UNION ALL
    
    -- Recursive Member: Add direct reports level by level
    SELECT 
        e.employee_id,
        e.full_name,
        e.manager_id,
        e.department,
        oh.hierarchy_level + 1,  -- Increment level (decomposition step)
        oh.hierarchy_path || ' > ' || e.full_name  -- Build interaction path
    FROM hr.employees e
    INNER JOIN org_structure oh 
        ON e.manager_id = oh.employee_id  -- Find subordinates of current level
)
SELECT 
    hierarchy_level,
    department,
    COUNT(*) as employee_count,
    AVG(hierarchy_level) as avg_depth
FROM org_structure
GROUP BY 
    hierarchy_level, 
    department
ORDER BY 
    hierarchy_level, 
    employee_count DESC;
```

### Detailed Workflow (How the Query Thinks):

**Phase 1: The Anchor (Base Case)**
- Snowflake identifies the "root nodes"—employees with no manager (usually the CEO).
- Creates the first temporary result set with just these top-level employees marked as `level 0`.
- **Reduction principle:** We're decomposing the complex network by identifying its foundation first.

**Phase 2: Recursion (Iterative Decomposition)**
- Snowflake looks at the anchor results and asks: "Who reports to these people?"
- It finds all employees where `manager_id` matches the `employee_id` from the previous level.
- Adds these to the result set as `level 1`, then repeats the process: "Who reports to the level 1 people?"
- **Maintaining Integrity:** The `hierarchy_path` column preserves the complete chain of command (e.g., "CEO > VP Sales > Manager A"), ensuring we don't lose the relationship structure during flattening.

**Phase 3: Aggregation of the Reduced Structure**
- Once all levels are mapped (recursion stops when no new matches found), we apply standard aggregation.
- **Why this matters:** Instead of navigating a complex tree structure, we've reduced it to a simple table showing "How many people are at each level in each department?"—the minimum information needed for organizational analysis.

## Use Case 3: Schema Normalization (Breaking the Wide Table)  
**The Complex System:** A single "kitchen sink" table with 80+ columns (customer info, order details, shipping data, payment methods) all duplicated across millions of rows.  
**The Reduction:** Decompose into focused dimension and fact tables—separating concerns while maintaining referential integrity.

```sql
-- Step 1: Reduce to Customer Dimension (Unique entities only)
CREATE OR REPLACE TABLE dim_customers AS
SELECT DISTINCT
    customer_id,
    first_name,
    last_name,
    email,
    customer_segment,
    registration_date,
    region,
    -- Maintain integrity: Keep only customer-at-birth attributes
    ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY registration_date) as dedup_rank
FROM messy_unified_table
QUALIFY dedup_rank = 1;  -- Reduce duplicates to single representative

-- Step 2: Reduce to Product Dimension
CREATE OR REPLACE TABLE dim_products AS
SELECT DISTINCT
    product_sku,
    product_name,
    category,
    subcategory,
    unit_cost
FROM messy_unified_table;

-- Step 3: Reduce to essential Fact table (measurements only)
CREATE OR REPLACE TABLE fact_orders AS
SELECT 
    order_id,
    customer_id,        -- Foreign key to dim_customers
    product_sku,        -- Foreign key to dim_products
    order_date,
    quantity,
    sale_price,
    discount_amount,
    (sale_price * quantity - discount_amount) as net_revenue  -- Calculated measure
FROM messy_unified_table
WHERE order_status != 'cancelled';  -- Filter to maintain data integrity
```

### Detailed Workflow (The Decomposition Process):

**Stage 1: Identifying Constituent Parts**
- We analyze the 80-column table and identify three distinct "systems":
  - **Customer system:** Demographics, contact info (slowly changing)
  - **Product system:** Catalog information (relatively static)
  - **Order system:** Transaction events (constantly changing measurements)

**Stage 2: Dimensional Reduction (CT #1)**
- `SELECT DISTINCT` tells Snowflake: "Show me only unique combinations of these columns."
- **Data reduction:** If customer "John Smith" placed 50 orders, he appears 50 times in the source table. After `DISTINCT`, he appears once.
- **Integrity safeguard:** `ROW_NUMBER()` identifies if the same customer appears with slightly different data (data quality check). We keep only the earliest record (`dedup_rank = 1`) to maintain a single source of truth.

**Stage 3: Fact Reduction (CT #3)**
- We strip away all descriptive attributes (names, categories) and keep only:
  - **Keys:** IDs that link to dimension tables (maintaining system connectivity)
  - **Measures:** Numeric values we can aggregate (quantity, price)
  - **Degenerate dimensions:** Order ID (fact-specific identifier)
- The `WHERE` clause ensures we don't include cancelled orders in revenue calculations—preserving analytical integrity.

**Result:** Instead of querying an 80-column monster, analysts now query focused 6-8 column tables, drastically simplifying analysis while maintaining the ability to rejoin data using the keys (relationships).

## Use Case 4: Time-Series Downsampling (IoT Sensor Data)  
**The Complex System:** High-frequency IoT sensors recording temperature every second (86,400 rows per day per sensor).  
**The Reduction:** Aggregate into 5-minute intervals with statistical summaries—preserving trends while eliminating noise.

```sql
WITH raw_sensor_data AS (
    SELECT 
        sensor_id,
        timestamp,
        temperature,
        humidity,
        -- Window function to check data quality before reduction
        LAG(temperature) OVER (PARTITION BY sensor_id ORDER BY timestamp) as prev_temp
    FROM iot.sensor_readings
    WHERE timestamp >= CURRENT_DATE - 7  -- Last 7 days only
),
quality_checked AS (
    SELECT *,
        CASE 
            WHEN ABS(temperature - prev_temp) > 50 THEN 'anomaly'  -- 50-degree jump = sensor error
            ELSE 'valid'
        END as data_quality_flag
    FROM raw_sensor_data
),
time_bucketed AS (
    SELECT 
        sensor_id,
        DATE_TRUNC('minute', timestamp) as minute_bucket,  -- Reduce precision: seconds → minutes
        temperature,
        data_quality_flag,
        ROW_NUMBER() OVER (
            PARTITION BY sensor_id, DATE_TRUNC('minute', timestamp) 
            ORDER BY timestamp
        ) as row_num
    FROM quality_checked
    WHERE data_quality_flag = 'valid'  -- Maintain integrity: remove bad readings
)
SELECT 
    sensor_id,
    minute_bucket,
    AVG(temperature) as avg_temperature,      -- Central tendency
    MIN(temperature) as min_temperature,      -- Range boundary
    MAX(temperature) as max_temperature,      -- Range boundary
    STDDEV(temperature) as temp_volatility,   -- Variability measure
    COUNT(*) as sample_count                  -- Confidence metric (how many points reduced)
FROM time_bucketed
GROUP BY 
    sensor_id, 
    minute_bucket
HAVING 
    COUNT(*) >= 3  -- Integrity check: ensure we have enough samples for statistical validity
ORDER BY 
    sensor_id, 
    minute_bucket;
```

### Detailed Workflow (Progressive Reduction):

**Layer 1: Raw Ingestion with Context**
- We select recent data but use `LAG()` to look backward one row per sensor.
- **Why:** Before reducing data volume, we check for integrity. If temperature jumps 50 degrees in 1 second, it's a sensor glitch, not reality.

**Layer 2: Quality Filtering (The "Must Not Be Violated" Rule)**
- We mark anomalous readings. This ensures our reduced model doesn't misrepresent the system (like the warning in your text about oversimplifying).

**Layer 3: Time Bucketing (Granularity Reduction)**
- `DATE_TRUNC('minute', timestamp)` chops off seconds—reducing 60 rows per minute into 1 representative timestamp.
- **Maintaining Relationships:** We keep `sensor_id` to ensure we don't mix readings from different locations.

**Layer 4: Statistical Aggregation**
- Instead of throwing away 59 seconds of data arbitrarily, we mathematically reduce them:
  - `AVG`: The expected value
  - `MIN/MAX`: The range of variation
  - `STDDEV`: How unstable the readings were
  - `COUNT`: How confident we are in the reduction (more samples = better average)

**Integrity Checkpoint:** The `HAVING` clause ensures we only keep minute-buckets with at least 3 original readings. If a sensor was offline (0 or 1 reading), we discard that bucket rather than present a misleading average based on insufficient data.

## Use Case 5: Query Complexity Reduction (The CTE Pipeline)  
**The Complex System:** A business question requiring 10-table joins, multiple filtering conditions, and complex calculations—impossible to write/read in one block.  
**The Reduction:** Decompose into sequential, named steps—each creating a simplified model for the next step.

```sql
WITH 
-- Step 1: Reduce universe to active customers only (system boundary)
active_customers AS (
    SELECT 
        customer_id,
        segment,
        region,
        account_created_date
    FROM customers
    WHERE status = 'active'  -- Remove churned/inactive components
      AND account_created_date <= CURRENT_DATE - 30  -- Must be 30+ days old for meaningful analysis
),

-- Step 2: Reduce transaction history to customer-level aggregates
customer_metrics AS (
    SELECT 
        t.customer_id,
        COUNT(DISTINCT t.order_id) as total_orders,
        SUM(t.amount) as lifetime_value,
        AVG(t.amount) as avg_order_value,
        MAX(t.transaction_date) as last_order_date,
        DATEDIFF('day', MAX(t.transaction_date), CURRENT_DATE) as days_since_last_order
    FROM transactions t
    INNER JOIN active_customers ac 
        ON t.customer_id = ac.customer_id  -- Maintain connection to reduced set
    WHERE t.transaction_date >= CURRENT_DATE - 365  -- Last year only
    GROUP BY 
        t.customer_id
),

-- Step 3: Reduce customers to behavioral segments (simplified model)
behavioral_segments AS (
    SELECT 
        cm.*,
        ac.region,
        ac.segment as original_segment,
        CASE 
            WHEN cm.lifetime_value > 10000 AND cm.days_since_last_order < 30 THEN 'Champions'
            WHEN cm.lifetime_value > 5000 AND cm.days_since_last_order < 60 THEN 'Loyal Customers'
            WHEN cm.days_since_last_order > 90 THEN 'At Risk'
            WHEN cm.total_orders = 1 THEN 'New Customers'
            ELSE 'Regular'
        END as rfm_segment,  -- Reduced: complex behavior → single category
        -- Calculate engagement score (reduce multiple metrics to one index)
        (cm.total_orders * 0.3) + 
        (cm.lifetime_value / 1000 * 0.5) + 
        ((90 - LEAST(cm.days_since_last_order, 90)) / 90 * 0.2) as engagement_score
    FROM customer_metrics cm
    JOIN active_customers ac ON cm.customer_id = ac.customer_id
),

-- Step 4: Final reduction to actionable insights
segment_summary AS (
    SELECT 
        rfm_segment,
        region,
        COUNT(*) as customer_count,
        AVG(lifetime_value) as avg_clv,
        SUM(lifetime_value) as total_revenue_at_risk,
        AVG(engagement_score) as avg_engagement
    FROM behavioral_segments
    GROUP BY 
        rfm_segment, 
        region
)
SELECT *
FROM segment_summary
ORDER BY 
    total_revenue_at_risk DESC;
```

### Detailed Workflow (Step-by-Step Decomposition):

**Stage 1: Boundary Definition (`active_customers`)**
- We start by reducing the 10-million-row customer table to only "active" and "established" customers.
- **System Integrity:** By filtering early, we ensure downstream calculations don't include test accounts or churned users who would skew our behavioral models.

**Stage 2: Dimensional Aggregation (`customer_metrics`)**
- We take the transaction table (potentially billions of rows) and reduce it to one row per customer.
- **Key reduction:** Millions of transactions become single summary statistics (LTV, frequency, recency).
- The `INNER JOIN` maintains system integrity by ensuring we only calculate metrics for customers who passed Step 1's quality filters.

**Stage 3: Behavioral Modeling (`behavioral_segments`)**
- We apply the RFM (Recency, Frequency, Monetary) model—a classic reduction technique that takes 5+ variables and reduces them to a single segment label.
- **Complexity Management:** Instead of analysts interpreting "Customer X: 5 orders, $3,200 total, 45 days ago," they see "Loyal Customer"—the minimum necessary descriptor for decision-making.
- The `engagement_score` further reduces multiple metrics to a 0-100 scale.

**Stage 4: Meta-Aggregation (`segment_summary`)**
- We reduce again: Instead of looking at individual customers, we look at segment/region combinations.
- **Outcome:** 10 million transactions → ~25 segment combinations. This is the ultimate reduction—making a complex system comprehensible to human decision-makers.

## Key Points
- Reduction involves decomposing complex systems into their fundamental parts to analyze and understand each component's role and interaction.
- The process of reduction helps in identifying the minimum set of elements necessary for a system to function, thereby simplifying the analysis and solution-finding process.
- Reduction can be applied at various levels, from molecular biology to socioeconomic systems, making it a versatile tool across different disciplines.
- The outcome of reduction can lead to the development of models, theories, or interventions that are more targeted and effective.

> [!important]
> The one rule that must not be violated in reduction is maintaining the integrity of the system's functionality and interactions during the decomposition process to ensure that the simplified model accurately represents the original complex system.

## Mental Model
- Identify the complex system or problem to be reduced.
- Decompose the system into its constituent parts, focusing on key components and their interactions.
- Analyze each part to understand its role, function, and how it contributes to the overall system, leading to a simplified model or solution.

## Observe
- A common mistake people make is oversimplifying complex systems, which can lead to loss of critical information or interactions necessary for a comprehensive understanding.
- A silent failure mode in reduction is the assumption that all components are equally important, which can result in overlooking crucial elements or relationships.
- A misleading assumption is that reduction always leads to a linear or straightforward solution, whereas in many cases, the interactions between components can introduce complexity that requires a more nuanced approach.

> [!Important]
> For example, in the field of medicine, reducing the human body to its organ systems allows for a better understanding of how diseases affect specific parts and how treatments can be targeted, demonstrating the practical application of reduction in solving complex health problems.
