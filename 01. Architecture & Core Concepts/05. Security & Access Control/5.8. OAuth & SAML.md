# Snowflake OAuth & SAML: Through and Through

## 1. Core Concepts

Modern identity management moves beyond static passwords. Snowflake supports two enterprise-grade standards to delegate authentication to trusted external providers: **OAuth 2.0** for delegated authorisation and **SAML 2.0** for federated single sign-on (SSO). These protocols allow users to access Snowflake using credentials managed by an external identity provider, eliminating the need for separate Snowflake passwords.

### 1.1. What is OAuth 2.0?

OAuth 2.0 is an authorisation framework that enables third-party applications to obtain limited access to a service on behalf of a resource owner (a user) without exposing credentials. In Snowflake, OAuth allows client applications (e.g., Tableau, Looker, custom apps) to obtain access tokens that grant them secure access to Snowflake data.

Snowflake offers **two distinct OAuth implementations**:

| Implementation | Description |
|----------------|-------------|
| **Snowflake OAuth (Managed)** | Snowflake acts as both the OAuth **authorisation server** and the **resource server**. Ideal for applications like Tableau, Looker, or custom clients where you want a simple, Snowflake‑native solution. |
| **External OAuth** | Delegates token issuance to an **external** OAuth 2.0 authorisation server (e.g., Okta, Microsoft Entra ID). Enables centralised token management and allows integration of corporate authentication policies (MFA, biometrics, etc.) directly into the token issuance process[reference:0]. |

### 1.2. What is SAML 2.0?

SAML 2.0 (Security Assertion Markup Language) is an XML‑based standard for exchanging authentication and authorisation data between an **Identity Provider (IdP)** and a **Service Provider (SP)**. In a federated environment:

- **Snowflake serves as the Service Provider (SP)**[reference:1].
- Your corporate directory (Okta, Microsoft Entra ID, Ping Identity, etc.) serves as the **Identity Provider (IdP)**[reference:2].

When a user attempts to access Snowflake, they are redirected to the IdP for authentication. After successful authentication (which may include MFA enforced by the IdP), the IdP issues a SAML assertion that Snowflake validates, creating a Snowflake session.

### 1.3. OAuth vs SAML: When to Use Which

| Aspect | OAuth 2.0 | SAML 2.0 |
|--------|-----------|----------|
| **Primary use case** | Authorising applications (BI tools, custom apps, APIs) to access Snowflake | Browser‑based SSO for human users (Snowsight, SnowSQL, drivers) |
| **Token format** | JSON Web Token (JWT) – compact and web‑friendly | XML – verbose, enterprise‑grade |
| **Typical client** | Machine‑to‑machine or application with browser redirection | Browser or any client capable of HTTP redirect |
| **Session management** | Short‑lived access tokens + optional refresh tokens | Session tied to IdP; Snowflake validates each login via SAML assertion |
| **Best for** | Modern, API‑driven integrations where an app acts on behalf of a user or itself | Traditional corporate SSO where users expect to log in once via a portal |

> **Important**: These are not mutually exclusive. Many organisations configure both: **SAML** for human users logging into Snowsight or SnowSQL, and **External OAuth** for BI tools and service accounts.

### 1.4. Federated Environment Components

A federated authentication setup consists of:
- **Service Provider (SP)**: Snowflake – the system the user wants to access.
- **Identity Provider (IdP)**: The external entity that authenticates users. Snowflake supports most SAML 2.0‑compliant vendors, including Okta, Microsoft Entra ID (Azure AD), Google G Suite, OneLogin, and Ping Identity[reference:3].
- **Security Integration**: A Snowflake object (`TYPE = SAML2` or `TYPE = OAUTH`/`EXTERNAL_OAUTH`) that stores the configuration linking Snowflake to the IdP or OAuth server.

## 2. Required Privileges & Key Commands

### 2.1. Required Privileges

| Privilege | Scope | Description |
|-----------|-------|-------------|
| `CREATE INTEGRATION` | Account | Allows creation of security integrations. Granted to `ACCOUNTADMIN` by default. |
| `OWNERSHIP` on integration | Integration | Required to alter or drop a security integration. |

Only account administrators (`ACCOUNTADMIN`) or roles with the `CREATE INTEGRATION` global privilege can execute these commands[reference:4].

## 3. OAuth Commands & Configuration

### 3.1. Create a Snowflake OAuth Integration (Custom Client)

```sql
CREATE OR REPLACE SECURITY INTEGRATION my_oauth_int
  TYPE = OAUTH
  OAUTH_CLIENT = CUSTOM
  OAUTH_CLIENT_TYPE = 'CONFIDENTIAL'
  OAUTH_REDIRECT_URI = 'https://your-app.com/callback'
  OAUTH_ISSUE_REFRESH_TOKENS = TRUE
  OAUTH_REFRESH_TOKEN_VALIDITY = 86400
  OAUTH_ENFORCE_PKCE = TRUE
  OAUTH_USE_SECONDARY_ROLES = IMPLICIT
  BLOCKED_ROLES_LIST = ('ACCOUNTADMIN', 'SYSADMIN')
  PRE_AUTHORIZED_ROLES_LIST = ('ANALYST', 'READONLY')
  NETWORK_POLICY = 'corporate_network_policy';
```

**Key parameters**:
- `OAUTH_CLIENT_TYPE`: `CONFIDENTIAL` for server‑side apps (can keep client secret); `PUBLIC` for single‑page apps.
- `OAUTH_ENFORCE_PKCE = TRUE`: Enforces Proof Key for Code Exchange – **highly recommended** to prevent authorisation code interception attacks[reference:5].
- `OAUTH_REDIRECT_URI`: Where the OAuth server sends the user after granting access.
- `PRE_AUTHORIZED_ROLES_LIST`: Roles that can be pre‑authorised for the client.
- `BLOCKED_ROLES_LIST`: Roles that cannot be used with this OAuth integration.

### 3.2. Create an External OAuth Integration (Okta / Entra ID)

```sql
CREATE OR REPLACE SECURITY INTEGRATION external_oauth_okta
  TYPE = EXTERNAL_OAUTH
  ENABLED = TRUE
  EXTERNAL_OAUTH_TYPE = OKTA
  EXTERNAL_OAUTH_ISSUER = '<okta_issuer_url>'
  EXTERNAL_OAUTH_TOKEN_USER_MAPPING_CLAIM = 'email'
  EXTERNAL_OAUTH_SNOWFLAKE_USER_MAPPING_ATTRIBUTE = 'login_name'
  EXTERNAL_OAUTH_GROUPS_MAPPING_ATTRIBUTE = 'groups'
  EXTERNAL_OAUTH_ANY_ROLE_MODE = 'DISABLE'
  EXTERNAL_OAUTH_SCOPE_DELIMITER = ' '
  EXTERNAL_OAUTH_JWS_KEYS_URL = '<okta_jwks_url>';
```

**Key parameters**:
- `EXTERNAL_OAUTH_TYPE`: `OKTA`, `AZURE`, `PING_FEDERATE`, or `CUSTOM`.
- `EXTERNAL_OAUTH_TOKEN_USER_MAPPING_CLAIM`: The claim in the access token that identifies the user (e.g., `email`, `sub`).
- `EXTERNAL_OAUTH_SNOWFLAKE_USER_MAPPING_ATTRIBUTE`: The Snowflake user attribute to match against (`login_name` or `email_address`).
- `EXTERNAL_OAUTH_ANY_ROLE_MODE`: Controls whether any role in the token can be used (`ENABLE`, `DISABLE`, `ENABLE_FOR_PRIVILEGE`).

### 3.3. Obtain OAuth Tokens (Custom Client Flow)

After creating the integration, retrieve the **authorisation endpoint, token endpoint, client ID, and client secret** via Snowsight or by describing the integration:

```sql
DESC SECURITY INTEGRATION my_oauth_int;
```

Use these values to configure your client application to request tokens from Snowflake.

### 3.4. Manage OAuth Integrations

```sql
-- Show all OAuth integrations
SHOW SECURITY INTEGRATIONS;

-- Alter an existing integration
ALTER SECURITY INTEGRATION my_oauth_int SET OAUTH_REFRESH_TOKEN_VALIDITY = 43200;

-- Drop an integration
DROP SECURITY INTEGRATION my_oauth_int;
```

## 4. SAML Commands & Configuration

### 4.1. Create a SAML2 Security Integration

```sql
CREATE OR REPLACE SECURITY INTEGRATION my_saml_idp
  TYPE = SAML2
  ENABLED = TRUE
  METADATA_URL = 'https://your-idp.com/metadata'
  SAML2_ISSUER = 'https://your-idp.com/entity-id'
  SAML2_SSO_URL = 'https://your-idp.com/sso/saml'
  SAML2_PROVIDER = 'CUSTOM'
  SAML2_X509_CERT = '<IdP_signing_certificate_in_PEM_format>'
  SAML2_SP_INITIATED_LOGIN_PAGE_LABEL = 'My Corporate SSO'
  SAML2_ENABLE_SP_INITIATED = TRUE
  SAML2_SNOWFLAKE_ACS_URL = 'https://<account>.snowflakecomputing.com/fed/login'
  SAML2_SNOWFLAKE_ISSUER_URL = 'https://<account>.snowflakecomputing.com'
  ALLOWED_USER_DOMAINS = ('mycompany.com')
  SAML2_REQUESTED_NAMEID_FORMAT = 'urn:oasis:names:tc:SAML:1.1:nameid-format:emailAddress';
```

**Key parameters**[reference:6]:
- `METADATA_URL`: IdP metadata URL – **preferred** method as Snowflake can refresh configuration dynamically, simplifying certificate rotation[reference:7].
- `SAML2_ISSUER`: The EntityID of the IdP.
- `SAML2_SSO_URL`: The IdP's SSO URL where Snowflake redirects users.
- `SAML2_PROVIDER`: `OKTA`, `ADFS`, or `CUSTOM`.
- `SAML2_X509_CERT`: The IdP's signing certificate (required if not using metadata).
- `SAML2_ENABLE_SP_INITIATED`: Allows users to start SSO from Snowflake (instead of only from the IdP portal).
- `ALLOWED_USER_DOMAINS`: Restricts login to email addresses from specified domains.
- `ALLOWED_EMAIL_PATTERNS`: Regular expression patterns to further restrict which users can authenticate.

### 4.2. Using Metadata URL for Dynamic Refresh

If you specify `METADATA_URL`, Snowflake can refresh the IdP configuration without altering the integration:

```sql
ALTER SECURITY INTEGRATION my_saml_idp REFRESH METADATA_URL;
```

This command pulls the latest IdP metadata, including updated certificates, which simplifies certificate rotation[reference:8].

### 4.3. Manage SAML Integrations

```sql
-- Show SAML integrations
SHOW SECURITY INTEGRATIONS;

-- Alter an integration (e.g., rotate certificate)
ALTER SECURITY INTEGRATION my_saml_idp SET SAML2_X509_CERT = '<new_certificate>';

-- Disable SAML integration temporarily
ALTER SECURITY INTEGRATION my_saml_idp SET ENABLED = FALSE;
```

## 5. Impactful Use Cases (4–8 with Queries & Explanations)

### Use Case 1: Configure Snowflake OAuth for Tableau (Partner Application)

**Scenario** – You want Tableau users to authenticate to Snowflake via OAuth without managing separate Snowflake passwords. Tableau is a supported partner application with a built‑in OAuth client.

```sql
-- Step 1: Create the security integration for Tableau
CREATE OR REPLACE SECURITY INTEGRATION tableau_oauth
  TYPE = OAUTH
  OAUTH_CLIENT = TABLEAU_DESKTOP  -- or TABLEAU_SERVER, TABLEAU_CLOUD
  OAUTH_REDIRECT_URI = 'https://tableau.example.com/callback'
  ENABLED = TRUE
  OAUTH_ISSUE_REFRESH_TOKENS = TRUE
  OAUTH_REFRESH_TOKEN_VALIDITY = 86400
  OAUTH_USE_SECONDARY_ROLES = IMPLICIT
  BLOCKED_ROLES_LIST = ('ACCOUNTADMIN', 'SECURITYADMIN');

-- Step 2: Grant necessary roles to the Tableau service user (if using a service account)
GRANT ROLE ANALYST TO USER tableau_service_user;

-- Step 3: In Tableau, configure the connection to use Snowflake OAuth using the integration details
-- (Client ID, Client Secret, etc., obtained from DESC SECURITY INTEGRATION)
```

**Explanation** – This integration creates an OAuth interface that Tableau can use. When a user connects from Tableau, they are redirected to Snowflake's OAuth authorisation page, then back to Tableau with an access token. No Snowflake password is ever stored in Tableau. `OAUTH_CLIENT = TABLEAU_DESKTOP` tells Snowflake to use Tableau's predefined OAuth client profile[reference:9].

---

### Use Case 2: External OAuth with Okta for SSO (Machine‑to‑Machine)

**Scenario** – Your organisation uses Okta as the central identity provider. You want a service (e.g., an ETL job) to authenticate to Snowflake via Okta without any browser interaction, using the **Client Credentials** flow.

```sql
-- Step 1: Configure Okta as the external OAuth provider
CREATE OR REPLACE SECURITY INTEGRATION okta_ext_oauth
  TYPE = EXTERNAL_OAUTH
  ENABLED = TRUE
  EXTERNAL_OAUTH_TYPE = OKTA
  EXTERNAL_OAUTH_ISSUER = 'https://mycompany.okta.com/oauth2/default'
  EXTERNAL_OAUTH_TOKEN_USER_MAPPING_CLAIM = 'sub'  -- 'sub' is the Okta user ID
  EXTERNAL_OAUTH_SNOWFLAKE_USER_MAPPING_ATTRIBUTE = 'login_name'
  EXTERNAL_OAUTH_JWS_KEYS_URL = 'https://mycompany.okta.com/oauth2/default/v1/keys'
  EXTERNAL_OAUTH_ANY_ROLE_MODE = 'DISABLE';

-- Step 2: Create a service user in Snowflake that matches the Okta user
CREATE USER etl_service_user LOGIN_NAME = 'etl_service@mycompany.com'
  DEFAULT_ROLE = ETL_ROLE
  MUST_CHANGE_PASSWORD = FALSE;

-- Step 3: Grant appropriate role
GRANT ROLE ETL_ROLE TO USER etl_service_user;
```

**Explanation** – External OAuth delegates token issuance to Okta. The service obtains an access token directly from Okta (Client Credentials flow) and then uses that token to connect to Snowflake. Snowflake validates the token with Okta's JWKS endpoint and maps the user based on the `sub` claim[reference:10]. No password is ever created for the Snowflake user – authentication is fully delegated.

---

### Use Case 3: SAML SSO with Microsoft Entra ID (Azure AD)

**Scenario** – Your company uses Microsoft Entra ID (Azure AD) for identity management. You want all employees to log into Snowsight using their corporate credentials, with MFA enforced by Entra ID.

```sql
-- Step 1: Create the SAML2 security integration
CREATE OR REPLACE SECURITY INTEGRATION azure_saml
  TYPE = SAML2
  ENABLED = TRUE
  SAML2_ISSUER = 'https://sts.windows.net/your-tenant-id/'
  SAML2_SSO_URL = 'https://login.microsoftonline.com/your-tenant-id/saml2'
  SAML2_PROVIDER = 'CUSTOM'
  SAML2_X509_CERT = '-----BEGIN CERTIFICATE-----...-----END CERTIFICATE-----'
  SAML2_SP_INITIATED_LOGIN_PAGE_LABEL = 'Azure AD SSO'
  SAML2_ENABLE_SP_INITIATED = TRUE
  ALLOWED_USER_DOMAINS = ('mycompany.com')
  SAML2_SNOWFLAKE_ACS_URL = 'https://myorg-myaccount.snowflakecomputing.com/fed/login'
  SAML2_SNOWFLAKE_ISSUER_URL = 'https://myorg-myaccount.snowflakecomputing.com';

-- Step 2: Create a Snowflake user for an existing Entra ID user
CREATE USER jane_doe EMAIL = 'jane.doe@mycompany.com'
  MUST_CHANGE_PASSWORD = FALSE
  DEFAULT_ROLE = ANALYST;

-- Step 3: Grant appropriate role
GRANT ROLE ANALYST TO USER jane_doe;
```

**Explanation** – This configuration establishes trust between Snowflake and Entra ID. When a user navigates to Snowsight, they are redirected to Entra ID for authentication. Upon successful authentication (including any MFA configured in Entra ID), Entra ID sends a SAML assertion back to Snowflake. Snowflake validates the assertion and creates a session for the matching Snowflake user (matched by email address). The user never enters a Snowflake password[reference:11].

### Use Case 4: Role Mapping with External OAuth (Any‑Role Mode)

**Scenario** – You want to map OAuth scopes or group memberships directly to Snowflake roles, allowing dynamic role assignment based on the user's entitlements in the external IdP.

```sql
-- Step 1: Create external OAuth integration with ANY_ROLE_MODE enabled
CREATE OR REPLACE SECURITY INTEGRATION oauth_role_mapping
  TYPE = EXTERNAL_OAUTH
  ENABLED = TRUE
  EXTERNAL_OAUTH_TYPE = OKTA
  EXTERNAL_OAUTH_ISSUER = 'https://mycompany.okta.com/oauth2/default'
  EXTERNAL_OAUTH_JWS_KEYS_URL = 'https://mycompany.okta.com/oauth2/default/v1/keys'
  EXTERNAL_OAUTH_TOKEN_USER_MAPPING_CLAIM = 'email'
  EXTERNAL_OAUTH_SNOWFLAKE_USER_MAPPING_ATTRIBUTE = 'login_name'
  EXTERNAL_OAUTH_GROUPS_MAPPING_ATTRIBUTE = 'groups'   -- Map group claims to roles
  EXTERNAL_OAUTH_ANY_ROLE_MODE = 'ENABLE';

-- Step 2: The external OAuth server must include a "scope" or "groups" claim that
-- matches Snowflake role names (e.g., 'FINANCE_ANALYST', 'DATA_ENGINEER').
-- Snowflake will automatically use the first role in the token that the user is
-- authorised to assume.
```

**Explanation** – `EXTERNAL_OAUTH_ANY_ROLE_MODE = 'ENABLE'` allows any Snowflake role listed in the OAuth token's scope or groups claim to be used for the session[reference:12]. This enables dynamic role assignment: a user who is a member of the "finance" group in Okta automatically gets the `FINANCE_ANALYST` role in Snowflake without needing a separate grant. The alternative `ENABLE_FOR_PRIVILEGE` mode grants the user all roles in the token that they are authorised for, creating a multi‑role session.

### Use Case 5: SAML SSO with Multiple Identity Providers (Okta + Entra ID)

**Scenario** – Your organisation has two subsidiaries, each using a different IdP (Okta for subsidiary A, Entra ID for subsidiary B). Both need SSO access to the same Snowflake account.

```sql
-- Step 1: Create SAML integration for Okta
CREATE SECURITY INTEGRATION okta_saml
  TYPE = SAML2
  ENABLED = TRUE
  SAML2_ISSUER = 'https://subsidiaryA.okta.com'
  SAML2_SSO_URL = 'https://subsidiaryA.okta.com/app/snowflake/sso/saml'
  SAML2_PROVIDER = 'OKTA'
  ALLOWED_USER_DOMAINS = ('subsidiarya.com');

-- Step 2: Create SAML integration for Entra ID
CREATE SECURITY INTEGRATION azure_saml
  TYPE = SAML2
  ENABLED = TRUE
  SAML2_ISSUER = 'https://sts.windows.net/tenant-id-2/'
  SAML2_SSO_URL = 'https://login.microsoftonline.com/tenant-id-2/saml2'
  SAML2_PROVIDER = 'CUSTOM'
  ALLOWED_USER_DOMAINS = ('subsidiaryb.com');

-- Step 3: Users from subsidiary A are redirected to Okta; users from subsidiary B to Entra ID
-- Snowflake selects the appropriate integration based on the user's email domain.
```

**Explanation** – Snowflake supports configuring multiple SAML security integrations simultaneously[reference:13]. When a user attempts to log in, Snowflake evaluates the available integrations and directs the user to the appropriate IdP, typically based on email domain or login hint. This is ideal for mergers, acquisitions, or multi‑tenant scenarios where different user groups have different IdPs. Note that only a subset of Snowflake drivers (JDBC, ODBC, Python) currently support multiple identity providers[reference:14].

### Use Case 6: Rotate SAML IdP Certificate Without Downtime

**Scenario** – Your corporate IdP certificate is expiring soon. You need to rotate it without disrupting user access.

```sql
-- Step 1: If using METADATA_URL, simply refresh
ALTER SECURITY INTEGRATION my_saml_idp REFRESH METADATA_URL;
-- Snowflake automatically pulls the new certificate from the IdP's metadata.

-- Step 2: If using manual certificate (SAML2_X509_CERT), add the new certificate as a secondary value
ALTER SECURITY INTEGRATION my_saml_idp SET SAML2_X509_CERT = '<new_certificate>';
-- Snowflake accepts both old and new certificates for a grace period.

-- Step 3: After confirming all users can authenticate with the new certificate (and the old one expires),
-- remove the old certificate (optional, as Snowflake will prioritise the new one)
```

**Explanation** – Rotating SAML certificates can be tricky. Snowflake allows **two certificates** to be active simultaneously during a transition period. When you set `SAML2_X509_CERT` to a new value, Snowflake retains the previous certificate for a period of time, allowing both to be used for validation. This ensures that users with sessions that have not yet refreshed can still authenticate. Using `METADATA_URL` simplifies this further: `ALTER ... REFRESH METADATA_URL` pulls the latest IdP metadata, including the updated certificate, without any manual certificate handling[reference:15].

### Use Case 7: Disable Password Authentication for SAML‑Enrolled Users

**Scenario** – Once SAML SSO is configured, you want to enforce that users **must** authenticate via the IdP and cannot fall back to Snowflake passwords.

```sql
-- Step 1: Identify users who should use SSO exclusively
SELECT name, has_password 
FROM snowflake.account_usage.users 
WHERE email LIKE '%@mycompany.com';

-- Step 2: Remove password authentication for each user
ALTER USER jane_doe SET MUST_CHANGE_PASSWORD = FALSE;
ALTER USER jane_doe UNSET PASSWORD;  -- Removes the password entirely

-- Step 3: (Optional) Set the user's default authentication method to SAML via authentication policy
CREATE AUTHENTICATION POLICY saml_only_policy
  SECURITY_INTEGRATIONS = ('my_saml_idp')
  CLIENT_TYPES = ('SNOWFLAKE_UI', 'SNOWSQL', 'JDBC');
  
ALTER USER jane_doe SET AUTHENTICATION POLICY saml_only_policy;
```

**Explanation** – Removing the password from a user forces them to authenticate via the configured SSO integration. This is a critical security step after SSO rollout to prevent password‑based attacks on accounts that should be federated[reference:16]. The authentication policy further restricts which authentication methods are allowed for specific users.

### Use Case 8: Monitor OAuth and SAML Usage for Compliance

**Scenario** – Your security team needs a monthly report showing all authentication events, including which users used OAuth vs. SAML vs. password, and any failures.

```sql
-- Query authentication history from the last 30 days
SELECT 
    user_name,
    authentication_method,
    reported_client_type,
    start_time,
    is_success,
    error_message
FROM snowflake.account_usage.login_history
WHERE start_time >= DATEADD('day', -30, CURRENT_TIMESTAMP())
ORDER BY start_time DESC;

-- Specifically check OAuth token usage
SELECT 
    user_name,
    COUNT(*) AS oauth_authentications,
    MIN(start_time) AS first_oauth_use,
    MAX(start_time) AS last_oauth_use
FROM snowflake.account_usage.login_history
WHERE authentication_method IN ('OAUTH', 'EXTERNAL_OAUTH')
  AND start_time >= DATEADD('month', -3, CURRENT_TIMESTAMP())
GROUP BY user_name
ORDER BY oauth_authentications DESC;

-- Track SAML SSO logins
SELECT 
    user_name,
    COUNT(*) AS saml_logins,
    MIN(start_time) AS first_saml_login,
    MAX(start_time) AS last_saml_login
FROM snowflake.account_usage.login_history
WHERE authentication_method = 'SAML'
GROUP BY user_name;

-- Identify users who are not using SSO (still using passwords)
SELECT 
    user_name,
    COUNT(*) AS password_logins,
    MAX(start_time) AS last_password_login
FROM snowflake.account_usage.login_history
WHERE authentication_method = 'PASSWORD'
GROUP BY user_name
HAVING MAX(start_time) >= DATEADD('day', -7, CURRENT_TIMESTAMP());
```

**Explanation** – The `snowflake.account_usage.login_history` view captures every authentication attempt, including the method used (`PASSWORD`, `SAML`, `OAUTH`, `EXTERNAL_OAUTH`, `KEY_PAIR`). Use this data to ensure SSO adoption, detect unexpected password usage, and troubleshoot authentication failures. Regular monitoring of this view is a key compliance requirement for SOC2 and ISO 27001.

## 6. Best Practices Summary

| Practice | Why |
|----------|-----|
| **Use External OAuth for service accounts** | Delegates authentication to corporate IdP; eliminates Snowflake passwords for automated processes. |
| **Enable PKCE for Snowflake OAuth custom clients** | Prevents authorisation code interception attacks. Highly recommended for public clients[reference:17]. |
| **Use `METADATA_URL` for SAML integrations** | Simplifies certificate rotation; Snowflake can refresh configuration dynamically[reference:18]. |
| **Block privileged roles in OAuth integrations** | Use `BLOCKED_ROLES_LIST` to prevent OAuth clients from authenticating as `ACCOUNTADMIN` or `SECURITYADMIN`[reference:19]. |
| **Remove passwords from SSO‑enrolled users** | Prevents password‑based authentication fallback; reduces attack surface. |
| **Regularly rotate OAuth client secrets** | Limits exposure if a client secret is compromised. |
| **Monitor `login_history` for anomalies** | Detect failed authentication attempts, unexpected password usage, or OAuth token misuse. |
| **Use `PRE_AUTHORIZED_ROLES_LIST` wisely** | Limits which roles an OAuth client can obtain tokens for, enforcing least privilege[reference:20]. |

## 7. Important Limitations & Considerations

| Limitation | Mitigation |
|------------|------------|
| **SAML integrations using deprecated `SAML_IDENTITY_PROVIDER` parameter must be migrated** | Migrate to `CREATE SECURITY INTEGRATION TYPE = SAML2` before further configuration[reference:21]. |
| **Multiple IdP support is limited to certain drivers** | JDBC, ODBC, and Python drivers support multiple IdPs; other clients may not[reference:22]. |
| **OAuth client secrets are stored securely but can be rotated** | Use `ALTER SECURITY INTEGRATION ... SET OAUTH_CLIENT_SECRET = '<new_secret>'` for rotation. |
| **Network policies interact with OAuth** | As of recent changes, Snowflake no longer considers user‑level network policies when restricting traffic from the OAuth client[reference:23]. Test thoroughly. |
| **SAML assertion size limits** | Ensure SAML assertions are not overly large; Snowflake imposes size limits. |
| **Refresh token validity** | Maximum refresh token validity is configurable; consider your organisation's security requirements. |

## 8. Key Takeaways

- **OAuth 2.0** in Snowflake comes in two flavours: **Snowflake OAuth** (Snowflake as authorisation server) for partner apps and custom clients, and **External OAuth** (delegated to Okta, Entra ID, etc.) for centralised token management[reference:24].
- **SAML 2.0** enables federated SSO where Snowflake is the Service Provider and an external IdP (Okta, Entra ID, Ping Identity, etc.) handles authentication[reference:25].
- **Security integrations** are the Snowflake objects that store the configuration linking Snowflake to external identity providers.
- **External OAuth supports dynamic role mapping** via `EXTERNAL_OAUTH_ANY_ROLE_MODE`, allowing roles to be assigned based on IdP groups or scopes[reference:26].
- **Multiple SAML IdPs** can be configured in a single Snowflake account, with Snowflake directing users to the appropriate IdP based on email domain or login hint[reference:27].
- **Rotate certificates gracefully** using `METADATA_URL` and the two‑certificate transition period for SAML integrations[reference:28].
- **Audit regularly** using `snowflake.account_usage.login_history` to track authentication methods and ensure SSO adoption.

This completes the thorough explanation of Snowflake's OAuth and SAML federation model with practical, actionable use cases.
