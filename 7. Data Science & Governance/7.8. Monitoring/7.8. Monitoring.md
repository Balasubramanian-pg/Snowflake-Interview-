# Snowflake Monitoring: Maintaining Visibility in the Automated Data Pipeline

## 1. Core Concepts

Monitoring Snowflake Streams and Tasks is the critical practice of tracking execution health, data freshness, and credit consumption. Without robust monitoring, automated pipelines can suffer from "silent failures"—where tasks appear successful but process no data—or "runaway costs" caused by inefficient schedules or excessive retries.

### 1.1. How it Works
Snowflake provides a dual-layered monitoring architecture leveraging the **Information Schema** for real-time operational data and the **Account Usage** share for long-term auditing.
1. **Event Capture:** The Snowflake controller logs every task attempt, state transition (e.g., `SCHEDULED` to `EXECUTING`), and stream offset change.
2. **Metadata Persistence:** Operational metadata is pushed to `TASK_HISTORY` and `SERVERLESS_TASK_HISTORY` views.
3. **Stream Evaluation:** Monitoring the `METADATA$IS_UPDATE` and `METADATA$ROW_COUNT` pseudocolumns allows for tracking the volume of change data.
4. **DAG Visualization:** Snowflake's Snowsight UI reconstructs task dependencies to provide a visual representation of pipeline flow and bottlenecks.



### 1.2. Monitoring Evolution
| Metric | Legacy Log Monitoring | Snowflake Native Monitoring |
| :--- | :--- | :--- |
| **Data Source** | Text-based log files | Relational Metadata Views |
| **Visibility** | Requires parsing/ETL | Instant SQL Querying |
| **Retention** | Limited by disk space | Up to 365 days (Account Usage) [reference:1] |
| **Latency** | Batch-delayed | Near Real-Time (Info Schema) |

### 1.3. Key Factors
1. **Metadata Latency:** Information Schema views provide data for the last 7 days with sub-minute latency, while Account Usage views can have a delay of 45 minutes to 3 hours [reference:2].
2. **State Identification:** Monitoring must distinguish between `FAILED`, `SKIPPED` (due to overlap), and `CANCELLED` states.
3. **Credit Attribution:** Serverless task compute is billed separately from virtual warehouses; monitoring must aggregate both for true TCO.
4. **Stream Staleness:** Streams become "stale" if not consumed within the underlying table's data retention period.

## 2. Basic Commands & Privileges

### 2.1. Required Privileges
| Privilege | Scope | Description |
| :--- | :--- | :--- |
| **MONITOR USAGE** | Account | Required to view account-level credit consumption. |
| **MONITOR** | Task | Allows viewing the history and status of specific tasks. |
| **SELECT** | SNOWFLAKE Database | Required to query the `ACCOUNT_USAGE` schema for long-term audits. |

### 2.2. Querying Task History
Retrieves a list of recent task executions to verify success or failure.
```sql
-- View the last 50 task executions in the current database
SELECT * FROM TABLE(information_schema.task_history(result_limit => 50));
```

### 2.3. Checking Stream Metadata
Evaluates whether a stream has pending changes before executing logic.
```sql
-- Check if the stream currently contains change data
SELECT SYSTEM$STREAM_HAS_DATA('my_cdc_stream');
```

### 2.4. Viewing DAG Dependencies
Identifies the parent-child relationships within a complex task graph.
```sql
-- Recursively find all tasks dependent on a root task
SELECT * FROM TABLE(information_schema.task_dependents(
  task_name => 'root_ingest_task', recursive => TRUE));
```

### 2.5. Monitoring Serverless Costs
Specifically tracks credits consumed by the Snowflake-managed compute layer.
```sql
-- View credits spent on serverless tasks in the last 24 hours
SELECT * FROM TABLE(information_schema.serverless_task_history(
  date_range_start => DATEADD('day', -1, CURRENT_TIMESTAMP())));
```

### 2.6. Auditing Task Definitions
Reviews the SQL code and schedules currently active in the metadata.
```sql
-- List all tasks, their schedules, and assigned warehouses
SHOW TASKS IN SCHEMA prod_reporting;
```

## 3. Impactful Use Cases

### Use Case 1: Identifying "Silent" Data Freshness Gaps
**Scenario** – A task is running successfully on a schedule, but the downstream table is not being updated because the source Stream is empty.

```sql
-- Compare Task Success with Stream consumption
SELECT 
    t.task_name,
    t.completed_time,
    t.state,
    s.stale_status
FROM TABLE(information_schema.task_history()) t
JOIN information_schema.streams s ON s.table_name = 'RAW_LANDING'
WHERE t.state = 'SUCCEEDED' 
  AND t.completed_time < DATEADD('hour', -24, CURRENT_TIMESTAMP());
```
**Explanation** – This query joins task execution history with stream status. It helps identify "Ghost Runs" where the pipeline is healthy from an infrastructure perspective, but no business data is moving, indicating an upstream ingestion failure.

### Use Case 2: Detecting DAG Bottlenecks (Longest Path)
**Scenario** – A complex DAG is taking 2 hours to finish, and the team needs to identify which specific node is causing the delay.

```sql
-- Identify the longest-running task in the most recent DAG execution
SELECT 
    name, 
    query_start_time, 
    completed_time,
    DATEDIFF('second', query_start_time, completed_time) AS duration_sec
FROM SNOWFLAKE.ACCOUNT_USAGE.TASK_HISTORY
WHERE root_task_id = '<your_root_task_id>'
ORDER BY duration_sec DESC
LIMIT 1;
```
**Explanation** – By focusing on the `duration_sec` for each node within a specific `root_task_id`, engineers can pinpoint exactly which transformation is the "Critical Path" and requires optimization or a larger warehouse.

### Use Case 3: Monitoring Stream Staleness Risk
**Scenario** – Ensuring a stream does not exceed its retention period, which would require a full table re-sync.

```sql
-- Check if any streams are approaching their retention limit
SELECT 
    database_name, 
    schema_name, 
    name as stream_name, 
    source_name as table_name,
    stale_after
FROM SNOWFLAKE.ACCOUNT_USAGE.STREAMS
WHERE stale_after < DATEADD('day', 2, CURRENT_TIMESTAMP())
  AND state = 'ACTIVE';
```
**Explanation** – This proactive query identifies streams that will become stale in the next 48 hours. It allows the team to trigger an emergency manual consumption if the automated task has been suspended or failing.

### Use Case 4: Pipeline Cost Attribution per Business Unit
**Scenario** – Attributing Snowflake credit spend to specific projects like "Project Aegis" based on task naming conventions.

```sql
-- Aggregate serverless and warehouse costs by task prefix
SELECT 
    CASE 
        WHEN task_name ILIKE 'AEGIS_%' THEN 'Project Aegis'
        WHEN task_name ILIKE 'FIN_%' THEN 'Finance'
        ELSE 'Other' 
    END AS business_unit,
    SUM(credits_used) as total_credits
FROM SNOWFLAKE.ACCOUNT_USAGE.SERVERLESS_TASK_HISTORY
GROUP BY 1;
```
**Explanation** – Utilizing string patterns on the `task_name` allows for dynamic cost allocation. This is essential for internal chargebacks and managing the ROI of specific automated data products.

### Use Case 5: Overlap and Skip Rate Analysis
**Scenario** – Determining if a task schedule is too aggressive, leading to skipped runs and data backlogs.

```sql
-- Calculate the percentage of skipped tasks due to overlapping runs
SELECT 
    name,
    COUNT(*) AS total_attempts,
    COUNT(CASE WHEN state = 'SKIPPED' THEN 1 END) AS skip_count,
    ROUND((skip_count / total_attempts) * 100, 2) AS skip_rate_pct
FROM TABLE(information_schema.task_history(
    scheduled_time_range_start => DATEADD('day', -7, CURRENT_TIMESTAMP())))
GROUP BY 1
HAVING skip_rate_pct > 10;
```
**Explanation** – A high skip rate (e.g., >10%) indicates that the warehouse is under-provisioned or the task logic is inefficient. This query provides a data-driven justification for increasing warehouse size or extending the cron interval.

## 4. Best Practices Summary

| Practice | Why |
| :--- | :--- |
| **Use Root Task IDs** | Always group monitoring queries by `ROOT_TASK_ID` to see the DAG as a single unit. |
| **Monitor 'SKIPPED' States** | A skipped task often means the previous run is hanging; investigate the SQL for locks. |
| **Check Stream 'STALE_AFTER'** | Prevent permanent data loss by ensuring tasks consume streams before they expire. |
| **Alert on Failed Retries** | Only trigger high-priority alerts after `TASK_AUTO_RETRY_ATTEMPTS` are exhausted. |
| **Prefer Account Usage for Cost** | Use `SNOWFLAKE.ACCOUNT_USAGE` for monthly financial reporting as it includes late-arriving metrics. |
| **Validate 'WHEN' Clauses** | Ensure `SYSTEM$STREAM_HAS_DATA` is working; otherwise, tasks may run fruitlessly. |
| **Leverage Resource Monitors** | Wrap task warehouses in Resource Monitors to auto-suspend if a bug causes a cost spike. |
| **Tagging for Context** | Apply Snowflake tags (e.g., `owner`, `env`) to tasks to filter monitoring dashboards easily. |
| **Audit Suspended Tasks** | Regularly check for tasks that are `SUSPENDED` but still have active schedules. |

## 5. Important Limitations & Considerations

| Limitation | Mitigation |
| :--- | :--- |
| **7-Day History Limit** | `INFORMATION_SCHEMA` only keeps 7 days. Use `ACCOUNT_USAGE` for long-term trends [reference:1]. |
| **45min Metadata Delay** | `ACCOUNT_USAGE` is not real-time. Use `INFORMATION_SCHEMA` for immediate debugging. |
| **Task Limit in View** | `TASK_HISTORY()` returns a max of 20,000 rows. Use specific time filters to narrow results. |
| **Stream Transactionality** | Monitoring cannot "see" uncommitted changes in a stream until the source transaction closes. |

## 6. Auditing & Monitoring Queries

### 6.1. Hourly Credit Consumption Trend
```sql
SELECT 
    DATE_TRUNC('hour', start_time) as hour,
    SUM(credits_used) as credits
FROM SNOWFLAKE.ACCOUNT_USAGE.SERVERLESS_TASK_HISTORY
GROUP BY 1 ORDER BY 1 DESC;
```

### 6.2. Detailed Error Message Breakdown
```sql
SELECT 
    error_code, 
    error_message, 
    count(*) as occurrence_count
FROM TABLE(information_schema.task_history())
WHERE state = 'FAILED'
GROUP BY 1, 2 ORDER BY 3 DESC;
```

### 6.3. Tasks Without Assigned Warehouses (Serverless)
```sql
SELECT name, database_name, schema_name 
FROM information_schema.tasks 
WHERE warehouse_name IS NULL AND state = 'STARTED';
```

### 6.4. Identifying Orphaned Child Tasks
```sql
-- Find tasks with an AFTER clause where the parent no longer exists
SELECT name FROM information_schema.tasks
WHERE predecessors IS NOT NULL 
  AND state = 'STARTED'; 
```

## 7. Key Takeaways

* **Information Schema** is for **real-time troubleshooting**, while **Account Usage** is for **strategic auditing** and cost analysis.
* Monitoring **Stream Staleness** is the most critical preventative measure to avoid expensive table re-syncs.
* High **Skip Rates** are a primary indicator of scheduling conflicts or under-provisioned compute resources.
* **Serverless monitoring** requires specific views to track credits that are not captured by standard warehouse monitors.
* Always correlate **Task Success** with **Stream Data Movement** to ensure pipelines are providing actual business value.
