# Snowflake Task Error Handling: Building Fail-Safe Pipelines

## 1. Core Concepts

Error handling in Snowflake Tasks ensures that automated data pipelines are resilient to transient failures and that engineers are notified immediately when a process breaks. Unlike standard SQL scripts, Task error handling is decoupled, utilizing **Notification Integrations** to push metadata to cloud-native messaging services.

### 1.1. How it Works:
The error handling framework in Snowflake revolves around the proactive interception of failure states:
1. **Notification Integration:** A cloud-level object connects Snowflake to a messaging service (AWS SNS, Azure Event Grid, or GCP Pub/Sub).
2. **Task Linkage:** The `ERROR_INTEGRATION` parameter is applied to a task, instructing Snowflake to generate a JSON payload upon failure.
3. **Automated Retries:** Snowflake can be configured to automatically retry a failed task instance before marking it as `FAILED`.
4. **DAG Termination:** If a predecessor task in a **Directed Acyclic Graph (DAG)** fails, all downstream child tasks are automatically skipped to prevent data corruption.

### 1.2. Error Management Comparison
| Metric | Standard SQL Script | Snowflake Task Error Handling |
| :--- | :--- | :--- |
| **Notification** | Manual (UDF/Polling) | Native (Push-based via Integrations) |
| **Retry Logic** | Custom Loop Logic | Declarative (`TASK_AUTO_RETRY_ATTEMPTS`) |
| **Visibility** | `QUERY_HISTORY` only | Dedicated `TASK_HISTORY` + Cloud Alerts |
| **Cleanup** | Manual Rollback | Transactional Task semantics |

### 1.3. Key Factors
1. **Error Integration Limit:** A task can be associated with exactly **one** error integration at a time [reference:1].
2. **Auto-Retry Range:** The `TASK_AUTO_RETRY_ATTEMPTS` can be set between **0 and 30** [reference:2].
3. **Payload Content:** The notification includes the `query_id`, `error_code`, `error_message`, and `scheduled_time`.
4. **State Transition:** A task that fails all retry attempts moves to a `FAILED` state, which prevents downstream child tasks from executing.

## 2. Basic Commands & Privileges

### 2.1. Required Privileges
| Privilege | Scope | Description |
| :--- | :--- | :--- |
| **CREATE INTEGRATION** | Account | Required to create the Notification Integration object (Accountadmin typically). |
| **USAGE** | Integration | Required by the task owner to link the integration to a task. |
| **OWNERSHIP** | Task | Required to alter task parameters like `ERROR_INTEGRATION`. |

### 2.2. Creating a Notification Integration
Defines the bridge between Snowflake and an AWS SNS topic for error broadcasting.
```sql
CREATE OR REPLACE NOTIFICATION INTEGRATION aws_error_sns
  TYPE = QUEUE
  NOTIFICATION_PROVIDER = AWS_SNS
  DEVICE_STG_TOPIC_ARN = 'arn:aws:sns:us-east-1:1234567890:snowflake-errors'
  ENABLED = TRUE;
```

### 2.3. Linking Integration to an Existing Task
Applies the error integration to a task so that failures trigger cloud alerts.
```sql
ALTER TASK tsk_load_daily_sales 
SET ERROR_INTEGRATION = 'aws_error_sns';
```

### 2.4. Configuring Automated Retries
Instructs Snowflake to attempt the task up to 5 times if a transient error occurs.
```sql
ALTER TASK tsk_load_daily_sales 
SET TASK_AUTO_RETRY_ATTEMPTS = 5;
```

### 2.5. Setting a Custom Task Timeout
Prevents a "hung" task from consuming excessive credits by killing it after 30 minutes.
```sql
ALTER TASK tsk_load_daily_sales 
SET USER_TASK_TIMEOUT_MS = 1800000; -- Value in milliseconds
```

### 2.6. Checking Task State and Error Metadata
Retrieves the most recent failure details directly from the information schema.
```sql
SELECT 
    name, state, error_code, error_message, query_id
FROM TABLE(information_schema.task_history())
WHERE state = 'FAILED'
ORDER BY scheduled_time DESC;
```

## 3. Impactful Use Cases



### Use Case 1: Automated Slack Alerting via AWS SNS
**Scenario** – A critical regulatory reporting task must notify the Data Engineering team via Slack immediately upon failure.

```sql
-- 1. Create Integration (Assumes AWS SNS to Lambda to Slack setup)
CREATE NOTIFICATION INTEGRATION slack_alerts
    TYPE = QUEUE
    NOTIFICATION_PROVIDER = AWS_SNS
    DEVICE_STG_TOPIC_ARN = 'arn:aws:sns:us-east-1:000:slack-topic'
    ENABLED = TRUE;

-- 2. Apply to Root Task
ALTER TASK root_regulatory_report 
SET ERROR_INTEGRATION = 'slack_alerts';
```
**Explanation** – By attaching the `ERROR_INTEGRATION` to the Root Task, any failure in the primary ingestion trigger sends a structured JSON payload to SNS. A downstream Lambda function can then parse this to send a formatted Slack message.

### Use Case 2: Custom Error Logging Table
**Scenario** – For auditing purposes, all task failures must be logged into a persistent internal table for long-term trend analysis.

```sql
-- Procedure to wrap logic and catch errors
CREATE OR REPLACE PROCEDURE prc_task_with_logging()
RETURNS STRING
LANGUAGE SQL
AS
$$
BEGIN
    -- Main Business Logic
    INSERT INTO prod_table SELECT * FROM stage_table;
EXCEPTION
    WHEN OTHER THEN
        -- Log failure to persistent audit table
        INSERT INTO task_error_log (task_name, error_code, error_msg, log_time)
        VALUES ('tsk_prod_load', SQLCODE, SQLERRM, CURRENT_TIMESTAMP());
        RAISE; -- Re-raise to ensure the task state is marked as FAILED
END;
$$;
```
**Explanation** – While `ERROR_INTEGRATION` handles external alerts, using a `TRY...CATCH` block within a Stored Procedure allows for internal database auditing. **Re-raising** the error is critical to ensure the Task Engine recognizes the failure and stops the DAG.

### Use Case 3: The "Circuit Breaker" DAG
**Scenario** – A pipeline that stops all downstream processing if a Data Quality (DQ) check fails.

```sql
-- Task 1: Data Quality Check
CREATE OR REPLACE TASK tsk_dq_validate
    WAREHOUSE = 'DQ_WH'
    AFTER tsk_ingestion
AS
    -- If count is 0, throw an error to stop the DAG
    CALL prc_validate_row_counts(); 

-- Task 2: Downstream Load (Only runs if Task 1 succeeds)
CREATE OR REPLACE TASK tsk_final_load
    AFTER tsk_dq_validate
AS
    INSERT INTO final_table SELECT * FROM curated_view;
```
**Explanation** – Snowflake DAGs are natively "gated." If `tsk_dq_validate` encounters an error (e.g., zero rows found), it enters a `FAILED` state. The scheduler then automatically skips `tsk_final_load`, protecting the gold-layer data from corruption.

### Use Case 4: Transient Error Resilience with Exponential Backoff
**Scenario** – A task calling an external API via an External Function frequently fails due to rate limits.

```sql
ALTER TASK tsk_call_external_api
SET TASK_AUTO_RETRY_ATTEMPTS = 10;
-- Note: Snowflake handles the backoff timing internally
```
**Explanation** – Setting a high `TASK_AUTO_RETRY_ATTEMPTS` value allows the task to survive transient network issues or 429 "Too Many Requests" errors without manual intervention, significantly reducing "on-call" noise for engineers.

### Use Case 5: Dead Letter Queue (DLQ) Pattern for Streams
**Scenario** – When a task fails, the data in the source Stream is not consumed. We need to identify these "stale" records.

```sql
-- Query to identify unconsumed data in a stream due to task failure
SELECT * FROM strm_source_data
WHERE metadata$action = 'INSERT'
  AND NOT EXISTS (
    SELECT 1 FROM task_history 
    WHERE task_name = 'tsk_consumer' 
    AND state = 'SUCCEEDED' 
    AND completed_time > metadata$create_time
  );
```
**Explanation** – Because Stream offsets only advance on a successful **DML** within a task, a failure leaves the data in the Stream. This query acts as a "Dead Letter Queue" monitor to find data that hasn't been processed due to upstream Task failures.

## 4. Best Practices Summary

| Practice | Why |
| :--- | :--- |
| **Centralize Integrations** | Use one `ERROR_INTEGRATION` per environment (Prod/Dev) to simplify cloud configuration. |
| **Always Use Timeouts** | Set `USER_TASK_TIMEOUT_MS` to prevent infinite loops or frozen queries from wasting credits. |
| **Re-raise in Procs** | If using `TRY...CATCH` in procedures, always `RAISE` the error so the Task Engine sees the failure. |
| **Idempotent Logic** | Ensure your SQL can run multiple times (retries) without creating duplicate records. |
| **Resume Root Last** | When fixing a DAG, resume child tasks first to ensure the error-handling chain is ready. |
| **Monitor 'SKIPPED' State** | A skipped child task is often a symptom of an unhandled parent failure. |
| **Tagging for Costs** | Apply Tags to tasks to associate error-related compute costs with specific business units. |
| **Separate Warehouses** | Use a dedicated warehouse for error-prone tasks to isolate resource contention. |

## 5. Important Limitations & Considerations

| Limitation | Mitigation |
| :--- | :--- |
| **Payload Size** | Notifications are limited to ~200KB [reference:1]. Do not include large data samples in error messages. |
| **External Alert Latency** | Cloud provider (SNS/PubSub) latency can delay alerts by seconds. Do not use for sub-second monitoring. |
| **Retry Limitation** | `TASK_AUTO_RETRY_ATTEMPTS` only works for transient errors. Logic/Syntax errors fail immediately. |
| **Manual Resume Required** | If a task is altered (e.g., adding an integration), it must be manually resumed. |
| **One Integration per Task** | Use a cloud-side fan-out (e.g., SNS to multiple SQS/Emails) if multiple alerts are needed. |

## 6. Auditing & Monitoring Queries

### 6.1. Retrieve Most Recent Error Payloads
```sql
SELECT 
    name, 
    query_id, 
    error_code, 
    error_message, 
    scheduled_time
FROM TABLE(information_schema.task_history())
WHERE state = 'FAILED'
ORDER BY scheduled_time DESC
LIMIT 20;
```

### 6.2. Summarize Failure Rates by Task (Last 7 Days)
```sql
SELECT 
    name, 
    count(CASE WHEN state = 'FAILED' THEN 1 END) as failure_count,
    count(CASE WHEN state = 'SUCCEEDED' THEN 1 END) as success_count,
    round(failure_count / nullif(count(*), 0) * 100, 2) as failure_rate_pct
FROM TABLE(information_schema.task_history(
    scheduled_time_range_start=>DATEADD('day',-7,CURRENT_TIMESTAMP())))
GROUP BY 1
ORDER BY failure_rate_pct DESC;
```

### 6.3. Find Tasks That Timed Out
```sql
SELECT 
    name, 
    query_id, 
    query_start_time, 
    completed_time,
    DATEDIFF('minute', query_start_time, completed_time) as duration_minutes
FROM TABLE(information_schema.task_history())
WHERE error_message ILIKE '%timeout%'
ORDER BY duration_minutes DESC;
```

### 6.4. Monitor Notification Integration Usage
```sql
SELECT * FROM TABLE(information_schema.notification_history(
    endpoint=>'aws_error_sns',
    start_time=>DATEADD('hour',-24,CURRENT_TIMESTAMP())));
```

## 7. Key Takeaways

* **Notification Integrations** provide a robust, push-based alerting mechanism that connects Snowflake to modern DevOps tools.
* **DAG Gating** is an essential architectural feature; failures in parent tasks protect downstream systems from processing **invalid data**.
* **TASK_AUTO_RETRY_ATTEMPTS** should be leveraged for tasks interacting with external APIs or busy warehouses to handle **transient contention**.
* Proactive monitoring via `TASK_HISTORY` and `NOTIFICATION_HISTORY` is required to maintain a **high-maturity** data platform.
* Always wrap complex task logic in **Stored Procedures** with explicit exception handling to ensure both internal auditing and external alerting.
