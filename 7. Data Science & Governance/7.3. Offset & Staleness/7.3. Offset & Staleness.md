# Snowflake Stream Offset & Staleness: Managing Time Travel and CDC Lifecycle

## 1. Core Concepts

In Snowflake, a Stream is fundamentally a serverless metadata pointer—an **Offset**—that tracks the exact version of a table's micro-partitions since the stream was created or last consumed. This offset mechanism relies entirely on Snowflake’s **Time Travel** architecture. As long as the unconsumed micro-partition changes remain within the source table’s Time Travel retention window, the stream remains active and queryable. However, if the stream is not consumed before those unconsumed changes fall out of the retention window, the stream enters a state of **Staleness**. A stale stream is permanently broken; querying it will result in a fatal error, and the unconsumed CDC data is effectively lost to that stream, requiring manual intervention to rebuild the pipeline.

### 1.1. How it Works
The lifecycle of a stream's offset is deeply intertwined with Snowflake's transactional mechanics and data retention engine.

1. **Initialization:** Creating a stream establishes an initial offset pointing to the current, active version of the source table.
2. **Delta Accumulation:** As DML operations alter the source table, new micro-partition versions are created. The stream dynamically calculates the delta between its offset version and the current table version.
3. **Offset Advancement:** When a downstream process consumes the stream via a `SELECT` nested inside a committed DML operation (e.g., `INSERT`, `MERGE`), the stream's offset is destructively advanced to the exact transaction timestamp of that consumption.
4. **Automatic Retention Extension:** To protect against staleness, Snowflake automatically extends the source table's Time Travel retention up to **14 days** specifically for unconsumed streams, regardless of the underlying Snowflake edition [reference: Snowflake Data Retention Limits].
5. **Staleness Trigger:** If the stream remains unconsumed past the table's configured retention period (or the 14-day automatic extension limit), the historical micro-partitions required to calculate the delta are purged by the Time Travel engine. The offset is now invalid, and the stream is marked as **stale**.

### 1.2. Offset Lifecycle State Comparison

| Metric | Active Stream Offset | Stale Stream Offset |
| :--- | :--- | :--- |
| **Pointer Status** | Points to a valid Time Travel micro-partition | Points to a purged/expired micro-partition |
| **Query Execution** | Succeeds; returns the net-change delta | Fails instantly with a `Stale Stream` error |
| **Compute Overhead** | Negligible metadata evaluation | None (Execution is blocked) |
| **Offset Advancement** | Advances upon successful DML commit | Cannot be advanced; requires `CREATE OR REPLACE` |
| **Data Loss Risk** | Zero (Transactions guarantee delivery) | High (Unconsumed delta is permanently lost to the stream) |

### 1.3. Key Factors
1. **Transactional Boundaries:** An offset only advances if the consuming DML statement successfully commits. If the statement fails or explicitly rolls back, the offset remains anchored to its original point, preventing data loss.
2. **Table Recreation Limitations:** Executing `CREATE OR REPLACE TABLE` on the source table destroys the internal table ID. This immediately breaks all dependent streams, rendering them irrevocably stale regardless of Time Travel settings.
3. **The 14-Day Safety Net:** Even if a standard table has `DATA_RETENTION_TIME_IN_DAYS = 1`, an active stream on that table will automatically force Snowflake to retain the delta for up to 14 days. However, you will incur storage costs for retaining those historical micro-partitions.
4. **Manual Advancement:** Executing `CREATE OR REPLACE STREAM` manually resets the offset to `CURRENT_TIMESTAMP()`, effectively "flushing" the stream and discarding the previous unconsumed delta.

## 2. Basic Commands & Privileges

### 2.1. Required Privileges

| Privilege | Scope | Description |
| :--- | :--- | :--- |
| `OWNERSHIP` | Stream | Required to view staleness status, alter, or replace the stream. |
| `OWNERSHIP` | Source Table | Required to modify the underlying `DATA_RETENTION_TIME_IN_DAYS` setting. |
| `USAGE` | Database & Schema | Required to access the environment containing the stream and target tables. |

### 2.2. Check Stream Staleness Status
Queries the system to determine if a stream has expired and can no longer be queried.
```sql
SHOW STREAMS LIKE 'customer_cdc_stream';
```

### 2.3. Extend Base Table Time Travel
Explicitly increases the data retention period for the base table, effectively extending the maximum allowable unconsumed lifespan of any dependent streams (Enterprise Edition only).
```sql
ALTER TABLE raw_customers SET DATA_RETENTION_TIME_IN_DAYS = 30;
```

### 2.4. Safely Consume and Advance Offset
Advances the offset by physically materializing the delta into a persistent target table.
```sql
INSERT INTO target_customers SELECT * FROM customer_cdc_stream;
```

### 2.5. Flush Stream (Reset Offset to Current Time)
Recreates the stream, destroying the old offset and establishing a new one at the exact current micro-second, purging the unconsumed backlog.
```sql
CREATE OR REPLACE STREAM customer_cdc_stream ON TABLE raw_customers;
```

### 2.6. Clone a Stream (Offset Inheritance)
Clones a stream into a new environment. The cloned stream inherits the exact offset pointer of the original stream at the moment of cloning.
```sql
CREATE STREAM dev_stream CLONE customer_cdc_stream;
```

### 2.7. Time Travel Stream Initialization
Creates a new stream with an offset artificially anchored to a specific historical timestamp in the past, enabling CDC recovery.
```sql
CREATE STREAM recovered_cdc_stream ON TABLE raw_customers 
AT (TIMESTAMP => '2023-10-01 12:00:00 -0700'::TIMESTAMP_TZ);
```

## 3. Impactful Use Cases

### Use Case 1: Synchronizing Multiple Offsets via Transactions
* **Scenario** – A single `transactions_stream` must be loaded into both an `audit_log` table and an `aggregated_metrics` table. Because a `SELECT` inside a DML immediately advances the offset, the first `INSERT` will steal the data from the second.
* ```sql
  -- Wrap both consumption queries in a single explicit transaction block
  BEGIN;

  -- Read #1: Does NOT advance the offset yet, because the transaction hasn't committed
  INSERT INTO audit_log (txn_id, amount) 
  SELECT txn_id, amount FROM transactions_stream;

  -- Read #2: Reads the exact same delta payload as Read #1
  MERGE INTO aggregated_metrics tgt
  USING (
      SELECT region, SUM(amount) as total FROM transactions_stream GROUP BY region
  ) src ON tgt.region = src.region
  WHEN MATCHED THEN UPDATE SET tgt.total = tgt.total + src.total;

  -- The offset is destructively advanced for BOTH operations simultaneously upon commit
  COMMIT;
  ```
* **Explanation** – The stream offset is tightly bound to transaction isolation levels. By wrapping multiple DML operations in an explicit `BEGIN; ... COMMIT;` block, Snowflake pins the stream's offset for the duration of the transaction. All statements within the block see the exact same snapshot of the stream. Once the `COMMIT` is issued, the offset advances exactly once, guaranteeing that both downstream pipelines receive a perfectly synchronized CDC payload.

### Use Case 2: Disaster Recovery of a Stale Stream
* **Scenario** – A task failed continuously over a holiday weekend. The stream on `raw_events` became stale because the table's Time Travel explicitly expired, causing data loss for the CDC pipeline. You must recover the missed delta.
* ```sql
  -- Step 1: Identify the last known successful ingestion timestamp from the target table
  SET last_success_ts = (SELECT MAX(ingested_at) FROM target_events);

  -- Step 2: Recreate the stale stream using Time Travel to anchor the offset to the failure point
  CREATE OR REPLACE STREAM raw_events_stream 
  ON TABLE raw_events 
  AT (TIMESTAMP => $last_success_ts);

  -- Step 3: Resume the task; the stream will now calculate the delta from the recovered timestamp
  ALTER TASK process_events_task RESUME;
  ```
* **Explanation** – If a stream goes stale, standard consumption fails. However, if the underlying base table still has Time Travel history covering the gap (e.g., Enterprise edition set to 90 days), you can execute a `CREATE OR REPLACE STREAM ... AT` command. This forcefully points the new stream's offset to the historical timestamp just before the pipeline broke. When next queried, the stream instantly recalculates the missed delta, providing a seamless, zero-data-loss recovery mechanism.

### Use Case 3: Flusing a Corrupted Stream Backlog
* **Scenario** – A buggy upstream application accidentally ingested 5 billion garbage records into a staging table, filling the stream's unconsumed delta. The bad records were subsequently `DELETE`d from the base table, but because it's an `APPEND_ONLY` stream, it will still try to process the 5 billion garbage inserts.
* ```sql
  -- Suspend the consuming task to stop ingestion
  ALTER TASK ingest_telemetry SUSPEND;

  -- Flush the stream by recreating it. This instantly advances the offset to CURRENT_TIMESTAMP.
  -- The 5 billion bad rows in the backlog are completely ignored and left unconsumed.
  CREATE OR REPLACE STREAM telemetry_append_stream 
  ON TABLE raw_telemetry 
  APPEND_ONLY = TRUE;

  -- Resume normal operations on clean data
  ALTER TASK ingest_telemetry RESUME;
  ```
* **Explanation** – While standard streams net-collapse operations (meaning an insert followed by a delete disappears), Append-Only streams ignore deletes entirely. To prevent the downstream data warehouse from processing billions of garbage records, you must manually "flush" the stream. Recreating the stream object inherently destroys the old offset and creates a new one at the present moment, acting as an instantaneous purge of the CDC queue without actually altering any base table data.

### Use Case 4: Preventing Staleness in Infrequent Batch Pipelines
* **Scenario** – A stream captures changes from a fast-moving `inventory` table. The data engineering team only runs the downstream aggregation task once every week (every 7 days), placing the stream at high risk of staleness.
* ```sql
  -- Ensure the base table's Time Travel accommodates the infrequent stream consumption schedule
  ALTER TABLE inventory SET DATA_RETENTION_TIME_IN_DAYS = 8;
  
  -- Create the stream (Snowflake will automatically guarantee 14 days, but explicit config is safer)
  CREATE STREAM inventory_weekly_stream ON TABLE inventory;
  ```
* **Explanation** – A stream's offset survival depends entirely on the base table's retained micro-partitions. While Snowflake automatically extends the table's retention up to 14 days for unconsumed streams, relying on this invisible extension inflates storage costs unexpectedly. By explicitly aligning the base table's `DATA_RETENTION_TIME_IN_DAYS` with the task execution cadence (plus a buffer day for failure retries), engineers maintain predictable control over both storage spend and offset staleness mechanics.

### Use Case 5: Conditional Task Execution Based on Offset Advancement
* **Scenario** – A heavy transformation warehouse consumes a stream hourly. If no underlying data changed, the stream offset has not moved, yet the warehouse spins up and burns credits just to find 0 rows.
* ```sql
  -- Create a task that conditionally checks the offset state before allocating compute
  CREATE OR REPLACE TASK transform_sales_task
    WAREHOUSE = 'HEAVY_ETL_WH'
    SCHEDULE = '60 MINUTE'
    WHEN SYSTEM$STREAM_HAS_DATA('sales_stream') -- Evaluates the offset metadata at zero cost
  AS
    INSERT INTO final_sales 
    SELECT * FROM sales_stream;
  ```
* **Explanation** – The `SYSTEM$STREAM_HAS_DATA` function checks if the stream's current offset differs from the table's current version, meaning new unconsumed DML has occurred. Utilizing this function in the `WHEN` clause of a Snowflake Task evaluates the offset entirely within the Cloud Services layer (Serverless). If false, the task aborts before spinning up the Virtual Warehouse, saving immense compute credits over empty pipeline executions.

## 4. Best Practices Summary

| Practice | Why |
| :--- | :--- |
| **Wrap Multi-Target Inserts in Transactions** | A stream advances upon the *first* committed DML. To load one stream into multiple tables, wrap all `INSERT`s in a single `BEGIN; ... COMMIT;` block. |
| **Align Time Travel with Task Cadence** | If a task consumes a stream every 30 days, the base table's Time Travel MUST be explicitly set to >= 30 days, or the stream will irreversibly go stale. |
| **Avoid TRUNCATE on Streamed Tables** | While `TRUNCATE` does not break a stream like `DROP TABLE` does, it registers as a massive set of `DELETE` actions, flooding the stream offset with deletion metadata. |
| **Monitor Staleness Programmatically** | Do not wait for tasks to fail. Actively monitor `information_schema.streams` for `stale = TRUE` and alert the engineering team. |
| **Use SYSTEM$STREAM_HAS_DATA** | Always gate your consuming tasks with this function to prevent spinning up Virtual Warehouses just to read an empty offset. |
| **Do Not Create Streams 'Just in Case'** | Unconsumed streams force Snowflake to artificially extend the base table's Time Travel up to 14 days, inflating your storage bill with retained micro-partitions. |
| **Handle Schema Evolution Carefully** | If you add a column to a base table, the stream offset is unaffected, but downstream `INSERT` statements expecting an exact column mapping will break. |
| **Recreate Streams to Flush** | If upstream data is heavily corrupted and you want to abandon the CDC backlog, use `CREATE OR REPLACE STREAM` to fast-forward the offset to the present. |

## 5. Important Limitations & Considerations

| Limitation | Mitigation |
| :--- | :--- |
| **Irreversible Staleness** | Once a stream is marked stale, it cannot be resumed or repaired. You must `CREATE OR REPLACE` it, permanently losing the unconsumed delta. |
| **14-Day Automatic Retention Limit** | Snowflake only protects an unconsumed stream for a maximum of 14 days automatically. Beyond 14 days, explicit Time Travel (`>14`) is required on Enterprise Edition to prevent staleness. |
| **Base Table Recreation Destroys Streams** | Executing `CREATE OR REPLACE TABLE` on a base table changes its internal object ID. All streams attached to the original table instantly become stale. Use `INSERT OVERWRITE` instead. |
| **Clone Offset Reset** | Cloning a schema or database clones the streams within it, but their offsets do NOT carry over unconsumed data from the source. The cloned streams start at the exact timestamp of the clone operation. |
| **Read-Only Offset Advancement** | You cannot manually "nudge" an offset without consuming data. `SELECT * FROM stream` does not advance it. It must be inside a DML (`INSERT`, `UPDATE`, `DELETE`, `MERGE`). |

## 6. Auditing & Monitoring Queries

### Identify Stale Streams Across the Account
```sql
SELECT 
    table_catalog, 
    table_schema, 
    name AS stream_name, 
    table_name AS base_object,
    created_on
FROM snowflake.information_schema.streams
WHERE stale = TRUE
ORDER BY table_catalog, table_schema;
```

### Audit Streams Nearing the 14-Day Staleness Cliff
```sql
SELECT 
    s.name AS stream_name,
    s.table_name,
    t.retention_time AS base_table_retention_days,
    DATEDIFF(day, s.created_on, CURRENT_TIMESTAMP()) AS days_since_creation
FROM snowflake.information_schema.streams s
JOIN snowflake.information_schema.tables t 
  ON s.table_name = t.table_name 
 AND s.table_schema = t.table_schema
WHERE s.stale = FALSE 
  AND t.retention_time < 14
ORDER BY days_since_creation DESC;
```

### Review Recent Stream Consumption via Task History
```sql
SELECT 
    name AS task_name,
    state,
    error_message,
    query_text,
    completed_time
FROM snowflake.information_schema.task_history
WHERE query_text ILIKE '%FROM %_stream%'
ORDER BY completed_time DESC
LIMIT 100;
```

### Detect Potential Orphaned Streams Inflating Storage
```sql
-- Finds streams that exist but haven't been successfully consumed by tasks recently
SELECT 
    s.name AS stream_name, 
    s.table_schema,
    s.created_on
FROM snowflake.information_schema.streams s
LEFT JOIN snowflake.information_schema.task_history t 
  ON t.query_text ILIKE '%' || s.name || '%'
 AND t.completed_time >= DATEADD(day, -7, CURRENT_TIMESTAMP())
WHERE t.task_id IS NULL;
```

## 7. Key Takeaways
* A stream is not a physical table; it is an **Offset** pointer that dynamically calculates the net-change delta by analyzing the source table's underlying **Time Travel** micro-partitions.
* An offset strictly advances when consumed inside a **successfully committed DML transaction**. If the transaction rolls back or fails, the offset remains exactly where it was, preventing **Data Loss**.
* If a stream remains unconsumed for longer than the source table's Time Travel window, the required micro-partitions are purged, and the stream permanently transitions into a state of **Staleness**.
* To prevent accidental staleness, Snowflake natively and invisibly extends the retention period of the base table up to **14 days** if a stream attached to it has not been consumed.
* To route a single stream's payload into multiple target tables simultaneously, all consuming `INSERT` or `MERGE` statements must be strictly wrapped inside an **explicit `BEGIN; ... COMMIT;` transaction block**.
* If a stream becomes stale and data recovery is necessary, engineers can leverage Enterprise **Time Travel** to recreate the stream with an offset artificially anchored to a historical timestamp (`AT (TIMESTAMP => ...)`).
