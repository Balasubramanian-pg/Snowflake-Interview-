# Snowflake UNION Types: Optimizing Set Operations and Deduplication Overhead

## 1. Core Concepts

In Snowflake, **Set Operations** (`UNION`, `UNION ALL`, `INTERSECT`, `EXCEPT`) are used to combine the result sets of two or more independent queries into a single output stream. From a Performance Optimization perspective, the distinction between `UNION` and `UNION ALL` is one of the most critical and frequently misunderstood concepts. While both operators merge datasets, a standard `UNION` implicitly performs a highly expensive, global **deduplication** process across the entire merged dataset. Conversely, `UNION ALL` strictly performs a lightweight **concatenation**, blindly appending rows without analyzing them. Misusing `UNION` when datasets are already mutually exclusive is a primary cause of unnecessary compute utilization, memory exhaustion, and remote disk spillage.

### 1.1. How it Works
When the Snowflake query optimizer encounters a set operation, it builds an execution plan based on the specific operator chosen.

1. **Query Parsing & Alignment:** Snowflake executes the independent `SELECT` statements in parallel. It ensures that both result sets have the exact same number of columns and compatible data types in the same ordinal positions.
2. **Result Concatenation:** The compute engine streams the results of the subsequent queries and appends them to the results of the first query. If `UNION ALL` is used, the operation **stops here** and immediately returns the data to the user or downstream operator.
3. **Global Hash Aggregation (UNION only):** If `UNION` is used, Snowflake treats the operation as an implicit `SELECT DISTINCT *`. It routes the concatenated dataset into a massive hash table in memory to identify and remove identical rows.
4. **Spillage & Resolution (UNION only):** If the merged dataset is too large to fit in the Virtual Warehouse's active RAM, the hash aggregation spills to Local SSDs and Remote Cloud Storage, heavily degrading performance before finally returning the deduplicated output.

**Primary Use Cases:**
* Merging historical, static partitions with active, real-time partitions (Lambda Architecture).
* Refactoring complex `OR` / `IN` filtering clauses to force isolated micro-partition pruning.
* Combining disparate multi-tenant databases into a unified, flattened semantic view.

### 1.2. Set Operation Performance Comparison

| Metric | UNION ALL | UNION (Implicit DISTINCT) |
| :--- | :--- | :--- |
| **Execution Operation** | Simple Pointer Append / Streaming | Hash Aggregation + Sorting |
| **Compute Cost** | **Zero to Low** (negligible overhead) | **Extremely High** (scales with data volume) |
| **Memory Requirement** | Minimal (rows are passed through) | High (maintains a global hash table of all columns) |
| **Network Shuffle** | None required for the merge | Massive network shuffle to group identical rows |
| **Disk Spillage Risk** | Zero | High (on multi-terabyte datasets) |

### 1.3. Key Factors
1. **Mutual Exclusivity:** If you are unioning `2023_sales` and `2024_sales`, the datasets are chronologically mutually exclusive. A standard `UNION` wastes immense compute looking for duplicates that mathematically cannot exist.
2. **Column Width:** `UNION` deduplicates based on the **entire row**. If you union 50 columns, the hash table must evaluate all 50 columns for every single row, exploding memory payloads.
3. **Implicit Casting:** If column data types don't match perfectly (e.g., `NUMBER(38,0)` vs `VARCHAR`), Snowflake will attempt an implicit cast, which can disable vectorization and slow down the underlying `SELECT` scans.
4. **First Query Supremacy:** The column names in the final output are strictly dictated by the column aliases defined in the **first** `SELECT` block of the union sequence.

## 2. Basic Commands & Privileges

### 2.1. Required Privileges

| Privilege | Scope | Description |
| :--- | :--- | :--- |
| `USAGE` | Virtual Warehouse | Required to execute the compute operations for the underlying `SELECT` queries. |
| `SELECT` | All Target Tables | The user must have explicit read access to every table referenced in every branch of the set operation. |
| `USAGE` | Target Schemas/DBs| Required to establish the routing context to the underlying tables. |

### 2.2. UNION ALL (Concatenation)
Merges two datasets quickly without checking for duplicates, retaining all rows from both queries.
```sql
SELECT employee_id, department FROM active_employees
UNION ALL
SELECT employee_id, department FROM terminated_employees;
```

### 2.3. UNION (Deduplication)
Merges datasets and removes any rows where all selected columns perfectly match an existing row.
```sql
SELECT customer_email FROM marketing_list_a
UNION
SELECT customer_email FROM marketing_list_b;
```

### 2.4. EXCEPT (Set Difference)
Returns distinct rows from the first query that do **not** exist in the second query (synonymous with `MINUS`).
```sql
SELECT product_id FROM all_products
EXCEPT
SELECT product_id FROM discontinued_products;
```

### 2.5. INTERSECT (Set Intersection)
Returns only the distinct rows that exist in **both** the first and second queries.
```sql
SELECT user_id FROM q1_active_users
INTERSECT
SELECT user_id FROM q2_active_users;
```

## 3. Impactful Use Cases

### Use Case 1: The Golden Rule of Mutually Exclusive Datasets
* **Scenario** – A data engineer needs to create a view combining historical, immutable transaction data with the current month's active transaction data.
* ```sql
  CREATE OR REPLACE VIEW all_transactions AS
  -- Read from the massive historical archive
  SELECT transaction_id, amount, transaction_date 
  FROM transactions_archive
  
  UNION ALL -- STRICTLY use UNION ALL here
  
  -- Read from the active, mutating current month table
  SELECT transaction_id, amount, transaction_date 
  FROM transactions_current;
  ```
* **Explanation** – Because a transaction cannot simultaneously occur in the past and the present, these two tables are mutually exclusive. Using a standard `UNION` here would force Snowflake to load billions of archive rows into memory just to confirm no duplicates exist in the current month's table, destroying query performance. `UNION ALL` simply streams both tables to the user, operating with zero deduplication overhead.

### Use Case 2: Refactoring Expensive OR Clauses for Partition Pruning
* **Scenario** – A massive 10TB table queries slowly because an `OR` clause prevents the optimizer from effectively pruning micro-partitions, forcing a full table scan.
* ```sql
  -- Anti-Pattern: The OR clause forces a full scan across both dimensions
  -- SELECT * FROM massive_logs WHERE status = 'ERROR' OR region = 'EU';

  -- Best Practice: Split into UNION ALL for isolated, parallel micro-partition pruning
  SELECT * 
  FROM massive_logs 
  WHERE status = 'ERROR'
  
  UNION ALL
  
  SELECT * 
  FROM massive_logs 
  WHERE region = 'EU' 
    AND status != 'ERROR'; -- Prevents duplicates without using expensive UNION
  ```
* **Explanation** – Snowflake evaluates micro-partitions linearly. When `OR` conditions span different clustering dimensions, pruning breaks. By splitting the query using `UNION ALL`, the optimizer spins up two independent execution threads. Thread 1 aggressively prunes for `ERROR` status. Thread 2 aggressively prunes for `EU` region. The `AND status != 'ERROR'` guarantees mutual exclusivity, achieving identical logical output to the `OR` clause but completing in a fraction of the time.

### Use Case 3: Deterministic Deduplication via QUALIFY vs. UNION
* **Scenario** – You are combining two customer lists from different systems. The `customer_id` might overlap, but the `last_updated` timestamp differs. A standard `UNION` fails to deduplicate them because the timestamps don't match exactly.
* ```sql
  -- Step 1: Merge the raw data instantly using UNION ALL
  WITH combined_lists AS (
      SELECT customer_id, email, last_updated FROM crm_system_a
      UNION ALL
      SELECT customer_id, email, last_updated FROM crm_system_b
  )
  -- Step 2: Use QUALIFY to deduplicate deterministically based on ID
  SELECT * 
  FROM combined_lists
  QUALIFY ROW_NUMBER() OVER (
      PARTITION BY customer_id 
      ORDER BY last_updated DESC
  ) = 1;
  ```
* **Explanation** – `UNION` only removes rows that are 100% identical across *all* columns. If a customer exists in both systems with slightly different timestamps or spellings, `UNION` fails to deduplicate them. By utilizing `UNION ALL` followed by a `QUALIFY` window function, you execute a highly controlled, deterministic deduplication. You dictate exactly what defines a duplicate (`customer_id`) and exactly which version wins (`ORDER BY last_updated DESC`), bypassing the blind limitations of a standard `UNION`.

### Use Case 4: Circumventing Join Skew via Multi-Threaded Appends
* **Scenario** – A heavy `JOIN` operation is experiencing severe Data Skew because 40% of the events belong to `client_id = 999`, causing a single thread to run out of memory.
* ```sql
  -- Branch A: Process only the heavily skewed client
  SELECT e.event_id, c.client_name
  FROM events e JOIN clients c ON e.client_id = c.client_id
  WHERE e.client_id = 999
  
  UNION ALL
  
  -- Branch B: Process all other perfectly distributed clients
  SELECT e.event_id, c.client_name
  FROM events e JOIN clients c ON e.client_id = c.client_id
  WHERE e.client_id != 999;
  ```
* **Explanation** – By splitting the exact same join logic into two branches divided by `UNION ALL`, you force the Snowflake optimizer to create two entirely separate execution plans. The well-distributed clients process extremely fast across all nodes in the warehouse. The skewed client still processes on a single node, but because it is isolated from the rest of the dataset, it demands less localized memory and is less likely to spill to disk. 

### Use Case 5: Recursive CTEs for Hierarchical Data
* **Scenario** – An organization table contains parent-child relationships (`employee_id` and `manager_id`). You need to flatten this tree to see every employee's depth in the reporting chain.
* ```sql
  WITH RECURSIVE org_chart AS (
      -- Anchor Query: Start with the CEO (No Manager)
      SELECT employee_id, manager_id, 1 as depth
      FROM employees 
      WHERE manager_id IS NULL
      
      UNION ALL -- Required by Snowflake for recursive CTEs
      
      -- Recursive Query: Join employees back to the CTE
      SELECT e.employee_id, e.manager_id, o.depth + 1
      FROM employees e
      JOIN org_chart o ON e.manager_id = o.employee_id
  )
  SELECT * FROM org_chart ORDER BY depth;
  ```
* **Explanation** – Snowflake strictly enforces the use of `UNION ALL` within a `RECURSIVE CTE`. The anchor query executes once. The recursive query loops iteratively, continually appending (`UNION ALL`) its output back into the working result set until no more joins are found. A standard `UNION` is syntactically invalid here because evaluating cross-iteration duplicates during a live recursive loop would infinitely lock the hash table.

## 4. Best Practices Summary

| Practice | Why |
| :--- | :--- |
| **Default to UNION ALL** | Always assume `UNION ALL` is the correct choice unless you specifically and mathematically *need* full-row deduplication. |
| **Enforce Mutual Exclusivity** | Write your `WHERE` clauses to ensure overlapping queries return disjoint sets, making `UNION ALL` completely safe to use. |
| **Use QUALIFY for True Dedup** | Standard `UNION` fails if even one column (like an ingestion timestamp) differs. `QUALIFY ROW_NUMBER()` allows business-logic deduplication. |
| **Align Data Types Explicitly** | Do not rely on implicit casting. If Query 1 has a `DATE` and Query 2 has a `TIMESTAMP_NTZ`, explicitly `CAST()` them so the scanner doesn't stall. |
| **Apply ORDER BY Once at the End** | `ORDER BY` applied inside the individual `SELECT` branches of a union is ignored. Only apply sorting at the very end of the final query. |
| **Evaluate View Definitions** | Regularly audit your `CREATE VIEW` scripts. Developers frequently use `UNION` out of habit, permanently injecting severe compute overhead into every downstream query. |
| **Limit Columns Extracted** | If you must use `UNION` (Distinct), only `SELECT` the columns you absolutely need. Reducing column count drastically shrinks the global hash table footprint. |
| **Push Filters Down** | Apply `WHERE` filters *inside* the individual union blocks rather than outside the combined dataset to maximize micro-partition pruning. |

## 5. Important Limitations & Considerations

| Limitation | Mitigation |
| :--- | :--- |
| **Column Count Mismatch** | `UNION` operators will fail instantly if Branch A returns 5 columns and Branch B returns 6. Inject `NULL AS missing_col` into the shorter branch to align them. |
| **Whole-Row Strictness** | `UNION` treats rows `[1, 'A', 10:00]` and `[1, 'A', 10:01]` as completely distinct. It cannot deduplicate based on a primary key subset natively. |
| **Global Order By Limits** | Applying an `ORDER BY` and `LIMIT` to a massive `UNION ALL` forces Snowflake to materialize the *entire* combined dataset before truncating it. |
| **Data Type Collisions** | Unioning a `VARCHAR` containing text with a `NUMBER` will fail if implicit casting errors out. Strictly define column typing per branch. |
| **Remote Spillage on UNION** | Because `UNION` acts like `DISTINCT`, a 10TB union will attempt to build a 10TB hash table in RAM. This *will* spill to disk. Rewrite the logic using `UNION ALL`. |

## 6. Auditing & Monitoring Queries

### Identify Costly UNION Operators (Without ALL)
```sql
SELECT 
    query_id, 
    user_name, 
    query_text, 
    total_elapsed_time,
    bytes_spilled_to_remote_storage
FROM snowflake.account_usage.query_history
WHERE start_time >= DATEADD(day, -7, CURRENT_TIMESTAMP())
  AND query_text ILIKE '%UNION%' 
  AND query_text NOT ILIKE '%UNION ALL%'
ORDER BY bytes_spilled_to_remote_storage DESC;
```

### Find Views Constructed with Dangerous UNIONs
```sql
SELECT 
    table_catalog, 
    table_schema, 
    table_name as view_name, 
    view_definition
FROM snowflake.information_schema.views
WHERE view_definition ILIKE '%UNION%' 
  AND view_definition NOT ILIKE '%UNION ALL%';
```

### Detect Spillage Caused by Heavy Set Operations
```sql
SELECT 
    query_id,
    operator_type,
    execution_time_breakdown:overall_percentage::FLOAT AS time_percent,
    operator_statistics:spilled_bytes_local::NUMBER AS local_spill,
    operator_statistics:spilled_bytes_remote::NUMBER AS remote_spill
FROM TABLE(GET_QUERY_OPERATOR_STATS('your_query_id_here'))
WHERE operator_type IN ('Aggregate', 'Join') -- UNIONs manifest as global aggregates
ORDER BY time_percent DESC;
```

### Evaluate Sub-Query Execution Times
```sql
-- Use this strictly in the UI/Snowsight Profile, but programmatically via table functions:
SELECT 
    node_id, 
    step_id, 
    operator_type, 
    execution_time_breakdown 
FROM TABLE(GET_QUERY_OPERATOR_STATS(LAST_QUERY_ID()))
WHERE operator_type = 'UnionAll';
```

## 7. Key Takeaways
* **`UNION ALL`** is a highly efficient, zero-compute append operation. It should be the **default choice** for 99% of data engineering workloads.
* A standard **`UNION`** implicitly triggers a global **`DISTINCT`** hash aggregation, resulting in massive memory footprints, extreme compute overhead, and high risks of remote disk spillage.
* Always enforce **mutual exclusivity** via `WHERE` clauses (e.g., `status = 'A'` in branch 1, `status != 'A'` in branch 2) to render deduplication entirely mathematically unnecessary, allowing the safe use of `UNION ALL`.
* **Refactoring `OR` clauses** into distinct `UNION ALL` statements frequently allows the query optimizer to execute independent, highly efficient **micro-partition pruning** threads that parallelize flawlessly.
* When deduplication is genuinely required across overlapping data streams, prefer using **`UNION ALL` paired with `QUALIFY ROW_NUMBER()`**. This shifts deduplication from a blind, whole-row hash to deterministic, business-logic-driven control.
