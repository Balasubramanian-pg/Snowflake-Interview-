# Snowpipe Throughput: Through and Through

## 1. Core Concepts

Snowpipe is Snowflake's serverless data ingestion service that continuously loads data from cloud storage as files arrive. Unlike traditional `COPY INTO` commands that require manual execution, Snowpipe automates the loading process through event notifications from cloud storage providers. The service is designed for **near‑real‑time**, low‑latency ingestion without requiring an active, user‑managed warehouse.

### 1.1. How Snowpipe Works

When a file lands in a designated cloud storage location (AWS S3, Azure Blob, or GCS), Snowpipe receives an event notification and triggers a `COPY INTO` command to load the data. Snowpipe operates on a **serverless** model, meaning Snowflake automatically provisions and scales the compute resources required for ingestion. This is fundamentally different from a standard `COPY INTO` executed on a user‑managed warehouse, where you pay for the compute time.

#### Classic Snowpipe Process Flow (File‑Based)

1. **File Lands**: Data files are placed into an external stage (e.g., S3 bucket).
2. **Event Notification**: Cloud provider sends an event (S3 → SNS → SQS for AWS, Event Grid for Azure, Pub/Sub for GCP)[reference:0].
3. **Snowpipe Triggers**: The notification triggers Snowpipe to execute a `COPY INTO` command.
4. **Data Loaded**: Snowpipe reads the file, validates data, and inserts rows into the target table.

#### Key Architecture Limitations (Classic)

- **File‑Centric**: The smallest unit of ingestion is a file. Each file is processed as a micro‑batch.
- **Parallelism Constraint**: The number of parallel load operations cannot exceed the number of files being loaded[reference:1]. A single large file will be loaded by a single node, regardless of warehouse size.
- **Latency**: Typical end‑to‑end latency is 1–2 minutes after file arrival[reference:2].
- **Cost Model**: You are charged based on the volume of data loaded (credits per GB) plus a per‑file fee. There is also a fixed charge for the underlying cloud messaging infrastructure.

#### Snowpipe Streaming: A Modern Alternative

To address the limitations of file‑based ingestion, Snowflake introduced **Snowpipe Streaming** – an API‑based approach that writes rows directly into tables without requiring intermediate files.

- **Row‑Level Ingestion**: Data is streamed directly from client applications (Java/Python SDKs or REST API)[reference:3].
- **Low Latency**: Data is typically available for querying in under 10 seconds.
- **Throughput**: Designed to support ingest speeds of up to **10 GB per second per table**.
- **Cost Model**: Simple throughput‑based pricing – charge per uncompressed gigabyte ingested.
- **Multilanguage Support**: Java 11+, Python 3.9+, and a REST API[reference:4].

### 1.2. Snowpipe Throughput vs. Latency

| Metric | Classic Snowpipe (File‑Based) | Snowpipe Streaming (High‑Performance) |
|--------|-------------------------------|----------------------------------------|
| **Maximum Throughput** | Limited by file parallelism | Up to **10 GB/s per table** |
| **End‑to‑End Latency** | ~1–2 minutes after file arrival | **5–10 seconds** (typically under 10s) |
| **Minimum Data Unit** | File (micro‑batch) | Row (or batch of rows) |
| **Compute Model** | Serverless (auto‑scaled) | Serverless with throughput‑based pricing |
| **Cloud Storage Required** | Yes (external stage) | No (direct API writes) |
| **Best For** | Near‑real‑time file ingestion | True real‑time streaming (IoT, logs, CDC) |

### 1.3. Throughput Factors

Several factors directly influence Snowpipe throughput:

1. **File Size**: The most critical factor. Files between 100–250 MB (compressed) enable optimal parallelism and load efficiency[reference:5]. Very large files (e.g., 100 GB) are strongly discouraged because they cannot be split across multiple nodes.
2. **Number of Files**: Since each file can be loaded in parallel, more files generally mean higher throughput. However, too many tiny files introduce metadata overhead.
3. **File Format**: CSV, JSON, Parquet, Avro, and ORC are supported. For maximum throughput, use compressed formats (gzip, zstd) and avoid complex nesting.
4. **Transformation Logic**: In‑flight transformations using `COPY INTO` syntax add processing overhead.
5. **Warehouse Size (for Classic Snowpipe)**: Larger warehouses allow more files to be processed in parallel. However, warehouse size does **not** affect the speed of a single file – each file is loaded by a single node.
6. **Internal Load Queue**: Snowpipe has a built‑in queue that can accumulate files during peak loads. Snowpipe service‑side scaling may cause brief latency spikes[reference:6].
7. **Data Types**: Complex nested data (e.g., deeply nested JSON) requires more parsing time.

## 2. Basic Commands & Privileges

### 2.1. Required Privileges

| Privilege | Scope | Description |
|-----------|-------|-------------|
| `CREATE STAGE` | Schema | Create external stage referencing cloud storage |
| `CREATE PIPE` | Schema | Create pipe object for continuous ingestion |
| `MONITOR` on pipe | Pipe | View pipe status and history |
| `OPERATE` on pipe | Pipe | Pause or resume the pipe |

### 2.2. Create an External Stage

```sql
-- Create storage integration (one‑time setup)
CREATE OR REPLACE STORAGE INTEGRATION s3_int
  TYPE = EXTERNAL_STAGE
  STORAGE_PROVIDER = S3
  STORAGE_AWS_ROLE_ARN = 'arn:aws:iam::001234567890:role/myrole'
  ENABLED = TRUE
  STORAGE_ALLOWED_LOCATIONS = ('s3://mybucket/load/');

-- Create stage referencing the bucket
CREATE OR REPLACE STAGE my_snowpipe_stage
  URL = 's3://mybucket/load/'
  STORAGE_INTEGRATION = s3_int
  FILE_FORMAT = (TYPE = 'CSV' SKIP_HEADER = 1);
```

### 2.3. Create a Pipe for Continuous Ingestion

```sql
CREATE OR REPLACE PIPE my_pipe
  AUTO_INGEST = TRUE
  AS
  COPY INTO my_table
  FROM @my_snowpipe_stage
  FILE_FORMAT = (TYPE = 'CSV');
```

### 2.4. Refresh Pipe Metadata (After Schema Changes)

```sql
ALTER PIPE my_pipe REFRESH;
```

### 2.5. Pause / Resume a Pipe

```sql
ALTER PIPE my_pipe SET PIPE_EXECUTION_PAUSED = TRUE;
ALTER PIPE my_pipe SET PIPE_EXECUTION_PAUSED = FALSE;
```

### 2.6. Check Pipe Status

```sql
SELECT SYSTEM$PIPE_STATUS('my_db.my_schema.my_pipe');
```

### 2.7. Show Load History

```sql
SELECT *
FROM TABLE(INFORMATION_SCHEMA.COPY_HISTORY(
    TABLE_NAME => 'MY_TABLE',
    START_TIME => DATEADD('hour', -24, CURRENT_TIMESTAMP())
));
```

### 2.8. Monitor Pipe Usage (ACCOUNT_USAGE)

```sql
SELECT *
FROM snowflake.account_usage.pipe_usage_history
WHERE pipe_name = 'MY_PIPE'
ORDER BY start_time DESC;
```

## 3. Impactful Use Cases (4–8 with Queries & Explanations)

### Use Case 1: Classic Snowpipe Setup for Real‑Time File Ingestion (IoT Data)

**Scenario** – IoT sensors continuously upload JSON files to an S3 bucket. You need these files to be loaded into Snowflake within 2–3 minutes of arrival without manual intervention.

```sql
-- Step 1: Create storage integration (AWS S3)
CREATE OR REPLACE STORAGE INTEGRATION iot_s3_int
  TYPE = EXTERNAL_STAGE
  STORAGE_PROVIDER = S3
  STORAGE_AWS_ROLE_ARN = 'arn:aws:iam::123456789012:role/snowflake_ingest_role'
  ENABLED = TRUE
  STORAGE_ALLOWED_LOCATIONS = ('s3://my-iot-bucket/sensor-data/');

-- Step 2: Create external stage
CREATE OR REPLACE STAGE iot_stage
  URL = 's3://my-iot-bucket/sensor-data/'
  STORAGE_INTEGRATION = iot_s3_int
  FILE_FORMAT = (TYPE = 'JSON')
  ENCRYPTION = (TYPE = 'AWS_SSE_S3');

-- Step 3: Create target table
CREATE OR REPLACE TABLE iot_sensor_readings (
  sensor_id INT,
  reading_timestamp TIMESTAMP_NTZ,
  temperature FLOAT,
  humidity FLOAT,
  ingested_at TIMESTAMP_NTZ DEFAULT CURRENT_TIMESTAMP()
);

-- Step 4: Create pipe with auto‑ingest enabled
CREATE OR REPLACE PIPE iot_pipe
  AUTO_INGEST = TRUE
  AS
  COPY INTO iot_sensor_readings (sensor_id, reading_timestamp, temperature, humidity)
  FROM (
    SELECT
      $1:sensor_id::INT,
      TO_TIMESTAMP_NTZ($1:timestamp::STRING),
      $1:temperature::FLOAT,
      $1:humidity::FLOAT
    FROM @iot_stage
  )
  FILE_FORMAT = (TYPE = 'JSON');
```

**Explanation** – This pipe listens for S3 event notifications (via SQS) and automatically loads new JSON files. Each file is processed as a micro‑batch. For optimal throughput, ensure files are **100–250 MB compressed** and that you have configured S3 bucket notifications to send events to the SQS queue that Snowflake monitors.

### Use Case 2: High‑Throughput Streaming with Snowpipe Streaming (Java SDK)

**Scenario** – A financial trading application generates 500,000 transactions per second. You need to ingest this data into Snowflake with sub‑10‑second latency and exactly‑once semantics.

```java
// Java SDK example (Snowpipe Streaming high‑performance)
import com.snowflake.ingest.streaming.*;

// 1. Create a StreamingClient
StreamingClient client = StreamingClient.builder("account", "user", "private_key")
    .clientName("trading_client")
    .build();

// 2. Open a channel to the target table
Channel channel = client.openChannel(
    OpenChannelRequest.builder("TRANSACTIONS_DB", "PUBLIC", "TRANSACTIONS_TABLE")
        .withOnErrorOption(OnErrorOption.CONTINUE)
        .build()
);

// 3. Insert rows in batches (10–16 MB compressed per batch)
List<Map<String, Object>> rows = new ArrayList<>();
for (Transaction txn : transactions) {
    Map<String, Object> row = new HashMap<>();
    row.put("txn_id", txn.getId());
    row.put("amount", txn.getAmount());
    row.put("timestamp", txn.getTimestamp());
    rows.add(row);
}
InsertRowsResponse response = channel.insertRows(rows, "batch_001");

// 4. Check for errors
if (response.hasErrors()) {
    // Handle errors (logged, retry, etc.)
}
```

**Explanation** – The Snowpipe Streaming SDK writes rows directly to Snowflake tables without intermediate files. The service automatically flushes data every second. For maximum throughput, aggregate data from multiple sources into fewer clients and batch rows into the optimal size (10–16 MB compressed). Use `OnErrorOption.CONTINUE` for better performance and manually check return values for errors[reference:7].

### Use Case 3: Monitor Throughput and Ingestion Health with SQL

**Scenario** – Your data engineering team needs a dashboard to monitor Snowpipe ingestion throughput, error rates, and latency over the last 24 hours.

```sql
-- Monitor pipe usage and credits consumed
SELECT 
    TO_DATE(start_time) AS date,
    pipe_name,
    SUM(credits_used) AS total_credits,
    SUM(credits_used_cloud_services) AS cloud_service_credits,
    SUM(credits_used_serverless) AS serverless_credits,
    COUNT(*) AS load_batches
FROM snowflake.account_usage.pipe_usage_history
WHERE start_time >= DATEADD('day', -1, CURRENT_TIMESTAMP())
GROUP BY date, pipe_name
ORDER BY total_credits DESC;

-- Check COPY history for specific files (error detection)
SELECT 
    file_name,
    file_size,
    rows_loaded,
    rows_parsed,
    errors_seen,
    first_error_message,
    status,
    last_load_time
FROM TABLE(INFORMATION_SCHEMA.COPY_HISTORY(
    TABLE_NAME => 'IOT_SENSOR_READINGS',
    START_TIME => DATEADD('hour', -6, CURRENT_TIMESTAMP())
))
WHERE status = 'LOAD_FAILURE'
ORDER BY last_load_time DESC;

-- View channel‑level metrics for Snowpipe Streaming
SELECT 
    pipe_name,
    table_name,
    created_on,
    rows_inserted,
    rows_parsed,
    rows_error_count,
    snowflake_processing_latency_ms
FROM snowflake.account_usage.snowpipe_streaming_channel_history
WHERE created_on >= DATEADD('hour', -24, CURRENT_TIMESTAMP())
ORDER BY created_on DESC;
```

**Explanation** – These queries leverage Snowflake’s `ACCOUNT_USAGE` schema and system functions to provide visibility into ingestion throughput, credit consumption, and error patterns. The `PIPE_USAGE_HISTORY` view shows credits used per pipe; the `COPY_HISTORY` function shows per‑file load details; the `SNOWPIPE_STREAMING_CHANNEL_HISTORY` view provides channel‑level metrics including average processing latency[reference:8].

```sql
-- Step 1: Identify average file size currently processed
SELECT 
    file_name,
    file_size / 1024 / 1024 AS file_size_mb

    TABLE_NAME => 'MY_TABLE',
    START_TIME => DATEADD('day', -1, CURRENT_TIMESTAMP())
))
ORDER BY file_size_mb DESC;

-- Step 2: (External to Snowflake) Implement file aggregation logic
-- Example: Using AWS Lambda to concatenate small JSON files into larger batches
-- Batch size target: 100–250 MB compressed
```

**Explanation** – Snowflake can process multiple files in parallel, but each file incurs a small fixed overhead. The recommended file size range for optimal throughput and cost is **100–250 MB compressed** per file[reference:9]. Files that are too small (e.g., <10 MB) waste resources on metadata overhead. Files that are too large (e.g., >5 GB) cannot be processed in parallel and are prone to timeout or failure. For continuous streaming, aggregate data for 1–2 minutes into a single file before uploading.

### Use Case 5: Achieve Exactly‑Once Semantics with Snowpipe Streaming

**Scenario** – Your CDC (Change Data Capture) pipeline processes financial transactions. Duplicate inserts are unacceptable. Snowpipe Streaming provides offset tokens to enable exactly‑once delivery.

```java
// Java SDK: Track offset tokens for exactly‑once semantics
String offsetToken = "transaction_batch_2025-03-15T10:00:00Z";

// Insert rows with a unique offset token
InsertRowsResponse response = channel.insertRows(rows, offsetToken);

if (response.hasErrors()) {
    // On retry, re‑send the same offset token – Snowflake will deduplicate
    channel.insertRows(rows, offsetToken);
}

// Retrieve current offset for checkpointing
String currentOffset = channel.getLatestCommittedOffsetToken();
// Store currentOffset externally (e.g., in Kafka or database) for recovery
```

**Explanation** – The high‑performance architecture for Snowpipe Streaming supports exactly‑once delivery through offset tokens[reference:10]. When you call `insertRows` with a unique offset token, Snowflake ensures that rows associated with that token are inserted exactly once, even if the client retries. After a failure, you can replay the same offset token, and Snowflake will skip duplicate rows. This is critical for financial, healthcare, and audit‑trail applications.

### Use Case 6: Apply In‑Flight Transformations with PIPE Object (Snowpipe Streaming)

**Scenario** – You need to cleanse, filter, or reshape data as it streams into Snowflake, without storing raw data in intermediate tables.

```sql
-- Create a pipe with in‑flight transformations (Snowpipe Streaming)
CREATE OR REPLACE PIPE transformed_ingestion_pipe
  AS
  COPY INTO analytics_table (
    user_id,
    event_type,
    event_timestamp,
    session_duration_seconds,
    normalized_country
  )
  FROM (
    SELECT
      $1:user_id::INT AS user_id,
      UPPER($1:event_type::STRING) AS event_type,
      TO_TIMESTAMP_NTZ($1:timestamp::STRING) AS event_timestamp,
      $1:duration_seconds::INT AS session_duration_seconds,
      CASE
        WHEN $1:country::STRING = 'USA' THEN 'United States'
        WHEN $1:country::STRING = 'UK' THEN 'United Kingdom'
        ELSE $1:country::STRING
      END AS normalized_country
    FROM @streaming_stage
  )
  WHERE session_duration_seconds > 0
  FILE_FORMAT = (TYPE = 'JSON');
```

**Explanation** – The high‑performance architecture for Snowpipe Streaming supports in‑flight transformations using the same `COPY INTO` syntax as batch loading[reference:11]. Transformations run on the Snowflake server side, reducing client‑side complexity and improving throughput. You can filter rows (using `WHERE`), cast data types, apply functions, and rename columns – all during ingestion.

### Use Case 7: Set Up Alerts for Ingestion Delays and Failures

**Scenario** – Your operations team needs to be alerted when a pipe fails to ingest files for more than 5 minutes.

```sql
-- Step 1: Create an event table for pipe events
CREATE OR REPLACE EVENT TABLE pipe_monitor_events;

-- Step 2: Configure pipe to log events to the event table
ALTER PIPE my_pipe SET EVENT_TABLE = pipe_monitor_events;

-- Step 3: Query the event table for recent errors
SELECT 
    TIMESTAMP,
    RECORD['name'] AS event_name,
    VALUE['state'] AS file_state,
    VALUE['file_content_key'] AS file_path
FROM pipe_monitor_events
WHERE RECORD['name'] = 'file_lifecycle'
  AND VALUE['state'] = 'error'
  AND TIMESTAMP >= DATEADD('minute', -30, CURRENT_TIMESTAMP());

-- Step 4: (External to Snowflake) Use a task to monitor and send alerts
CREATE OR REPLACE TASK check_pipe_errors
  WAREHOUSE = my_warehouse
  SCHEDULE = '5 MINUTE'
AS
  CALL system$send_alert_if_pipe_stalled('my_pipe', 300);  -- Custom stored procedure
```

**Explanation** – Snowpipe can emit detailed events to an active event table, including file lifecycle events (`RECEIVED`, `INGESTED`, `ERRORED`)[reference:12]. By configuring an event table and querying it, you can track ingestion progress and detect failures. For automated alerting, you can use a Snowflake Task to run a custom procedure that checks the event table and sends notifications (e.g., via email, Slack, or PagerDuty) using external functions.

### Use Case 8: Scale Classic Snowpipe for Burst Throughput with Multi‑File Parallelism

**Scenario** – You have a batch of 1,000 files arriving every 15 minutes. You want to maximise parallel load throughput by splitting files appropriately.

```sql
-- Step 1: Ensure files are split into the optimal size range (100–250 MB)
-- (Perform this in your ETL pipeline before uploading to cloud storage)

-- Step 2: Use a larger warehouse to process more files in parallel
-- The number of parallel loads equals the number of files (or warehouse nodes, whichever is lower)

-- Step 3: Monitor parallelism and queue depth
SELECT 
    SYSTEM$PIPE_STATUS('my_pipe') AS pipe_status_json;

-- Step 4: Verify that files are being processed concurrently
SELECT 
    file_name,
    last_load_time,
    DATEDIFF('second', last_load_time, CURRENT_TIMESTAMP()) AS seconds_since_load
FROM TABLE(INFORMATION_SCHEMA.COPY_HISTORY(
    TABLE_NAME => 'MY_TABLE',
    START_TIME => DATEADD('minute', -15, CURRENT_TIMESTAMP())
))
ORDER BY last_load_time DESC;
```

**Explanation** – Classic Snowpipe’s throughput is directly tied to the **number of files** being loaded in parallel. If you have 1,000 files of optimal size, Snowpipe can load many of them concurrently, dramatically reducing total processing time. However, if you have a single large file, only one node processes it, regardless of warehouse size[reference:13]. The best practice is to split large datasets into many files of **100–250 MB each** to leverage full parallelism.

## 4. Best Practices Summary

| Practice | Why |
|----------|-----|
| **Optimise file size (100–250 MB compressed)** | Enables full parallelism; balances metadata overhead with load efficiency |
| **Split large files into many smaller ones** | Allows multiple warehouse nodes to process data concurrently |
| **Use Snowpipe Streaming for sub‑10‑second latency** | Direct row‑level ingestion eliminates file staging overhead |
| **Aggregate data from multiple sources into fewer clients** | Reduces client‑side overhead and per‑client costs[reference:14] |
| **Batch rows into 10–16 MB chunks (Streaming SDK)** | Minimises lock contention and improves throughput[reference:15] |
| **Set `ON_ERROR = CONTINUE` for streaming workloads** | Avoids exception‑handling overhead and allows manual error checking[reference:16] |
| **Use offset tokens for exactly‑once delivery** | Prevents duplicate rows in financial and audit‑critical pipelines |
| **Monitor `PIPE_USAGE_HISTORY` and `COPY_HISTORY` daily** | Detects bottlenecks, error spikes, and cost anomalies early |
| **Keep channels long‑lived; do not close after each insert** | Retains offset token information; data flushes automatically every second[reference:17] |
| **Use event tables for proactive alerting** | Receive notifications for file errors or ingestion delays |

## 5. Important Limitations & Considerations

| Limitation | Mitigation |
|------------|------------|
| **Classic Snowpipe: One file → one node** | Split large files into 100–250 MB chunks to enable parallelism |
| **File size >5 GB not recommended** | Split into smaller files before loading; large files risk timeout |
| **Maximum pipe objects per account (high‑performance)** | 1,000 pipes per account; 10 pipes per table[reference:18] |
| **Maximum channels per pipe** | 2,000 channels by default; contact Snowflake Support for more[reference:19] |
| **SDK throughput limits (high‑performance)** | 12 MBps (uncompressed) per channel; REST: 1 MBps observed[reference:20] |
| **REST payload limit** | 4 MB per request; use compression (Gzip/ZSTD) to fit more data[reference:21] |
| **ON_ERROR = CONTINUE only (high‑performance)** | ABORT is not supported; must manually check errors via SDK return value[reference:22] |
| **Burst latency spikes** | Sudden throughput increases may cause brief latency increases as the service scales elastically[reference:23] |
| **MATCH_BY_COLUMN_NAME not supported with auto‑increment columns** | Explicitly list columns in `COPY INTO`; omit auto‑generated columns from the target list[reference:24] |
| **Shared databases** | Snowpipe cannot load into tables in shared databases (Snowflake Marketplace) |
| **Region availability (high‑performance)** | GA on AWS; Azure and GCP support coming soon[reference:25] |

## 6. Auditing & Monitoring Queries

### View pipe status (JSON)

```sql
SELECT SYSTEM$PIPE_STATUS('my_db.my_schema.my_pipe');
```

### Track file load history per pipe

```sql
SELECT 
    pipe_name,
    file_name,
    rows_loaded,
    errors_seen,
    last_load_time
FROM snowflake.account_usage.copy_history
WHERE pipe_name = 'MY_PIPE'
  AND last_load_time >= DATEADD('day', -7, CURRENT_TIMESTAMP())
ORDER BY last_load_time DESC;
```

### Monitor daily Snowpipe credits by pipe

```sql
SELECT 
    TO_DATE(start_time) AS ingestion_day,
    pipe_name,
    SUM(credits_used) AS total_credits,
    SUM(credits_used_serverless) AS serverless_credits,
    COUNT(*) AS loads
FROM snowflake.account_usage.pipe_usage_history
WHERE start_time >= DATEADD('month', -1, CURRENT_TIMESTAMP())
GROUP BY ingestion_day, pipe_name
ORDER BY ingestion_day DESC, total_credits DESC;
```

### Identify stalled pipes (no activity for 2 hours)

```sql
SELECT 
    pipe_name,
    MAX(last_load_time) AS last_load
FROM snowflake.account_usage.copy_history
GROUP BY pipe_name
HAVING MAX(last_load_time) < DATEADD('hour', -2, CURRENT_TIMESTAMP());
```

### View channel‑level latency for Snowpipe Streaming

```sql
SELECT 
    pipe_name,
    table_name,
    snowflake_processing_latency_ms,
    rows_inserted,
    rows_error_count,
    created_on
FROM snowflake.account_usage.snowpipe_streaming_channel_history
WHERE created_on >= DATEADD('hour', -1, CURRENT_TIMESTAMP())
ORDER BY snowflake_processing_latency_ms DESC;
```

### Detect files that failed ingestion

```sql
SELECT 
    file_name,
    rows_loaded,
    rows_parsed,
    errors_seen,
    first_error_message,
    last_load_time
FROM TABLE(INFORMATION_SCHEMA.COPY_HISTORY(
    TABLE_NAME => 'MY_TABLE',
    START_TIME => DATEADD('day', -1, CURRENT_TIMESTAMP())

WHERE errors_seen > 0;
```

## 7. Key Takeaways

- **Classic Snowpipe** is file‑based, serverless, and ideal for near‑real‑time ingestion from cloud storage. Throughput is maximised by using many files of **100–250 MB compressed**.

- **Snowpipe Streaming high‑performance architecture** provides direct row‑level ingestion, supports up to **10 GB/s per table**, and achieves **5–10 second latency** with throughput‑based pricing.

- **File size and file count** are the most critical factors for Classic Snowpipe throughput. Small files create metadata overhead; large files cannot be parallelised.

- **Snowpipe Streaming** eliminates file management entirely – data is written directly to tables using Java/Python SDKs or a REST API.

- **Exactly‑once delivery** is supported via offset tokens in the high‑performance Streaming SDK – essential for financial and audit workloads.

- **Monitor regularly** using `PIPE_USAGE_HISTORY`, `COPY_HISTORY`, and `SNOWPIPE_STREAMING_CHANNEL_HISTORY` to detect bottlenecks, errors, and cost anomalies.

- **In‑flight transformations** (filtering, casting, renaming) can be applied during streaming ingestion, reducing client‑side complexity.

- **Automatic scaling** is handled server‑side. During sudden throughput spikes, you may observe brief latency increases as the service scales elastically.

This completes the thorough explanation of Snowpipe throughput with practical, actionable use cases.
