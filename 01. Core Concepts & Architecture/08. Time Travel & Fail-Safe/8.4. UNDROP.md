# Snowflake UNDROP: Reinstating Dropped Objects Instantly

## 1. Core Concepts

The `UNDROP` command in Snowflake is a **metadata restoration operation** that reactivates previously dropped objects within the **Time Travel retention window**. It does not reload data or reconstruct from backups. Instead, it **rebinds the object’s metadata to its preserved micro-partitions**, making the object immediately accessible again.

`UNDROP` operates at the **object level**, not the data level. It restores the latest dropped version of an object, not arbitrary historical states.

---

### 1.1. How it Works

**Architecture Overview:**

* Dropping an object removes active metadata references
* Underlying micro-partitions remain intact during retention
* Object metadata is preserved in a recoverable state
* `UNDROP` restores metadata pointers to original partitions

**Process Flow:**

1. Object is dropped using `DROP TABLE/SCHEMA/DATABASE`
2. Snowflake marks object as deleted but retains metadata + data
3. Object enters Time Travel retention window
4. User executes `UNDROP`
5. Metadata is restored and object becomes active again
6. Object resumes normal query operations

**Primary Use Cases:**

* Recover accidentally dropped tables
* Restore schemas/databases after cleanup operations
* Rapid recovery during deployment failures
* Maintain operational continuity without data reloads

---

### 1.2. UNDROP vs Other Recovery Methods

| Metric                   | UNDROP               | Time Travel (SELECT) | CLONE             |
| ------------------------ | -------------------- | -------------------- | ----------------- |
| Recovery Type            | Object-level         | Data-level           | Snapshot          |
| Speed                    | Instant              | Moderate             | Instant           |
| Storage Impact           | None                 | New storage          | Zero-copy         |
| Precision                | Latest dropped state | Any point in time    | Any point in time |
| Complexity               | Low                  | Medium               | Medium            |
| Requires Object Dropped? | Yes                  | No                   | No                |

---

### 1.3. Key Factors

1. **Retention Window Constraint**
   `UNDROP` is only valid within Time Travel retention

2. **Latest Version Only**
   Cannot restore earlier historical states directly

3. **Object Scope Matters**
   Supports tables, schemas, and databases

4. **Cascading Behavior**
   Dropped parent objects retain child lineage for full restoration

5. **Name Rebinding**
   Object name is restored as-is, unless conflict exists

6. **Storage Cost Continuation**
   Data remains billable until retention expires

## 2. Basic Commands & Privileges

### 2.1. Required Privileges

| Privilege | Scope           | Description            |
| --------- | --------------- | ---------------------- |
| OWNERSHIP | Object          | Required to undrop     |
| USAGE     | Database/Schema | Access container       |
| MONITOR   | Account         | Inspect object history |

---

### 2.2. Undrop a Table

```sql id="v19k2p"
-- Restore most recently dropped table
UNDROP TABLE sales_data;
```

Reinstates the table with its latest dropped state.

---

### 2.3. Undrop a Schema

```sql id="b8f4x1"
-- Restore schema and all contained objects
UNDROP SCHEMA analytics_db.public;
```

Recovers entire schema hierarchy.

---

### 2.4. Undrop a Database

```sql id="c3m9w7"
-- Restore database with all schemas and tables
UNDROP DATABASE analytics_db;
```

Restores full database structure.

---

### 2.5. Show Dropped Objects

```sql id="d7q2z5"
-- List dropped objects and recovery window
SHOW TABLES HISTORY;
```

Displays recoverable objects.

---

### 2.6. Resolve Name Conflicts

```sql id="k6p4y2"
-- Rename existing table before undrop
ALTER TABLE sales_data RENAME TO sales_data_backup;

-- Now restore original
UNDROP TABLE sales_data;
```

Prevents conflicts during restoration.

---

### 2.7. Validate Recoverability

```sql id="t5r8n3"
-- Check retention settings
SHOW PARAMETERS LIKE 'DATA_RETENTION_TIME_IN_DAYS' IN TABLE sales_data;
```

Ensures object is still within retention.

## 3. Impactful Use Cases

### Use Case 1: Recover Dropped Production Table

**Scenario** – A deployment script drops a critical table.

```sql id="u1x9z4"
-- Identify dropped table
SHOW TABLES HISTORY LIKE 'sales_data';

-- Restore immediately
UNDROP TABLE sales_data;
```

**Explanation** – Instant metadata restoration avoids downtime and eliminates need for data reload.

---

### Use Case 2: Restore Entire Schema After Cleanup

**Scenario** – A cleanup job removes an entire schema unintentionally.

```sql id="w2c7m8"
-- Restore schema and all tables
UNDROP SCHEMA analytics_db.public;
```

**Explanation** – Snowflake preserves hierarchical lineage, enabling full recovery of nested objects.

---

### Use Case 3: Recover Database After Environment Reset

**Scenario** – A database is dropped during environment refresh.

```sql id="p9s4d6"
-- Restore database
UNDROP DATABASE analytics_db;
```

**Explanation** – Database-level undrop restores all schemas and tables in one operation.

---

### Use Case 4: Handle Name Conflict During Restore

**Scenario** – A new table was created with the same name after drop.

```sql id="z8k1x3"
-- Rename conflicting table
ALTER TABLE sales_data RENAME TO sales_data_new;

-- Restore original table
UNDROP TABLE sales_data;
```

**Explanation** – Snowflake requires unique object names, so conflicts must be resolved before undrop.

---

### Use Case 5: Validate Recovery Window Before Undrop

**Scenario** – Ensure object is still recoverable.

```sql id="f7q2r5"
-- Check history for dropped objects
SHOW TABLES HISTORY LIKE 'sales_data';
```

**Explanation** – Prevents failed recovery attempts after retention expiry.

---

### Use Case 6: Combine UNDROP with Time Travel Validation

**Scenario** – Verify correctness after restoration.

```sql id="m4y8v2"
-- Restore table
UNDROP TABLE sales_data;

-- Validate historical state
SELECT COUNT(*)
FROM sales_data
AT (OFFSET => -600);
```

**Explanation** – Combines object restoration with data validation for reliability.

## 4. Best Practices Summary

| Practice                                     | Why                             |
| -------------------------------------------- | ------------------------------- |
| Use UNDROP immediately after accidental drop | Minimizes downtime              |
| Monitor dropped objects regularly            | Avoid missing retention window  |
| Avoid name reuse after drop                  | Prevent conflicts               |
| Combine UNDROP with validation queries       | Ensure correctness              |
| Restrict DROP privileges                     | Reduce accidental deletion risk |
| Automate alerts for DROP operations          | Improve response time           |
| Use schema-level undrop cautiously           | Prevent unintended restores     |
| Track retention policies                     | Ensure recoverability           |
| Test recovery procedures periodically        | Maintain operational readiness  |

---

## 5. Important Limitations & Considerations

| Limitation                                  | Mitigation                         |
| ------------------------------------------- | ---------------------------------- |
| Only latest dropped version can be restored | Use Time Travel for earlier states |
| Limited to retention window                 | Increase retention or clone        |
| Cannot undrop after Fail-safe               | Contact Snowflake support          |
| Name conflicts block restore                | Rename conflicting objects         |
| Temporary tables cannot be undropped        | Avoid for critical workloads       |
| Storage billed during retention             | Optimize retention policies        |

## 6. Auditing & Monitoring Queries

### 6.1. List Dropped Tables


---

### 6.2. Identify DROP Operations

```sql id="k3f8m1"
-- Track drop statements
SELECT query_id, query_text, start_time
FROM SNOWFLAKE.ACCOUNT_USAGE.QUERY_HISTORY
WHERE query_text ILIKE 'DROP%';
```

---

### 6.3. Check Retention Settings

```sql id="q7p1n4"
-- Verify retention configuration
SHOW PARAMETERS LIKE 'DATA_RETENTION_TIME_IN_DAYS' IN ACCOUNT;
```

---

### 6.4. Monitor Recoverable Objects

```sql id="r2t5y8"
-- Identify objects still within retention
SELECT table_name, deleted
FROM SNOWFLAKE.ACCOUNT_USAGE.TABLES
WHERE deleted IS NOT NULL;
```

---

### 6.5. Validate Query History for Recovery

```sql id="w6m9x3"
-- Retrieve query IDs for debugging
SELECT query_id, query_text
FROM SNOWFLAKE.ACCOUNT_USAGE.QUERY_HISTORY
ORDER BY start_time DESC;
```

## 7. Key Takeaways

* **UNDROP restores objects instantly using metadata, not data copies**
* **Only the most recent dropped version is recoverable via UNDROP**
* **Retention window strictly governs recoverability**
* **Schema and database undrop restore full object hierarchies**
* **Name conflicts must be resolved before restoration**
* **Temporary tables cannot be recovered using UNDROP**
* **Combine UNDROP with Time Travel for complete recovery strategies**
