# Snowflake Restoring Tables: Reconstructing Lost State with Time Travel

## 1. Core Concepts

**Restoring tables** in Snowflake is a metadata-driven operation that leverages **Time Travel** to recover dropped or modified tables without relying on backups. Instead of copying data back, Snowflake **rebinds metadata pointers to historical micro-partitions**, effectively resurrecting prior states.

Restoration is not a single mechanism. It spans three distinct patterns:

* **UNDROP** → Restore a dropped table
* **Time Travel SELECT** → Reconstruct data into a new table
* **CLONE** → Create a zero-copy snapshot at a past point

---

### 1.1. How it Works

**Architecture Overview:**

* Data is stored in immutable micro-partitions
* Table definitions map to partition sets via metadata
* Historical versions are preserved within retention window
* Restoration reassigns metadata to previous partition states

**Process Flow:**

1. A table is modified or dropped
2. Snowflake retains historical partitions and metadata lineage
3. User initiates restore via:

   * `UNDROP` (object-level recovery)
   * `AT | BEFORE` (data-level reconstruction)
   * `CLONE` (snapshot duplication)
4. Query engine resolves correct partition set
5. Restored table becomes active with prior state

**Primary Use Cases:**

* Recover dropped tables
* Rollback bad updates or deletes
* Reconstruct historical datasets
* Create recovery checkpoints for pipelines

---

### 1.2. Restoration Methods Comparison

| Metric                   | UNDROP          | Time Travel (SELECT)     | CLONE             |
| ------------------------ | --------------- | ------------------------ | ----------------- |
| Use Case                 | Dropped objects | Data recovery            | Snapshot creation |
| Storage Cost             | No extra        | New storage              | Zero-copy         |
| Granularity              | Entire object   | Row-level reconstruction | Full object       |
| Speed                    | Instant         | Moderate                 | Instant           |
| Flexibility              | Low             | High                     | High              |
| Requires Object Dropped? | Yes             | No                       | No                |

---

### 1.3. Key Factors

1. **Retention Window Dependency**
   All restoration methods rely on Time Travel availability

2. **Object State vs Data State**
   UNDROP restores object metadata, SELECT reconstructs data

3. **Storage Implications**
   SELECT creates new storage, CLONE shares existing partitions

4. **Precision Requirements**
   Use `BEFORE` for exact rollback, `AT` for time-based recovery

5. **Concurrency Considerations**
   Restoration does not block active queries due to MVCC model

6. **Edition Constraints**
   Retention duration varies by Snowflake edition [reference:3]

---

### 2.2. Restore Dropped Table (UNDROP)

```sql id="j4y27t"
-- Restore most recently dropped table
UNDROP TABLE sales_data;
```

Reactivates table metadata within retention window.

---

### 2.3. Restore Table to Previous State (CREATE AS SELECT)

```sql id="ybg9h6"
-- Recreate table from historical snapshot
CREATE OR REPLACE TABLE sales_data AS
SELECT *
FROM sales_data
AT (OFFSET => -600); -- 10 minutes ago
```

Overwrites current table with past data state.

---

### 2.4. Clone Table at Historical Point

```sql id="l5f1xf"
-- Create zero-copy snapshot
CREATE TABLE sales_backup
CLONE sales_data
AT (TIMESTAMP => '2026-04-04 09:00:00');
```

Creates a new table referencing historical partitions.

---

### 2.5. Restore Using BEFORE Clause

```sql id="a5cbxy"
-- Restore state before a destructive query
CREATE OR REPLACE TABLE sales_data AS
SELECT *
FROM sales_data
BEFORE (STATEMENT => 'query_id_here');
```

Provides precise rollback tied to query execution.

---

### 2.6. Verify Table History

```sql id="s5mznu"
-- Check table history and deletion timestamps
SHOW TABLES HISTORY LIKE 'sales_data';
```

Used to confirm recoverability window.

---

### 2.7. Check Retention Settings

```sql id="l68c4l"
-- Validate retention period
SHOW PARAMETERS LIKE 'DATA_RETENTION_TIME_IN_DAYS' IN TABLE sales_data;
```

Ensures restoration is still possible.

## 3. Impactful Use Cases

### Use Case 1: Recover Dropped Production Table

**Scenario** – A critical table is dropped during deployment.

```sql id="u9q5w3"
-- Step 1: Identify dropped table
SHOW TABLES HISTORY LIKE 'sales_data';

-- Step 2: Restore table
UNDROP TABLE sales_data;
```

**Explanation** – UNDROP restores metadata instantly without scanning or copying data, making recovery near-instantaneous.

---

### Use Case 2: Rollback Corrupted Data

**Scenario** – An UPDATE operation corrupts key columns.

```sql id="r7f4vq"
-- Restore previous state
CREATE OR REPLACE TABLE sales_data AS
SELECT *
FROM sales_data
AT (OFFSET => -300);
```

**Explanation** – This approach reconstructs the table using historical partitions, effectively performing a rollback.

---

### Use Case 3: Precise Recovery Using Query ID

**Scenario** – A DELETE statement removed valid records.

```sql id="s5p9k1"
-- Find query ID
SELECT query_id, query_text
FROM SNOWFLAKE.ACCOUNT_USAGE.QUERY_HISTORY
WHERE query_text ILIKE '%DELETE%';

-- Restore state before deletion
CREATE OR REPLACE TABLE sales_data AS
SELECT *
FROM sales_data
BEFORE (STATEMENT => 'identified_query_id');
```

**Explanation** – Statement-level recovery eliminates ambiguity compared to timestamp-based methods.

---

### Use Case 4: Create Backup Before Risky Operation

**Scenario** – A migration requires a fallback snapshot.

```sql id="q3d9x2"
-- Create backup clone
CREATE TABLE sales_backup
CLONE sales_data;
```

**Explanation** – Zero-copy cloning provides instant backup without storage duplication.

---

### Use Case 5: Restore Subset of Data

**Scenario** – Only a subset of rows needs recovery.

```sql id="6hwt0x"
-- Recover specific rows
CREATE TABLE recovered_subset AS
SELECT *
FROM sales_data
AT (OFFSET => -600)
WHERE region = 'APAC';
```

**Explanation** – Time Travel enables selective recovery, avoiding full table restoration.

---

### Use Case 6: Compare Current vs Historical State

**Scenario** – Validate impact of recent transformations.

```sql id="a0f8e9"
-- Current state
SELECT COUNT(*) FROM sales_data;

-- Historical state
SELECT COUNT(*)
FROM sales_data
AT (OFFSET => -3600);
```

**Explanation** – Enables quick validation and anomaly detection without external systems.

## 4. Best Practices Summary

| Practice                                    | Why                            |
| ------------------------------------------- | ------------------------------ |
| Use UNDROP for dropped tables               | Fastest recovery method        |
| Prefer CLONE for backups                    | Zero storage overhead          |
| Use BEFORE for precise rollback             | Eliminates timestamp ambiguity |
| Avoid overwriting without validation        | Prevent cascading errors       |
| Monitor retention settings                  | Ensure recoverability          |
| Use selective recovery where possible       | Reduce compute and storage     |
| Automate backup clones before major changes | Create safety checkpoints      |
| Audit destructive queries                   | Improve governance             |
| Test recovery workflows regularly           | Ensure operational readiness   |

---

## 5. Important Limitations & Considerations

| Limitation                                  | Mitigation                         |
| ------------------------------------------- | ---------------------------------- |
| Restoration limited by retention window     | Increase retention or clone data   |
| UNDROP restores only latest dropped version | Use Time Travel for earlier states |
| SELECT-based restore creates new storage    | Use CLONE where possible           |
| Cannot restore after Fail-safe              | Contact Snowflake support          |
| Query ID dependency for BEFORE              | Log query history                  |
| High churn increases storage footprint      | Optimize DML patterns              |

### 6.2. Track Destructive Queries

```sql id="h3t7y9"

FROM SNOWFLAKE.ACCOUNT_USAGE.QUERY_HISTORY
```

```sql id="1z6f9q"

SELECT query_id, query_text
FROM SNOWFLAKE.ACCOUNT_USAGE.QUERY_HISTORY
ORDER BY start_time DESC;
```

## 7. Key Takeaways

* **Restoration in Snowflake is metadata-driven, not data-copy based**
* **UNDROP is fastest for recovering dropped tables**
* **AT and BEFORE enable flexible and precise recovery strategies**
* **CLONE provides instant, zero-copy backups**
* **Retention window defines the boundary of recoverability**
* **SELECT-based restoration allows granular recovery but incurs storage cost**
* **Combining multiple restoration methods yields robust recovery workflows**
