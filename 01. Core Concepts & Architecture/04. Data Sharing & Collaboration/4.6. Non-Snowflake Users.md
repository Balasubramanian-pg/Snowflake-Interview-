# Sharing with Non-Snowflake Users: The Power of Reader Accounts

When we talk about "Non-Snowflake Users" in the context of data sharing, we're primarily referring to individuals or organizations that:

1.  **Do not have their own Snowflake account.**
2.  **Do not want to incur the costs or administrative overhead of setting up and managing a full Snowflake account.**
3.  **Primarily need read-only access to specific datasets.**

For these scenarios, Snowflake provides **Reader Accounts**. These are dedicated, limited-access Snowflake accounts that you (the data provider) create and manage on behalf of your data consumers.

### What is a Reader Account?

A **Reader Account** is essentially a stripped-down Snowflake account provided by a data provider to a data consumer. It allows the consumer to query shared data without having to sign up for or manage their own full Snowflake account.

**Key Characteristics:**

*   **Provider-Managed:** The data provider creates, configures, and manages the Reader Account. This includes setting up users and roles within the Reader Account.
*   **Cost-Borne by Provider:** All compute costs (i.e., virtual warehouse usage) incurred by queries run in the Reader Account are billed to the data provider's Snowflake account. The consumer incurs no direct Snowflake costs.
*   **Read-Only Access:** Consumers using a Reader Account can only query the data that has been explicitly shared with them. They cannot modify the data, create new objects (databases, tables, etc.), or load their own data into the Reader Account.
*   **Simplified Access:** Consumers access the Reader Account using standard Snowflake clients (web interface, JDBC/ODBC drivers, connectors), just like a regular Snowflake user.
*   **Live Data:** Just like other Snowflake sharing mechanisms, the data shared with a Reader Account is live. Any updates made by the provider to the original data are instantly available to the Reader Account.

### Why Use Reader Accounts for Non-Snowflake Users?

1.  **Removes Barriers to Entry:** This is the biggest advantage. Non-Snowflake users don't need to understand Snowflake pricing, contracts, or administration. They just get a username, password, and account URL.
2.  **Cost Control for Consumer:** The consumer pays nothing directly to Snowflake. This makes it ideal for partners, vendors, or clients where you want to provide data as a service.
3.  **Cost Visibility for Provider:** While the provider pays for the compute, they have full visibility into the usage and can manage the virtual warehouse associated with the Reader Account (e.g., suspend it when not in use, size it appropriately).
4.  **Security and Governance:** The provider maintains full control over what data is shared (via the share object) and can audit all activity within the Reader Account from their own provider account.
5.  **Seamless Experience:** Consumers get a familiar SQL querying experience within a secure environment.

### How to Set Up and Use Reader Accounts (with Example Queries)

Let's use an example: "Analytics Co." (the provider) wants to share a weekly sales report with "Retail Partner Inc." (the non-Snowflake user).

**Steps for the Data Provider ("Analytics Co."):**

#### Step 1: Prepare the Data to be Shared

Ensure the data exists in your provider account and create a `SHARE` object with the data you want to expose. Using secure views is highly recommended for fine-grained control and abstraction.

```sql
-- As Analytics Co. (Provider Account)

-- Assume we have a database and table with sales data
USE DATABASE ANALYTICS_DB;
USE SCHEMA SALES;

CREATE SECURE VIEW WEEKLY_SALES_REPORT_VIEW AS
SELECT
    SALE_DATE,
    PRODUCT_CATEGORY,
    SUM(REVENUE) AS TOTAL_REVENUE,
    COUNT(DISTINCT ORDER_ID) AS TOTAL_ORDERS
FROM SALES.DAILY_SALES_TABLE
WHERE SALE_DATE >= DATEADD(day, -7, CURRENT_DATE()) -- Example: last 7 days data
GROUP BY SALE_DATE, PRODUCT_CATEGORY;

-- Create the SHARE object (if not already created)
CREATE SHARE RETAIL_PARTNER_SALES_SHARE
COMMENT = 'Weekly sales report for Retail Partner Inc.';

-- Grant necessary privileges to the SHARE
GRANT USAGE ON DATABASE ANALYTICS_DB TO SHARE RETAIL_PARTNER_SALES_SHARE;
GRANT USAGE ON SCHEMA ANALYTICS_DB.SALES TO SHARE RETAIL_PARTNER_SALES_SHARE;
GRANT SELECT ON VIEW ANALYTICS_DB.SALES.WEEKLY_SALES_REPORT_VIEW TO SHARE RETAIL_PARTNER_SALES_SHARE;
```

#### Step 2: Create the Reader Account

You create the Reader Account and define an initial administrator user for it. You'll need to specify the region where you want the Reader Account to reside (it must be the same as your provider account).

```sql
-- Create the Reader Account
CREATE MANAGED ACCOUNT RETAIL_PARTNER_RA
  ADMIN_NAME = retail_admin -- The username for the admin of the reader account
  ADMIN_PASSWORD = 'StrongPassword123!' -- The password for the reader_admin
  COMMENT = 'Reader account for Retail Partner Inc. to access sales reports'
  REGION = 'AWS_US_EAST_1'; -- Example region, must match provider account region
```
*   Snowflake will return the `account_locator` for this newly created Reader Account. You'll need this.

#### Step 3: Grant the Share to the Reader Account

Now, link your `SHARE` object to the newly created Reader Account.

```sql
-- Grant the SHARE to the Reader Account
ALTER SHARE RETAIL_PARTNER_SALES_SHARE ADD ACCOUNT = RETAIL_PARTNER_RA;
```

#### Step 4: Provide Credentials to the Non-Snowflake User

You provide "Retail Partner Inc." with:
*   The `account_locator` for `RETAIL_PARTNER_RA` (e.g., `XYZ12345`).
*   The username `retail_admin`.
*   The password `StrongPassword123!`.
*   Instructions on how to access Snowflake (e.g., via the web UI using the account locator, or configuring a JDBC/ODBC driver).

---

**Steps for the Non-Snowflake User ("Retail Partner Inc."):**

The consumer logs into the Reader Account using the provided credentials.

#### Step 1: Log In

*   They navigate to `https://<account_locator>.snowflakecomputing.com/console/login` (e.g., `https://xyz12345.snowflakecomputing.com/console/login`).
*   They enter `retail_admin` and `StrongPassword123!`.

#### Step 2: Query the Shared Data

Once logged in, they will see a database in their Reader Account that represents the shared data (often named after the original shared database).

```sql
-- As Retail Partner Inc. (in the Reader Account)

-- The shared database will appear in their account.
-- Its name might be `ANALYTICS_DB` or similar, reflecting the provider's database name.
USE DATABASE ANALYTICS_DB;
USE SCHEMA SALES;

-- Query the shared view
SELECT
    SALE_DATE,
    PRODUCT_CATEGORY,
    TOTAL_REVENUE
FROM WEEKLY_SALES_REPORT_VIEW
WHERE PRODUCT_CATEGORY = 'Apparel'
ORDER BY SALE_DATE DESC;
```

### Managing the Reader Account (Provider's Perspective)

As the provider, you have ongoing responsibilities:

*   **Virtual Warehouse Management:** A default virtual warehouse is created for each Reader Account. You can manage this warehouse from your provider account. You can resize it, suspend it when inactive, or monitor its usage.
    ```sql
    -- In Provider Account
    -- To see warehouses associated with managed accounts
    SHOW WAREHOUSES IN ACCOUNT; -- Look for warehouses with type 'MANAGED_ACCOUNT'

    -- To suspend a reader account's warehouse (e.g., if no one is using it)
    ALTER WAREHOUSE RETAIL_PARTNER_RA_WH SUSPEND;

    -- To resume
    ALTER WAREHOUSE RETAIL_PARTNER_RA_WH RESUME;

    -- To resize
    ALTER WAREHOUSE RETAIL_PARTNER_RA_WH SET WAREHOUSE_SIZE = 'SMALL';
    ```
*   **User Management within Reader Account:** You can log into the Reader Account as `retail_admin` to create additional users, roles, and manage their permissions for your partner's internal team.
    ```sql
    -- In the Reader Account (logged in as retail_admin)
    CREATE USER retail_analyst PASSWORD = 'AnotherStrongPassword!';
    CREATE ROLE retail_viewer;
    GRANT USAGE ON WAREHOUSE RETAIL_PARTNER_RA_WH TO ROLE retail_viewer;
    GRANT USAGE ON DATABASE ANALYTICS_DB TO ROLE retail_viewer;
    GRANT USAGE ON SCHEMA ANALYTICS_DB.SALES TO ROLE retail_viewer;
    GRANT SELECT ON VIEW ANALYTICS_DB.SALES.WEEKLY_SALES_REPORT_VIEW TO ROLE retail_viewer;
    GRANT ROLE retail_viewer TO USER retail_analyst;
    ```
*   **Monitoring and Auditing:** All queries run in the Reader Account are visible in your provider account's query history, allowing for full auditing.
*   **Updating Shared Data:** As you update `ANALYTICS_DB.SALES.DAILY_SALES_TABLE` in your provider account, the `WEEKLY_SALES_REPORT_VIEW` in the Reader Account will instantly reflect those changes.
*   **Terminating Access:** If the partnership ends, you can simply drop the Reader Account.
    ```sql
    -- In Provider Account
    DROP MANAGED ACCOUNT RETAIL_PARTNER_RA;
    ```
    This immediately revokes all access and deallocates its resources.

### Limitations of Reader Accounts:

It's important to remember that Reader Accounts are simplified and have limitations compared to full Snowflake accounts:

*   **No Data Ingress/Egress:** Cannot load data (e.g., via `COPY INTO`) or unload data to stages.
*   **No Object Creation:** Cannot create new databases, schemas, tables, views, stages, etc.
*   **No Data Marketplace Access:** Cannot subscribe to data products from the Snowflake Data Marketplace.
*   **No Cloning/Time Travel:** Cannot clone databases/tables or use Time Travel features.

These limitations are by design, ensuring that Reader Accounts remain focused on secure, read-only data consumption while keeping administrative and cost burden on the provider.

Reader Accounts are an incredibly powerful tool for extending the reach of your data to external partners and non-Snowflake users, fostering collaboration without the typical hurdles of traditional data sharing methods.
