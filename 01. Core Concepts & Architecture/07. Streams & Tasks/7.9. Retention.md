Snowflake Retention: Safeguarding Change Tracking Integrity

## 1. Core Concepts

In the Snowflake ecosystem, **Retention** specifically refers to the temporal window during which Change Data Capture (CDC) records in a **Stream** remain valid and consumable. Unlike standard table retention, which focuses on Time Travel recovery, Stream retention is a proactive "use-it-or-lose-it" mechanism designed to prevent the indefinite growth of change tracking metadata.

### 1.1. How it Works:
The retention logic for Streams is fundamentally tied to the source table's configuration but includes a specialized safety buffer:
1.  **Retention Linkage:** A stream tracks changes by pinning the transaction version of the source table.
2.  **Extended Retention Period:** Snowflake automatically extends the retention period of the source table to ensure stream data remains consumable. This period is the **maximum** of the table’s `DATA_RETENTION_TIME_IN_DAYS` or **14 days** [reference:1].
3.  **Staleness Trigger:** If the stream is not consumed (via a DML operation like `INSERT` or `MERGE`) before it reaches the `STALE_AFTER` timestamp, the stream becomes **Stale**.
4.  **Metadata Cleanup:** Once stale, the change tracking version is released, and the stream can no longer access the historical delta, requiring a full re-initialization.

### 1.2. Retention Comparison
| Metric | Standard Table Retention | Stream Consumption Retention |
| :--- | :--- | :--- |
| **Primary Purpose** | Point-in-time recovery (Time Travel) | CDC Consumability |
| **Duration (Standard)** | 0 or 1 Day | 14 Days (Default Minimum) |
| **Duration (Enterprise)** | Up to 90 Days | Max(Table Retention, 14 Days) |
| **Failure Result** | Data cannot be restored | Stream becomes unusable (Stale) |

### 1.3. Key Factors
1.  **Table Retention Setting:** Increasing `DATA_RETENTION_TIME_IN_DAYS` on a source table directly increases the stream's lifespan up to **90 days** [reference:2].
2.  **Consumption Frequency:** The "clock" for staleness does not reset until the stream is successfully used in a transaction that moves its offset forward.
3.  **DML Activity:** High-churn tables generate more metadata; while this doesn't change the *time* limit, it increases the underlying storage overhead.
4.  **Task History Retention:** Separate from streams, Task execution metadata is retained for **7 days** in the Information Schema and **365 days** in Account Usage [reference:3].

## 2. Basic Commands & Privileges

### 2.1. Required Privileges
| Privilege | Scope | Description |
| :--- | :--- | :--- |
| **OWNERSHIP** | Stream | Required to view stream properties or drop/recreate. |
| **OWNERSHIP** | Table | Required to modify `DATA_RETENTION_TIME_IN_DAYS` on the source. |
| **MONITOR** | Task | Required to view task history retention metrics. |

### 2.2. Checking Stream Staleness Status
Returns the metadata for a stream, specifically the `stale` and `stale_after` columns.
```sql
SHOW STREAMS LIKE 'strm_clinical_trials';
-- Check the 'stale' column (boolean) and 'stale_after' (timestamp)
```

### 2.3. Extending Source Table Retention
Increasing the table's retention period to 30 days to provide a larger buffer for a monthly batch task.
```sql
ALTER TABLE raw_safety_data 
SET DATA_RETENTION_TIME_IN_DAYS = 30;
```

### 2.4. Detecting "Stale" Tasks in Information Schema
Queries the task history to identify execution metadata that is approaching the 7-day cleanup limit.
```sql
SELECT * FROM TABLE(information_schema.task_history(
    scheduled_time_range_start => DATEADD('day', -7, CURRENT_TIMESTAMP())
));
```

### 2.5. Validating Stream Buffer via System Functions
Uses the system function to evaluate if a stream is currently at risk of staleness based on current offsets.
```sql
DESCRIBE STREAM strm_patient_vitals;
-- View 'invalid_reason' to diagnose why a stream may have become stale
```

## 3. Impactful Use Cases



### Use Case 1: Aligning Dimension Loads with Source Retention
**Scenario** – A clinical trial dimension table is updated via a Task every 10 days. If the source table has a default 1-day retention, the stream might become stale if the Task fails once.

```sql
-- 1. Ensure the source table has a safety buffer
ALTER TABLE src_clinical_sites SET DATA_RETENTION_TIME_IN_DAYS = 14;

-- 2. Define the stream
CREATE OR REPLACE STREAM strm_site_delta ON TABLE src_clinical_sites;

-- 3. Task consumes the stream every 10 days
CREATE OR REPLACE TASK tsk_load_sites
  WAREHOUSE = 'TRANSFORM_WH'
  SCHEDULE = '14400 MINUTE' -- 10 Days
AS
  MERGE INTO dim_sites d USING strm_site_delta s ON d.id = s.id
  WHEN MATCHED THEN UPDATE SET d.name = s.name
  WHEN NOT MATCHED THEN INSERT (id, name) VALUES (s.id, s.name);
```
**Explanation** – By explicitly setting the table retention to 14 days, we ensure the Stream's `STALE_AFTER` date provides enough headroom to survive a single Task failure/retry cycle in a 10-day window.

### Use Case 2: Emergency Recovery of a Stale Stream
**Scenario** – A production Stream has become stale due to an extended system outage. You must re-initialize the CDC process without missing the current state.

```sql
-- 1. Identify the stream is stale
-- SHOW STREAMS; -> stale = 'true'

-- 2. Re-create the stream to reset the offset to the current table version
CREATE OR REPLACE STREAM strm_inventory_cdc ON TABLE inventory_master;

-- 3. Perform a manual 'Catch-up' sync
INSERT INTO target_inventory
SELECT * FROM inventory_master; 
```
**Explanation** – Once a stream is stale, it is unrecoverable. The only solution is to recreate the stream (which sets the offset to the current timestamp) and perform a manual synchronization to align the target table with the source.

### Use Case 3: Monthly Audit Archive via Task History
**Scenario** – For compliance, you need to store task execution details for 2 years, exceeding the 1-year limit of `ACCOUNT_USAGE`.

```sql
-- 1. Create a persistent audit table
CREATE TABLE IF NOT EXISTS task_execution_archive (
    task_name STRING,
    query_id STRING,
    state STRING,
    completed_time TIMESTAMP_LTZ
);

-- 2. Create a task to archive 'Account Usage' data into the persistent table
CREATE OR REPLACE TASK tsk_archive_task_metadata
  WAREHOUSE = 'ADMIN_WH'
  SCHEDULE = 'USING CRON 0 0 1 * * UTC' -- Monthly
AS
  INSERT INTO task_execution_archive
  SELECT name, query_id, state, completed_time
  FROM SNOWFLAKE.ACCOUNT_USAGE.TASK_HISTORY
  WHERE completed_time >= DATEADD('month', -1, CURRENT_TIMESTAMP());
```
**Explanation** – This pattern circumvents Snowflake’s internal retention limits for metadata. By moving the data into a standard table, you maintain control over the retention period indefinitely.

### Use Case 4: Protecting Streams During Table Replacements
**Scenario** – You are performing a `CREATE OR REPLACE TABLE` on a source. This operation can break Stream retention because the underlying object ID changes.

```sql
-- 1. Create a temporary 'buffer' stream
CREATE OR REPLACE STREAM strm_buffer ON TABLE source_data;

-- 2. Swap tables instead of 'Replace' to preserve stream logic
ALTER TABLE source_data RENAME TO source_data_old;
ALTER TABLE source_data_new RENAME TO source_data;

-- 3. Re-point or re-create stream as necessary
CREATE OR REPLACE STREAM strm_active ON TABLE source_data;
```
**Explanation** – `REPLACE` operations are destructive to stream retention. Using `RENAME` or `SWAP` helps maintain the lineage, though streams often still require recreation. Monitoring the `STALE_AFTER` property during structural changes is a critical best practice.

## 4. Best Practices Summary

| Practice | Why |
| :--- | :--- |
| **Set Retention >= 14 Days** | Ensures the stream utilizes the maximum automatic extension buffer. |
| **Monitor 'stale_after'** | Proactive alerts on this timestamp prevent the need for expensive full re-syncs. |
| **Avoid 'Replace Table'** | Replacing a source table drops the stream's metadata pin; use `TRUNCATE`/`INSERT` or `SWAP` instead. |
| **Consume Streams Regularly** | Even if a task does nothing, a successful DML move the offset forward, resetting the staleness risk. |
| **Use Serverless for Monitoring** | Serverless tasks are ideal for lightweight "Staleness Check" queries to avoid minimum warehouse billing. |
| **Document Retention SLAs** | Ensure the batch frequency of tasks is at least 30% shorter than the stream retention period. |
| **Audit Account Usage** | Use the `SNOWFLAKE` database to monitor long-term task performance trends beyond 7 days. |
| **Recreate Stale Streams Fast** | Delaying recreation after staleness leads to a wider gap between source and target data. |

## 5. Important Limitations & Considerations

| Limitation | Mitigation |
| :--- | :--- |
| **90-Day Hard Limit** | Streams cannot track changes older than 90 days, regardless of table settings. Design for more frequent consumption. |
| **Information Schema Lag** | `TASK_HISTORY` is real-time but only lasts 7 days. Use `ACCOUNT_USAGE` for historical analysis. |
| **Cloned Streams** | Cloning a database clones the stream but **not** its offset history; cloned streams start empty [reference:4]. |
| **Time Travel Interaction** | Dropping a table also drops its streams; while the table can be undropped, the stream often must be recreated. |

## 6. Auditing & Monitoring Queries

### 6.1. Identify Streams At Risk of Staleness (< 3 Days)
```sql
SELECT 
    database_name, 
    schema_name, 
    name AS stream_name, 
    stale_after
FROM SNOWFLAKE.ACCOUNT_USAGE.STREAMS
WHERE stale = 'false'
  AND stale_after <= DATEADD('day', 3, CURRENT_TIMESTAMP());
```

### 6.2. Verify Table vs. Stream Retention Alignment
```sql
SELECT 
    t.table_name, 
    t.retention_time, 
    s.name AS stream_name,
    s.stale_after
FROM information_schema.tables t
JOIN information_schema.streams s ON s.table_name = t.table_name
WHERE t.table_schema = 'RAW_DATA';
```

### 6.3. Audit Long-Term Task History (Past 6 Months)
```sql
SELECT 
    name, 
    COUNT(*) AS run_count, 
    AVG(duration_seconds) AS avg_runtime
FROM SNOWFLAKE.ACCOUNT_USAGE.TASK_HISTORY
WHERE query_start_time >= DATEADD('month', -6, CURRENT_TIMESTAMP())
GROUP BY 1;
```

### 6.4. Monitor Stream Retention Metadata Charges
```sql
-- Change tracking adds hidden storage overhead; monitor for high-churn tables
SELECT 
    table_name, 
    active_bytes, 
    time_travel_bytes, 
    failsafe_bytes
FROM SNOWFLAKE.ACCOUNT_USAGE.TABLE_STORAGE_METRICS
WHERE table_name IN (SELECT source_name FROM SNOWFLAKE.ACCOUNT_USAGE.STREAMS);
```

## 7. Key Takeaways

* **Stream Retention** is a dynamic window that defaults to 14 days but can be extended to 90 days via table settings.
* The **`STALE_AFTER`** property is the single most important metric for pipeline health; crossing this date results in unrecoverable CDC metadata.
* Tasks have **two-tier retention**: 7 days for operational debugging (Information Schema) and 365 days for auditing (Account Usage).
* **Transactional consumption** is the only way to advance a stream offset and maintain its "Active" status.
* **Data Engineers** must align the cadence of their DAGs with the retention buffers of their source tables to ensure zero data loss.
