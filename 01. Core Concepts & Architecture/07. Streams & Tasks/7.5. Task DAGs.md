Snowflake Task DAGs: Architecting Complex Data Orchestration

## 1. Core Concepts

A **Directed Acyclic Graph (DAG)** in Snowflake is a series of tasks composed of a single root task and a set of child tasks, organized by their dependencies. Unlike simple task pairs, a DAG allows for complex, multi-layered execution logic where tasks can trigger in parallel (**fan-out**) or wait for multiple predecessors to complete (**fan-in**) before proceeding.



### 1.1. How it Works
The orchestration of a Snowflake Task DAG follows a strict relational hierarchy:
1.  **Root Task Authority:** The Root Task is the only node with a `SCHEDULE`. It acts as the "trigger" for the entire graph.
2.  **Predecessor Mapping:** Child tasks are defined using the `AFTER` keyword, referencing one or more parent tasks. 
3.  **Graph Traversal:** When the Root Task completes successfully, the Snowflake scheduler identifies all immediate children.
4.  **Parallel Execution:** If multiple tasks share the same parent, Snowflake attempts to run them concurrently, provided compute resources (warehouses) are available.
5.  **State Evaluation:** A task only enters the `RUNNING` state if its predecessor(s) reached a `SUCCEEDED` state.

### 1.2. Orchestration Evolution
| Metric | Standalone Tasks | Task DAGs |
| :--- | :--- | :--- |
| **Logic Flow** | Linear / Sequential | Multi-path / Branched |
| **Trigger Mechanism** | Individual Schedules | Single Root Schedule |
| **Concurrency** | Manual / Scripted | Native Parallelism |
| **Management** | High Overhead (N tasks) | Unified (1 Graph) |

### 1.3. Key Factors
1.  **Node Limit:** A single DAG is limited to a maximum of **1,000 tasks** [reference:1].
2.  **Strict Directionality:** Loops are strictly prohibited; a child cannot eventually reference its own ancestor.
3.  **Compute Isolation:** Each task in a DAG can point to a different warehouse, allowing heavy compute for transforms and serverless for lightweight DDLs.
4.  **Overlapping Protection:** If a DAG run is still in progress when the Root Task's next schedule fires, the new run is skipped to prevent resource contention [reference:2].

## 2. Basic Commands & Privileges

### 2.1. Required Privileges
| Privilege | Scope | Description |
| :--- | :--- | :--- |
| **CREATE TASK** | Schema | Necessary to define the nodes of the DAG. |
| **EXECUTE TASK** | Account | Global privilege required for the task engine to initiate runs. |
| **OPERATE** | Task | Allows resuming/suspending specific tasks within the graph. |

### 2.2. Initializing the Root Task
The entry point must be scheduled and is typically the first object created.
```sql
CREATE OR REPLACE TASK dag_root_master
  WAREHOUSE = 'INGEST_WH'
  SCHEDULE = 'USING CRON 0 2 * * * UTC' -- Runs daily at 2 AM
AS
  TRUNCATE TABLE staging_land_zone;
```

### 2.3. Defining a Fan-Out (Parallel) Step
Creating multiple tasks that depend on the same parent to enable parallel processing.
```sql
-- Child A
CREATE OR REPLACE TASK transform_sales AFTER dag_root_master
  WAREHOUSE = 'TRANSFORM_WH'
AS
  CALL prc_process_sales();

-- Child B (Runs in parallel with Child A)
CREATE OR REPLACE TASK transform_inventory AFTER dag_root_master
  WAREHOUSE = 'TRANSFORM_WH'
AS
  CALL prc_process_inventory();
```

### 2.4. Defining a Fan-In (Join) Step
A task that waits for multiple predecessors to finish before executing.
```sql
CREATE OR REPLACE TASK generate_final_report
  WAREHOUSE = 'REPORT_WH'
  AFTER transform_sales, transform_inventory -- Depends on BOTH
AS
  INSERT INTO final_bi_report SELECT * FROM v_integrated_data;
```

### 2.5. Resuming the Graph
Tasks must be resumed in a specific order (bottom-up) or via a recursive function.
```sql
-- Resuming children first, root last
ALTER TASK generate_final_report RESUME;
ALTER TASK transform_sales RESUME;
ALTER TASK transform_inventory RESUME;
ALTER TASK dag_root_master RESUME;
```

### 2.6. Viewing Graph Structure
Using the account usage view to visualize the relationship of a specific DAG.
```sql
SELECT 
    name, 
    predecessors, 
    condition 
FROM TABLE(information_schema.task_dependents(task_name => 'dag_root_master', recursive => true));
```

## 3. Impactful Use Cases



### Use Case 1: Parallel Dimension and Fact Loading
**Scenario** – A standard Star Schema refresh where multiple dimensions must be updated in parallel before the Fact table can be calculated.

```sql
-- Parallel Dimension Loads
CREATE TASK load_dim_cust AFTER root_task AS CALL sp_load_customers();
CREATE TASK load_dim_prod AFTER root_task AS CALL sp_load_products();
CREATE TASK load_dim_geo  AFTER root_task AS CALL sp_load_geography();

-- Fact Load (The Fan-In)
CREATE TASK load_fact_sales 
    AFTER load_dim_cust, load_dim_prod, load_dim_geo 
AS 
    INSERT INTO fact_sales SELECT * FROM v_sales_stage;
```
**Explanation** – This maximizes warehouse utilization by processing non-dependent dimensions simultaneously. The Fact table load is "gated," ensuring referential integrity is maintained by waiting for all dimensions to succeed.

### Use Case 2: Multi-Region Data Aggregation
**Scenario** – Data arrives from three different regional buckets (EMEA, AMER, APAC). Each requires unique cleaning before a global aggregate is created.

```sql
-- Regional processing
CREATE TASK clean_emea AFTER root_sync AS CALL prc_clean('EMEA');
CREATE TASK clean_amer AFTER root_sync AS CALL prc_clean('AMER');
CREATE TASK clean_apac AFTER root_sync AS CALL prc_clean('APAC');

-- Global union
CREATE TASK global_aggregate 
    AFTER clean_emea, clean_amer, clean_apac 
AS 
    CALL prc_generate_global_view();
```
**Explanation** – By splitting the logic into regional tasks, a failure in the APAC file does not stop EMEA or AMER from completing. Engineers can then troubleshoot only the failed branch.

### Use Case 3: ML Model Scoring Pipeline
**Scenario** – A daily process that extracts fresh features, joins them with reference data, and runs a Python-based scoring procedure.

```sql
-- Feature Engineering
CREATE TASK fe_extract AFTER root_trigger AS CALL sp_extract_features();

-- Reference Join (Branch A)
CREATE TASK join_ref_data AFTER fe_extract AS CALL sp_join_geo_ref();

-- ML Scoring (Branch B - uses output of extraction)
CREATE TASK run_scoring 
    WAREHOUSE = 'ML_XSMALL'
    AFTER join_ref_data 
AS 
    CALL sp_predict_churn();
```
**Explanation** – This encapsulates the ML lifecycle. By separating extraction from scoring, you can assign a specialized high-memory warehouse to the `run_scoring` task while keeping the earlier steps on standard compute.

### Use Case 4: Data Quality Gating
**Scenario** – Implementing a "Circuit Breaker" where downstream analytics are aborted if data quality checks fail.

```sql
-- Transformation
CREATE TASK t_transform AFTER t_root AS INSERT INTO curated_table SELECT * FROM raw_table;

-- Data Quality Check
CREATE TASK t_dq_check AFTER t_transform AS CALL sp_verify_nulls_and_duplicates();

-- Gated Consumer Task
CREATE TASK t_refresh_dashboard AFTER t_dq_check AS CALL sp_refresh_mv();
```
**Explanation** – Because tasks only trigger on the `SUCCESS` of a predecessor, any exception raised in `sp_verify_nulls_and_duplicates` will naturally stop the `t_refresh_dashboard` task from running, preventing bad data from reaching executives.

## 4. Best Practices Summary

| Practice | Why |
| :--- | :--- |
| **Atomic Tasks** | Keep each task doing one thing. Smaller tasks are easier to debug and retry. |
| **Serverless for DDL** | Use serverless compute for tasks that only truncate tables or move files to avoid warehouse overhead. |
| **Error Integration** | Set `ERROR_INTEGRATION` on the Root task to catch graph-level failures immediately. |
| **Explicit Timeouts** | Set `USER_TASK_TIMEOUT_MS` to prevent a hanging query from blocking the entire DAG for hours. |
| **Bottom-Up Resuming** | Always resume children before the root to ensure the dependency chain is active when the root fires. |
| **Idempotency** | Ensure scripts can be re-run safely. If a task fails and is retried, it shouldn't create duplicate data. |
| **Naming Conventions** | Prefix tasks with `TSK_` and include their level in the DAG (e.g., `TSK_L1_ROOT`, `TSK_L2_TRANSFORM`). |
| **Metadata Tagging** | Apply Snowflake Tags to tasks to track which business unit or project (e.g., 'Project Aegis') they belong to. |

## 5. Important Limitations & Considerations

| Limitation | Mitigation |
| :--- | :--- |
| **1,000 Task Limit** | For extremely vast pipelines, nest your logic within Stored Procedures or use a dedicated orchestrator. |
| **Single Root** | A DAG cannot have multiple scheduled roots. Use separate DAGs or a "Controller" task if needed. |
| **Manual Re-wiring** | Changing the `AFTER` clause requires the task to be suspended first. |
| **Cross-Database DAGs** | Tasks can span databases, but the role executing them must have permissions across all involved schemas. |
| **No Dynamic DAGs** | Snowflake DAGs are static. For dynamic branching, use Python stored procedures to generate SQL. |

## 6. Auditing & Monitoring Queries

### 6.1. Monitor the Current Status of a DAG
```sql
SELECT 
    NAME, 
    STATE, 
    LAST_COMMITTED_TIME, 
    NEXT_SCHEDULED_TIME
FROM TABLE(information_schema.task_history())
WHERE root_task_id = (SELECT id FROM TABLE(information_schema.task_history()) WHERE name = 'dag_root_master' LIMIT 1)
ORDER BY scheduled_time DESC;
```

### 6.2. Detect Broken Chains (Suspended Children)
```sql
SELECT 
    NAME, 
    DATABASE_NAME, 
    SCHEMA_NAME 
FROM TABLE(information_schema.task_dependents(task_name => 'dag_root_master', recursive => true))
WHERE state != 'STARTED';
```

### 6.3. Average Execution Time per DAG Node
```sql
SELECT 
    task_name, 
    AVG(DATEDIFF('second', query_start_time, completed_time)) as avg_duration_sec
FROM snowflake.account_usage.task_history
WHERE root_task_name = 'dag_root_master'
  AND state = 'SUCCEEDED'
GROUP BY 1
ORDER BY 2 DESC;
```

### 6.4. Failed Tasks and Error Messages
```sql
SELECT 
    scheduled_time, 
    name, 
    error_code, 
    error_message
FROM TABLE(information_schema.task_history())
WHERE state = 'FAILED'
  AND scheduled_time >= DATEADD('day', -3, CURRENT_TIMESTAMP());
```

## 7. Key Takeaways

* A **DAG** is the standard for building **resilient, non-linear data pipelines** directly within the Snowflake ecosystem.
* **Parallelism** is a native feature; tasks with the same parent execute concurrently to optimize **pipeline latency**.
* The **Root Task** is the single source of scheduling truth; if it is suspended, the entire graph is dormant.
* **Fan-in architectures** provide a powerful mechanism for "joining" asynchronous data streams before final processing.
* Reliability is enhanced by **modularizing code** into small, idempotent tasks that can be individually monitored and retried.
