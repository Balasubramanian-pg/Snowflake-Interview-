# Snowflake FLATTEN: Exploding Semi-Structured Data into Relational Form

## 1. Core Concepts

The `FLATTEN` function in Snowflake is a **table function** that transforms **nested arrays and objects inside VARIANT columns into row sets**, enabling relational operations on semi-structured data.

It is not just a utility. It is a **cardinality multiplier**. Every invocation reshapes data, often dramatically increasing row count.

Used well, it unlocks analytics. Used carelessly, it detonates your warehouse.

---

### 1.1. How it Works

**Architecture Overview:**

* VARIANT stores arrays and objects as nested structures
* `FLATTEN` iterates over elements and emits rows
* Implemented as a **lateral table function**
* Produces metadata columns (`KEY`, `VALUE`, `INDEX`, `PATH`)

**Process Flow:**

1. JSON/VARIANT data is stored with nested arrays/objects
2. Query invokes `LATERAL FLATTEN(input => <variant_path>)`
3. Snowflake iterates through elements
4. Each element becomes a row in result set
5. Output is joined with original table via lateral join

**Primary Use Cases:**

* Expand arrays into rows for aggregation
* Traverse nested JSON structures
* Normalize semi-structured data
* Perform joins on nested elements

---

### 1.2. FLATTEN vs Traditional Normalization

| Metric              | Traditional Normalization | FLATTEN               |
| ------------------- | ------------------------- | --------------------- |
| Data Transformation | ETL upfront               | Query-time            |
| Flexibility         | Low                       | High                  |
| Storage Impact      | Increased tables          | No additional storage |
| Query Complexity    | Lower                     | Higher                |
| Performance         | High                      | Depends on usage      |
| Schema Evolution    | Hard                      | Easy                  |

---

### 1.3. Key Factors

1. **Row Explosion Risk**
   Arrays multiply rows; nested arrays multiply exponentially

2. **Lateral Join Behavior**
   Each row is processed independently with correlation

3. **Metadata Columns**
   Provides `KEY`, `VALUE`, `INDEX`, `PATH`, `SEQ`

4. **Selective Flattening**
   Target only required paths to avoid unnecessary expansion

5. **Filter Pushdown**
   Apply filters before flattening to reduce data volume

6. **Multi-Level Flattening Complexity**
   Nested FLATTEN calls increase query cost

## 2. Basic Commands & Privileges

### 2.1. Required Privileges

| Privilege    | Scope           | Description               |
| ------------ | --------------- | ------------------------- |
| SELECT       | Table           | Query VARIANT data        |
| USAGE        | Database/Schema | Access objects            |
| CREATE TABLE | Schema          | Persist flattened results |

---

### 2.2. Basic FLATTEN Usage

```sql
-- Expand array into rows
SELECT 
    f.value
FROM events,
LATERAL FLATTEN(input => payload:items) f;
```

Transforms array elements into rows.

---

### 2.3. Access Metadata Columns

```sql
-- Inspect FLATTEN metadata
SELECT 
    f.key,
    f.value,
    f.index,
    f.path
FROM events,
LATERAL FLATTEN(input => payload:items) f;
```

Provides structural context.

---

### 2.4. Filter Before Flatten

```sql
-- Apply filter before flattening
SELECT *
FROM events
WHERE payload:type = 'purchase';
```

Reduces data processed by FLATTEN.

---

### 2.5. Flatten Nested Arrays

```sql
-- Multi-level flatten
SELECT 
    f2.value
FROM events,
LATERAL FLATTEN(input => payload:items) f1,
LATERAL FLATTEN(input => f1.value:sub_items) f2;
```

Handles deep nesting.

---

### 2.6. Flatten with OUTER Option

```sql
-- Preserve rows with empty arrays
SELECT *
FROM events,
LATERAL FLATTEN(input => payload:items, OUTER => TRUE);
```

Ensures rows are retained even if array is empty.

---

### 2.7. Persist Flattened Data

```sql
-- Materialize flattened data
CREATE TABLE items_flat AS
SELECT 
    e.id,
    f.value:id::INT AS item_id
FROM events e,
LATERAL FLATTEN(input => e.payload:items) f;
```

Converts semi-structured data into relational table.

## 3. Impactful Use Cases

### Use Case 1: Expand Order Items

**Scenario** – Each order contains multiple items in an array.

```sql
SELECT 
    e.order_id,
    f.value:item_id::INT AS item_id
FROM orders e,
LATERAL FLATTEN(input => e.payload:items) f;
```

**Explanation** – Converts nested array into relational rows for aggregation and joins.

---

### Use Case 2: Aggregate Nested Data

**Scenario** – Count items per order.

```sql
SELECT 
    order_id,
    COUNT(*) AS item_count
FROM orders,
LATERAL FLATTEN(input => payload:items)
GROUP BY order_id;
```

**Explanation** – Enables aggregation over semi-structured arrays.

---

### Use Case 3: Join Nested Data with External Tables

**Scenario** – Join item IDs with product dimension.

```sql
SELECT 
    o.order_id,
    p.product_name
FROM orders o,
LATERAL FLATTEN(input => o.payload:items) f
JOIN products p
ON f.value:item_id::INT = p.product_id;
```

**Explanation** – Enables relational joins on extracted nested values.

---

### Use Case 4: Handle Deeply Nested Structures

**Scenario** – Extract sub-items within items.

```sql
SELECT 
    f2.value:detail::STRING AS detail
FROM orders,
LATERAL FLATTEN(input => payload:items) f1,
LATERAL FLATTEN(input => f1.value:sub_items) f2;
```

**Explanation** – Sequential flattening traverses nested hierarchies.

---

### Use Case 5: Preserve Empty Arrays

**Scenario** – Some records have no items.

```sql
SELECT *
FROM orders,
LATERAL FLATTEN(input => payload:items, OUTER => TRUE);
```

**Explanation** – Ensures no data loss for empty arrays.

---

### Use Case 6: Optimize Flatten with Filtering

**Scenario** – Only process relevant records.

```sql
SELECT *
FROM orders
WHERE payload:type = 'purchase',
LATERAL FLATTEN(input => payload:items);
```

**Explanation** – Reduces computational overhead by filtering early.

---

## 4. Best Practices Summary

| Practice                               | Why                        |
| -------------------------------------- | -------------------------- |
| Filter before flattening               | Reduce row explosion       |
| Flatten only required paths            | Minimize computation       |
| Use OUTER option carefully             | Avoid unintended null rows |
| Persist frequently used flattened data | Improve performance        |
| Limit multi-level flattening           | Control query complexity   |
| Extract key fields into columns        | Optimize analytics         |
| Monitor row counts post-flatten        | Detect explosion           |
| Use clustering on flattened tables     | Improve performance        |
| Combine with structured schema         | Balance flexibility        |

---

## 5. Important Limitations & Considerations

| Limitation                            | Mitigation                   |
| ------------------------------------- | ---------------------------- |
| Row explosion risk                    | Filter early and selectively |
| Complex queries for deep nesting      | Normalize data               |
| Performance overhead                  | Persist flattened results    |
| Difficult debugging                   | Use metadata columns         |
| No indexing on nested fields          | Extract to columns           |
| Multiple flatten layers increase cost | Limit depth                  |
