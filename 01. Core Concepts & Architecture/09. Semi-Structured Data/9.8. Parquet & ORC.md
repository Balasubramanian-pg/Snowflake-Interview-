# Snowflake Parquet & ORC: Columnar File Formats for Efficient Semi-Structured Ingestion

## 1. Core Concepts

**Parquet and ORC** are columnar file formats that integrate deeply with Snowflake to enable **high-performance ingestion and querying of semi-structured and structured data**.

Unlike JSON, which is row-oriented and verbose, these formats are **columnar, compressed, and metadata-rich**, making them ideal for analytics at scale.

Snowflake treats them as **externally structured sources** that can be mapped directly into tables or VARIANT columns with minimal transformation.

---

### 1.1. How it Works

**Architecture Overview:**

* Parquet/ORC store data in columnar blocks
* Each file contains metadata (schema, statistics)
* Snowflake reads metadata to optimize ingestion and query pruning
* Data can be loaded into structured columns or VARIANT

**Process Flow:**

1. Data is stored in Parquet/ORC files in external/internal stage
2. Snowflake reads file metadata (schema, column stats)
3. During load:

   * Columns mapped directly OR
   * Entire row stored as VARIANT
4. Query engine uses metadata for pruning
5. Only required columns are scanned

**Primary Use Cases:**

* Data lake ingestion (S3, ADLS, GCS)
* High-performance ETL pipelines
* Interoperability with Spark/Hadoop ecosystems
* Large-scale analytics workloads

---

### 1.2. Parquet vs ORC vs JSON

| Metric             | JSON      | Parquet   | ORC       |
| ------------------ | --------- | --------- | --------- |
| Format Type        | Row-based | Columnar  | Columnar  |
| Compression        | Low       | High      | High      |
| Schema             | Flexible  | Embedded  | Embedded  |
| Query Performance  | Moderate  | High      | High      |
| Storage Efficiency | Low       | High      | Very High |
| Ecosystem Support  | Universal | Very High | High      |

---

### 1.3. Key Factors

1. **Columnar Storage Efficiency**
   Only required columns are scanned, reducing I/O

2. **Embedded Schema**
   Schema is stored within file, enabling automatic mapping

3. **Compression Advantage**
   Significantly reduces storage footprint

4. **Predicate Pushdown**
   File-level metadata enables filtering before full scan

5. **Interoperability**
   Works seamlessly with Spark, Hive, and other big data tools

6. **Semi-Structured Compatibility**
   Nested structures supported natively

## 2. Basic Commands & Privileges

### 2.1. Required Privileges

| Privilege    | Scope  | Description               |
| ------------ | ------ | ------------------------- |
| USAGE        | Stage  | Access files              |
| CREATE TABLE | Schema | Create destination tables |
| INSERT       | Table  | Load data                 |
| SELECT       | Table  | Query loaded data         |

---

### 2.2. Create File Format for Parquet

```sql id="rb1hh7"
-- Define Parquet file format
CREATE FILE FORMAT parquet_format
TYPE = PARQUET;
```

Specifies how Snowflake reads Parquet files.

---

### 2.3. Create File Format for ORC

```sql id="4kh3xv"
-- Define ORC file format
CREATE FILE FORMAT orc_format
TYPE = ORC;
```

Defines ORC ingestion format.

---

### 2.4. Query Parquet Directly from Stage

```sql id="ws7gnr"
-- Query Parquet file without loading
SELECT *
FROM @my_stage (FILE_FORMAT => parquet_format);
```

Enables external table-like querying.

---

### 2.5. Load Parquet into Table

```sql id="4jrl7s"
-- Load structured data
COPY INTO events
FROM @my_stage
FILE_FORMAT = (TYPE = PARQUET);
```

Performs bulk ingestion.

---

### 2.6. Load into VARIANT

```sql id="3ox1d9"
-- Load entire row as VARIANT
INSERT INTO events
SELECT $1
FROM @my_stage (FILE_FORMAT => parquet_format);
```

Stores data as semi-structured.

---

### 2.7. Create External Table

```sql id="a9fz3g"
-- Create external table on Parquet files
CREATE EXTERNAL TABLE ext_events
WITH LOCATION = @my_stage
FILE_FORMAT = parquet_format;
```

Enables querying without loading.

## 3. Impactful Use Cases

### Use Case 1: Ingest Data Lake Files

**Scenario** – Load Parquet files from data lake into Snowflake.

```sql id="qtpb2t"
COPY INTO analytics.events
FROM @s3_stage/events/
FILE_FORMAT = (TYPE = PARQUET);
```

**Explanation** – Efficient ingestion leveraging columnar format and compression.

---

### Use Case 2: Query Without Loading (External Table)

**Scenario** – Analyze data without ingestion.

```sql id="hyk9pr"
SELECT *
FROM ext_events
WHERE event_type = 'click';
```

**Explanation** – External tables allow direct querying with minimal latency.

---

### Use Case 3: Hybrid Semi-Structured Storage

**Scenario** – Store Parquet data as VARIANT.

```sql id="ds13g7"
INSERT INTO raw_events
SELECT $1
FROM @stage (FILE_FORMAT => parquet_format);
```

**Explanation** – Enables flexible schema-on-read analytics.

---

### Use Case 4: Optimize Column-Level Queries

**Scenario** – Query only required columns.

```sql id="fb67dg"
SELECT user_id, event_type
FROM events;
```

**Explanation** – Columnar format ensures only relevant data is scanned.

---

### Use Case 5: Integrate with Spark Pipelines

**Scenario** – Data generated in Spark is consumed in Snowflake.

```sql id="65dru0"
COPY INTO snowflake_table
FROM @spark_output_stage
FILE_FORMAT = (TYPE = PARQUET);
```

**Explanation** – Seamless interoperability across platforms.

---

### Use Case 6: Reduce Storage Costs

**Scenario** – Optimize storage footprint.

```sql id="8dsuv3"
-- Load compressed Parquet data
COPY INTO events
FROM @stage
FILE_FORMAT = (TYPE = PARQUET);
```

**Explanation** – Compression reduces storage and improves performance.

---

## 4. Best Practices Summary

| Practice                            | Why                                |
| ----------------------------------- | ---------------------------------- |
| Prefer Parquet for large datasets   | Better compression and performance |
| Use external tables for exploration | Avoid unnecessary ingestion        |
| Load only required columns          | Reduce storage and compute         |
| Combine with clustering             | Improve pruning                    |
| Use VARIANT for flexible ingestion  | Handle schema evolution            |
| Monitor file sizes                  | Optimize ingestion performance     |
| Partition data in storage layer     | Improve query pruning              |
| Avoid small files                   | Reduce overhead                    |
| Align schema with query patterns    | Maximize performance               |

---

## 5. Important Limitations & Considerations

| Limitation                        | Mitigation                       |
| --------------------------------- | -------------------------------- |
| Small files degrade performance   | Consolidate files                |
| Schema mismatch issues            | Validate schema before load      |
| External tables have latency      | Load data for critical workloads |
| Complex nested structures         | Use VARIANT or flatten           |
| ORC less widely used than Parquet | Prefer Parquet                   |
| File-level errors during load     | Validate data upstream           |

## 7. Key Takeaways

* **Parquet and ORC are columnar formats optimized for analytics**
* **Snowflake leverages file metadata for pruning and performance**
* **Parquet is generally preferred due to ecosystem support**
* **External tables enable querying without ingestion**
* **Columnar storage reduces I/O and improves query efficiency**
* **File size and partitioning strategy significantly impact performance**
* **These formats are critical for modern data lake integration with Snowflake**
