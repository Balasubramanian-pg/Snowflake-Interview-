# Snowflake Column-Level Security: Through and Through

## 1. Core Concepts

Column‑level security in Snowflake controls access to **specific columns** within tables and views based on the executing user’s role or other contextual attributes. Unlike traditional RBAC that grants access to entire tables, column‑level security allows organisations to protect sensitive fields (PII, PHI, financial data) while still permitting access to non‑sensitive columns in the same table[reference:0].

### 1.1. Key Principles

- **Policy‑driven**: Access decisions are defined in reusable policies attached to columns, not hard‑coded into queries or views.
- **Query‑time enforcement**: Sensitive data is **not modified** in storage; masking or tokenisation is applied dynamically when a query executes[reference:1].
- **Transparent to users**: Authorised users see plain text; unauthorised users see masked, tokenised, or redacted values – the underlying table remains unchanged.

### 1.2. The Two Pillars of Column‑Level Security

| Feature | Description | Use Case |
|---------|-------------|----------|
| **Dynamic Data Masking (DDM)** | Masks data **at query time** using built‑in SQL functions or custom logic. Data is never altered on disk. | Hide email addresses, phone numbers, or salary figures based on user role. |
| **External Tokenisation** | Replaces sensitive data with **indecipherable tokens** before loading into Snowflake, and detokenises at query runtime using external functions. | Zero‑trust environments where even Snowflake admins should never see plaintext PII. |

### 1.3. Masking Policy Types

- **Column‑Based Masking Policy**: Attached directly to a specific column in a specific table or view. Provides tight, localised control[reference:2].
- **Tag‑Based Masking Policy (Recommended)**: A masking policy assigned to a Snowflake tag. Any column carrying that tag automatically inherits the masking logic across all tables and schemas. This approach is scalable, classification‑driven, and ideal for compliance with regulations like GDPR or HIPAA[reference:3].

### 1.4. Execution Order and Evaluation

Masking policies follow a defined evaluation order:
1. Policies on **tables** are executed first.
2. Policies on **views** are applied next.
3. For nested views, policies are applied sequentially left to right[reference:4].

**Anti‑pattern warning**: When a masked column appears in a `WHERE` clause or `JOIN` condition, Snowflake masks **only the column side** of the comparison, not the literal or the other column. This can produce unexpected results[reference:5]. For example:

```sql
-- Query intention: find email 'user@example.com'
SELECT email FROM customers WHERE email = 'user@example.com';

-- What actually executes (email is masked)
SELECT <masked_expression>(email) FROM customers 
WHERE <masked_expression>(email) = 'user@example.com';
```

Because the literal `'user@example.com'` is **not** masked, the comparison may never match. Snowflake recommends avoiding the use of protected columns in predicates or join conditions unless the policy is designed accordingly.

## 2. Basic Commands & Privileges

### 2.1. Required Privileges

| Privilege | Scope | Description |
|-----------|-------|-------------|
| `CREATE MASKING POLICY` | Schema | Allows creation of masking policies within a schema. |
| `APPLY MASKING POLICY` | Account | Controls who can set or unset masking policies on columns. Granted to `ACCOUNTADMIN` by default[reference:6]. |
| `APPLY ON MASKING POLICY` | Policy | Allows a policy owner to delegate [un]set operations to object owners[reference:7]. |
| `OWNERSHIP` on policy | Policy | Required to alter or drop a masking policy. |

**Recommendation**: Create a dedicated `MASKING_ADMIN` role and grant it `CREATE MASKING POLICY` on relevant schemas and `APPLY MASKING POLICY` on the account.

```sql
CREATE ROLE masking_admin;
GRANT CREATE MASKING POLICY ON SCHEMA my_db.my_schema TO ROLE masking_admin;
GRANT APPLY MASKING POLICY ON ACCOUNT TO ROLE masking_admin;
```

### 2.2. Create a Masking Policy

```sql
CREATE OR REPLACE MASKING POLICY mask_email AS (val STRING) RETURNS STRING ->
    CASE WHEN CURRENT_ROLE() IN ('HR_MANAGER', 'ACCOUNTADMIN') THEN val
         ELSE 'REDACTED'
    END;
```

### 2.3. Apply a Masking Policy to a Column

```sql
ALTER TABLE employees MODIFY COLUMN email SET MASKING POLICY mask_email;
```

### 2.4. Unset a Masking Policy from a Column

```sql
ALTER TABLE employees MODIFY COLUMN email UNSET MASKING POLICY;
```

### 2.5. Show Existing Masking Policies

```sql
SHOW MASKING POLICIES;
```

### 2.6. View Policies Applied to Columns

```sql
SELECT * FROM snowflake.account_usage.masking_policies;
SELECT * FROM snowflake.account_usage.policy_references;
```

## 3. Impactful Use Cases (4–8 with Queries & Explanations)

### Use Case 1: Role‑Based Email Masking (Basic DDM)

**Scenario** – The `employees` table contains an `email` column. Only the `HR_MANAGER` role should see full email addresses; all other roles should see `REDACTED`.

```sql
-- Create the masking policy
CREATE OR REPLACE MASKING POLICY mask_email AS (val STRING) RETURNS STRING ->
    CASE WHEN CURRENT_ROLE() = 'HR_MANAGER' THEN val
         ELSE 'REDACTED'
    END;

-- Apply to column
ALTER TABLE hr_db.employees MODIFY COLUMN email SET MASKING POLICY mask_email;

-- Test: As a regular analyst
USE ROLE ANALYST;
SELECT email FROM hr_db.employees;   -- All emails show 'REDACTED'

-- Test: As HR manager
USE ROLE HR_MANAGER;
SELECT email FROM hr_db.employees;   -- Shows actual email addresses
```

**Explanation** – The policy is evaluated for every query. The `CURRENT_ROLE()` function determines the active role of the user. This is the most common implementation of dynamic data masking[reference:8].

---

### Use Case 2: Tag‑Based Masking for PII Across Multiple Tables (Scalable Governance)

**Scenario** – Your organisation has 50 tables across 10 schemas, each containing columns with PII (email, phone, SSN). Instead of attaching policies to each column individually, use **tag‑based masking** with a tag named `PII='HIGH'`.

```sql
-- Step 1: Create a tag
CREATE TAG pii_classification;

-- Step 2: Apply the tag to columns across tables
ALTER TABLE sales_db.customers MODIFY COLUMN email SET TAG pii_classification = 'HIGH';
ALTER TABLE sales_db.customers MODIFY COLUMN phone SET TAG pii_classification = 'HIGH';
ALTER TABLE hr_db.employees MODIFY COLUMN ssn SET TAG pii_classification = 'HIGH';
-- ... repeat for all sensitive columns

-- Step 3: Create one masking policy for the tag
CREATE OR REPLACE MASKING POLICY mask_pii_high AS (val STRING) RETURNS STRING ->
    CASE WHEN CURRENT_ROLE() IN ('DATA_GOVERNANCE', 'AUDIT') THEN val
         ELSE '***MASKED***'
    END;

-- Step 4: Attach the policy to the tag
ALTER TAG pii_classification SET MASKING POLICY mask_pii_high;

-- Now, any column tagged with PII='HIGH' automatically enforces the masking logic
```

**Explanation** – Tag‑based masking reduces management overhead dramatically. When a new table with PII columns is added, you simply apply the existing tag, and the policy is enforced automatically. This approach is recommended by Snowflake for large‑scale compliance[reference:9].

### Use Case 3: Conditional Masking Based on a Different Column Value

**Scenario** – The `patients` table has a `diagnosis` column that should only be visible to doctors assigned to that patient. A `doctor_id` column indicates which doctor is responsible.

```sql
-- Lookup table mapping doctors to patients
CREATE TABLE patient_doctor (
    patient_id INT,
    doctor_name STRING
);
INSERT INTO patient_doctor VALUES (1, 'DR_SMITH'), (2, 'DR_JONES');

-- Create conditional masking policy
CREATE OR REPLACE MASKING POLICY conditional_diagnosis AS 
    (val STRING, patient_id INT) RETURNS STRING ->
    CASE WHEN CURRENT_ROLE() IN (
            SELECT doctor_name FROM patient_doctor 
            WHERE patient_doctor.patient_id = patient_id
         ) THEN val
         ELSE 'RESTRICTED'
    END;

-- Apply to diagnosis column (policy now references two columns)
ALTER TABLE patients MODIFY COLUMN diagnosis 
    SET MASKING POLICY conditional_diagnosis USING (patient_id);
```

**Explanation** – Conditional masking allows the policy to reference **other columns** in the same row to decide whether to reveal the protected column. In this example, the `diagnosis` column is masked unless the current role matches the `doctor_name` associated with that `patient_id`. This provides fine‑grained, row‑dependent column protection[reference:10].

**Important limitation**: A column cannot be both masked **and** used as the conditional argument at the same time[reference:11].

### Use Case 4: Partial Masking – Show Only Last 4 Digits

**Scenario** – Customer support representatives need to verify credit card numbers but should only see the last 4 digits for security.

```sql
-- Create partial masking policy
CREATE OR REPLACE MASKING POLICY mask_credit_card AS (val STRING) RETURNS STRING ->
    CASE WHEN CURRENT_ROLE() = 'SUPPORT_REP' 
         THEN CONCAT('****-****-****-', RIGHT(val, 4))
         WHEN CURRENT_ROLE() IN ('FRAUD_TEAM', 'FINANCE') THEN val
         ELSE 'INVALID'
    END;

-- Apply to credit_card column
ALTER TABLE payments.modify COLUMN credit_card_number 
    SET MASKING POLICY mask_credit_card;
```

**Explanation** – Partial masking preserves data utility (e.g., for verification) while hiding the full sensitive value. The `SUPPORT_REP` role sees `****-****-****-1234` instead of the full 16‑digit number. This is a common requirement for PCI‑DSS compliance.

### Use Case 5: Hashing Instead of Masking (Irreversible Obfuscation)

**Scenario** – For analytics use cases where you need consistent identifiers but must never reveal the original PII (e.g., user email for join operations), use a **deterministic hash**.

```sql
CREATE OR REPLACE MASKING POLICY hash_email AS (val STRING) RETURNS STRING ->
    CASE WHEN CURRENT_ROLE() IN ('ANALYTICS_ENGINE', 'DATA_SCIENCE') 
         THEN SHA2(val, 256)   -- Returns a fixed-length hash
         WHEN CURRENT_ROLE() = 'COMPLIANCE' THEN val
         ELSE NULL
    END;

ALTER TABLE user_activity MODIFY COLUMN user_email SET MASKING POLICY hash_email;
```

**Explanation** – Hashing is **one‑way** (irreversible) and deterministic: the same email always produces the same hash. This allows analysts to join tables on the hashed value without ever seeing the original email. Use `SHA2(val, 256)` for secure, collision‑resistant hashing. Note that hashing is not encryption – you cannot recover the original value.

### Use Case 6: External Tokenisation with an External Function (Zero‑Trust Architecture)

**Scenario** – Your organisation operates a zero‑trust model where even Snowflake admins should never see plaintext PII. Data is tokenised **before** loading into Snowflake, and detokenised only for authorised roles at query time via an external tokenisation service (e.g., AWS KMS, Fortanix, Protegrity).

```sql
-- Step 1: Create an external function pointing to your tokenisation service
-- (Configuration depends on your cloud provider and tokenisation vendor)
CREATE OR REPLACE EXTERNAL FUNCTION detokenize(VAL STRING)
    RETURNS STRING
    API_INTEGRATION = tokenization_api
    AS 'https://your-tokenization-service.com/detokenize';

-- Step 2: Create a masking policy that calls the external function
CREATE OR REPLACE MASKING POLICY tokenize_ssn AS (val STRING) RETURNS STRING ->
    CASE WHEN CURRENT_ROLE() = 'AUDITOR' 
         THEN detokenize(val)   -- External function reverses the token
         ELSE val               -- Stored token value remains unchanged
    END;

-- Step 3: Apply to the column
ALTER TABLE citizens MODIFY COLUMN ssn_token SET MASKING POLICY tokenize_ssn;
```

**Explanation** – External tokenisation leverages **masking policies combined with external functions** to tokenise data before it ever enters Snowflake and detokenise it at query runtime[reference:12][reference:13]. The actual plaintext never resides in Snowflake storage. The external function calls out to a third‑party tokenisation vault (e.g., Fortanix, Protegrity, or a custom AWS Lambda) to perform the detokenisation[reference:14].

**Requirements**: Enterprise Edition or higher, and an external tokenisation partner integration[reference:15].

### Use Case 7: Combine Column‑Level Security with Row‑Level Security

**Scenario** – You need both row‑level filtering (sales reps see only their own region) **and** column‑level masking (salary column hidden from junior reps).

```sql
-- Create a row access policy (filters rows)
CREATE OR REPLACE ROW ACCESS POLICY region_filter AS (region STRING) RETURNS BOOLEAN ->
    CASE WHEN CURRENT_ROLE() = 'SALES_EXEC' THEN TRUE
         ELSE region = CURRENT_REGION()
    END;

-- Create a column masking policy (masks salary)
CREATE OR REPLACE MASKING POLICY mask_salary AS (val NUMBER) RETURNS NUMBER ->
    CASE WHEN CURRENT_ROLE() = 'SALES_EXEC' THEN val
         ELSE NULL
    END;

-- Apply both to the same table
ALTER TABLE sales_data ADD ROW ACCESS POLICY region_filter ON (region);
ALTER TABLE sales_data MODIFY COLUMN salary SET MASKING POLICY mask_salary;
```

**Explanation** – Snowflake evaluates **row‑level policies first**, then **column‑level policies** on the remaining rows. This layered approach provides defence in depth. A junior sales rep cannot see rows outside their region **and** cannot see salary values even for rows they are allowed to access.

### Use Case 8: Audit and Monitor Masking Policy Usage

**Scenario** – Your compliance team needs a monthly report showing which columns have masking policies, who created them, and when they were last modified.

```sql
-- List all masking policies in the account
SELECT 
    policy_name,
    policy_schema,
    policy_database,
    created_on,
    last_altered,
    owner
FROM snowflake.account_usage.masking_policies
ORDER BY created_on DESC;

-- Show which columns are protected by which policies
SELECT 
    p.policy_name,
    ref.object_name AS table_name,
    ref.column_name,
    ref.tag_name,
    ref.policy_status
FROM snowflake.account_usage.policy_references ref
JOIN snowflake.account_usage.masking_policies p 
    ON ref.policy_id = p.policy_id
WHERE ref.policy_kind = 'MASKING_POLICY';

-- Track changes to masking policies
SELECT 
    query_id,
    query_text,
    user_name,
    role_name,
    start_time
FROM snowflake.account_usage.query_history
WHERE query_text ILIKE '%MASKING POLICY%'
  AND start_time >= DATEADD(day, -30, CURRENT_TIMESTAMP())
ORDER BY start_time DESC;
```

**Explanation** – Snowflake provides system views (`account_usage.masking_policies`, `account_usage.policy_references`) to track all masking policies across your account. Use these for compliance audits, change management, and troubleshooting.

## 4. Best Practices Summary

| Practice | Why |
|----------|-----|
| **Prefer tag‑based masking over column‑based** | Scalable, centralised management; policy changes automatically apply to all tagged columns[reference:16]. |
| **Create a dedicated `MASKING_ADMIN` role** | Separates policy management from object ownership; follows least‑privilege principle. |
| **Avoid using masked columns in `WHERE` or `JOIN` predicates** | Masking only one side of a comparison leads to unexpected results[reference:17]. |
| **Test policies thoroughly before production** | Masking can break existing queries; use a staging environment. |
| **Do not rely solely on masking for compliance** | Masking is query‑time only; users with `OWNERSHIP` can alter or remove policies. Use managed access schemas for enforcement. |
| **Monitor policy references after schema changes** | Dropping a table or column does not automatically drop policy references; review periodically. |
| **Use external tokenisation for zero‑trust requirements** | Plaintext never stored in Snowflake; ideal for highly regulated industries (healthcare, finance). |
| **Document all masking policies** | Include business justification, authorised roles, and masking logic for auditors. |

## 5. Important Limitations & Considerations

| Limitation | Mitigation |
|------------|------------|
| Masking policies cannot be applied to columns in shared databases (e.g., Snowflake Sample Data)[reference:18]. | Copy the data into your own database before applying policies. |
| A column cannot be both masked **and** used as a conditional argument. | Use a separate column for conditional logic. |
| Future grants (`GRANT ... ON FUTURE`) are not supported for masking policies[reference:19]. | Grant `APPLY MASKING POLICY` to roles at the account level. |
| External tokenisation requires Enterprise Edition or higher[reference:20]. | Upgrade edition or use dynamic data masking instead. |
| Masking policies can impact query performance. | Test with representative workloads; avoid complex subqueries in policy bodies. |

---

## 6. Key Takeaways

- **Column‑level security** in Snowflake is implemented via **masking policies** – schema‑level objects that dynamically transform data at query time.
- Two approaches: **Dynamic Data Masking** (built‑in, no external dependencies) and **External Tokenisation** (zero‑trust, requires external function).
- **Tag‑based masking** is the recommended pattern for scalable, classification‑driven governance across hundreds of columns.
- Masking policies are **transparent to authorised users** and **invisible to unauthorised users** – the underlying table never changes.
- Always test masking policies on a copy of production data and **avoid using masked columns in WHERE/JOIN clauses**.
- Use Snowflake’s `account_usage` views to **audit and monitor** all masking policy activity for compliance.

This completes the thorough explanation of Snowflake’s column‑level security model with practical, actionable use cases.
