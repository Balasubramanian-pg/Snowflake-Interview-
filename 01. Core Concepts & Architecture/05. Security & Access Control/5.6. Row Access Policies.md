# Snowflake Row Access Policies: Through and Through

## 1. Core Concepts

Row Access Policies (RAP) control **which rows** of a table or view a user can see **at query time**. Unlike a `WHERE` clause embedded in a view, a row access policy is a **reusable, schema‑level object** that enforces row‑level security (RLS) automatically on every query against the protected table – regardless of how the user queries it (SELECT, UPDATE, DELETE, or JOIN).

### 1.1. Key Principles

- **Dynamic filtering**: Policy returns `TRUE` or `FALSE` for each row; if `FALSE`, the row is excluded from the result set.
- **Policy arguments**: At least one argument (usually a column from the table) used to decide access.
- **Context‑aware**: Can use `CURRENT_ROLE()`, `CURRENT_USER()`, `SESSION_USER()`, or any other function.
- **Automatic enforcement**: Cannot be bypassed by the user (except by roles with `APPLY ROW ACCESS POLICY` or `OWNERSHIP` with `OVERRIDE` privileges).
- **Evaluation order**: Applied after `WHERE` clause of the user’s query, but before aggregation (`GROUP BY`, `HAVING`), window functions, and column masking policies.

### 1.2. Policy Structure

```sql
CREATE [ OR REPLACE ] ROW ACCESS POLICY <name>
  AS ( <arg_name> <arg_type> [, ...] ) RETURNS BOOLEAN ->
  <body>
```

- **Arguments**: One or more columns from the table (or expressions) used to determine access.
- **Return type**: Must be `BOOLEAN`.
- **Body**: SQL expression that returns `TRUE` (row visible) or `FALSE` (row hidden).

### 1.3. How It Works

When a user queries a table with a row access policy:

1. Snowflake rewrites the query to include the policy logic.
2. The policy function is evaluated **for each row** (or as an optimized predicate).
3. Rows where the policy returns `FALSE` are filtered out **before** column masking, aggregations, or joins.

### 1.4. Row Access Policy vs Secure View

| Aspect | Row Access Policy | Secure View |
|--------|------------------|--------------|
| **Reusability** | Attached to table; works for all queries | Only works when users query the specific view |
| **Bypass risk** | Cannot be bypassed (except explicit override) | Users can query the underlying table directly if they have privileges |
| **Maintenance** | One policy per table; changes propagate instantly | Each view must be maintained separately |
| **Performance** | Generally faster because policy is pushed down | May be slower if view contains complex logic |
| **Transparency** | User unaware of filtering | User knows they are querying a view |

**Recommendation**: Use row access policies when you need mandatory, unbypassable row filtering on a table that users must access directly. Use secure views when you need different filtering logic for different use cases (e.g., sales rep view vs finance view).

## 2. Basic Commands & Privileges

### 2.1. Required Privileges

| Privilege | Scope | Description |
|-----------|-------|-------------|
| `CREATE ROW ACCESS POLICY` | Schema | Allows creation of RAP in a schema. |
| `APPLY ROW ACCESS POLICY` | Account | Allows a role to attach/detach RAP to tables/views. |
| `APPLY ON ROW ACCESS POLICY` | Policy | Allows a policy owner to delegate [un]set operations. |
| `OWNERSHIP` | Policy | Required to alter or drop the policy. |
| `OVERRIDE ROW ACCESS POLICY` | Table | Allows a role to bypass the policy (rare, for admin). |

### 2.2. Create a Row Access Policy

```sql
CREATE OR REPLACE ROW ACCESS POLICY sales_region_policy
  AS (region STRING) RETURNS BOOLEAN ->
    CASE WHEN CURRENT_ROLE() = 'SALES_ADMIN' THEN TRUE
         ELSE region = CURRENT_REGION()
    END;
```

### 2.3. Apply Policy to a Table

```sql
ALTER TABLE sales_data ADD ROW ACCESS POLICY sales_region_policy ON (region);
```

### 2.4. Remove Policy from a Table

```sql
ALTER TABLE sales_data DROP ROW ACCESS POLICY sales_region_policy;
```

### 2.5. Show Row Access Policies

```sql
SHOW ROW ACCESS POLICIES;
```

### 2.6. View Policy References

```sql
SELECT * FROM snowflake.account_usage.row_access_policies;
SELECT * FROM snowflake.account_usage.policy_references 
WHERE policy_kind = 'ROW_ACCESS_POLICY';
```

### 2.7. Bypass Policy for a Role (Emergency)

```sql
-- Grant override privilege to a specific role
GRANT OVERRIDE ROW ACCESS POLICY ON TABLE sales_data TO ROLE audit_role;
```

## 3. Impactful Use Cases (4–8 with Queries & Explanations)

### Use Case 1: Region‑Based Row Filtering (Basic RAP)

**Scenario** – Sales representatives should only see rows belonging to their assigned region. The `sales` table has a `region` column.

```sql
-- Create a mapping from role to region (simplified; in practice, use a mapping table)
CREATE OR REPLACE ROW ACCESS POLICY region_policy
  AS (region STRING) RETURNS BOOLEAN ->
    CASE WHEN CURRENT_ROLE() = 'SALES_EMEA' THEN region = 'EMEA'
         WHEN CURRENT_ROLE() = 'SALES_NA' THEN region = 'NA'
         WHEN CURRENT_ROLE() = 'SALES_APAC' THEN region = 'APAC'
         WHEN CURRENT_ROLE() = 'SALES_ADMIN' THEN TRUE
         ELSE FALSE
    END;

-- Apply to table
ALTER TABLE sales ADD ROW ACCESS POLICY region_policy ON (region);

-- Test as SALES_EMEA
USE ROLE SALES_EMEA;
SELECT * FROM sales;  -- Only rows where region = 'EMEA'
```

**Explanation** – The policy inspects the current role and filters rows accordingly. `SALES_ADMIN` sees all rows; any other role sees nothing. This is the simplest and most common RAP pattern.

---

### Use Case 2: Policy Based on a Mapping Table (Dynamic Role Assignment)

**Scenario** – You have a mapping table that links roles to regions. Instead of hardcoding roles in the policy, query the mapping table dynamically.

```sql
-- Mapping table
CREATE TABLE region_role_map (role_name STRING, region STRING);
INSERT INTO region_role_map VALUES 
  ('SALES_EMEA', 'EMEA'), ('SALES_NA', 'NA'), ('SALES_APAC', 'APAC');

-- Row access policy referencing the mapping table
CREATE OR REPLACE ROW ACCESS POLICY dynamic_region_policy
  AS (region STRING) RETURNS BOOLEAN ->
    EXISTS (
      SELECT 1 FROM region_role_map
      WHERE role_name = CURRENT_ROLE()
        AND region = dynamic_region_policy.region
    ) OR CURRENT_ROLE() = 'SALES_ADMIN';

ALTER TABLE sales ADD ROW ACCESS POLICY dynamic_region_policy ON (region);
```

**Explanation** – The policy uses a correlated subquery to check the mapping table. This centralises region assignments; adding a new region or role only requires updating the mapping table, not the policy. Note that the policy will be evaluated for each row, so ensure the mapping table is small or indexed.

---

### Use Case 3: Row Access Based on User Attributes (Using a Lookup Table)

**Scenario** – Each user can see only records where `user_id` equals their own session user (e.g., for a personal dashboard). The table `user_data` contains a `user_id` column.

```sql
CREATE OR REPLACE ROW ACCESS POLICY user_self_policy
  AS (user_id INT) RETURNS BOOLEAN ->
    user_id = CURRENT_USER()  -- or SESSION_USER()
    OR CURRENT_ROLE() = 'ADMIN';

ALTER TABLE user_data ADD ROW ACCESS POLICY user_self_policy ON (user_id);
```

**Explanation** – `CURRENT_USER()` returns the username (e.g., `'ALICE'`). If the `user_id` column stores numeric IDs, you might need a mapping table. Adjust the policy accordingly:

```sql
-- If user_id is integer, map username to ID
CREATE OR REPLACE ROW ACCESS POLICY user_self_policy
  AS (user_id INT) RETURNS BOOLEAN ->
    user_id = (SELECT id FROM user_accounts WHERE username = CURRENT_USER())
    OR CURRENT_ROLE() = 'ADMIN';
```

This pattern is ideal for multi‑tenant applications where each user owns their own data.

---

### Use Case 4: Combine Row Access Policy with Dynamic Data Masking

**Scenario** – Managers see all rows and full salary columns; employees see only their own row and a masked salary.

```sql
-- Row access policy: employees see only their own row
CREATE OR REPLACE ROW ACCESS POLICY emp_row_policy
  AS (emp_id INT) RETURNS BOOLEAN ->
    CURRENT_ROLE() = 'HR_MANAGER' 
    OR emp_id = (SELECT id FROM employees WHERE username = CURRENT_USER());

-- Column masking policy: employees see salary as NULL
CREATE OR REPLACE MASKING POLICY salary_mask AS (val NUMBER) RETURNS NUMBER ->
    CASE WHEN CURRENT_ROLE() = 'HR_MANAGER' THEN val
         ELSE NULL
    END;

-- Apply both to the employees table
ALTER TABLE employees ADD ROW ACCESS POLICY emp_row_policy ON (emp_id);
ALTER TABLE employees MODIFY COLUMN salary SET MASKING POLICY salary_mask;
```

**Explanation** – The row policy limits which rows are visible, and the masking policy further restricts the salary column on the visible rows. Evaluation order: row policy first, then column masking. An employee sees only their own row and with `NULL` salary; the manager sees all rows and real salaries.

### Use Case 5: Time‑Based Row Access (e.g., Only Future Orders)

**Scenario** – A shipping dashboard should only show orders scheduled for the next 7 days. Older orders should be hidden.

```sql
CREATE OR REPLACE ROW ACCESS POLICY future_orders_policy
  AS (delivery_date DATE) RETURNS BOOLEAN ->
    delivery_date BETWEEN CURRENT_DATE() AND CURRENT_DATE() + 7
    OR CURRENT_ROLE() = 'SHIPPING_ADMIN';

ALTER TABLE orders ADD ROW ACCESS POLICY future_orders_policy ON (delivery_date);
```

**Explanation** – The policy dynamically filters based on a date column and the current date. This is useful for operational dashboards, SLA monitoring, or data retention windows.

---

### Use Case 6: Complex Policy Using Multiple Columns (AND/OR Logic)

**Scenario** – A support ticket system: agents can see tickets they own OR tickets escalated to their team. The `tickets` table has `owner_id` and `escalated_team` columns.

```sql
-- Assume a mapping from role to agent_id and team
CREATE OR REPLACE ROW ACCESS POLICY ticket_policy
  AS (owner_id INT, escalated_team STRING) RETURNS BOOLEAN ->
    CASE 
      WHEN CURRENT_ROLE() = 'SUPPORT_ADMIN' THEN TRUE
      WHEN CURRENT_ROLE() = 'AGENT_1' THEN owner_id = 1
      WHEN CURRENT_ROLE() = 'TEAM_LEAD' THEN escalated_team = 'TIER2'
      ELSE FALSE
    END;

ALTER TABLE tickets ADD ROW ACCESS POLICY ticket_policy ON (owner_id, escalated_team);
```

**Explanation** – The policy can reference **multiple columns** by listing them in the `ON` clause. This allows complex business rules. The policy function must accept all those columns as arguments.

---

### Use Case 7: Override Policy for Auditing (Emergency Access)

**Scenario** – An internal auditor needs to see all rows of a protected table for compliance checks, but you don’t want to modify the policy.

```sql
-- Grant OVERRIDE privilege to the audit role
GRANT OVERRIDE ROW ACCESS POLICY ON TABLE sensitive_table TO ROLE audit_role;

-- Auditor can now see all rows
USE ROLE audit_role;
SELECT * FROM sensitive_table;  -- bypasses row access policy

-- Regular user still sees only filtered rows
USE ROLE analyst;
SELECT * FROM sensitive_table;  -- policy enforced
```

**Explanation** – `OVERRIDE` allows specific roles to ignore the row access policy. Use this sparingly and audit its usage. To revoke:

```sql
REVOKE OVERRIDE ROW ACCESS POLICY ON TABLE sensitive_table FROM ROLE audit_role;
```

---

### Use Case 8: Row Access on Shared Data (Secure Data Sharing)

**Scenario** – You share a table with a consumer account, but you want the consumer to see only rows where `share_flag = TRUE`.

```sql
-- In provider account
CREATE OR REPLACE ROW ACCESS POLICY share_policy
  AS (share_flag BOOLEAN) RETURNS BOOLEAN ->
    share_flag = TRUE;

ALTER TABLE shared_data ADD ROW ACCESS POLICY share_policy ON (share_flag);

-- Create a share and add the table
CREATE SHARE customer_share;
GRANT USAGE ON DATABASE shared_db TO SHARE customer_share;
GRANT SELECT ON TABLE shared_data TO SHARE customer_share;
```

**Explanation** – Row access policies are **enforced even on shared tables**. The consumer account cannot bypass the policy. This is a powerful way to provide row‑filtered data to external partners without creating separate views or tables.

## 4. Best Practices Summary

| Practice | Why |
|----------|-----|
| **Keep policies simple** | Complex policies with subqueries can degrade performance; evaluate per row. |
| **Use mapping tables for role–attribute relationships** | Avoid hardcoding roles inside policies; easier to maintain and audit. |
| **Test policies with representative roles** | Ensure that unintended roles see too little (or too much). |
| **Combine with column masking** | Defence in depth – hide sensitive columns even on visible rows. |
| **Use `OVERRIDE` sparingly** | Grant only to trusted audit/admin roles; log all override usage. |
| **Prefer row access policies over secure views** | When you need mandatory, unbypassable filtering on base tables. |
| **Monitor policy performance** | Use `QUERY_HISTORY` to spot queries with high evaluation cost. |
| **Document policy logic** | Future administrators need to understand the business rules. |

## 5. Important Limitations & Considerations

| Limitation | Mitigation |
|------------|------------|
| **One active policy per table** | A table can have only one row access policy at a time. If you need multiple rules, combine them inside the policy body using `AND`/`OR`. |
| **Policy arguments must be columns** | You cannot use expressions directly in the `ON` clause (e.g., `UPPER(country)`). Create a derived column or use a view. |
| **No `JOIN` inside policy body?** | Policy body can contain subqueries and `EXISTS` but may impact performance. Avoid correlated subqueries on large tables. |
| **Views inherit table policies** | If you create a view on a table with a row access policy, the view automatically enforces the policy. |
| **Materialised views and row policies** | Materialised views may not be refreshable if the underlying table has a row access policy. Test carefully. |
| **Streams and tasks** | Row access policies apply to streams and tasks; ensure the task’s execution role has necessary visibility. |
| **Policy changes require table alteration** | Changing a policy definition (`CREATE OR REPLACE`) does not automatically affect attached tables; you must re‑apply or use `ALTER TABLE ... MODIFY ROW ACCESS POLICY`. |

## 6. Auditing & Monitoring Queries

### View all row access policies
```sql
SHOW ROW ACCESS POLICIES;
```

### Which tables have which policies?
```sql
SELECT 
    p.policy_name,
    ref.object_name AS table_name,
    ref.object_schema,
    ref.object_database,
    ref.argument_signature
FROM snowflake.account_usage.policy_references ref
JOIN snowflake.account_usage.row_access_policies p 
  ON ref.policy_id = p.policy_id
WHERE ref.policy_kind = 'ROW_ACCESS_POLICY';
```

### Track queries that were filtered by a row access policy
```sql
SELECT 
    query_id,
    user_name,
    role_name,
    start_time,
    rows_produced,
    rows_available
FROM snowflake.account_usage.query_history
WHERE rows_produced < rows_available  -- Some rows filtered
AND query_text NOT LIKE '%OVERRIDE%'
LIMIT 100;
```

### Who has OVERRIDE privileges on tables?
```sql
SELECT * FROM snowflake.account_usage.grants_to_roles
WHERE privilege = 'OVERRIDE ROW ACCESS POLICY';
```

## 7. Key Takeaways

- **Row Access Policies** provide mandatory, unbypassable row‑level filtering directly on tables.
- They are **dynamic** – evaluated at query time based on role, user, or any column value.
- **Single policy per table** – combine multiple conditions inside the policy body.
- Use **mapping tables** to decouple role‑to‑attribute assignments from policy logic.
- **Combine with column masking** for comprehensive, multi‑dimensional security.
- **Audit carefully** – `OVERRIDE` privilege bypasses all row policies; monitor its use.
- **Performance matters** – simple equality checks are fast; correlated subqueries can be slow.
- **Row policies apply to shares, views, streams** – consistent security across all access paths.

This completes the thorough explanation of Snowflake’s Row Access Policies with practical, actionable use cases.
