# Snowflake Network Policies: Through and Through

## 1. Core Concepts

Network policies in Snowflake control which **IP addresses or CIDR ranges** can access your Snowflake account. They are a fundamental layer of defence, restricting incoming connections to a trusted network – for example, your corporate VPN, office IP ranges, or specific cloud NAT gateways.

### 1.1. Key Principles

- **IP allowlisting / blocklisting**: Define a set of allowed IPs and optionally a set of blocked IPs.
- **Account‑level vs User‑level**:
  - **Account‑level policy** applies to **all** users of the account (unless overridden by a user‑level policy).
  - **User‑level policy** applies only to a specific user, overriding the account policy for that user.
- **IPv4 only**: Snowflake network policies support IPv4 addresses (CIDR notation). IPv6 is not supported.
- **Connection‑time enforcement**: The policy is evaluated when a connection is initiated – before authentication. If the source IP is not allowed, the connection is refused with a “policy violation” error.

### 1.2. How It Works

1. User initiates a connection (Snowsight, SnowSQL, JDBC, etc.).
2. Snowflake extracts the client’s **public IP address** (the one seen by Snowflake).
3. If the account has a network policy, Snowflake checks:
   - If a **user‑level policy** exists for that user, it is used.
   - Otherwise, the **account‑level policy** is used.
4. The policy applies:
   - `ALLOWED_IP_LIST` – only IPs in this list can connect (allowlist).
   - `BLOCKED_IP_LIST` – IPs in this list are denied, even if also in the allowlist (blocklist takes precedence).
5. If the source IP matches neither (or is blocked), the connection is rejected.

### 1.3. Policy Precedence

```
User‑level policy (if exists)
    └─ If no user‑level policy, use account‑level policy.
```

A user‑level policy completely **replaces** the account policy for that user – it does not merge with it.

### 1.4. Interaction with Other Security Features

| Feature | Interaction |
|---------|-------------|
| **MFA** | Network policies work alongside MFA. A user must come from an allowed IP **and** satisfy MFA (if required). |
| **OAuth / SAML** | Network policies are evaluated after the IdP redirect? **Important**: For federated authentication (SAML, OAuth), the source IP is the **client** IP, not the IdP IP. The policy still applies. |
| **Private connectivity (AWS PrivateLink, Azure Private Link, GCP Private Service Connect)** | Network policies are evaluated **after** the private link termination. The client IP seen is the **private IP** (if NAT is not used) or the NAT IP. Test carefully. |

## 2. Basic Commands & Privileges

### 2.1. Required Privileges

| Privilege | Scope | Description |
|-----------|-------|-------------|
| `CREATE NETWORK POLICY` | Account | Allows creation of network policies. Granted to `ACCOUNTADMIN` by default. |
| `OWNERSHIP` on policy | Policy | Required to alter or drop a network policy. |
| `APPLY NETWORK POLICY` | Account | Allows attaching a network policy to the account. |
| `APPLY NETWORK POLICY` on user | User | Allows attaching a network policy to a specific user (or `ALL USERS`). |

**Recommended setup**: Create a `NETWORK_ADMIN` role.

```sql
CREATE ROLE network_admin;
GRANT CREATE NETWORK POLICY ON ACCOUNT TO ROLE network_admin;
GRANT APPLY NETWORK POLICY ON ACCOUNT TO ROLE network_admin;
```

### 2.2. Create a Network Policy

```sql
CREATE OR REPLACE NETWORK POLICY corporate_policy
  ALLOWED_IP_LIST = ('203.0.113.0/24', '198.51.100.5')
  BLOCKED_IP_LIST = ('203.0.113.100');
```

- `ALLOWED_IP_LIST` – max 1000 entries (each an IPv4 address or CIDR).
- `BLOCKED_IP_LIST` – max 1000 entries. Entries in `BLOCKED_IP_LIST` take precedence.

### 2.3. Attach Policy to the Account

```sql
ALTER ACCOUNT SET NETWORK POLICY = corporate_policy;
```

### 2.4. Attach Policy to a Specific User (Override)

```sql
ALTER USER service_user SET NETWORK POLICY = service_network_policy;
```

### 2.5. Remove Policy from Account or User

```sql
ALTER ACCOUNT UNSET NETWORK POLICY;
ALTER USER service_user UNSET NETWORK POLICY;
```

### 2.6. Show and Describe Policies

```sql
SHOW NETWORK POLICIES;
DESC NETWORK POLICY corporate_policy;
```

### 2.7. Modify an Existing Policy

```sql
ALTER NETWORK POLICY corporate_policy
  SET ALLOWED_IP_LIST = ('203.0.113.0/24', '198.51.100.5', '192.0.2.10');
```

**Note**: `ALTER` replaces the entire list – you must provide the full list of allowed IPs each time.

## 3. Impactful Use Cases (4–8 with Queries & Explanations)

### Use Case 1: Restrict Access to Corporate VPN (Account‑Level)

**Scenario** – All employees must connect through the corporate VPN, which has a public IP range `203.0.113.0/24`. No other IPs should be allowed.

```sql
-- Create the policy
CREATE OR REPLACE NETWORK POLICY corp_vpn_only
  ALLOWED_IP_LIST = ('203.0.113.0/24')
  BLOCKED_IP_LIST = ();  -- empty blocklist

-- Apply to account
ALTER ACCOUNT SET NETWORK POLICY = corp_vpn_only;

-- Verify
SHOW PARAMETERS LIKE 'network_policy' IN ACCOUNT;
```

**Explanation** – Users who try to connect from home (without VPN) will receive error `250008 (42501): Network policy 'CORP_VPN_ONLY' violation: IP '<home_ip>' is not allowed`. This forces all traffic through the VPN, simplifying logging and compliance.

---

### Use Case 2: Allow External Service Account from a Fixed NAT IP (User‑Level Override)

**Scenario** – An ETL service running on AWS Lambda (which uses a dynamic IP range) must connect to Snowflake. You cannot allow the entire AWS IP range. Instead, route the Lambda through a **NAT Gateway** with a static IP `34.120.100.50`. Only this service user should be allowed from that IP; other users must still use the corporate VPN.

```sql
-- Step 1: Create a network policy for the service user
CREATE OR REPLACE NETWORK POLICY lambda_etl_policy
  ALLOWED_IP_LIST = ('34.120.100.50')
  BLOCKED_IP_LIST = ();

-- Step 2: Attach it to the service user (overrides account policy)
ALTER USER etl_service_user SET NETWORK POLICY = lambda_etl_policy;

-- Step 3: Ensure the account policy still applies to other users
ALTER ACCOUNT SET NETWORK POLICY = corp_vpn_only;

-- Step 4: Verify user-level override
SHOW PARAMETERS LIKE 'network_policy' IN USER etl_service_user;
```

**Explanation** – The service user now can connect from `34.120.100.50`, even though the account policy would normally reject it. Other users are still restricted to the corporate VPN. User‑level policies are ideal for service accounts, break‑glass accounts, or external partners.

---

### Use Case 3: Block Malicious IPs While Keeping Allowlist

**Scenario** – You have a broad allowlist (e.g., `0.0.0.0/0` – all IPs) but want to block a specific range known for attack attempts.

```sql
CREATE OR REPLACE NETWORK POLICY block_attackers
  ALLOWED_IP_LIST = ('0.0.0.0/0')   -- Allow everything
  BLOCKED_IP_LIST = ('192.0.2.0/24', '198.51.100.10');  -- Block specific malicious IPs

ALTER ACCOUNT SET NETWORK POLICY = block_attackers;
```

**Explanation** – `BLOCKED_IP_LIST` entries always take precedence. Even if `0.0.0.0/0` allows all IPs, those in the blocklist are denied. This is useful when you want a deny‑by‑default approach with exceptions.

---

### Use Case 4: Temporary Allowlisting for Remote Incident Response

**Scenario** – A security consultant needs temporary access from their home IP `203.0.113.200` for one day. You don’t want to modify the permanent account policy.

```sql
-- Step 1: Create a temporary network policy
CREATE OR REPLACE NETWORK POLICY temp_consultant
  ALLOWED_IP_LIST = ('203.0.113.200');

-- Step 2: Apply to the consultant's user
ALTER USER consultant_user SET NETWORK POLICY = temp_consultant;

-- Step 3: After 24 hours, remove the policy
ALTER USER consultant_user UNSET NETWORK POLICY;
DROP NETWORK POLICY temp_consultant;
```

**Explanation** – User‑level policies can be applied and removed without affecting the account policy. This provides granular, time‑limited access. For automation, you could combine with a scheduled task (Snowflake Task) to unset the policy automatically.

---

### Use Case 5: Multi‑Environment Segregation (Dev vs Prod)

**Scenario** – You have two Snowflake accounts: `PROD` and `DEV`. Production must only accept connections from a hardened set of IPs (corporate VPN + jump host). Development can accept a wider range.

```sql
-- In PROD account
CREATE OR REPLACE NETWORK POLICY prod_policy
  ALLOWED_IP_LIST = ('203.0.113.0/24', '10.0.0.5');  -- VPN + jump host
ALTER ACCOUNT SET NETWORK POLICY = prod_policy;

-- In DEV account
CREATE OR REPLACE NETWORK POLICY dev_policy
  ALLOWED_IP_LIST = ('203.0.113.0/24', '192.0.2.0/24', '0.0.0.0/0');  -- wider access
ALTER ACCOUNT SET NETWORK POLICY = dev_policy;
```

**Explanation** – Network policies are account‑specific. Different environments can have different risk postures. Production is locked down; development is more permissive for developer agility.

---

### Use Case 6: Combine Network Policy with MFA and SAML for High‑Security Access

**Scenario** – A financial services company requires that users accessing sensitive tables must come from a **specific internal IP range** AND authenticate via SAML with MFA. Network policy enforces the IP constraint.

```sql
-- Step 1: Create network policy
CREATE OR REPLACE NETWORK POLICY finance_network
  ALLOWED_IP_LIST = ('10.88.0.0/16');   -- Internal finance subnet

ALTER ACCOUNT SET NETWORK POLICY = finance_network;

-- Step 2: SAML integration already configured (see 5.8)
-- Step 3: Enforce MFA via authentication policy (see 5.7)
CREATE AUTHENTICATION POLICY mfa_required_policy
  MFA_ENROLLMENT = REQUIRED
  CLIENT_TYPES = ('SNOWFLAKE_UI', 'SNOWSQL', 'JDBC');

ALTER ACCOUNT SET AUTHENTICATION POLICY = mfa_required_policy;
```

**Explanation** – Defence in depth: the user must be on the finance subnet (network policy), must authenticate via SAML (no password), and must have MFA enrolled. This meets stringent compliance requirements.

---

### Use Case 7: Audit Network Policy Violations (Security Monitoring)

**Scenario** – Your security team wants to track every attempted connection from an unauthorised IP, including the source IP and the user (if identifiable).

```sql
-- Query login history for network policy rejections
SELECT 
    user_name,
    reported_client_type,
    client_ip,
    error_message,
    start_time
FROM snowflake.account_usage.login_history
WHERE error_message LIKE '%Network policy%violation%'
  AND start_time >= DATEADD('day', -7, CURRENT_TIMESTAMP())
ORDER BY start_time DESC;

-- Aggregate by IP address to spot attack patterns
SELECT 
    client_ip,
    COUNT(*) AS failed_attempts,
    MIN(start_time) AS first_attempt,
    MAX(start_time) AS last_attempt
FROM snowflake.account_usage.login_history
WHERE error_message LIKE '%Network policy%violation%'
GROUP BY client_ip
ORDER BY failed_attempts DESC;
```

**Explanation** – The `login_history` view includes `error_message` for failed logins. Network policy violations produce a specific error string. Monitoring these helps detect brute‑force attempts or misconfigured applications.

---

### Use Case 8: Gradually Roll Out a Network Policy (Dry‑Run Mode)

**Scenario** – You want to test a new network policy before enforcing it, to avoid locking out legitimate users.

```sql
-- Step 1: Create a network policy with a very broad allowlist (e.g., 0.0.0.0/0)
-- but also a blocklist that includes nothing. Then attach it to a test user.
CREATE OR REPLACE NETWORK POLICY test_policy
  ALLOWED_IP_LIST = ('0.0.0.0/0');   -- Allows everything

ALTER USER test_user SET NETWORK POLICY = test_policy;

-- Step 2: After testing, modify the policy to the intended allowlist
ALTER NETWORK POLICY test_policy
  SET ALLOWED_IP_LIST = ('203.0.113.0/24');

-- Step 3: Have the test user verify connectivity from the allowed range.
-- Step 4: Apply to all users (account level)
ALTER ACCOUNT SET NETWORK POLICY = test_policy;
```

**Explanation** – You cannot “dry‑run” a network policy without applying it, but you can start with a permissive policy attached to a single user, tighten it, and only then apply to the whole account. This reduces the risk of production lockout.

## 4. Best Practices Summary

| Practice | Why |
|----------|-----|
| **Start with a restrictive allowlist and expand only as needed** | Least privilege: only known, trusted IPs can connect. |
| **Use user‑level policies for service accounts and emergency access** | Prevents service accounts from needing broad network access for all users. |
| **Regularly audit `login_history` for policy violations** | Detect misconfigurations and potential attacks. |
| **Combine with MFA and authentication policies** | Defence in depth: network policies alone do not stop credential theft. |
| **Test policy changes on a non‑production account first** | Avoid accidental lockout of production. |
| **Keep a break‑glass user with a very broad network policy (or none)** | Ensure you have an emergency access path if your IP ranges change unexpectedly. |
| **Document your IP ranges and update policies when your network changes** | Stale policies can lock out legitimate users. |
| **Use CIDR notation efficiently** | Prefer `/24` or larger aggregates to reduce the number of entries. |

## 5. Important Limitations & Considerations

| Limitation | Mitigation |
|------------|------------|
| **IPv4 only** | IPv6 addresses are not supported. Use IPv4 ranges or a proxy that provides IPv4 egress. |
| **Max 1000 entries per list** | Use CIDR aggregation (e.g., `10.0.0.0/8` instead of many `/24`s). If you exceed, consider a proxy or VPN with a single egress IP. |
| **Policy changes are immediate** | Existing sessions are not terminated; only new connections are affected. If you lock yourself out, you must use another user with a different policy or contact Snowflake support. |
| **No “allowlist only” without a blocklist** | You can have an empty blocklist. Blocklist is optional. |
| **Private connectivity (PrivateLink, etc.)** | The client IP seen may be the private IP of the NAT. Ensure your policy includes that private IP range. |
| **Snowflake UI (Snowsight) uses CDN IPs** | If you restrict access to corporate IPs, Snowsight may still work because the UI is served from a CDN. The actual query execution still uses the client IP. Test thoroughly. |
| **Network policy does not apply to Snowflake’s internal services** | Policy only applies to client connections, not to replication, cross‑account data sharing, or Snowflake’s own management traffic. |

## 6. Auditing & Monitoring Queries

### Show current account‑level network policy
```sql
SHOW PARAMETERS LIKE 'network_policy' IN ACCOUNT;
```

### List all network policies and their definitions
```sql
SHOW NETWORK POLICIES;
SELECT * FROM snowflake.account_usage.network_policies;
```

### Find users with user‑level network policies
```sql
SELECT user_name, network_policy 
FROM snowflake.account_usage.users
WHERE network_policy IS NOT NULL;
```

### Track policy changes over time
```sql
SELECT 
    query_id,
    query_text,
    user_name,
    role_name,
    start_time
FROM snowflake.account_usage.query_history
WHERE query_text ILIKE '%NETWORK POLICY%'
ORDER BY start_time DESC;
```

### Monitor connections rejected by network policy (detailed)
```sql
SELECT 
    login_history.user_name,
    login_history.client_ip,
    login_history.reported_client_type,
    network_policies.policy_name AS blocking_policy,
    login_history.error_message,
    login_history.start_time
FROM snowflake.account_usage.login_history
JOIN snowflake.account_usage.network_policies
    ON login_history.error_message LIKE '%' || network_policies.policy_name || '%'
WHERE login_history.is_success = 'NO'
  AND login_history.error_message LIKE '%Network policy%violation%'
ORDER BY login_history.start_time DESC
LIMIT 100;
```

## 7. Key Takeaways

- **Network policies** allow or block access based on the client’s IPv4 address.
- Two levels: **account** (applies to all users) and **user** (overrides account).
- Blocklist entries take precedence over allowlist entries.
- Use user‑level policies for **service accounts, emergency access, or temporary allowlisting**.
- Always **test** policy changes with a test user before applying to the whole account.
- Combine with **MFA, SAML, and authentication policies** for layered security.
- **Monitor** `snowflake.account_usage.login_history` for violation events.
- Keep a **break‑glass user** with a separate network policy (or none) to avoid permanent lockout.

This completes the thorough explanation of Snowflake’s Network Policies with practical, actionable use cases.
