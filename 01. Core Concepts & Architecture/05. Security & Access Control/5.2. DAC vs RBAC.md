# Snowflake DAC vs RBAC: Through and Through

## 1. Core Concepts

Snowflake implements **Discretionary Access Control (DAC) using Role‑Based Access Control (RBAC)** as its security model. This hybrid approach combines the flexibility of DAC with the manageability of RBAC.

### 1.1. Discretionary Access Control (DAC)

- **Definition**: The owner of an object decides who can access it and what they can do.
- **Key principle**: “Ownership grants discretion” – the owner can grant privileges to other users or roles at their own discretion.
- **In Snowflake**: The role that holds the `OWNERSHIP` privilege on an object can grant any privilege on that object to any other role.

### 1.2. Role‑Based Access Control (RBAC)

- **Definition**: Permissions are assigned to roles, and users are assigned to roles. Access decisions are based on a user’s role membership.
- **Key principle**: “Access follows role” – administrators grant privileges to roles, not directly to users.
- **In Snowflake**: All privileges (including `OWNERSHIP`) are granted to roles. Users activate roles to obtain permissions.

### 1.3. Snowflake’s Hybrid Model

| Aspect               | DAC                                      | RBAC                                      |
|----------------------|------------------------------------------|-------------------------------------------|
| **Permission assignment** | Owner grants directly to others.      | Admin grants privileges to roles.         |
| **Who can grant**    | Object owner (discretion).               | Roles with `MANAGE GRANTS` or `OWNERSHIP`.|
| **Flexibility**      | High – owners can delegate quickly.      | Controlled – centralised policy.          |
| **Auditability**     | Harder – many individual grants.         | Easier – role‑based grants are aggregated.|
| **Snowflake default** | Object owner can grant (DAC).            | All grants go to roles (RBAC).            |

**How Snowflake combines them**:
- Privileges (e.g., `SELECT`, `INSERT`) are always granted **to a role** – that’s RBAC.
- But the role that owns an object can grant privileges on that object to **any other role** – that’s DAC.
- Central administrators (e.g., `SECURITYADMIN`) can also grant/revoke any privilege, overriding owner discretion.

Thus: **“DAC implemented via RBAC”** – you get the control of RBAC with the optional decentralisation of DAC.

## 2. Basic Commands (DAC vs RBAC)

### Grant a privilege (RBAC syntax)
```sql
GRANT SELECT ON TABLE sales TO ROLE analyst;
```

### Transfer ownership (DAC action)
```sql
GRANT OWNERSHIP ON TABLE sales TO ROLE finance_lead;
```

### Revoke owner’s ability to grant (move to centralised RBAC)
```sql
-- Create a managed access schema – owner cannot grant further
CREATE SCHEMA secure_schema WITH MANAGED ACCESS;
```

### Show who can grant what
```sql
SHOW GRANTS ON TABLE sales;
```

## 3. Impactful Use Cases (4–8 with Queries & Explanations)

### Use Case 1: Object Owner Grants SELECT to Another Role (Pure DAC)

**Scenario** – The role `data_owner` owns table `customers`. A new analyst role `analyst_junior` needs read access. The owner grants it directly.

```sql
-- Assume data_owner already has OWNERSHIP on customers
USE ROLE data_owner;

-- DAC: owner decides to grant SELECT
GRANT SELECT ON TABLE customers TO ROLE analyst_junior;

-- Verify
SHOW GRANTS ON TABLE customers;
-- Output includes: analyst_junior, SELECT, granted_by = data_owner
```

**Explanation** – This is discretionary because `data_owner` acted on its own authority, without involving `SECURITYADMIN`. This speeds up access but can lead to “grant sprawl”.

---

### Use Case 2: Centralise All Grants with SECURITYADMIN (Strict RBAC)

**Scenario** – Company policy says only `SECURITYADMIN` can grant privileges. Object owners have no grant rights.

```sql
-- Remove grant privilege from object owners (by revoking OWNERSHIP from them)
-- First, create a central admin role
CREATE ROLE central_admin;
GRANT MANAGE GRANTS ON ACCOUNT TO ROLE central_admin;

-- Transfer ownership of all tables in a schema to central_admin
SELECT 'GRANT OWNERSHIP ON TABLE ' || table_name || ' TO ROLE central_admin;'
FROM information_schema.tables
WHERE table_schema = 'HR_SCHEMA';
-- Execute the generated statements

-- Now, only central_admin can grant on those tables
USE ROLE central_admin;
GRANT SELECT ON TABLE hr_schema.employees TO ROLE hr_analyst;
```

**Explanation** – This disables DAC. Even the original object owner cannot grant access anymore. All grants must go through `central_admin`. This is pure RBAC with central control.

---

### Use Case 3: Managed Access Schema – Remove DAC Entirely

**Scenario** – You want a schema where **no object owner** can grant privileges. Only `SECURITYADMIN` (or a custom admin) can grant.

```sql
-- Create a managed access schema
CREATE SCHEMA finance_secure WITH MANAGED ACCESS;

-- Create a table inside (the creating role gets OWNERSHIP, but cannot grant)
CREATE TABLE finance_secure.salaries (id INT, amount NUMBER);

-- Try to grant from the owner role (fails)
USE ROLE owner_role;  -- the role that created the table
GRANT SELECT ON finance_secure.salaries TO ROLE auditor;
-- ERROR: Insufficient privileges to grant on managed access schema object

-- Only a role with MANAGE GRANTS can grant
USE ROLE securityadmin;
GRANT SELECT ON finance_secure.salaries TO ROLE auditor;  -- succeeds
```

**Explanation** – Managed access schemas remove the discretionary part of DAC. Object ownership becomes “technical ownership” without delegation rights. This is ideal for regulated data (finance, healthcare).

---

### Use Case 4: Role Hierarchy to Mimic DAC Delegation (RBAC Style)

**Scenario** – A team lead wants to delegate access to a junior, but without giving `OWNERSHIP`. Use role hierarchy.

```sql
-- Create roles
CREATE ROLE team_lead;
CREATE ROLE team_junior;

-- Grant junior to lead (lead inherits junior’s privileges)
GRANT ROLE team_junior TO ROLE team_lead;

-- Grant SELECT on table to the junior role
GRANT SELECT ON TABLE projects TO ROLE team_junior;

-- Now team_lead also has SELECT (inherited), but no need to grant directly.
-- The lead can "delegate" by assigning the junior role to new users.
GRANT ROLE team_junior TO USER new_analyst;
```

**Explanation** – This is RBAC’s answer to DAC delegation: instead of granting privileges on objects, you grant roles. The team lead gets the ability to assign the junior role (if lead has `MANAGE GRANTS` on that role), not the table privilege directly.

---

### Use Case 5: Transfer Ownership to Another Role (Changing Discretion)

**Scenario** – A data engineer leaves the company. Their role `eng_old` owns critical tables. Transfer ownership to `eng_new`.

```sql
-- As ACCOUNTADMIN or role with MANAGE GRANTS
GRANT OWNERSHIP ON TABLE critical_table TO ROLE eng_new;
-- The old role loses OWNERSHIP automatically.

-- Verify
SHOW GRANTS ON TABLE critical_table;
-- Now eng_new shows as owner with grant privilege.
```

**Explanation** – Transferring ownership moves the discretionary power from one role to another. This is essential for employee offboarding and reorganisation.

---

### Use Case 6: Limit DAC by Revoking Grant Option (Not Directly Supported)

**Scenario** – You want an object owner to have `OWNERSHIP` but **not** be able to grant privileges. Snowflake does not allow that – `OWNERSHIP` always includes grant privilege. However, you can use a **managed access schema** (Use Case 3) or **remove OWNERSHIP** entirely.

```sql
-- Instead of OWNERSHIP, grant all other privileges but not OWNERSHIP
USE ROLE securityadmin;
GRANT ALL PRIVILEGES ON TABLE sensitive_data TO ROLE data_operator;
REVOKE OWNERSHIP ON TABLE sensitive_data FROM ROLE data_operator;
-- Now data_operator can SELECT, INSERT, UPDATE, DELETE, but cannot grant.
```

**Explanation** – This gives operational control without delegation rights. The object is technically owned by `securityadmin`, but `data_operator` can fully manage data. This is a common pattern for ETL roles.

---

### Use Case 7: Future Grants – Enforcing RBAC Without DAC Interference

**Scenario** – You want every new table in a schema to automatically have `SELECT` for `analyst_role`, regardless of who creates it (even if the creator might try to deny access).

```sql
-- As SECURITYADMIN
GRANT SELECT ON FUTURE TABLES IN SCHEMA raw_data TO ROLE analyst_role;
GRANT USAGE ON SCHEMA raw_data TO ROLE analyst_role;

-- Now a data engineer creates a new table
USE ROLE data_engineer;
CREATE TABLE raw_data.new_logs (id INT, message STRING);

-- Even though engineer owns the table, analyst_role already has SELECT
-- The engineer cannot revoke it unless they have MANAGE GRANTS.
```

**Explanation** – Future grants override DAC. The central administrator pre‑defines access, and object owners cannot easily remove those grants (they would need `MANAGE GRANTS`). This is a powerful way to enforce RBAC policies.

---

### Use Case 8: Auditing DAC vs RBAC – Find Objects Where Owner Is Not a Central Admin

**Scenario** – Security audit wants to find all objects where the owner is a non‑administrative role (potential DAC risk).

```sql
-- List all tables where the owner is not in a set of safe roles
SELECT 
    table_catalog,
    table_schema,
    table_name,
    owner AS owning_role
FROM information_schema.tables
WHERE owner NOT IN ('SYSADMIN', 'SECURITYADMIN', 'ACCOUNTADMIN', 'CENTRAL_ADMIN')
AND table_schema NOT IN ('INFORMATION_SCHEMA')
ORDER BY owner;

-- For each such table, show who has granted what (possible DAC actions)
SELECT *
FROM information_schema.table_privileges
WHERE table_name IN (
    SELECT table_name FROM information_schema.tables 
    WHERE owner NOT IN ('SYSADMIN', 'SECURITYADMIN', 'ACCOUNTADMIN')
)
AND grantee != owner;
```

**Explanation** – This identifies “decentralised” ownership where DAC is active. The second query shows grants made by those owners (discretionary grants). Use this to decide whether to move objects into managed access schemas or transfer ownership.

## 4. Best Practices Summary

| Practice                                                                 | Why                                                                 |
|--------------------------------------------------------------------------|---------------------------------------------------------------------|
| **Use managed access schemas for sensitive data**                        | Removes DAC completely – only central admins can grant.             |
| **Prefer role hierarchies over direct grants**                           | Reduces grant sprawl; easier to audit.                              |
| **Limit OWNERSHIP to service roles, not user roles**                     | Prevents individuals from having discretionary power.               |
| **Use future grants to enforce RBAC policies**                           | Overrides DAC for new objects.                                      |
| **Regularly audit `information_schema.tables` for non‑admin owners**    | Detect unauthorised DAC.                                            |
| **Transfer ownership during role changes/offboarding**                   | Prevent orphaned DAC rights.                                        |
| **Avoid granting `OWNERSHIP` to end‑user roles**                         | Users should not have grant discretion.                             |

---

## 5. DAC vs RBAC – Decision Table

| If you need…                                      | Use…                         |
|---------------------------------------------------|------------------------------|
| Fast, decentralised access for agile teams       | DAC (default Snowflake)      |
| Central control, compliance, least privilege     | Managed access schemas + RBAC|
| Temporary delegation without permanent grant     | Role hierarchy               |
| Owner cannot grant                                | Revoke OWNERSHIP (give ALL PRIVILEGES except OWNERSHIP) |
| Consistent access on future objects               | Future grants (RBAC)         |

## 6. Complete Example: Moving from DAC to RBAC

```sql
-- Step 1: Identify all DAC owners
SELECT DISTINCT owner FROM information_schema.tables WHERE owner NOT LIKE '%ADMIN%';

-- Step 2: Create a central admin role
CREATE ROLE data_governance;
GRANT MANAGE GRANTS ON ACCOUNT TO ROLE data_governance;

-- Step 3: Transfer ownership of all tables in a risky schema
SELECT 'GRANT OWNERSHIP ON TABLE ' || table_name || ' TO ROLE data_governance;'
FROM information_schema.tables
WHERE table_schema = 'USER_SCHEMA';
-- Execute generated SQL

-- Step 4: Convert schema to managed access (prevents future DAC)
ALTER SCHEMA USER_SCHEMA ENABLE MANAGED ACCESS;

-- Step 5: Grant back necessary privileges via RBAC (role‑based)
GRANT SELECT, INSERT ON ALL TABLES IN SCHEMA USER_SCHEMA TO ROLE analyst;
GRANT UPDATE, DELETE ON ALL TABLES IN SCHEMA USER_SCHEMA TO ROLE data_engineer;
```

**Result** – The schema is now fully RBAC‑controlled. No object owner can grant privileges discretionarily.

---

## 7. Key Takeaways

- Snowflake’s security model is **DAC implemented via RBAC**.
- By default, object owners have **discretion** to grant privileges (DAC).
- You can **disable DAC** with managed access schemas or by removing `OWNERSHIP`.
- Use **role hierarchies** and **future grants** to enforce RBAC policies.
- Audit regularly to find where DAC is active and decide if it’s appropriate.

This completes the thorough explanation of DAC vs RBAC in Snowflake with practical, actionable use cases.
