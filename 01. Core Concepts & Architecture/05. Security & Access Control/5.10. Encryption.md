# Snowflake Encryption: Through and Through

## 1. Core Concepts

Snowflake encrypts **all data by default**—both at rest and in transit—with no action required by the user. Encryption is transparent, automatic, and deeply integrated into the architecture.

### 1.1. Encryption at Rest

- **All data files** (micro‑partitions) stored in cloud storage (S3, Azure Blob, GCS) are encrypted.
- **Metadata** (table schemas, query history, etc.) is also encrypted.
- Encryption uses **AES‑256** (Advanced Encryption Standard with 256‑bit keys).

### 1.2. Encryption in Transit

- **All communication** between clients (drivers, Snowflake UI, connectors) and Snowflake uses TLS 1.2+ (HTTPS).
- **Internal service‑to‑service** traffic inside Snowflake’s cloud account also uses TLS.

### 1.3. Snowflake’s Hierarchical Key Model

Snowflake uses a **four‑tier key hierarchy** to balance security, performance, and key rotation:

| Level | Key Type                        | Purpose                                                                 |
|-------|--------------------------------|-------------------------------------------------------------------------|
| 1     | **Root Key** (account‑level)   | Encrypts the account’s composite master key. Never leaves HSM.         |
| 2     | **Account Master Key**         | Encrypts table master keys. Rotated every ~90 days.                     |
| 3     | **Table Master Key**           | Unique per table, encrypts file keys. Rotated on table schema changes.  |
| 4     | **File Key**                   | Unique per encrypted file (micro‑partition). Rotated automatically.     |

- **Key derivation**: File keys are derived from the table master key using a unique nonce per file.
- **Key encryption keys (KEK)**: Each higher‑level key encrypts the next level down.

### 1.4. Key Management Options

| Option                           | Description                                                                 |
|----------------------------------|-----------------------------------------------------------------------------|
| **Snowflake‑managed keys**       | Default. Snowflake generates, rotates, and stores keys in its HSMs.         |
| **Customer‑managed keys (CMK)**  | You control a key in an external KMS (AWS KMS, Azure Key Vault, GCP KMS). Snowflake uses it to wrap the account master key. Called **Tri‑Secret Secure** (formerly “Customer‑Managed Key” with key rotation). |
| **Client‑side encryption**       | You encrypt data before sending to Snowflake. Snowflake sees only ciphertext. |

## 2. Basic Commands & Concepts

### 2.1. View current encryption settings (no command needed – default)
```sql
-- No explicit encryption command – everything is encrypted by default.
SHOW TABLES;   -- All tables encrypted automatically
```

### 2.2. Configure customer‑managed key (Tri‑Secret Secure)
```sql
-- Requires ACCOUNTADMIN
ALTER ACCOUNT SET USE_KEY_STORAGE_FOR_ACCOUNT_MASTER_KEY = TRUE;
ALTER ACCOUNT SET KEY_STORAGE_MASTER_KEY = '<key_id_from_kms>';
ALTER ACCOUNT SET KEY_STORAGE_MASTER_KEY_AUTO_ROTATION = TRUE;
```

### 2.3. Re‑encrypt a table with new keys (manual)
```sql
-- Forces table master key rotation
ALTER TABLE mytable REKEY;
```

### 2.4. Monitor key rotation status
```sql
SHOW TABLES HISTORY;   -- Includes last key rotation timestamp
```

## 3. Impactful Use Cases (4–8 with Queries & Explanations)

### Use Case 1: Verify That All Data Is Encrypted at Rest

**Scenario** – Compliance audit needs proof that no unencrypted data exists in Snowflake.

```sql
-- There is no “unencrypted” flag because encryption is mandatory.
-- But you can show that external stage files are also encrypted.
LIST @my_s3_stage;
-- Snowflake automatically encrypts staged files using your configured KMS.
-- For a self‑managed external stage, you can enforce encryption:
CREATE OR REPLACE STAGE my_secure_stage
  URL = 's3://mybucket/'
  STORAGE_INTEGRATION = my_integration
  ENCRYPTION = (TYPE = 'AWS_SSE_S3');
```

**Explanation** – Snowflake does not allow creating an unencrypted table. The only potential gap is an external stage pointing to a bucket with server‑side encryption disabled. Use `ENCRYPTION` parameter to enforce encryption.

---

### Use Case 2: Implement Tri‑Secret Secure for Compliance (Bring Your Own Key)

**Scenario** – Your company requires that Snowflake cannot access data without your cloud KMS key (e.g., for regulatory reasons like GDPR, HIPAA).

```sql
-- As ACCOUNTADMIN
-- Step 1: Grant Snowflake access to your AWS KMS key (done via IAM role)
-- Step 2: Configure the key storage
ALTER ACCOUNT SET USE_KEY_STORAGE_FOR_ACCOUNT_MASTER_KEY = TRUE;

-- Step 3: Provide the key ID from AWS KMS
ALTER ACCOUNT SET KEY_STORAGE_MASTER_KEY = 'arn:aws:kms:us-east-1:123456789012:key/abcd1234-...';

-- Step 4: Enable automatic rotation (Snowflake will ask KMS for new key versions)
ALTER ACCOUNT SET KEY_STORAGE_MASTER_KEY_AUTO_ROTATION = TRUE;

-- Verify
SHOW PARAMETERS LIKE 'KEY_STORAGE_MASTER_KEY' IN ACCOUNT;
```

**Explanation** – With Tri‑Secret Secure, the account master key is wrapped by a key in your own KMS. Snowflake’s HSMs never see your plaintext key. Even if Snowflake’s storage is compromised, your data cannot be decrypted without your KMS key. The “Tri‑Secret” refers to the three components: Snowflake HSM, your KMS, and a passphrase (optional legacy).

---

### Use Case 3: Force Table Re‑encryption After a Security Incident

**Scenario** – A table master key may have been exposed (hypothetical). You want to re‑encrypt the table immediately with a new key.

```sql
-- Re‑encrypt the table (generates new table master key, rewrites all micro‑partitions)
ALTER TABLE sensitive_customers REKEY;

-- To re‑encrypt all tables in a schema:
SELECT 'ALTER TABLE ' || table_name || ' REKEY;'
FROM information_schema.tables
WHERE table_schema = 'HR_SCHEMA'
AND table_type = 'BASE TABLE';
-- Execute the generated statements.
```

**Explanation** – `REKEY` triggers a full table rewrite with a new table master key. This is a metadata‑only operation for the key itself, but physically rewrites data files, which consumes credits and time for large tables. Use only when necessary.

---

### Use Case 4: Monitor Key Rotation Compliance with SQL

**Scenario** – You need to report which tables have not had their master key rotated in the last 90 days.

```sql
-- Information schema tables include key rotation timestamp
SELECT 
    table_catalog,
    table_schema,
    table_name,
    last_altered,
    DATEDIFF(day, last_key_rotation, CURRENT_TIMESTAMP()) AS days_since_key_rotation
FROM information_schema.tables
WHERE table_schema NOT IN ('INFORMATION_SCHEMA')
AND last_key_rotation IS NOT NULL
AND DATEDIFF(day, last_key_rotation, CURRENT_TIMESTAMP()) > 90
ORDER BY days_since_key_rotation DESC;
```

**Explanation** – Snowflake automatically rotates table master keys every ~90 days for tables that are modified. This query identifies tables that might need a manual `REKEY` or investigation.

### Use Case 5: Client‑Side Encryption for “Zero‑Knowledge” Architecture

**Scenario** – You want Snowflake to never see plaintext for a specific sensitive column (e.g., SSN). Encrypt on the client before loading.

```sql
-- On client side (Python example using cryptography, then load into Snowflake)
from cryptography.fernet import Fernet
key = Fernet.generate_key()
cipher = Fernet(key)
encrypted_ssn = cipher.encrypt(b"123-45-6789")

-- Load into Snowflake as a VARIANT or BINARY column
CREATE TABLE clients (
    id INT,
    ssn_encrypted BINARY   -- store the ciphertext
);

-- Later, decrypt within Snowflake (if you import the key as a secure UDF)
CREATE OR REPLACE FUNCTION decrypt_ssn(encrypted BINARY)
RETURNS STRING
LANGUAGE JAVASCRIPT
AS '
  // Use the key stored in a secure view or external function
  const key = ...; // NOT RECOMMENDED – key would be in plaintext
';
```

**Explanation** – True client‑side encryption means Snowflake never sees the key. However, you then cannot filter, join, or aggregate on that column inside Snowflake. This is for **storage‑only** of secrets. Use a dedicated external tokenization service instead for usability.

### Use Case 6: Verify Encryption of Data in Transit (TLS)

**Scenario** – Security team wants proof that all connections from your BI tool use TLS.

```sql
-- In Snowflake, check the connection history for a user
SELECT user_name, client_environment, authentication_method, 
       start_time, end_time
FROM snowflake.account_usage.login_history
WHERE user_name = 'powerbi_user'
ORDER BY start_time DESC
LIMIT 10;
-- The `client_environment` column will show TLS version (e.g., "TLSv1.2")
```

**Explanation** – Snowflake rejects non‑TLS connections. You can audit the `login_history` view to confirm that all clients negotiate TLS 1.2 or higher. There is no option to disable TLS.

---

### Use Case 7: Rotate Account Master Key Manually (Tri‑Secret Secure)

**Scenario** – Your company policy requires you to rotate the top‑level customer key every 30 days (more aggressive than Snowflake’s 90‑day default).

```sql
-- Step 1: Create a new key version in your external KMS (AWS KMS, Azure Key Vault)
-- Step 2: Update Snowflake to use the new key version
ALTER ACCOUNT SET KEY_STORAGE_MASTER_KEY = 'arn:aws:kms:us-east-1:123456789012:key/abcd1234-...';

-- Step 3: Force re‑wrap of the account master key (no data rewrite needed)
ALTER ACCOUNT RESET KEY_STORAGE_MASTER_KEY_AUTO_ROTATION;  -- toggles
```

**Explanation** – When you change `KEY_STORAGE_MASTER_KEY` to a different key ID (or same ID with new version), Snowflake re‑encrypts the account master key with the new customer key. This is a fast metadata operation. No data is re‑encrypted.

---

### Use Case 8: Export Encrypted Data Without Decrypting (for Cross‑Account Sharing)

**Scenario** – You want to share encrypted data with another Snowflake account without exposing plaintext, and let the other account decrypt using its own key.

```sql
-- Source account: Create an encrypted stage with customer‑managed key
CREATE STAGE cross_account_stage
  URL = 's3://shared-bucket/encrypted-export/'
  STORAGE_INTEGRATION = my_integration
  ENCRYPTION = (TYPE = 'AWS_KMS' , KMS_KEY_ID = 'arn:aws:kms:...');

-- Copy data into stage – stays encrypted
COPY INTO @cross_account_stage/export/
FROM sales_table
FILE_FORMAT = (TYPE = PARQUET);

-- Target account: Has its own KMS key and can read the stage (if bucket policy allows)
-- The target must have the same KMS key permissions.
```

**Explanation** – By using an external stage with encryption (AWS SSE‑KMS or Azure Key Vault), you control the encryption key separately from Snowflake. The receiving account needs access to the same KMS key to decrypt. This is useful for **encrypted data sharing** without a Snowflake data share.

## 4. Best Practices Summary

| Practice                                             | Why                                                                 |
|------------------------------------------------------|---------------------------------------------------------------------|
| **Use Snowflake‑managed keys** for most workloads   | Zero config, automatic rotation, industry‑standard security.       |
| **Enable Tri‑Secret Secure** for regulated industries | You retain ultimate control; Snowflake cannot access data without your KMS. |
| **Never store keys in tables or UDFs**              | Keys in plaintext defeat encryption. Use external KMS or Snowflake’s secure key storage. |
| **Use `REKEY` sparingly**                           | Causes full table rewrite; can be expensive. Rely on automatic rotation. |
| **Audit key rotation regularly**                    | Use `information_schema.tables.last_key_rotation` to verify compliance. |
| **Enforce TLS on external stages**                  | Always use `ENCRYPTION = (TYPE = 'AWS_SSE_S3' or 'AZURE_SSE' or 'GCS_SSE')`. |
| **Do not rely solely on client‑side encryption**    | You lose search, joins, aggregation. Use for secrets only.          |

## 5. Auditing & Monitoring Queries

```sql
-- Show when each table’s master key was last rotated
SELECT table_name, last_key_rotation 
FROM information_schema.tables 
WHERE table_schema = 'MY_SCHEMA';

-- Check account‑level key storage configuration
SHOW PARAMETERS LIKE 'KEY_STORAGE_%' IN ACCOUNT;

-- List stages and their encryption settings
SHOW STAGES;

-- Find all external stages that might not enforce encryption
SELECT stage_name, stage_type, encryption
FROM information_schema.stages
WHERE stage_type = 'EXTERNAL'
AND encryption NOT LIKE '%AWS_SSE%' 
AND encryption NOT LIKE '%AZURE_SSE%';
```

## 6. Key Takeaways

- Snowflake **encrypts everything by default** – no configuration needed for basic security.
- The **hierarchical key model** provides efficient rotation and fine‑grained key management.
- **Tri‑Secret Secure** gives customers control over the top‑level key for maximum compliance.
- Use the provided SQL commands to **audit, rotate, and enforce** encryption policies.
- Encryption in Snowflake is **transparent to queries** – you don’t need to change application logic.

This completes the thorough explanation of Snowflake’s encryption model with practical, impactful use cases.
