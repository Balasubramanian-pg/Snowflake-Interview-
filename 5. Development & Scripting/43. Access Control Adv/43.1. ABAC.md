# ABAC (Attribute-Based Access Control) in Snowflake

**Definition**

ABAC grants access based on attributes of the user, resource, or environment rather than static roles. It uses tags and masking policies to dynamically enforce rules without creating multiple roles.

**Applications (SQL Queries)**

Create tag:
```sql
CREATE TAG sensitivity;
```

Apply to column:
```sql
ALTER TABLE customers MODIFY COLUMN ssn SET TAG sensitivity = 'high';
```

Create policy:
```sql
CREATE MASKING POLICY ssn_mask AS (val string) RETURNS string ->
CASE WHEN CURRENT_ROLE() = 'ADMIN' THEN val ELSE '***-**-****' END;
```

Attach policy:
```sql
ALTER TABLE customers MODIFY COLUMN ssn SET MASKING POLICY ssn_mask;
```

Row-level:
```sql
CREATE ROW ACCESS POLICY region_filter AS (region string) RETURNS boolean ->
region = CURRENT_REGION();
```

**Edge Cases**

- Tag inheritance conflicts: Explicit column tags override table tags. Verify with `SHOW TAGS`.
- Policy ordering: Multiple masking policies on same column use most restrictive. Use `PRIORITY` if needed.
- Performance: Complex `CASE` statements in policies slow queries. Pre-calculate attributes in lookup tables.
- NULL handling: Policies returning `NULL` may break joins. Explicitly handle `NULL` in policy logic.

**Common Pitfalls**

- Assuming tags propagate automatically to views. Re-apply tags on views or use `ALTER VIEW`.
- Forgetting that account admins bypass all policies. Test with actual end-user roles.
- Over-engineering policies with too many `CASE` branches. Split into multiple policies instead.
- Not auditing policy changes. Enable `access_history` and review tag modifications regularly.
