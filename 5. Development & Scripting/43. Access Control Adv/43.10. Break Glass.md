# Break Glass in Snowflake

**Definition**

Emergency access mechanism that bypasses normal security controls when standard authentication fails, critical systems are down, or immediate administrative action is required to prevent data loss or service outage.

**Applications (SQL Queries)**

Create emergency role:
```sql
CREATE ROLE break_glass_role;
GRANT ALL PRIVILEGES ON ACCOUNT TO ROLE break_glass_role;
```

Stored procedure activation:
```sql
CREATE PROCEDURE activate_break_glass()
RETURNS STRING
LANGUAGE SQL
EXECUTE AS OWNER
AS
BEGIN
  GRANT ROLE break_glass_role TO USER CURRENT_USER();
  INSERT INTO audit.emergency_access_log (user, timestamp, reason) 
  VALUES (CURRENT_USER(), CURRENT_TIMESTAMP(), 'BREAK GLASS ACTIVATED');
  RETURN 'Emergency access granted. Incident logged.';
END;
```

Network policy bypass:
```sql
CREATE NETWORK POLICY break_glass_policy ALLOWED_IP_LIST = ('0.0.0.0/0');
```

**Edge Cases**

- MFA failure: Pre-generate backup codes or maintain hardware token with C-level executive.
- Network isolation: Store emergency credentials in physical safe, not password manager.
- Account lockout: Designate secondary account admin in different identity provider.
- Audit trail gaps: Break glass usage must trigger immediate alerts to security team.

**Common Pitfalls**

- Using break glass for routine maintenance. It is for emergencies only.
- Single person holding credentials. Requires dual control or sealed envelope protocol.
- Not testing the procedure. Run drills quarterly to verify it works under pressure.
- Missing automatic expiration. Implement session timeouts and mandatory re-authentication.
- No post-incident review. Every activation requires documented root cause analysis.
