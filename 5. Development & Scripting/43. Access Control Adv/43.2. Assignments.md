# Role Assignments in Snowflake

**Definition**

Mechanism to grant privileges by assigning roles to users or other roles, establishing inheritance hierarchies and access paths without direct privilege grants.

**Applications (SQL Queries)**

Grant role to user:
```sql
GRANT ROLE analyst TO USER john_doe;
```

Grant role to role:
```sql
GRANT ROLE junior_analyst TO ROLE senior_analyst;
```

Set default role:
```sql
ALTER USER john_doe SET DEFAULT_ROLE = analyst;
```

Enable secondary roles:
```sql
USE SECONDARY ROLES ALL;
```

**Edge Cases**

- Circular inheritance: Role A granted to B, B granted to A. System blocks this.
- Grant cycles: A→B→C→A. Use SYSTEM$ROLE_GRANTS_TO_ROLE_TREE to diagnose.
- Privilege escalation: User with SECURITYADMIN granting ACCOUNTADMIN to themselves. Enable SCIM or disable direct grants.
- Role explosion: Users accumulating 50+ roles. Implement role consolidation reviews.

**Common Pitfalls**

- Forgetting that grants are not transitive for privileges. Only role membership inherits.
- Using `GRANT ROLE` instead of `GRANT <privilege>`. Confusing membership with capability.
- Not revoking inherited roles. `REVOKE ROLE` only removes direct grants, not hierarchy.
- Default role mismatch. User logs in with wrong role, gets access denied. Verify with `SHOW GRANTS TO USER`.
- Secondary roles disabled by default. Queries fail across databases. Set `USE_SECONDARY_ROLES = ALL` in session parameters or user defaults.
