# Snowflake System Roles: Through and Through

## 1. Core Concepts

Snowflake provides a set of **system-defined roles** that come built‑in with every account. These roles establish the foundation of the RBAC model and cannot be dropped or renamed. They are designed to separate administrative responsibilities and enforce the principle of least privilege from the very start.

### 1.1. The System Role Hierarchy

System roles follow a specific hierarchical structure. Privileges flow **downward** from higher‑level roles to lower‑level roles:

```
ORGADMIN (Organization‑level, across multiple accounts)
  │
  └─ GLOBALORGADMIN (Newer, recommended for org‑level tasks)
  │
  └─ ACCOUNTADMIN (Top‑level within a single account)
        │
        ├─ SECURITYADMIN (Security & access management)
        │      │
        │      └─ USERADMIN (User & role management)
        │
        └─ SYSADMIN (System object management – warehouses, databases, schemas)
              │
              └─ (Custom functional roles you create)
```

**Important**: `ACCOUNTADMIN` does **not** automatically have access to all data. It only has the privileges to grant those access rights. To actually view data, `ACCOUNTADMIN` must be granted the necessary privileges on objects, or a lower role in the hierarchy must have them.[reference:0][reference:1]

### 1.2. System Role Summary

| System Role | Primary Responsibility | Key Privileges |
|-------------|------------------------|----------------|
| **ORGADMIN** | Organization‑level account management (phasing out) | Manage multiple accounts, view org usage |
| **ACCOUNTADMIN** | Account‑level administration | Configure account parameters, manage billing, stop queries |
| **SECURITYADMIN** | Security & access control | Manage users, roles, grants globally (`MANAGE GRANTS`) |
| **SYSADMIN** | System object management | Create warehouses, databases, schemas, tables |
| **USERADMIN** | User & role management | Create/modify users and roles |
| **PUBLIC** | Basic access | Automatically granted to every user/role |

## 2. Basic Commands for System Roles

### Show all system roles
```sql
SHOW ROLES;
```

### Show grants for a system role
```sql
SHOW GRANTS TO ROLE SYSADMIN;
```

### Grant a system role to a user
```sql
GRANT ROLE SYSADMIN TO USER alice;
```

### Grant a system role to another role (build hierarchy)
```sql
GRANT ROLE SECURITYADMIN TO ROLE ACCOUNTADMIN;  -- Already default
```

### Set a system role as active
```sql
USE ROLE SYSADMIN;
```

### Show applicable roles for current user
```sql
SELECT * FROM INFORMATION_SCHEMA.APPLICABLE_ROLES;
```

## 3. Impactful Use Cases (4–8 with Queries & Explanations)

### Use Case 1: Bootstrap Your Account Using System Roles (Initial Setup)

**Scenario** – When your Snowflake account is first provisioned, the initial user is assigned `ACCOUNTADMIN`. This user must create at least one additional user with `USERADMIN` role for daily administration, and then restrict `ACCOUNTADMIN` to break‑glass scenarios.

```sql
-- Step 1: As ACCOUNTADMIN (initial user)
USE ROLE ACCOUNTADMIN;

-- Step 2: Create a dedicated admin user
CREATE USER admin_user PASSWORD = 'StrongP@ssw0rd' 
  MUST_CHANGE_PASSWORD = TRUE
  DEFAULT_ROLE = USERADMIN;

-- Step 3: Grant USERADMIN role to this user
GRANT ROLE USERADMIN TO USER admin_user;

-- Step 4: Create a second admin user (redundancy – Snowflake recommends at least two)
CREATE USER backup_admin PASSWORD = 'AnotherStrongP@ssw0rd'
  DEFAULT_ROLE = USERADMIN;
GRANT ROLE USERADMIN TO USER backup_admin;

-- Step 5: Now admin_user can create all remaining users
USE ROLE USERADMIN;
CREATE USER data_engineer PASSWORD = '...' DEFAULT_ROLE = SYSADMIN;
CREATE USER security_officer PASSWORD = '...' DEFAULT_ROLE = SECURITYADMIN;
```

**Explanation** – Snowflake strongly recommends assigning `ACCOUNTADMIN` to only a limited number of people and requiring MFA for them.[reference:2] By creating dedicated `USERADMIN` users, you separate the power of account‑level configuration from day‑to‑day user management.

### Use Case 2: Build a Custom Role Hierarchy Under SYSADMIN (Best Practice)

**Scenario** – Your organisation has data engineers and data analysts. All custom roles should roll up to `SYSADMIN` so it can manage the objects they create.

```sql
-- As SECURITYADMIN (to create roles and grants)
USE ROLE SECURITYADMIN;

-- Create functional roles
CREATE ROLE data_engineer;
CREATE ROLE data_analyst;

-- Grant all custom roles to SYSADMIN (this is the recommended pattern)
GRANT ROLE data_engineer TO ROLE SYSADMIN;
GRANT ROLE data_analyst TO ROLE SYSADMIN;

-- Create warehouses as SYSADMIN (so they are owned by SYSADMIN)
USE ROLE SYSADMIN;
CREATE WAREHOUSE analytics_wh WAREHOUSE_SIZE = 'XSMALL';
CREATE WAREHOUSE etl_wh WAREHOUSE_SIZE = 'MEDIUM';

-- Grant warehouse USAGE to the custom roles
GRANT USAGE ON WAREHOUSE analytics_wh TO ROLE data_analyst;
GRANT USAGE ON WAREHOUSE etl_wh TO ROLE data_engineer;

-- Create databases and schemas
CREATE DATABASE analytics_db;
CREATE SCHEMA analytics_db.staging;

-- Grant schema USAGE to data_analyst
GRANT USAGE ON DATABASE analytics_db TO ROLE data_analyst;
GRANT USAGE ON SCHEMA analytics_db.staging TO ROLE data_analyst;

-- Assign roles to users
GRANT ROLE data_engineer TO USER bob;
GRANT ROLE data_analyst TO USER alice;
```

**Explanation** – By placing all custom roles under `SYSADMIN`, you ensure that `SYSADMIN` (or roles granted to it) can manage all system objects. This is Snowflake’s recommended best practice.[reference:3][reference:4] The `SECURITYADMIN` role is used only for creating roles and managing grants, not for day‑to‑day object creation.

### Use Case 3: Enforce Separation of Duties with SECURITYADMIN and SYSADMIN

**Scenario** – Your security team (`SECURITYADMIN`) should manage access policies, but never create or modify data warehouses. The data engineering team (`SYSADMIN`) should create objects but never grant privileges.

```sql
-- As ACCOUNTADMIN (initial configuration)
USE ROLE ACCOUNTADMIN;

-- Verify current grants on SECURITYADMIN
SHOW GRANTS TO ROLE SECURITYADMIN;
-- This role has MANAGE GRANTS by default – it can grant/revoke any privilege globally.

-- Ensure SECURITYADMIN does NOT have CREATE WAREHOUSE
REVOKE CREATE WAREHOUSE ON ACCOUNT FROM ROLE SECURITYADMIN;  -- Already not granted by default

-- Ensure SYSADMIN does NOT have MANAGE GRANTS
REVOKE MANAGE GRANTS ON ACCOUNT FROM ROLE SYSADMIN;  -- Already not granted by default

-- Now test the separation:
USE ROLE SECURITYADMIN;
CREATE WAREHOUSE test_wh;  -- ERROR: Insufficient privileges
GRANT SELECT ON TABLE analytics_db.staging.sales TO ROLE data_analyst;  -- SUCCEEDS

USE ROLE SYSADMIN;
CREATE WAREHOUSE test_wh;  -- SUCCEEDS
GRANT SELECT ON TABLE analytics_db.staging.sales TO ROLE data_analyst;  -- ERROR: No MANAGE GRANTS
```

**Explanation** – By default, `SECURITYADMIN` has the `MANAGE GRANTS` privilege but not `CREATE WAREHOUSE`. `SYSADMIN` has `CREATE WAREHOUSE` but not `MANAGE GRANTS`. This natural separation of duties is a key security feature of Snowflake’s system roles.[reference:5]

### Use Case 4: Monitor Who Is Using ACCOUNTADMIN (Security Audit)

**Scenario** – Your compliance team needs a report of every action performed by the `ACCOUNTADMIN` role, to detect unauthorised usage.

```sql
-- As ACCOUNTADMIN
USE ROLE ACCOUNTADMIN;

-- View all query history for ACCOUNTADMIN in the last 7 days
SELECT 
    query_id,
    query_text,
    user_name,
    role_name,
    start_time,
    end_time,
    execution_status,
    error_message
FROM snowflake.account_usage.query_history
WHERE role_name = 'ACCOUNTADMIN'
  AND start_time >= DATEADD(day, -7, CURRENT_TIMESTAMP())
ORDER BY start_time DESC;

-- Also check which users have ACCOUNTADMIN granted
SELECT 
    grantee_name AS user_name,
    granted_by,
    created_on
FROM snowflake.account_usage.grants_to_users
WHERE role = 'ACCOUNTADMIN';

-- Alert if any ACCOUNTADMIN user does not have MFA enabled
SELECT user_name, has_mfa
FROM snowflake.account_usage.users
WHERE user_name IN (
    SELECT grantee_name 
    FROM snowflake.account_usage.grants_to_users 
    WHERE role = 'ACCOUNTADMIN'
);
```

**Explanation** – `ACCOUNTADMIN` is the most powerful role and should be used sparingly.[reference:6] This audit query helps enforce the “break‑glass” principle by identifying any unexpected usage or users without MFA.

### Use Case 5: Grant Minimal Privileges to PUBLIC Role (Security Hardening)

**Scenario** – By default, the `PUBLIC` role has access to the `SNOWFLAKE_SAMPLE_DATA` database. For security compliance, you should revoke all unnecessary privileges from `PUBLIC`.

```sql
-- As ACCOUNTADMIN or SECURITYADMIN
USE ROLE SECURITYADMIN;

-- Check what PUBLIC currently has access to
SHOW GRANTS TO ROLE PUBLIC;

-- Revoke access to sample data (if not needed)
REVOKE USAGE ON DATABASE SNOWFLAKE_SAMPLE_DATA FROM ROLE PUBLIC;

-- Revoke any other grants that might exist
-- (Typically, PUBLIC should have nothing granted in production)

-- Create a specific role for basic read‑only access instead
CREATE ROLE basic_user;
GRANT USAGE ON DATABASE core_db TO ROLE basic_user;
GRANT USAGE ON SCHEMA core_db.public TO ROLE basic_user;

-- Grant only the basic_user role to new users
GRANT ROLE basic_user TO USER new_hire;

-- Verify PUBLIC now has no grants
SHOW GRANTS TO ROLE PUBLIC;
-- Should return nothing or minimal system grants
```

**Explanation** – The `PUBLIC` role is automatically granted to every user and every role.[reference:7] Security best practice is to grant the absolute minimum to `PUBLIC` – usually nothing. Instead, create explicit roles for baseline access.[reference:8]

### Use Case 6: Delegate User and Role Management with USERADMIN

**Scenario** – You want your HR team to be able to create new users and assign basic roles, but not manage security policies or create warehouses.

```sql
-- As SECURITYADMIN
USE ROLE SECURITYADMIN;

-- Create an HR admin role
CREATE ROLE hr_admin;

-- Grant USERADMIN role to hr_admin (so it inherits user/role creation privileges)
GRANT ROLE USERADMIN TO ROLE hr_admin;

-- Grant only the privileges needed for HR to create users and assign basic roles
GRANT CREATE USER ON ACCOUNT TO ROLE hr_admin;
GRANT CREATE ROLE ON ACCOUNT TO ROLE hr_admin;

-- Do NOT grant MANAGE GRANTS or CREATE WAREHOUSE to hr_admin

-- Create a basic role that HR can assign to new employees
CREATE ROLE employee_basic;
GRANT USAGE ON WAREHOUSE analytics_wh TO ROLE employee_basic;
GRANT SELECT ON DATABASE hr_db TO ROLE employee_basic;

-- Now HR can create users and assign the basic role
USE ROLE hr_admin;
CREATE USER jane_doe PASSWORD = '...' DEFAULT_ROLE = employee_basic;
GRANT ROLE employee_basic TO USER jane_doe;

-- HR cannot grant any additional privileges beyond what they already have
GRANT SELECT ON DATABASE finance_db TO ROLE employee_basic;  -- ERROR
```

**Explanation** – `USERADMIN` is a child of `SECURITYADMIN` in the default hierarchy.[reference:9] By granting `USERADMIN` to a custom role, you delegate user management without giving full `SECURITYADMIN` power.

### Use Case 7: Create a Break‑Glass Account with ACCOUNTADMIN Only

**Scenario** – For emergency access (e.g., when all other admins are unavailable), you maintain a “break‑glass” account that has `ACCOUNTADMIN` but is heavily restricted.

```sql
-- As ACCOUNTADMIN
USE ROLE ACCOUNTADMIN;

-- Create a break‑glass user with a very strong password
CREATE USER break_glass_user 
  PASSWORD = 'ExtremelyLongAndComplexP@ssw0rd!@#$%'
  MUST_CHANGE_PASSWORD = FALSE
  DEFAULT_ROLE = ACCOUNTADMIN
  DISABLED = TRUE;  -- Initially disabled

-- Enable only during emergencies
ALTER USER break_glass_user SET DISABLED = FALSE;

-- Grant ACCOUNTADMIN explicitly (though DEFAULT_ROLE already does this)
GRANT ROLE ACCOUNTADMIN TO USER break_glass_user;

-- Log every time this user is enabled (manual process outside Snowflake)

-- After emergency, disable again
ALTER USER break_glass_user SET DISABLED = TRUE;

-- Audit all actions taken by break‑glass user
SELECT query_text, start_time, execution_status
FROM snowflake.account_usage.query_history
WHERE user_name = 'BREAK_GLASS_USER'
ORDER BY start_time DESC;
```

**Explanation** – Snowflake recommends assigning `ACCOUNTADMIN` to at least two users to avoid lockout, but also advises limiting its use.[reference:10] A break‑glass account that is normally disabled provides an extra layer of security.

### Use Case 8: Migrate from ORGADMIN to GLOBALORGADMIN (Future‑Proofing)

**Scenario** – Snowflake is phasing out `ORGADMIN` in favour of `GLOBALORGADMIN`. You need to migrate your organisation‑level administration.

```sql
-- Step 1: Identify which accounts currently have ORGADMIN enabled
-- (This requires ORGADMIN access)

-- Step 2: Sign in to the organisation account (a special account for org management)
-- Use the GLOBALORGADMIN role in the organisation account
USE ROLE GLOBALORGADMIN;

-- Step 3: Perform organisation‑level tasks (e.g., create a new account)
CREATE ACCOUNT my_new_account
  ADMIN_NAME = 'account_admin'
  ADMIN_PASSWORD = '...'
  EMAIL = 'admin@company.com'
  FIRST_NAME = 'Admin'
  LAST_NAME = 'User';

-- Step 4: Remove ORGADMIN from legacy accounts (when ready)
USE ROLE ORGADMIN;  -- On an ORGADMIN‑enabled account
ALTER ACCOUNT my_legacy_account SET IS_ORG_ADMIN = FALSE;
```

**Explanation** – Snowflake will send a notification at least three months before phasing out `ORGADMIN`. You should strongly consider using `GLOBALORGADMIN` in the organisation account instead.[reference:11] This use case prepares your team for the transition.

## 4. Best Practices Summary

| Practice | Why |
|----------|-----|
| **Never use ACCOUNTADMIN for daily work** | Blast radius too large; audit logs become unreadable; compliance violations.[reference:12] |
| **Assign ACCOUNTADMIN to at least two users with MFA** | Prevents lockout; adds security layer.[reference:13] |
| **Build all custom roles under SYSADMIN** | Centralises object management; follows Snowflake recommendation.[reference:14] |
| **Use SECURITYADMIN only for grants and role management** | Maintains separation of duties from SYSADMIN. |
| **Grant minimal privileges to PUBLIC** | Every user automatically gets PUBLIC – keep it empty.[reference:15] |
| **Use USERADMIN for delegated user creation** | Avoids giving full SECURITYADMIN to HR or team leads. |
| **Regularly audit system role usage** | Detect privilege creep and unauthorised access. |
| **Plan migration from ORGADMIN to GLOBALORGADMIN** | ORGADMIN is being phased out.[reference:16] |

## 5. Auditing & Monitoring Queries

### List all users and their granted system roles
```sql
SELECT 
    grantee_name AS user_name,
    role,
    granted_by,
    created_on
FROM snowflake.account_usage.grants_to_users
WHERE role IN ('ACCOUNTADMIN', 'SECURITYADMIN', 'SYSADMIN', 'USERADMIN', 'PUBLIC')
ORDER BY role, user_name;
```

### Show current hierarchy of system roles
```sql
SHOW GRANTS OF ROLE ACCOUNTADMIN;
SHOW GRANTS OF ROLE SECURITYADMIN;
SHOW GRANTS OF ROLE SYSADMIN;
```

### Check which users have ACCOUNTADMIN but no MFA
```sql
SELECT u.user_name, u.has_mfa
FROM snowflake.account_usage.users u
WHERE u.user_name IN (
    SELECT grantee_name 
    FROM snowflake.account_usage.grants_to_users 
    WHERE role = 'ACCOUNTADMIN'
);
```

### View query history for any system role
```sql
SELECT 
    user_name,
    role_name,
    COUNT(*) AS query_count,
    MIN(start_time) AS first_query,
    MAX(start_time) AS last_query
FROM snowflake.account_usage.query_history
WHERE role_name IN ('ACCOUNTADMIN', 'SECURITYADMIN', 'SYSADMIN')
  AND start_time >= DATEADD(day, -30, CURRENT_TIMESTAMP())
GROUP BY user_name, role_name
ORDER BY role_name, query_count DESC;
```

## 6. Key Takeaways

- System roles are **pre‑defined, immutable, and form the foundation** of Snowflake’s RBAC.
- The hierarchy is: `ORGADMIN` (phasing out) → `ACCOUNTADMIN` → `SECURITYADMIN`/`SYSADMIN` → `USERADMIN`.
- **Separation of duties** is built in: `SECURITYADMIN` manages access, `SYSADMIN` manages objects.
- **`ACCOUNTADMIN` is not a superuser** – it cannot view data without explicit grants.
- **`PUBLIC` should have minimal grants** – use custom roles for baseline access.
- Always **build custom roles under `SYSADMIN`** and use `SECURITYADMIN` for grant management.
- **Audit regularly** using `snowflake.account_usage` views to enforce least privilege.

This completes the thorough explanation of Snowflake’s system roles with practical, actionable use cases.
