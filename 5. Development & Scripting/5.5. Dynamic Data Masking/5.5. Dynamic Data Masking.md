# Snowflake Dynamic Data Masking: Through and Through

## 1. Core Concepts

Dynamic Data Masking (DDM) is Snowflake's **column-level security feature** that selectively obscures sensitive data **at query time** without altering the underlying stored data[reference:0]. Unlike static masking, which permanently transforms stored data, DDM applies transformations on‑the‑fly based on the querying user's role and permissions[reference:1].

### 1.1. Key Principles

- **Query‑time enforcement**: Sensitive data is never modified on disk; masking happens dynamically when a query executes.
- **Policy‑driven**: Masking policies are schema‑level objects that encapsulate conditional transformation logic[reference:2].
- **Transparent to authorised users**: Users with appropriate roles see plain text; unauthorised users see masked, redacted, or hashed values.
- **Role‑aware**: Policies evaluate `CURRENT_ROLE()`, `SESSION_USER()`, or other identity functions to determine visibility[reference:3].

### 1.2. Masking Policy Structure

A masking policy is defined with:

| Component | Description |
|-----------|-------------|
| **Signature** | The column(s) to mask and any conditional columns |
| **Return type** | Must match the input column's data type |
| **Body** | SQL expression (CASE, built‑in functions, UDFs) that transforms the value |
| **Comment** | Optional description for documentation |

The first argument (`arg_name_to_mask`) always represents the column whose data is to be masked. Additional arguments (`arg_1`, `arg_2`, ...) are **conditional columns** that determine *whether* masking occurs for each row[reference:4].

### 1.3. Dynamic Data Masking vs External Tokenisation

| Aspect | Dynamic Data Masking | External Tokenisation |
|--------|---------------------|----------------------|
| **Data stored in Snowflake** | Plain text | Tokens |
| **Masking location** | Query‑time within Snowflake | Query‑time via external function call |
| **External dependencies** | None | Requires external KMS/tokenisation service |
| **Zero‑trust** | Snowflake admins could potentially see plaintext | Plaintext never enters Snowflake[reference:5] |
| **Edition requirement** | Enterprise Edition+ | Enterprise Edition+ |

### 1.4. Tag‑Based Masking (Recommended for Scale)

A **tag‑based masking policy** combines object tagging with masking policies, allowing a single policy to protect all columns carrying a specific tag across your entire account[reference:6]. The tag can support **one masking policy per data type** (STRING, NUMBER, TIMESTAMP, etc.)[reference:7]. Tags can be set at database, schema, table, or column level, with inheritance automatically protecting child objects[reference:8]. If a column has both a direct masking policy and a tag‑based policy, the **direct policy takes precedence**[reference:9].

## 2. Basic Commands & Privileges

### 2.1. Required Privileges

| Privilege | Scope | Description |
|-----------|-------|-------------|
| `CREATE MASKING POLICY` | Schema | Allows creation of masking policies within a schema. |
| `APPLY MASKING POLICY` | Account | Controls who can set or unset masking policies on columns. |
| `APPLY ON MASKING POLICY` | Policy | Allows policy owner to delegate [un]set operations. |
| `OWNERSHIP` | Policy | Required to alter or drop a masking policy. |

**Recommended setup** – Create a dedicated `MASKING_ADMIN` role:

```sql
CREATE ROLE masking_admin;
GRANT CREATE MASKING POLICY ON SCHEMA my_db.my_schema TO ROLE masking_admin;
GRANT APPLY MASKING POLICY ON ACCOUNT TO ROLE masking_admin;
```

### 2.2. Create a Masking Policy

```sql
CREATE OR REPLACE MASKING POLICY mask_email AS (val STRING) RETURNS STRING ->
    CASE WHEN CURRENT_ROLE() = 'HR_MANAGER' THEN val
         ELSE 'REDACTED'
    END;
```

### 2.3. Apply a Masking Policy to a Column

```sql
ALTER TABLE employees MODIFY COLUMN email SET MASKING POLICY mask_email;
```

### 2.4. Unset a Masking Policy

```sql
ALTER TABLE employees MODIFY COLUMN email UNSET MASKING POLICY;
```

### 2.5. Apply to All Tables in a Schema (Future Grants)

```sql
-- Note: Future grants for masking policies are not supported directly.
-- You must apply policies to each table, or use tag‑based masking.
```

### 2.6. Show Existing Masking Policies

```sql
SHOW MASKING POLICIES;
```

### 2.7. View Policy References

```sql
SELECT * FROM snowflake.account_usage.masking_policies;
SELECT * FROM snowflake.account_usage.policy_references;
```

## 3. Impactful Use Cases (4–8 with Queries & Explanations)

### Use Case 1: Role‑Based Email Masking (Basic DDM)

**Scenario** – The `employees` table contains an `email` column. Only the `HR_MANAGER` role should see full email addresses; all other roles should see `REDACTED`.

```sql
-- Create the masking policy
CREATE OR REPLACE MASKING POLICY mask_email AS (val STRING) RETURNS STRING ->
    CASE WHEN CURRENT_ROLE() = 'HR_MANAGER' THEN val
         ELSE 'REDACTED'
    END;

-- Apply to column
ALTER TABLE hr_db.employees MODIFY COLUMN email SET MASKING POLICY mask_email;

-- Test: As a regular analyst
USE ROLE ANALYST;
SELECT email FROM hr_db.employees;   -- All emails show 'REDACTED'

-- Test: As HR manager
USE ROLE HR_MANAGER;
SELECT email FROM hr_db.employees;   -- Shows actual email addresses
```

**Explanation** – The policy is evaluated for every query. The `CURRENT_ROLE()` function determines the active role. Authorised users see plain text; unauthorised users see a fixed masked value[reference:10].

---

### Use Case 2: Tag‑Based Masking for PII Across Multiple Tables (Scalable Governance)

**Scenario** – Your organisation has 50 tables across 10 schemas, each containing PII columns (email, phone, SSN). Instead of attaching policies individually, use **tag‑based masking**.

```sql
-- Step 1: Create a tag (as TAG_ADMIN)
CREATE TAG pii_classification;

-- Step 2: Apply the tag to columns across tables
ALTER TABLE sales_db.customers MODIFY COLUMN email SET TAG pii_classification = 'HIGH';
ALTER TABLE sales_db.customers MODIFY COLUMN phone SET TAG pii_classification = 'HIGH';
ALTER TABLE hr_db.employees MODIFY COLUMN ssn SET TAG pii_classification = 'HIGH';

-- Step 3: Create one masking policy for the tag
CREATE OR REPLACE MASKING POLICY mask_pii_high AS (val STRING) RETURNS STRING ->
    CASE WHEN CURRENT_ROLE() IN ('DATA_GOVERNANCE', 'AUDIT') THEN val
         ELSE '***MASKED***'
    END;

-- Step 4: Attach the policy to the tag
ALTER TAG pii_classification SET MASKING POLICY mask_pii_high;

-- Now, any column tagged with PII='HIGH' automatically enforces the masking logic
```

**Explanation** – Tag‑based masking reduces management overhead dramatically. When a new table with PII columns is added, you simply apply the existing tag, and the policy is enforced automatically. The tag can support one masking policy per data type, allowing different masking behaviours for STRING, NUMBER, and TIMESTAMP columns[reference:11].

### Use Case 3: Conditional Masking Based on Another Column Value

**Scenario** – The `patients` table has a `diagnosis` column that should only be visible to doctors assigned to that patient. A `doctor_id` column indicates which doctor is responsible.

```sql
-- Create conditional masking policy (uses two arguments)
CREATE OR REPLACE MASKING POLICY conditional_diagnosis AS 
    (val STRING, patient_id INT) RETURNS STRING ->
    CASE WHEN CURRENT_ROLE() IN (
            SELECT doctor_name FROM patient_doctor 
            WHERE patient_doctor.patient_id = patient_id
         ) THEN val
         ELSE 'RESTRICTED'
    END;

-- Apply to diagnosis column (policy references patient_id as conditional argument)
ALTER TABLE patients MODIFY COLUMN diagnosis 
    SET MASKING POLICY conditional_diagnosis USING (patient_id);
```

**Explanation** – Conditional masking allows the policy to reference **other columns** in the same row to decide whether to reveal the protected column. The first argument is the column to mask; additional arguments are **conditional columns** that determine masking behaviour per row[reference:12]. This provides fine‑grained, row‑dependent column protection.

**Critical limitation**: A column cannot be both masked **and** used as a conditional argument at the same time[reference:13].

### Use Case 4: Partial Masking – Show Only Last 4 Digits

**Scenario** – Customer support representatives need to verify credit card numbers but should only see the last 4 digits for PCI‑DSS compliance.

```sql
-- Create partial masking policy
CREATE OR REPLACE MASKING POLICY mask_credit_card AS (val STRING) RETURNS STRING ->
    CASE WHEN CURRENT_ROLE() = 'SUPPORT_REP' 
         THEN CONCAT('****-****-****-', RIGHT(val, 4))
         WHEN CURRENT_ROLE() IN ('FRAUD_TEAM', 'FINANCE') THEN val
         ELSE 'INVALID'
    END;

-- Apply to credit_card column
ALTER TABLE payments MODIFY COLUMN credit_card_number 
    SET MASKING POLICY mask_credit_card;
```

**Explanation** – Partial masking preserves data utility (e.g., for verification) while hiding the full sensitive value. The `SUPPORT_REP` role sees `****-****-****-1234` instead of the full 16‑digit number.

### Use Case 5: Hashing Instead of Masking (Irreversible Obfuscation)

**Scenario** – For analytics use cases where you need consistent identifiers but must never reveal the original PII (e.g., user email for join operations), use a **deterministic hash**.

```sql
CREATE OR REPLACE MASKING POLICY hash_email AS (val STRING) RETURNS STRING ->
    CASE WHEN CURRENT_ROLE() IN ('ANALYTICS_ENGINE', 'DATA_SCIENCE') 
         THEN SHA2(val, 256)   -- Returns a fixed-length hash
         WHEN CURRENT_ROLE() = 'COMPLIANCE' THEN val
         ELSE NULL
    END;

ALTER TABLE user_activity MODIFY COLUMN user_email SET MASKING POLICY hash_email;
```

**Explanation** – Hashing is **one‑way** (irreversible) and deterministic: the same email always produces the same hash. This allows analysts to join tables on the hashed value without ever seeing the original email. Use `SHA2(val, 256)` for secure, collision‑resistant hashing. Note that hashing is not encryption – you cannot recover the original value[reference:14].

### Use Case 6: External Tokenisation with an External Function (Zero‑Trust Architecture)

**Scenario** – Your organisation operates a zero‑trust model where even Snowflake admins should never see plaintext PII. Data is tokenised **before** loading into Snowflake, and detokenised only for authorised roles at query time.

```sql
-- Step 1: Create an external function pointing to your tokenisation service
CREATE OR REPLACE EXTERNAL FUNCTION detokenize(VAL STRING)
    RETURNS STRING
    API_INTEGRATION = tokenization_api
    AS 'https://your-tokenization-service.com/detokenize';

-- Step 2: Create a masking policy that calls the external function
CREATE OR REPLACE MASKING POLICY tokenize_ssn AS (val STRING) RETURNS STRING ->
    CASE WHEN CURRENT_ROLE() = 'AUDITOR' 
         THEN detokenize(val)   -- External function reverses the token
         ELSE val               -- Stored token value remains unchanged
    END;

-- Step 3: Apply to the column
ALTER TABLE citizens MODIFY COLUMN ssn_token SET MASKING POLICY tokenize_ssn;
```

**Explanation** – External tokenisation ensures that sensitive data never enters Snowflake in cleartext[reference:15]. The external function calls out to a third‑party tokenisation vault (e.g., Fortanix, Protegrity, or a custom AWS Lambda) to perform the detokenisation. This approach is ideal for zero‑trust requirements and strict compliance demands[reference:16].

**Requirements**: Enterprise Edition or higher, plus integration with a tokenisation provider[reference:17].

### Use Case 7: Combine Dynamic Masking with Row Access Policies

**Scenario** – You need both row‑level filtering (sales reps see only their own region) **and** column‑level masking (salary column hidden from junior reps).

```sql
-- Create a row access policy (filters rows)
CREATE OR REPLACE ROW ACCESS POLICY region_filter AS (region STRING) RETURNS BOOLEAN ->
    CASE WHEN CURRENT_ROLE() = 'SALES_EXEC' THEN TRUE
         ELSE region = CURRENT_REGION()
    END;

-- Create a column masking policy (masks salary)
CREATE OR REPLACE MASKING POLICY mask_salary AS (val NUMBER) RETURNS NUMBER ->
    CASE WHEN CURRENT_ROLE() = 'SALES_EXEC' THEN val
         ELSE NULL
    END;

-- Apply both to the same table
ALTER TABLE sales_data ADD ROW ACCESS POLICY region_filter ON (region);
ALTER TABLE sales_data MODIFY COLUMN salary SET MASKING POLICY mask_salary;
```

**Explanation** – Snowflake evaluates **row‑level policies first**, then **column‑level policies** on the remaining rows. A junior sales rep cannot see rows outside their region **and** cannot see salary values even for rows they are allowed to access[reference:18].

### Use Case 8: Audit and Monitor Masking Policy Usage for Compliance

**Scenario** – Your compliance team needs a monthly report showing which columns have masking policies, who created them, and when they were last modified.

```sql
-- List all masking policies in the account
SELECT 
    policy_name,
    policy_schema,
    policy_database,
    created_on,
    last_altered,
    owner
FROM snowflake.account_usage.masking_policies
ORDER BY created_on DESC;

-- Show which columns are protected by which policies
SELECT 
    p.policy_name,
    ref.object_name AS table_name,
    ref.column_name,
    ref.tag_name,
    ref.policy_status
FROM snowflake.account_usage.policy_references ref
JOIN snowflake.account_usage.masking_policies p 
    ON ref.policy_id = p.policy_id
WHERE ref.policy_kind = 'MASKING_POLICY';

-- Track who accessed masked columns (Access History)
SELECT 
    query_id,
    user_name,
    role_name,
    columns_accessed,
    start_time
FROM snowflake.account_usage.access_history
WHERE columns_accessed LIKE '%MASKING_POLICY%'
  AND start_time >= DATEADD(day, -30, CURRENT_TIMESTAMP())
ORDER BY start_time DESC;
```

**Explanation** – Audit logs in Snowflake record which roles accessed masked columns, offering traceability without extra tools[reference:19]. The `account_usage.masking_policies` and `policy_references` views provide full visibility into your masking policy estate.

## 4. Best Practices Summary

| Practice | Why |
|----------|-----|
| **Prefer tag‑based masking over column‑based** | Scalable, centralised management; policy changes automatically apply to all tagged columns[reference:20]. |
| **Create a dedicated `MASKING_ADMIN` role** | Separates policy management from object ownership; follows least‑privilege principle. |
| **Test policies thoroughly before production** | Masking can break existing queries; use a staging environment. |
| **Combine with row access policies** | Defence in depth: control both which rows and which columns are visible[reference:21]. |
| **Avoid using masked columns in `WHERE` or `JOIN` predicates** | Masking only one side of a comparison leads to unexpected results[reference:22]. |
| **Version control masking policies** | Track changes for auditing and rollback using CI/CD pipelines[reference:23]. |
| **Monitor performance on deep role hierarchies** | On accounts with thousands of roles, test query performance after applying policies[reference:24]. |
| **Document all masking policies** | Include business justification, authorised roles, and masking logic for auditors. |

## 5. Important Limitations & Considerations

| Limitation | Mitigation |
|------------|------------|
| **Enterprise Edition requirement** | DDM is only available in Enterprise Edition and above. Upgrading from Standard results in a 50% compute cost increase[reference:25]. |
| **One policy per column** | A column can only be associated with one masking policy at a time[reference:26]. |
| **Input/output types must match** | You cannot define a policy that targets a TIMESTAMP and returns a STRING[reference:27]. |
| **Cannot mask and condition on same column** | A column cannot be both masked **and** used as a conditional argument simultaneously[reference:28]. |
| **No future grants for masking policies** | `GRANT ... ON FUTURE` is not supported. Use tag‑based masking for automatic protection of new objects. |
| **External tokenisation requires Enterprise Edition** | Tokenisation integration requires Enterprise Edition or higher[reference:29]. |
| **Policy management at scale can be complex** | Managing policies across hundreds of columns can become challenging without proper processes and tooling[reference:30]. |
| **Masking policies cannot reference other tables directly** | Conditional logic must use columns within the same row; cross‑table lookups are not supported[reference:31]. |

---

## 6. Key Takeaways

- **Dynamic Data Masking** is a column‑level security feature that masks sensitive data **at query time** without altering stored data[reference:32].
- **Masking policies** are schema‑level objects that use conditional logic based on `CURRENT_ROLE()` or other identity functions[reference:33].
- **Tag‑based masking** is the recommended pattern for scalable, classification‑driven governance across hundreds of columns[reference:34].
- **Conditional masking** allows policies to reference other columns to determine per‑row masking behaviour[reference:35].
- **External tokenisation** provides zero‑trust protection by keeping plaintext data out of Snowflake entirely[reference:36].
- **Audit** using `snowflake.account_usage` views to track policy changes and access history for compliance.
- **Limitations** include Enterprise Edition requirement, one policy per column, and the inability to mask and condition on the same column.

This completes the thorough explanation of Snowflake's Dynamic Data Masking model with practical, actionable use cases.
